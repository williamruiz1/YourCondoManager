# Coverage Matrix — YourCondoManager (YCM) Codebase Audit

**Protocol:** codebase-audit-protocol-v1 — Phase 6 (Synthesis)
**Ref audited:** `origin/main` — baseline snapshot `1e6f941` (Phase 1/2 mapping); the three specialist lanes independently re-cloned and executed against `bbb49aa` (a slightly later `main` commit). All findings below are cross-checked against the ref each lane actually read; no finding depends on a diff between `1e6f941` and `bbb49aa` — see §Scope note.
**Mode:** 100% static, read-only. No app code was modified by this audit. No `npm install` was run (esbuild postinstall SIGKILL risk — `node_modules` symlinked from the existing checkout instead). No runtime execution, no production database, no Stripe/Plaid dashboard, no GCP console, no Fly.io runtime state.

---

## Scope note — commit drift between phases

Phase 1/2 (baseline + system map) pinned the audit to `1e6f941`. The three Phase-3 specialist lanes each independently created a fresh worktree and landed on `bbb49aa` — `main` had moved forward by the time the lanes claimed their dispatches (normal for a live, actively-shipping repo). This is recorded as a limitation, not a defect: every finding is evidence-quoted with a file+line from the ref the discovering lane actually read, and no lane reported the diff between the two commits as material to any finding. Re-baselining exact-line-numbers against current `main` before remediation is recommended as a first step of each fix (line numbers may have shifted).

## Severity resolutions

One direct severity conflict was found and resolved during synthesis:

- **`A-REL-006` (lane-a: LOW) vs. `CQ-001` (lane-c: HIGH)** — both findings assert the identical fact: `"lint": "eslint client/src"` means the entire `server/` tree (38.5k lines, all payment/webhook/tenant-isolation code) receives zero static analysis. **Resolved severity: HIGH.** Rationale: lane-c's framing is the correct one — this is not a single reliability defect, it is the total absence of a structural quality gate over the highest-risk code in the application, applying to every line of `server/` written past, present, and future (no-floating-promises, no-unused-vars, no-shadow, exhaustive-switch, etc. never run there). The severity tally in this report and in `remediation-backlog.csv` uses HIGH for this item and lists it once, cross-referencing both IDs.

No other severity conflicts were found between lanes; all other duplicate/overlapping findings agreed on severity.

## What this audit does NOT tell you (explicit non-coverage, portfolio-wide)

- **No runtime execution anywhere.** Every "needs runtime verification" / "candidate" finding (10 of 51) is a structural defect that is plausible and code-evident but not confirmed against a live system. These should be the FIRST checks run post-audit — several (A-WEBHOOK-001's Stripe dashboard check, A-SEC-002/COST-B-004/CQ-012's GCP console check, A-OPS-003/CQ-009's Fly secret check) take minutes, not days.
- **No production database access.** All schema/data facts (183 `pgTable`s, nullable columns, index counts) are read from `shared/schema.ts` migration source, not confirmed against actual row-level data. Findings whose blast radius depends on "how many rows actually have `association_id = NULL`" or "how many tenants actually exist" (A-AUTHZ-001, A-AUTHZ-002/003/004) are bounded in CURRENT impact by the fact that YCM has ~1 live tenant (Cherry Hill) today — but this is a fact stated in the audit brief/charter, not independently re-verified against a live DB query.
- **No dead-code / unused-dependency findings.** Lane-c explicitly declined to assert any dead-code findings to avoid false positives, noting that reliable detection needs dynamic/route/config/flag-use tracing it did not have budget to complete. This is a genuine, unfilled gap — not a "we checked and found nothing" result.
- **The React client was out of scope for the security lane.** Portal-token storage medium, XSS surface, and CSRF posture on the browser side are unverified (A-AUTH-001's client-storage claim is a hypothesis).
- **No live CI/CD runtime observation beyond one first-hand incident** (the #10172 self-hosted-runner outage lane-c cites directly).
