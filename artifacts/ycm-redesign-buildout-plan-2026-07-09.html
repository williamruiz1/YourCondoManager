YCM Redesign — Build-Out Plan

Three surfaces on the new brand, in priority order: Manager app → Owner portal → Owner app. Grounded in the gated redesign preview and the current codebase.

1Executive summary

Build order (priority): 1) Manager app (paying users — drives adoption, build first) → 2) Owner portal → 3) Owner app (greenfield native).

Owner-not-resident reframing: the redesign's "Resident App" is built as an OWNER app — for condo unit OWNERS, not renters. Owner-centric features only (pay dues, statements/ledger, book amenities, submit requests, documents/announcements). Reframed throughout.

The honest framing: the platform is mature, not greenfield — 141 routes, 169 tables, a working Owner Portal, and live Stripe/Connect payments already exist. So Manager & Portal are mostly REBRAND + IA cleanup (one net-new module: Violations); the Owner app is a greenfield native shell with maximum reuse. Stack today: React + Vite + wouter + react-query + Tailwind + drizzle + express + passport/session.

0Cross-cutting foundation — build ONCE, all 3 consume

F1 — Shared design system (@ycm/design-system) — CRITICAL PATH. Light-first — dark mode stays SHELVED (current force-light default).

F2 — Auth/session parity across manager (/app/*), portal (/portal/*), owner app — one session model; migrate the full surface, never lock out a class of users.

F3 — Gated review → William signoff per surface: publish a wireframe round to the YCM admin hub, get explicit "build this" before porting to live code. Wireframe-first, never reactive tweaks in production.

4Surface 3 — Owner app (priority 3 · greenfield)

5Top 3 sequencing / dependency risks

1. The design system (F1) is the whole critical path. Build + lock @ycm/design-system and get signoff before any surface starts, or you get three drifting rebrands. F1 is Phase 0; nothing else starts until it lands.
2. Owner app depends on the portal being restyled first. The Capacitor app reuses portal screens/logic — building the native shell before the portal restyle wraps soon-to-change UI. Enforce Manager → Portal → Owner-app order.
3. Money surfaces + IA route de-dup. Payments/Ledger/Pay-Dues touch live Stripe + Cherry Hill's books, and consolidating duplicate /app/financial(s)/* routes can break payment paths / deep links. Test-mode-only in dev, flag-gate, keep redirects, William ratify before any live-money change.
