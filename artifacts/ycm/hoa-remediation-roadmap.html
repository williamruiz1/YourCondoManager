YourCondoManager — HOA Financial Remediation Roadmap

Turning the best-practices audit into a sequenced, buildable plan. Every step is grounded in the real YCM codebase, and every data-touching step carries a migration + rollback + verification plan because Cherry Hill Court is a live customer with real books. This is scoping — no code is changed and no git is run here.

Companion to artifacts/ycm/hoa-best-practices-audit.html · First customer: Cherry Hill Court (18-unit condo) · Ratified model (2026-07-01): the balance attaches to the UNIT, not a person · Grounded in shared/schema.ts, server/services/gl/*, server/services/reconciliation/*, server/services/account-statement.ts

Start here → Phase 1: Unit-Centric Ledger + Unique Payment Reference (P0-1 + P0-3, shipped together)

This is the architectural pivot the owner already ratified, and it's the one thing everything else compounds off. It fixes the liability model (the receivable belongs to the unit, co-owners jointly & severally liable) AND makes reconciliation deterministic (match on a per-unit reference first, fuzzy-name last). It's also the lowest data-migration risk of the P0s, because the winning move is additive: add a nullable unit_account_ref and a payer roster beside the existing rows, backfill from data that's already there (owner_ledger_entries.unitId is already notNull; Stripe metadata already carries unit_id), and re-scope statements to the unit behind a flag — without deleting or rewriting a single ledger row.

2 · Sequencing & dependencies

3 · The plan, gap by gap

PHASE 1 Unit-centric receivable + unique payment reference P0-1 P0-3

Make the unit the balance-bearing entity; issue each unit a stable payment reference; aggregate all co-owner names into a per-unit payer roster with a designated primary contact. Shipped together because they are the same pivot.

P0-1 Ledger is person-centric, not unit-centric

Current state (cited)
owner_ledger_entries (schema L729) requires both unitId AND personId as notNull(). account-statement.ts (L71–154) scopes the canonical statement to personId ("an owner may hold multiple units under one personId"), with unitId only an optional narrower. ownerships (L86) already models many-persons-per-unit with ownershipPercentage + startDate/endDate.
Target state (CIRA / ASC 972)
The assessment receivable is an obligation of the unit; co-owners are jointly & severally liable; the lien attaches to the property. personId becomes payment metadata (who tendered), not the balance owner. A per-unit payer roster (all authorized co-owner names + descriptors) sits beside the unit balance; one co-owner is flagged the primary contact.

Concrete build steps

  • Schema: make owner_ledger_entries.personId nullable (it becomes optional "tendered-by" metadata); unitId stays notNull and becomes the sole balance key. Add index on (associationId, unitId, postedAt) for unit-scoped statement reads.
  • Schema — payer roster: add unit_payer_roster (unitId, personId nullable for non-owner third-party payers, displayName, normalizedName, isPrimaryContact, role enum {owner|co-owner|tenant|third-party}, active). This is the deterministic name source the matcher reads, derived from ownerships but explicitly editable.
  • Service: add buildUnitAccountStatement({associationId, unitId, from, to}) in account-statement.ts that scopes to unitId and aggregates ALL co-owner ledger rows for that unit. Keep buildAccountStatement(personId) as a per-owner view (a filter of the unit statement), not the source of truth.
  • Service: the owner-ledger summary / balance readers (server/storage.ts getOwnerLedgerSummary, referenced by gl/reconcile.ts) already sum by Σ amount — re-key their grouping from personId to unitId. This is the balance-of-record change.
  • UI: financial-ledger.tsx + financial-statement.tsx group by unit with the payer roster shown as "authorized payers for Unit X"; the owner portal still shows the owner their unit's balance.
Data migration plan (CRITICAL — live CHC books):
  • Additive-first, never destructive. No ledger row is deleted or its unitId rewritten — every existing row already carries a valid unitId (it's notNull today), so the unit balance is already fully derivable. The migration only (a) relaxes the personId NOT NULL constraint and (b) creates the roster table + backfills it from ownerships ⋈ persons.
  • Staged: (1) ship the roster table + backfill in one migration (reversible: drop table); (2) ship the nullable-personId constraint change in a second migration (reversible: the existing rows all still have personId, so re-adding NOT NULL succeeds as long as no null rows were inserted — gate new null-personId inserts behind the flag until verified); (3) flip statement reads to unit-scoped behind UNIT_CENTRIC_LEDGER flag, CHC-only allowlist mirroring GL_ENABLED_ASSOCIATIONS.
  • Verification (must equal, to the cent): a read-only script asserts Σ(owner_ledger by unit) == Σ(owner_ledger by person, mapped to unit) for CHC, and that every unit's new unit-scoped statement closing balance equals the sum of its owners' old per-person statements. Run before the flag flip; block the flip on any variance.
  • Rollback: flip UNIT_CENTRIC_LEDGER off → statement reads revert to person-scoped instantly (the person path is never removed, only demoted to a view). The roster table + nullable constraint are harmless when the flag is off.

P0-3 No unique payment reference — matching keys on name+amount

Current state (cited)
reconciliation/auto-matcher.ts scores amount-exactness (0.55) + date-proximity + fuzzy payor-name (Levenshtein tokens, 0.25) + the learned bank_descriptor_aliases map (schema L3930). Primary signal is name+amount → collides by design (two owners, same dues, same day → both to manual review). Good news: Stripe metadata (stripe-charge-metadata.ts §3.1) already sets unit_id on every charge, and payment_transactions (schema L3429) already has a unique receiptReference column.
Target state (lockbox / AAB / Vantaca)
Each unit gets a stable, unique reference number that must appear on the remittance. Payments resolve to the unit by that key deterministically (confidence 1.0, no review) before any name matching. Name/alias matching is the last resort only.

Concrete build steps

  • Schema: add units.unitAccountRef (text, unique per association) — a short, human-readable per-unit reference (e.g. CHC-0007). Generated once per unit; never reused.
  • Reference on the remittance (two rails): (Stripe rail) put unit_account_ref into the Stripe metadata object already built in stripe-charge-metadata.ts (it already carries unit_id) and into the statement-descriptor suffix vocabulary; (Lockbox/check rail) print the ref as a coupon line on the owner statement so mailed checks carry it in the memo.
  • Matcher: add a Tier-0 exact-reference pass in auto-matcher.ts that runs BEFORE the scored pass — if the bank descriptor / Stripe metadata contains a known unitAccountRef, match to that unit at confidence 1.0, skip review. Fall through to amount+alias, then fuzzy-name as today.
  • UI: financial-payments.tsx + owner portal show the unit's reference on the statement/coupon; admin-reconciliation.tsx shows "matched by reference" as a distinct, no-review tier.
Data migration plan:
  • Additive, non-destructive: one migration adds the nullable unitAccountRef column + a backfill that assigns a generated ref to every existing unit (deterministic from association slug + unit number, so re-running is idempotent). No financial row is touched.
  • No behavior change until wired: the reference is issued and printed first (owners start putting it on payments); the Tier-0 matcher pass ships behind a flag and is proven against historical CHC bank transactions in a dry-run (does it match what the humans already matched?) before it auto-applies.
  • Verification: replay the last N months of CHC bank transactions through the Tier-0 pass in report-only mode; assert it never contradicts an existing confirmed match, only adds confidence. Rollback: flag off → matcher reverts to the current scored-only behavior; the column is inert.
Why P0-1 + P0-3 ship together: the unique reference (P0-3) is what makes the unit-centric balance (P0-1) resolvable from a bank feed without name-guessing. Built separately, each is half a fix. The roster (P0-1) also gives the fuzzy fallback a unit-scoped name list instead of an association-wide one, cutting false positives even before the reference lands.

PHASE 2 Segregation of duties + dual-approval on disbursements P1-2

The #1 HOA embezzlement control. No single person authorizes, disburses, reconciles, and records. Independent of everything else and low risk (purely additive AP tables — touches no existing books), so it can run in parallel with Phase 1.

P1-2 No maker-checker on money-out

Current state (cited)
vendors (schema L559) and vendor_invoices (L578) tables already exist (expense capture), but there is no disbursement / check-run / payment-approval model and no maker-checker gate anywhere in schema. audit_logs (actor/action/before/after) records changes but does not enforce a second approver.
Target state (CAI / Hovland / HOAleader)
Money-out over a threshold ($1k–$2.5k) requires a second approver; reserve transfers require two approvers. One person cannot author + approve + disburse the same payment. Both approvers captured in the audit log.

Concrete build steps (all additive — new tables only)

  • Schema: add disbursements (associationId, vendorInvoiceId nullable, fund enum reused from GL, amountCents, status {draft|pending-approval|approved|paid|void}, createdBy, requiresSecondApproval, approvalThresholdCents). Add disbursement_approvals (disbursementId, approverPersonId, decision {approve|reject}, at) — the maker-checker record.
  • Schema: add approval_policies (associationId, dualApprovalThresholdCents default $1000, reserveTransferRequiresTwo bool). Board-configurable per association.
  • Service: a disbursement state machine that refuses the paid transition unless the approvals satisfy the policy (maker ≠ checker enforced in code, not just UI). Reserve-fund transfers always require two regardless of amount.
  • UI: new financial-expenses.tsx disbursement queue → "needs your approval" surface for the second approver → paid. The audit log entry carries both approver identities.
Data migration plan: zero risk to live books — this is entirely new tables and a new UI flow; it does not read or write owner_ledger_entries or any existing financial row. Migration = create tables + seed a default approval_policy per association. Rollback = drop the tables (nothing else references them). Ships behind a DUAL_APPROVAL flag; when off, the disbursement flow simply isn't surfaced.
Why do this early despite being P1: it's the highest-value control per dollar of effort, it de-risks the moment CHC starts paying vendors through the platform, and it has no data-migration exposure at all — the exact opposite of the GL flip. Building it in parallel with Phase 1 means the money-out controls are live before Phase 3 turns on the expense side.

PHASE 3 Turn the real GL on for CHC P0-2

Build the expense/vendor/bank side so journals balance both ways, pass the reconcile-to-the-cent gate, then flip GL_ENABLED_ASSOCIATIONS for Cherry Hill. The GL is already built and correct — this is the expense side + the flip, not a rewrite. Highest care: this is when the GL becomes the association's real books.

P0-2 The live books are a single-sided dues subledger; the real GL is OFF

Current state (cited)
A correct, idempotent, fund-aware GL exists: gl_accounts + gl_entries (schema L3983/L4018, integer cents, Σdebit==Σcredit, unique source-leg index). gl/flag.ts gates it behind GL_ENABLED / GL_ENABLED_ASSOCIATIONS (default OFF), and gl/runtime-sync.ts adds a reconcile-to-the-cent gate on top. BUT: CHART_OF_ACCOUNTS (gl/posting.ts L60) has only income/asset/liability/equity accounts — no expense accounts, and the posting core maps only owner_ledger_entry. The vendor_invoice / bank_transaction source types are declared but unwired.
Target state (GAAP ASC 972)
Accrual-basis, fund-aware, double-entry GL that posts BOTH the dues side (already built) AND the expense side (vendor invoice → AP → cash), producing a real balance sheet + income/expense statement by fund. The owner subledger becomes a view of the GL, not a parallel truth.

Concrete build steps

  • Chart of accounts: extend CHART_OF_ACCOUNTS in gl/posting.ts with the expense accounts (5xxx: utilities, R&M, management, insurance, legal, admin) + 2000 Accounts Payable (liability) per fund. The audit notes "the full 40+ expense accounts arrive when vendor_invoices are wired (later)" — this is that.
  • Posting core: add the vendor_invoice mapping (accrue: Dr Expense / Cr AP) and the disbursement/bank_transaction mapping (pay: Dr AP / Cr Cash) to the pure posting.ts mapper — reusing the existing balanced-legs + idempotent-source-leg machinery. Wire it to the vendors/vendor_invoices tables and the Phase-2 disbursements.
  • Opening balances: a one-time opening_balance journal (source type already declared) that establishes CHC's starting cash, AR, AP, and fund balances as of the go-live date, so the GL doesn't start at zero.
  • Reconcile-to-the-cent: run script/verify-gl-reconcile.ts (via gl/reconcile.ts) — it already proves Σ owner-ledger == GL Accounts Receivable to the cent. Extend it to also assert the expense side ties out (AP + cash balance sanity).
  • Flip: set GL_ENABLED_ASSOCIATIONS = CHC's association id. The runtime-sync reconcile gate then refuses to post if the ledger doesn't reconcile — so a bad flip is self-blocking.
Data migration plan (the highest-risk step — treat with maximum care):
  • The GL is forward-only & parallel by design — per runtime-sync.ts comments, the owner ledger STAYS the system of record; the GL runs alongside and is derived. So "turning it on" does not mutate a single owner-ledger row. This is what makes the flip reversible.
  • Staged flip: (1) build the expense-side posting + extended chart in code, off; (2) run the GL in shadow for CHC (post derived legs, but statements still served from the subledger) and let the reconcile gate run for a full close cycle; (3) only after the reconcile-to-the-cent gate passes across a full month AND the expense side ties out, flip statement rendering to GL-sourced.
  • Verification (the gate IS the verification): the reconcile-to-cent gate (gl/reconcile.ts) is already the acceptance test — differenceCents MUST be 0 and invariantViolations MUST be empty, or postings are refused. Add the expense-side tie-out (bank cash balance vs GL cash, AP aging vs vendor invoices outstanding) as a second assertion before flipping statement rendering.
  • Rollback: remove CHC from GL_ENABLED_ASSOCIATIONS → GL sync stops, statements revert to the subledger instantly. Because GL legs are additive with a unique source-leg index, re-enabling later re-derives cleanly with no duplicates. The GL tables can be truncated for CHC without touching the books.

PHASE 4 Three-way reconciliation tie-out P1-1

Prove bank balance = GL cash = sum of owner subledger balances at period close, itemize reconciling items (deposits-in-transit, outstanding checks), and block locking a period on an unexplained variance. Depends on Phase 3 for the GL cash side.

P1-1 Reconciliation is one-way counts, not a balance proof

Current state (cited)
reconciliation/report.ts produces bankDepositsCents, ledgerPaymentsCents, matched/unmatched counts, and a gapCents = ledger − bank. It's a payment-matching inventory. reconciliation_periods (schema L2708) already has open → closed → locked with closedBy/lockedBy. There is no assertion that the books equal the bank, and no deposits-in-transit / outstanding-items model.
Target state (CAI / AICPA)
A monthly three-way tie-out: bank statement balance ± in-transit/outstanding items = GL cash = Σ subledger. Any difference is itemized and investigated; a period can't lock with an unexplained variance.

Concrete build steps

  • Schema: add reconciling_items (periodId, type {deposit-in-transit|outstanding-check|bank-error|book-error}, amountCents, note, clearedAt) hung off reconciliation_periods.
  • Service: extend report.ts into a threeWayTieOut(periodId) that computes bankBalance ± Σ(open reconciling items) vs GL cash (from gl_entries cash accounts, Phase 3) vs Σ subledger, and returns the variance + the itemized bridge.
  • Gate: add a period-lock precondition in the reconciliation_periods close/lock path that refuses locked when the tie-out variance ≠ 0 and no reconciling item explains it.
  • UI: admin-reconciliation.tsx / financial-reconciliation.tsx render the tie-out bridge (three columns that must agree) and the "can't lock — $X unexplained" block.
Data migration plan: additive — new reconciling_items table + read-only computation over existing GL + subledger + bank rows. No existing financial row is modified. The lock-gate is behavior-gated behind a THREE_WAY_TIEOUT flag so existing periods can still close during rollout. Verification: run the tie-out over the last few closed CHC periods in report-only mode; confirm they tie (or the variance is a real, explainable item) before making the gate blocking. Rollback: flag off → period locking reverts to today's behavior; the table is inert.

PHASE 5 Maturity layer P1-3 P2-1 P2-2 P2-3

Fund-separated special assessments, AR aging buckets + lien lifecycle, reserve study + funding plan, and the annual CPA audit export. Sequence by board demand. The GL-dependent items (5a, 5c) ride on Phase 3; the ledger items (5b) ride on Phase 1.

P2-1 Fund-separate the special assessment rides on Phase 3

Current state (cited)
special_assessments (schema L427) is a solid model (allocation methods, installments, loan-style terms, auto-post) and posts to the ledger — but with GL off there is no fund tag, so Cherry Hill's driveway money isn't accounted as restricted activity separate from operating cash.
Target / build
Post special-assessment charges/receipts to a dedicated fund/GL account (a reserve or project fund) so the driveway project has its own visible balance and can't be spent on operations. Small once Phase 3's GL + fund dimension is live — it's a mapping in the posting core + a fund tag on the assessment.
Migration: add an optional special_assessments.fund tag (defaults to operating for backward compat); the GL posting maps flagged assessments to the reserve/project fund. Additive, reversible (fund tag defaults to today's behavior). Verify the driveway assessment's fund balance in the GL equals its outstanding receivable.

P2-2 AR aging buckets + lien lifecycle rides on Phase 1

Current state (cited)
Strong staged notices exist: delinquency_settings, delinquency_notices (30/60/90), delinquency_escalations, late_fee_rules/events, collections_handoffs. Missing: an aging-bucket report (current/30/60/90+) and a lien object (notice-of-intent → filed → released).
Target / build
A unit-centric aging report (current/30/60/90+) computed off Phase-1's unit ledger, showing delinquency % against the CAI benchmark (≤3% excellent). A new liens table modeling the statutory pre-lien → lien-filed → foreclosure ladder, linked to the unit.
Migration: the aging report is read-only over the unit ledger (no migration). The liens table is additive (unitId, status enum, filedAt, releasedAt, amountCents, docRefs). Reversible = drop the table. Depends on Phase 1 so aging ages the unit, not the person.

P1-3 Reserve study & funding plan rides on Phase 1/GL

Current state (cited)
The GL can hold a reserve fund, but there is no reserve-study data model — no component inventory, no % funded, no contribution driver. A CT reserve-disclosure checklist exists (migration 0049_ct_reserve_disclosure.sql) but no funding math.
Target / build
A reserve_studies + reserve_components model (component, useful life, remaining life, replacement cost, current reserves) that computes the recommended monthly reserve contribution and % funded (target 70–100%), surfaced on the dashboard and feeding the budget's reserve line.
Migration: purely additive new tables; no existing financial row touched. Reversible = drop tables. Independent of the GL flip for the study itself (it's planning data); the % funded figure reads the reserve-fund balance, which is best sourced from the live GL (Phase 3) but can fall back to the reserve bank account balance.

P2-3 Annual CPA audit/review export rides on Phase 3

Current state (cited)
No audit workflow or CPA export package anywhere. server/services/gl/statements.ts already derives financial statements (read-only, GL-gated).
Target / build
A year-end export package — trial balance, financials by fund, reserve disclosure, GL detail — generated from the live GL (Phase 3) for the outside CPA, per the AICPA CIRA guide. Lowest priority: get the books right first.
Migration: none — pure read-only export over the live GL. No schema change. Depends on Phase 3 (needs the GL to be source-of-truth) and benefits from Phase 4 (the tie-out proof is part of a clean package).

4 · The biggest data-migration risk, and how the plan de-risks it

The single biggest risk is Phase 3 — flipping the GL to source-of-truth on a live customer's books. A wrong GL that becomes authoritative would silently misstate what the association owns, owes, and holds by fund. The audit even warns the GL "may NOT become source-of-truth until the reconcile-to-the-cent gate passes."

The plan de-risks it structurally, not by hoping:

  • The GL is forward-only & parallel by construction. Per runtime-sync.ts, the owner ledger STAYS the system of record and the GL is derived alongside it. Turning the GL on mutates zero owner-ledger rows — so the flip is inherently reversible (remove CHC from the allowlist and statements revert instantly).
  • The reconcile-to-the-cent gate is a hard, code-enforced precondition (gl/reconcile.ts → runtime-sync.ts): even an allowlisted association gets no GL postings unless differenceCents === 0 and there are zero invariant violations. A ledger that can't reconcile is refused a GL — "surfacing nothing beats surfacing a wrong statement."
  • Shadow-then-flip. Run the GL in shadow for a full close cycle (post derived legs, keep serving statements from the subledger), let the gate run, and only flip statement rendering after a full month reconciles AND the expense side ties out.
  • Idempotent, additive legs. The unique source-leg index means re-running the sync never double-posts; the GL tables can be truncated for CHC and re-derived cleanly. Nothing is destructive.

Phase 1 (the pivot) is de-risked the same way: it's additive — add unitAccountRef + the payer roster beside the existing rows, backfill from data already present (owner_ledger_entries.unitId is already notNull; Stripe already carries unit_id), and re-scope statements behind a UNIT_CENTRIC_LEDGER flag with a before-flip verification that every unit's new balance equals the sum of its owners' old balances, to the cent. No ledger row is deleted or rewritten; rollback is a flag flip.

5 · Recommended starting point

Start with Phase 1 (P0-1 + P0-3), and run Phase 2 (dual-approval) in parallel.

  • Phase 1 first because it's the ratified architectural pivot, it fixes the liability model AND makes reconciliation deterministic, and nothing else should be built on the person-centric model — every later phase (aging, tie-out, statements) reads cleaner off the unit ledger. It's also the lowest-risk P0 (fully additive migration).
  • Phase 2 in parallel because it depends on nothing, is the highest-value control per unit of effort (the #1 embezzlement vector), and has zero live-book exposure (new AP tables only) — so it can proceed while Phase 1 is verified.
  • Do NOT start with Phase 3 (the GL flip) despite it being tempting — it's the highest-risk step and it's more valuable after Phase 1, because a unit-centric, reference-matched ledger is exactly the clean input the reconcile-to-the-cent gate wants.
