New-Client Onboarding Template

The repeatable, low-effort setup path for every future YourCondoManager association — connect the bank, let the engine identify the feed, confirm only the ambiguous few, and let it learn. Cherry Hill Court is the worked example throughout.

Playbook · reusable template · Cherry Hill Court = reference build

The promise

Setting up a new association should be five clean steps, not a data-entry marathon. YourCondoManager already has the matching brain (server/services/reconciliation/auto-matcher.ts). Once the association's records are wired correctly and the bank is connected, the engine does the heavy lifting: it auto-identifies confident deposits, routes the ambiguous few to a review queue, and learns from every human confirmation so the effort drops every month. This template captures that path — plus a data-readiness checklist built from the exact bugs Cherry Hill hit, so a new client is set up clean from day one.

1

Register the association, its units, and its owners

The foundation. Every later step (especially reconciliation) depends on this being wired correctly.

Create the association, add each unit, and attach the owner(s) of record to each unit. This is where the whole system either works cleanly or silently breaks later — so it carries the tightest checklist.

Data-readiness checklist — get this right and reconciliation just works:
  • Every owner has an association_id set (NOT NULL). This is the bug that broke Cherry Hill: the matcher's owner-directory query filters on persons.association_id, so a blank association means the engine can't "see" that owner and matches nothing.
  • Every unit has an active owner. An ownerless unit can't be billed dues and produces no ledger entries to match against.
  • No duplicate person records. One real owner split across two or three person IDs splits their ledger — payments land on one ID while charges sit on another, and the balance never reconciles.
  • primary_contact_person_id is set on every unit. Without it, statements, notices, and payment reminders have no addressee.
2

Set the dues / assessment plan and the payment reference

This creates the expected charges — the ledger entries a bank deposit gets matched to.

For each unit, set the assessment plan and the current dues rate, plus a unique per-unit payment reference.

  • Current dues rate per unit. Rate changes are first-class — Cherry Hill moved from $280 → $330/mo, so the plan carries the effective-dated rate and the matcher's amount signal tracks the current expected amount, not a stale one.
  • A unique per-unit payment reference (unitAccountRef, e.g. CHC-0007). When an owner puts that code on their remittance, the engine's Tier-0 reference pass matches the deposit to that exact unit at confidence 1.0 — no review, no name-guessing. This is the single highest-leverage field for hands-off reconciliation.
  • Opening balances loaded (if migrating mid-year) so the engine has real balances to match deposits against.

The matcher compares each deposit's amount to the unit's expected/open balance within a ±$1 "exact" tolerance (and a ±$50 window for balance-based suggestions), so a clean current-rate plan is what lets confident deposits auto-apply.

3

Connect the bank feed

Link the operating account; the transaction feed becomes the reconciliation source.

On the Bank Accounts page, link the association's operating account (Stripe Financial Connections). The feed pulls in — for Cherry Hill this was 91 Chase transactions (70 owner deposits + 21 debits). Owner payments arrive as credits; the engine only considers credits from the last 30 days for the live matcher (older activity is handled by the report period filter).

One reconciliation surface, not two lists. The Bank Accounts page shows a single table where every transaction is identified by the engine — who it's from, what for (dues vs. assessment), matched or not, and a confidence — rather than a raw dump. The human works only the rows that need it.
4

Run the auto-matcher — confirm only the ambiguous few

The engine identifies the confident deposits; you touch just the rest, and it learns as you go.

The matcher scores every deposit against every open charge using a multi-signal confidence (amount exactness, date proximity, fuzzy payor-name, and the unique reference). It sorts each row into a disposition:

DispositionWhenWho acts
Matched by referenceThe unique per-unit ref (CHC-0007) is in the descriptor — confidence 1.0Nobody — auto-applied
Auto-matchedConfidence ≥ 0.85 (exact amount + name, single candidate)Nobody — auto-applied
Suggested — confirmOne-name match / near amount — a strong guess below the barOne-click confirm (teaches an alias)
Needs youNo name in the descriptor, or two owners plausibly match (ambiguous)Human picks the owner
The learning loop this is why effort drops
Every human-confirmed match writes a descriptor→owner alias. The next deposit carrying that same bank descriptor auto-matches at confidence 0.99 — no review needed. So a payer whose descriptor was confirmed once in month 1 flows through untouched in month 2. The system gets more hands-off the longer it runs.

It refuses to guess wrong: if two owners could plausibly match the same deposit (tied confidence above the bar), neither auto-applies — both surface for a human. False-positive protection is built in.

On Cherry Hill's 70 unapplied deposits: 38 name-matched, 24 would auto-match cleanly, and the remaining ~32 surfaced in the review queue for a quick human pass — instead of a treasurer hand-keying all 70.

5

Enable the General Ledger + statements

Forward-only, parallel, flag-gated — with a balance banner that surfaces any discrepancy.

  • GL runs forward-only and in parallel, behind a feature flag — it doesn't rewrite history; it accrues from the go-live point alongside the existing ledger, so switching it on is safe and reversible.
  • The out-of-balance banner surfaces any discrepancy between the books and the reconciled feed, so nothing drifts silently — a discrepancy becomes a visible review item, not a hidden gap.
  • Statements go out to each unit's primary contact (set in Step 1) — which is why that field matters.

Effort over time

Month 1

Highest human touch. The engine auto-matches the obvious deposits (exact amount + name, or a unique ref), but first-time payers with odd descriptors land in Suggested / Needs-you. You confirm those once — and each confirmation writes an alias.

Month 3+

Most recurring payers now carry a learned alias, so their deposits auto-match at 0.99 with zero clicks. The only rows that still need you are genuinely new payers or truly opaque descriptors (e.g. DEPOSIT ID NUMBER 95016). Effort drops sharply as aliases accumulate.

Run this checklist at onboarding and a new client starts clean: the engine sees every owner, deposits carry a reference, and reconciliation is hands-off from month one.

Why this is the standard path

Built once, every new association inherits it. The same five steps and the same data-readiness checklist onboard the next client — no per-client custom engineering. Connect the bank → the engine identifies the feed → the human confirms only the ambiguous few → the system learns. That's the low-effort, reconciliation-driven setup Cherry Hill proved, made repeatable.

Scope note: this is a playbook/design artifact. It documents the existing deterministic + descriptor-alias-learning engine (auto-matcher.ts) and the data conditions it needs. No live database writes, code changes, or money movement are part of this document.
