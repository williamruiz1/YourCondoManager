Auto-Reconciliation System — Design

Cherry Hill as the reference build · a reusable matching system for every future client + migration

The headline (the important part)

YourCondoManager already HAS a sophisticated auto-reconciliation engine. The matching brain is built (server/services/reconciliation/auto-matcher.ts) and it does most of what you asked for. The problem you hit isn't a missing engine — it's that the Bank Accounts page doesn't surface the engine's intelligence. It shows two raw transaction lists with no identification, instead of the engine's answer. So this is connect-the-brain-to-the-screen, not build-a-brain.

What's actually broken the gap

The Bank Accounts page is the problem, not the engine.
  • It shows a "Connected bank → Transactions" raw list and a separate "Pending reconciliation" list — the duplicate lists you saw.
  • Its transaction rows are a thin {amount, description, date} — no column says who the deposit is from, what it's for (dues vs assessment), or whether it's matched.
  • The engine's rich output (owner, confidence, suggestions, learned aliases) isn't wired to this page — so it reads as a dumb dump, not reconciliation.

The design — one clear reconciliation surface build

Replace the two confusing lists with ONE table where every bank transaction is identified by the engine.

DateFrom (bank descriptor)AmountIdentified asForConfidenceStatus

  • One table, columns that identify each item: who, what for, matched-or-not, confidence. The duplicate raw list is gone (it becomes a filter, not a second tab).
  • The human only touches the few that need it — Suggested (one-click confirm, which teaches the alias) and Needs-you. Auto-matched rows just show the answer.
  • Dues vs assessment is read from the matched ledger entry (now that income is split) so each deposit lands in the right account.

Why this is reusable for every future client

The same surface is the standard onboarding path: connect the bank → the engine auto-identifies the feed → the human confirms only the ambiguous few → each confirmation teaches the system. By the second month per client, most deposits auto-match from learned aliases. This is the low-effort migration path you described — built once, every new association inherits it.

Build plan (Cherry Hill = reference)

  1. Surface the engine on the Bank Accounts page — one reconciliation table with the identification columns; kill the duplicate list. (UI; the engine already returns the data.)
  2. Load the assessment — per-owner opening balances + chosen plan from your spreadsheet, so the engine has ledger entries to match the 91 Chase deposits against.
  3. Run the matcher on the 91 transactions — auto-match the confident ones, surface the rest; you confirm a handful, the system learns the descriptors.
  4. That's the template — the same three steps onboard the next client.

Honest scope note: steps 1–3 use the existing deterministic + learning engine. A heavier statistical/ML layer (clustering recurring payers, anomaly flags) can sit on top later — but the deterministic + descriptor-alias-learning engine already gives most of the "minimal effort once integrated" you want, without the risk of a black-box guess on a live customer's money.
