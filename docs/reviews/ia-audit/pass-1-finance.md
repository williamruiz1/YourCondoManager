## DEMOTE-ADMIN handovers

None. All 13 pages in scope have a primary persona of `manager` (or `board-admin`/`viewer` as secondary consumers). No page in the Finance section is platform-operator-only. Role checks in `server/routes.ts` show consistent multi-role access (`platform-admin, board-admin, manager, viewer` on GET endpoints; `platform-admin, board-admin, manager` on write endpoints). No DEMOTE-ADMIN verdicts are warranted.

## Cross-refs

- **`/app/financial/foundation` forward link to `/app/financial/recurring-charges`** (line 450 of `financial-foundation.tsx`): the JSX renders `<Link href="/app/financial/recurring-charges">` in the "forward guidance" card. This link resolves via the `<RouteRedirect>` back to `/app/financial/foundation` — creating a navigation loop to the same page. Phase 6 should update this link to anchor to the `recurring-charges` tab, e.g. `/app/financial/foundation?tab=recurring-charges`, once the dead file is retired.
- **`financial-payments.tsx` — partial-payment exceptions tab:** The Payments page handles configuration of partial-payment rules and exceptions. This implicitly overlaps with the "collections" concern surfaced in `financial-late-fees.tsx` (payment plans / delinquency tracking). The two are co-located in different hubs (Payments vs. Billing). A future Phase 5 nav proposal may want to consider whether Payments and Billing should be merged into a single "Collections & Payments" hub to avoid conceptual split, but that is a proposal-phase question, not a Phase 1 verdict.
- **Owner-facing payment surface:** `financial-payments.tsx` generates owner payment links that direct residents to a `/pay/:token` or similar route. That consumer surface is outside `/app/*` scope and should be tracked in a follow-on audit.

## Legacy coverage analysis

For each of the 8 legacy files, whether the redirect-target hub actually covers the content:

- **`financial-budgets.tsx` → `/app/financial/expenses`:** COVERED. `FinancialExpensesPage` directly imports `FinancialBudgetsContent` and renders it as the "Budgets" tab. The legacy file is the content component; the hub is the container. Verdict: **KILL**.

- **`financial-utilities.tsx` → `/app/financial/expenses`:** COVERED. `FinancialExpensesPage` directly imports `FinancialUtilitiesContent` and renders it as the "Utilities" tab. Full utility payment lifecycle (log, filter by status, attach receipts) is present. Verdict: **KILL**.

- **`financial-recurring-charges.tsx` → `/app/financial/foundation`:** COVERED. `FinancialFoundationPage` directly imports `FinancialRecurringChargesContent` and renders it as the "Recurring Charges" tab. Full schedule management + run history + retry is present. Verdict: **KILL**.

- **`financial-reconciliation.tsx` → `/app/financial/reports`:** COVERED. `FinancialReportsPage` directly imports `FinancialReconciliationContent` and renders it as the "Reconciliation" tab. Full CSV import → auto-match → manual-match → period-close workflow is present. Verdict: **KILL**.

- **`financial-invoices.tsx` → `/app/financial/expenses`:** COVERED. `FinancialExpensesPage` directly imports `FinancialInvoicesContent` and renders it as the "Invoices" tab. Full vendor invoice lifecycle (draft → received → approved → paid → void) plus attachments and bulk actions is present. Verdict: **KILL**.

- **`financial-late-fees.tsx` → `/app/financial/billing`:** COVERED. `FinancialBillingPage` directly imports `FinancialLateFeesContent` and renders it as the "Late Fees" tab. The content is notably feature-rich (rules, bulk run, payment plans, delinquency thresholds, collections handoffs), but it is all present inside the billing hub. Verdict: **KILL**.

- **`financial-ledger.tsx` → `/app/financial/billing`:** COVERED. `FinancialBillingPage` directly imports `FinancialLedgerContent` and renders it as the "Owner Ledger" tab. Ledger entry CRUD, CSV import/export, financial alerts, and audit log are present. Verdict: **KILL**.

- **`financial-assessments.tsx` → `/app/financial/billing`:** COVERED. `FinancialBillingPage` directly imports `FinancialAssessmentsContent` and renders it as the "Assessments" tab. Special assessment creation, per-unit exclusion editing, run-and-post workflow are present. Verdict: **KILL**.

**Summary finding:** All 8 legacy financial files function exclusively as exported content-components that are imported and rendered inside their respective hub pages. They are not standalone pages in any meaningful sense — they exist on disk as `.tsx` files with standalone `WorkspacePageHeader` scaffolding that is never rendered via a live route, and as content components embedded in hubs. The architectural pattern is clean and the hubs do cover all legacy content. All 8 are unambiguous KILL candidates pending Phase 6 file deletion.

**Note on C2 (Triplet Rule):** No KILL verdict here requires retiring any API routes. Every `/api/financial/*` route backing a legacy concept is actively consumed by the hub tab that absorbed that content. All routes should be retained as-is. No API deprecation or deletion is warranted for any of the 8 legacy files.
