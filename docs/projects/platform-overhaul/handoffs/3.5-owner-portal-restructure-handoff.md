## Rollback notes

Large refactor = harder to revert than single-file changes. Mitigation per 3.5 Q7 incremental plan: each migration step is its own commit/PR, so revert chain is ordered.

**Per-step revert:**
1. `git revert` the step's merge commit.
2. Deploy the revert.
3. Re-run the portal manual smoke (every portal URL × `owner` session).
4. File a post-mortem issue in PPM documenting root cause.
5. Fix forward in a new PR; re-land.

**Mega-file deletion (step 7) is the point of no return.** After step 7 ships, reverting a single migration step requires re-adding the mega-file from git history AND reverting the step. Mitigation: hold off on step 7 until steps 1–6 have baked in production for at least one release cycle, or until a regression-free confidence threshold is met.

**Big-bang fallback (if 3.5 Q7 locks big-bang):** revert is one PR, but the blast radius is all 8 routes at once. Validation burden at deploy time is higher.

**Data-side caveats:** 3.5 introduces no data migrations. All changes are client-side React route restructure. No SQL migrations, no backend enum changes, no data rollback risk.
