## Rollback + stop-the-line policy (Plan ¬ß8 R7)

Each zone landing is **serially revert-dependent**. Phase 12 Financials is the foundation ‚Äî Phase 13 Operations depends on its `canAccessTab` null-fix + Financials RouteGuard wrappers being live. If Phase 12 reverts, Phase 13 halts until Phase 12 re-lands. The cascade continues: Phase 13 halts Phase 14 halts Phase 15 halts Phase 16.

**Revert trigger (stop-the-line signal, per Q10):**
- Any 401/403 on a route that should be permitted for any persona (Q11 gate 1 fails).
- Any unguarded access discovered on a route that should be blocked (Q11 gate 2 fails).
- Two or more JS console errors not present in the baseline (Q11 gate 3 fails).
- A Tier 1 parity-harness test that was passing pre-deploy starts failing in CI.

**Revert procedure:**
1. `git revert` the zone's merge commit.
2. Deploy the revert.
3. Re-run the zone's manual smoke (every persona √ó every route in the zone).
4. File a post-mortem issue in PPM documenting root cause.
5. Fix forward in a new PR; re-run all Q9 gates + Q11 observability; re-land.

**Escalation (AC-Q10-4):** Two consecutive zone failures trigger a full pause of all subsequent zone landings and a cross-zone root-cause analysis. Escalate to William immediately; do not proceed on a third zone.

Data-side caveats:
- Phase 8a portal-enum-collapse migration is **not** revertible by PR revert alone ‚Äî the reverse migration script restores original role values from `role_migration_log`. This is why Q2 ships behind `PORTAL_ROLE_COLLAPSE`.
- Phase 14 `normalizeAdminRole` deletion is **not** revertible by PR revert alone ‚Äî restore the function from git history if needed. The data migration is idempotent; no data rollback is required.
