## Scope summary

Decouple the role-coupled hub-visibility vocabulary (`public | resident | owner | board | admin`) from the role model. Migrate to the role-agnostic vocabulary `public | residents | unit-owners | board-only | operator-only` adopted verbatim from 2.1 Q11. The `public` literal is preserved unchanged so the anonymous `routes.ts` public hub endpoint (`routes.ts:15319-15337`) does not break.

Two columns migrate together:
- `hub_map_issues.visibility_level` (pgEnum)
- `community_announcements.visibility_level` (text)

Three-phase migration gated by the `HUB_VISIBILITY_RENAME` feature flag (modeled on `PORTAL_ROLE_COLLAPSE`):

- **HV-1** — additive ADD VALUE migration; flag default OFF; commit (#23 `766ae32` `feat(1.5): hub-visibility rename HV-1 — additive enum values + translation helper + flag`).
- **HV-2** — backfill + dual-read; per-association flag flip to begin emitting new vocab; **default flipped to ON** after prod-data audit confirmed zero old-vocab rows in either column. SHIPPED.
- **HV-3** — retire old enum values via enum-recreate + column recast; remove flag from code. **DEFERRED** — flag retained as belt-and-suspenders kill-switch until every call site uses new vocab exclusively.

## Acceptance Criteria

From decision doc:

1. New vocabulary literals `public | residents | unit-owners | board-only | operator-only` are accepted at every read path and emitted at every write path. (HV-1 + HV-2)
2. The `public` literal is bit-identical to the pre-1.5 value. The anonymous public-hub endpoint at `server/routes.ts:15319-15337` continues to return rows where `visibility_level = 'public'` with no behavior change. (HV-1)
3. Both `hub_map_issues.visibility_level` and `community_announcements.visibility_level` migrate together; no partial-state where one is on new vocab and the other is on old. (HV-2)
4. `shared/feature-flags.ts` reserves `HUB_VISIBILITY_RENAME` (default OFF in HV-1 / ON in HV-2). Per-association override supported via `getFeatureFlagForAssociation()`. (HV-1)
5. `shared/hub-visibility.ts` exports `normalizeHubVisibility()` + `oldToNewVisibility()` + `newToOldVisibility()` translation helpers. Reads accept either vocabulary; writes emit the active vocab. (HV-1)
6. Layer-2 primitives AC 9 ("hub-visibility vocabulary decoupled from role model") becomes satisfiable on flag flip. (HV-2)
7. After HV-3, the old enum values (`resident | owner | board | admin`) are dropped from `hub_visibility_level` pgEnum; both columns are recast to use the new enum exclusively; the feature flag is removed from code. (HV-3 — deferred)

## Risks + mitigations

- **Risk:** dropping old enum values in HV-3 with stale data still in the table → write fails. **Mitigation:** HV-2 backfill ran during the per-association flag flip; HV-3 starts only after analytics confirm zero rows still emitting old vocab for one full notice cycle.
- **Risk:** the anonymous public-hub endpoint at `routes.ts:15319-15337` accepts `visibility_level = 'public'` without normalization. **Mitigation:** `public` literal preserved bit-identical across all phases.
- **Risk:** `community_announcements.visibility_level` is `text`, not enum, so accepts arbitrary strings. **Mitigation:** HV-3 column recast to use the new enum will reject non-conforming values; backfill must clean up any drift first.
