## Rollback notes

Route-table changes are broadly reversible by reverting the Phase 11 PR — it is a single PR touching `client/src/App.tsx` + four new hub page files + two placeholder stubs + one deletion (the `/app/admin` alias declaration). No data migrations. No feature flags. No schema changes.

- **Full Phase 11 revert:** `git revert` the merge commit; redeploy. All 61 routes return to the pre-3.2 52-route state. Downstream Phases 12–16 halt per Plan §8 R7 — they cannot land `<RouteGuard>` wrappers on routes that no longer exist.
- **Amendment revert (billing or inbox):** Tiny, independently revertible. Each amendment is one route-declaration line plus one placeholder component file. Revert the amendment commit without touching the Phase 11 hub work.
- **Individual hub revert (not recommended):** The four hubs can be reverted individually in principle, but this produces a mixed state where the sidebar (3.1) links to a hub URL that returns 404. Prefer full Phase 11 revert over partial.
- **Governance/Communications content relocation revert:** The `/overview` relocation is a route-declaration change, not a code move. Reverting the route declaration returns the content to the zone-root URL — no content loss.

Data-side caveats: **none**. 3.2 is a pure client-side router change. No `admin_users.role` or `portal_accesses.role` mutations. No SQL migrations. No feature-flag lever.
