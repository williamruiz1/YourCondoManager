## Acceptance Criteria

Verified during downstream Layer 2 and Layer 3 implementation — not by this handoff directly. Verbatim from the decision doc:

1. No route inside `/app/*` is accessible to a user authenticated only through the Owner Portal portal-access-id flow.
2. No route inside `/portal/*` requires an operator workspace session.
3. A `board-officer` or `assisted-board` user navigating to `/app/portfolio` is redirected to `/app` (preserves 0.1).
4. An `assisted-board` user does not see a "Portfolio Health" nav item in the sidebar.
5. An `assisted-board` user does not see a Settings nav item in the sidebar.
6. An `assisted-board` user sees only the sections in the PM-Managed Default Access Table by default, with PM toggle state applied at runtime.
7. A `board-officer` user has full Manager-equivalent access — no section restrictions, including Settings.
8. No PM toggle may grant `assisted-board` or `pm-assistant` access to `/app/platform/*`, `/app/admin/*`, or `/app/settings/*`.
9. Owner Portal sidebar contains exactly the sections listed under "Current live capabilities" — no HOA payment processing or bill history nav items.
10. Layer 2 permission boundary corrections treat the 0.2 matrix as the authoritative gating spec — deviation requires a new decision doc.
11. A `platform-admin` user is the only persona with access to `/app/platform/*`, `/app/admin/*`, `/app/ai/*`.
12. No `/app/platform/*`, `/app/admin/*`, or `/app/ai/*` API endpoint accepts a role other than `platform-admin` without a locked decision doc citing 0.3 Q5 AC 22.
13. No operator-side nav surface contains a link to `/portal`.
14. A `pm-assistant` user sees only surfaces configured by Manager via PM toggle.
15. A `pm-assistant` user does not see Settings unless Manager explicitly grants access via toggle.
