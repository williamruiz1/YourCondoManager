## Objective

Governance-only handoff. 0.3 locks the navigation framework rules — shape (single role-filtered sidebar), grouping (functional), entry points, breadcrumb convention, role-gating principle (strict isolation), section count ceiling, cross-association navigation, default landings, and the central inbox model. **This spec has no direct build**; it is referenced by every Layer 1, 2, and 3 handoff for boundary conformance checks.

## Acceptance Criteria

Verified during Layer 1, 2, and 3 implementation. Verbatim from the decision doc (43 ACs consolidated from Q1–Q9):

### Q1 — Single canonical navigation
1. The navigation model defines one canonical zone structure.
2. Each persona sees only the sections relevant to that role.
3. Owner, Board Officer, Assisted Board, Manager experiences differ in visible items but not underlying taxonomy.
4. No module introduces a persona-specific top-level taxonomy without a decision doc update.

### Q2 — Functional grouping
5. All pages assigned to a functional zone.
6. Sidebar grouping is not user-toggleable.
7. Zone labels are functional and stable across personas.
8. Lifecycle/audience framing may appear in supporting copy but not as canonical sidebar taxonomy.

### Q3 — Distinct landings, shared shell
9–15. Manager and Board Officer / Assisted Board share `/app`; Owner in `/portal`. 0.3 does not introduce conflicting route prefixes.

### Q4 — Shallow breadcrumbs
16. Breadcrumbs never exceed three visible levels.
17. Association context appears when user is in an association-scoped view.
18. Breadcrumb labels are human-readable and match page titles/section labels.
19. Mobile views may collapse or suppress breadcrumbs.
20. Breadcrumb structure does not imply multiple alternative parent paths for the same page.

### Q5 — Strict role isolation
21. Pages and navigation items are hidden when the role lacks access.
22. Read-only exposure to cross-role content requires explicit approval in spec.
23. Permissions are role-based, not individually customized by default.
24. Role-gating rules documented for each primary persona.

### Q6 — Section count
25–28. Owner navigation stays lean; Manager/Board may exceed five top-level sections only with functional justification.

### Q7 — Cross-association navigation
29. Manager workspace includes a persistent association switcher in the header.
30. Switching association updates the active app context.
31. A portfolio/associations page exists as a secondary navigation surface.
32. Cross-association views clearly distinguished from association-scoped views.
33. Tab-style association switching is not the core navigation model.

### Q8 — Canonical home per persona
34–38. Manager, Board Officer / Assisted Board, and Owner each have one canonical home.

### Q9 — Central updates/inbox
39–43. A central updates/inbox concept exists; does not replace section-native workflows.
