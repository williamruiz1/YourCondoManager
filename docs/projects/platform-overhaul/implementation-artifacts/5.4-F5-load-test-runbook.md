# 5.4-F5 — Load-test runbook (Wave 19)

**Spec:** `docs/projects/platform-overhaul/implementation-artifacts/5.4-performance-audit.md` §6 F5
**Harness:** `script/load-test.ts` (npm run load-test)
**Status:** harness shipped; **not executed in CI** — Wave 19 ships the runner only.

---

## What this is

A small autocannon-based load runner that probes the 5 hot endpoints called out
in the 5.4 audit §2:

1. `GET /api/alerts/cross-association`  — cross-association alert fan-out (Wave 16b batched)
2. `GET /api/operations/dashboard`        — single-association ops snapshot
3. `GET /api/portal/financial-dashboard`  — owner portal landing dashboard
4. `GET /api/financial/assessment-run-log`— paginated audit trail (Wave 19 indexed)
5. `GET /api/portal/me`                   — portal session bootstrap

It prints p50 / p95 / p99 latency, throughput (rps), and error rate per endpoint.

## Why we don't run it in CI

CI runs against a Vitest-mocked server today; the audit endpoints sit behind
real Postgres + sessions. Until we wire a non-prod stack with a stable seed,
this script is for manual investigations on a developer machine or a staging
box that someone has explicitly stood up.

## Running it

### 1. Install the optional devDep

`autocannon` is not part of the lockfile yet (skipped to keep CI installs lean).
Install it locally on demand:

```bash
npm i -D autocannon
```

### 2. Boot a dev server

```bash
npm run dev
# server listens on http://localhost:5000
```

### 3. Mint a session token

The endpoints all require an authenticated session. Two options:

- **Cookie-based session (recommended):** sign in via the dev login flow, open
  DevTools → Application → Cookies → `connect.sid`, copy the full
  `name=value` pair into `AUTH_TOKEN`.
- **Bearer token:** if you've wired a dev API token, set `AUTH_TOKEN` to the
  raw token; the script auto-prepends `Bearer `.

### 4. Run the harness

```bash
BASE_URL=http://localhost:5000 \
AUTH_TOKEN='connect.sid=s%3AAbCdEf...' \
CONNECTIONS=10 \
DURATION_SEC=10 \
npm run load-test
```

Defaults: `BASE_URL=http://localhost:5000`, `CONNECTIONS=10`, `DURATION_SEC=10`.

### 5. Read the output

```
=== Load-test report ===

Endpoint                                                p50       p95       p99      rps   errors
----------------------------------------------------------------------------------------------
/api/alerts/cross-association                          12.4ms    34.1ms    61.2ms     482   0 (0.0%)
/api/operations/dashboard                               9.2ms    18.7ms    32.0ms     613   0 (0.0%)
...
```

Anything over 500 ms p99 on a single-tenant endpoint is a red flag — file a
follow-up.

## Safety rails

The script refuses to run against URLs that look prod-tier:

- `yourcondomanager.com`
- `*.replit.app`
- `*.fly.dev`
- `*.vercel.app`
- `*.netlify.app`

Override only by setting `ALLOW_NON_LOCAL=1` on the same command line. The
override gets logged on every run so it's never silent.

The script also refuses to start without `AUTH_TOKEN`.

## What's covered

- 5 GET endpoints (read-only — no writes hit the DB).
- p50 / p95 / p99 / max latency.
- Total requests + computed RPS (wall-clock).
- HTTP errors + timeouts + non-2xx responses.

## What's NOT covered (yet)

- Cold-start vs warm-cache distinction (the cross-association endpoint has a
  60s in-memory cache; first request after process boot is the true cold path).
- POST/PUT endpoints (e.g. `POST /api/financial/rules/:ruleId/run` from §2 is
  intentionally excluded — it writes ledger entries).
- Headed browser flows (k6/Artillery would be needed for chained workflows).
- Multi-region latency.
