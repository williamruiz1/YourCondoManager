## 2. Server Endpoint Complexity Notes

Traffic heuristic: endpoints that ship in Layer 0–4 recent waves + the handful any logged-in operator hits per page load.

| Endpoint | Worst-case shape | DB round-trips | Cache | Follow-up |
|---|---|---|---|---|
| `GET /api/alerts/cross-association` | **N associations × M alert sources (7)** — fans out to budget-variance, insurance-expiry, unpaid-late-fees, vendor-contract-renewals, overdue-work-orders, due-maintenance, active-elections. Each source runs its own query per association. | ≈ N × 7 per cold hit; cached 60s per user | 60s in-memory per user | If an operator has 50+ associations, 350 queries per cold hit. Consider batching via `WHERE associationId IN (…)` inside each source and a single materialized view. File: `5.4-F1`. |
| `GET /api/operations/dashboard` | Single-association; N units × status joins. | ~6 queries | none | Fine today. Document cold-path timings once prod. |
| `GET /api/portal/financial-dashboard` | Single owner; joins ledger, payments, assessments. | ~4 queries | none | Fine. |
| `GET /api/financial/assessment-run-log` | Paginated log table; `limit=50` default. | 1 query + count | none | Ensure composite index on `(associationId, runStartedAt DESC)`. Verify. File: `5.4-F2`. |
| `POST /api/financial/rules/:ruleId/run` | Single rule × all in-scope units (per `unitScopeMode`). With `dryRun=false` writes ledger entries per unit. | 1 rule fetch + N unit fetches + N ledger inserts | none | Long-lived. Consider backgrounding via a job queue if > 500 units. File: `5.4-F3`. |
| `GET /api/portal/me` | Single user; portal access + unit joins. | 2 queries | none | Fine. |
| `GET /api/admin/billing/subscription` | Single subscription fetch + planCatalog lookup. | 1–2 queries | none | Fine. |
| `GET /api/portal/maintenance-requests` | Single owner; filtered list. | 1 query | none | Fine. |
| `GET /api/portal/notices` | Single association; paginated. | 1 query | none | Fine. |
| `GET /api/portal/communications` | Single owner; joined with comm history. | 1 query | none | Fine. |

### Biggest risk: `/api/alerts/cross-association` fan-out

The endpoint is the only one that scales per association in a practical way. Today the 60s cache soaks most hits, but a newly-assigned Manager with 50 associations gets a ~350-query cold load. The Layer-4 spec accepted this as the price of the shared-endpoint architecture; 5.4 surfaces it as a real scaling concern for portfolios > 20 associations.

## 5. Top 3 Recommendations (future work)

1. **Batch the alert fan-out** (`5.4-F1`). Convert each alert source to accept an `associationIds: string[]` array and run one query with `IN (...)` instead of one per association. Expected: drop a 50-association cold hit from ~350 queries to ~7.
2. **Dynamic-import `recharts`** on the first chart render in `dashboard.tsx` / `financial-reports.tsx` (`5.4-F6`). Ship ~108 KB gzip out of the initial critical path for Managers who don't open a chart on first nav.
3. **Ledger table virtualization** for `portal-finances/ledger` and operator ledger tables (`5.4-F7`). Owners with long histories render hundreds of rows today; use `@tanstack/react-virtual` (no new top-level dep — it's already pulled transitively for table primitives in some codebases; verify before adding).
