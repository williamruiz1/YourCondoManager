## Purpose

This document is the canonical zone-assignment rubric output for 1.1. It transcribes the Q2 assignment rubric verbatim, lists the six zone definitions (Q3) and the Owner Portal zones (Q5), and provides the per-page zone assignment table for every `client/src/pages/*.tsx` page component in the codebase.

Layer 2 (permission boundary corrections) and Layer 3 (navigation restructure) consume this table.

## Q2 Assignment Rubric (verbatim from 1.1)

> Pages are assigned by primary user intent. The rubric is: (1) What is the user trying to accomplish on this page? (2) Which zone label best describes that intent? If a page serves two zones, it belongs to the zone of its primary action, not its secondary context. Ambiguous cases are escalated — not silently assigned.

**Application steps (for each page file):**

1. Identify the primary user intent on the page.
2. Map that intent to one of the six `/app` zones (or four `/portal` zones).
3. If two zones apply, pick the zone of the primary action (not the secondary context).
4. If intent is genuinely ambiguous, flag the page in the Open Items section below — do NOT silently assign.

## Q3 Six `/app` Zone Definitions (verbatim from 1.1)

- **Home** — Portfolio-level and association-level overview. Landing pages, summary dashboards, health indicators.
- **Financials** — All money-related pages. Budgets, assessments, owner ledgers, payment tracking, financial reports.
- **Operations** — All work and property-related pages. Maintenance requests, work orders, vendors, inspections.
- **Governance** — All board and compliance pages. Meetings, elections, violations, governing documents, rules.
- **Communications** — All messaging and notice pages. Announcements, notices, updates inbox, direct messaging.
- **Platform** — All YCM-internal operator tooling. Platform Controls (tenant config, permission envelopes, provider configuration and diagnostics), Admin Roadmap, Admin Users, Executive Deck, AI Ingestion lifecycle. Platform Admin persona only.

## Q5 Four `/portal` Zones (verbatim from 1.1)

- **Home** — Owner Portal landing / dashboard.
- **My Finances** — Owner-scoped balance, ledger, payments, autopay.
- **My Requests** — Owner-scoped maintenance requests and status tracking.
- **My Community** — Owner-scoped notices, documents, elections, announcements, amenity booking, community hub.

## Q4 Zone Tag Format (verbatim from 1.1)

Every page file carries the following comment block at the top of the file, below imports:

```
// zone: [Zone Label]
// persona: [comma-separated: Manager | Board Officer | Assisted Board | PM Assistant | Owner | Platform Admin]
```

Rules:
- Zone label is one of: Home, Financials, Operations, Governance, Communications, Platform (for `/app`) or Home, My Finances, My Requests, My Community (for `/portal`).
- Persona list reflects the 0.2 Persona Boundary Matrix row for that surface.
- `zone: Platform` entries have `persona: Platform Admin` as the only persona entry.
- Portal zone tags use `persona: Owner`.

## Open items — ambiguous pages

Per Q2, genuinely ambiguous pages are flagged here rather than silently tagged.

### `help-center.tsx` (`/app/help-center`)

- **Ambiguity:** Help Center is cross-zone — it documents Financials, Operations, Governance, Communications, and Platform topics. No single zone matches its primary intent. 2.3 Q12 grants access to all roles (including Platform Admin).
- **Resolution taken (pending founder review):** Tagged as `zone: Platform` with `persona: Manager, Board Officer, Assisted Board, PM Assistant, Platform Admin`. Rationale: Help Center is an operator-tooling surface (documentation/support for YCM platform features); no persona is excluded. `zone: Platform` deviates from the "Platform = Platform Admin only" rule on a per-persona basis, so this assignment is flagged as an open question for founder.
- **Alternative under consideration:** Introduce a seventh "Support" zone (requires decision doc amendment per Q1 AC), or assign to `zone: Home` as an infrastructure landing for documentation access.
- **Flagged for founder decision via `ppm_human_task_create`:** see follow-up task.

### `user-settings.tsx` (`/app/settings`)

- **Ambiguity:** Account settings (profile, notifications, communication preferences) is cross-cutting. 2.3 Q6 gates to Manager + Board Officer + Platform Admin. Not clearly Platform (Platform is YCM-internal operator tooling, not user account config).
- **Resolution taken (pending founder review):** Tagged as `zone: Platform` with `persona: Manager, Board Officer, Platform Admin`. Rationale: Settings is an operator-level cross-zone configuration surface; Home feels wrong because it's not an overview. Platform is the closest fit under the six-zone model, but conflicts with the "Platform = Platform Admin only" rule.
- **Alternative under consideration:** Introduce a "Settings" cross-zone category outside the six zones, similar to how 2.4 treats Platform as a separate vocabulary. Requires decision doc amendment.
- **Flagged for founder decision via `ppm_human_task_create`:** see follow-up task.
