## 3. Cross-association alerts scope (plan §3 Phase 5 frozen signals)

**Signal 1 (overdue WOs):** `status IN ('open','in-progress','pending') AND due_date < NOW()`.
Current alerts endpoint surfaces: `urgent` (priority=urgent), `stalledOpen` (createdAt < 7d ago AND status=open). These are NOT the Phase-5 overdue signal. **Requires a new filter branch** in `/api/dashboard/alerts` using `wo.dueDate`.

**Signal 2 (due MSIs within 7 days):** `due_date < NOW() + 7 days AND status != 'completed'`.
Current alerts endpoint does NOT surface maintenance schedule instances at all. `storage.getMaintenanceScheduleInstances` exists (`storage.ts:13260`) but is not called in the alerts handler. **Requires adding a new aggregation branch** to the handler.

**Cross-association support in current code:**
- Client: `dashboard.tsx:544-550` uses `[` /api/dashboard/alerts?associationId=X`]` when activeAssociationId is set, `[` /api/dashboard/alerts`]` otherwise.
- Server: when `associationId` absent, storage returns all rows. BUT: no `adminScopedAssociationIds` filter is applied (`/api/dashboard/stats` does this at line 1279; `/api/dashboard/alerts` does not). This is a **pre-existing latent bug** — a non-platform-admin manager has effectively portfolio-wide visibility here regardless of their access matrix. Phase 5 must either fix this or explicitly not-regress it.

**Delta size:** **Substantial, not small.**
- ~40 LOC of new server code (two new aggregation branches in `/api/dashboard/alerts` + scoping block).
- ~30 LOC of new client code (two new `AlertRow` entries, possibly a parallel query that always fetches cross-association for signals 1+2 while scoped queries continue for other signals).
- One TypeScript interface addition (extend `DashboardAlerts`).
- No new endpoint, no schema changes. Handoff "no backend changes" is technically violated (additive handler work) — but not in spirit; the endpoint contract extends, no new routes are added. Flag for William review (⚠️ FLAG #2 above).

**Backend readiness:** Storage functions exist. Handler needs additive logic. No schema work.
