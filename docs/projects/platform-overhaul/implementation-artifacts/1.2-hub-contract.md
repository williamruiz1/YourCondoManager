## 1. Hub definition (per 1.2 Q1)

A section hub is the top-level landing page for a zone. It is reached by clicking a zone label in the sidebar. It is navigational and summary only — it is **not** a dashboard and **not** a workflow page.

A hub's job is to orient a user within a zone and provide direct access to that zone's sub-pages.

Mandatory properties:

- **Exactly one hub per zone.** No zone may have two hubs. No page may serve as a hub for more than one zone.
- **Hubs are not workflow pages.** No forms. No deep data tables.
- **Hubs are sidebar-reachable.** Clicking the zone label in the sidebar navigates directly to the hub URL.
- **Hubs are not the Home dashboard.** `/app` and `/portal` are dashboards, not hubs, and are exempt from hub content rules.

---

## 2. Hub composition (per 1.2 Q2)

Every hub page contains, in order:

1. **Zone title.** Exactly matches the sidebar zone label (see §4 naming rule).
2. **Zone description.** One sentence, plain language, describing what the zone is for.
3. **Sub-page link list.** Direct links to the primary sub-pages within the zone, filtered by the active persona (see §5).
4. **Quick-action shortcuts (optional).** Up to **three** shortcuts for the highest-frequency actions in that zone. Omit this block entirely if there is nothing worth featuring.
5. **Summary widget (optional).** **At most one** lightweight summary widget where it adds genuine value (e.g., outstanding balance on Financials, open requests count on Operations). Omit this block entirely if no value is being added.

Prohibited on hub pages:

- No data tables.
- No workflow forms.
- No more than one summary widget.
- No duplicated content from Home (hubs are not mini-dashboards).
- No "Hub" / "Overview" suffix anywhere in the hub title (see §4).

## 4. Naming rule (per 1.2 Q3 / Q7, cross-referenced 1.4)

Hub page `<h1>` = zone label exactly. Document title follows 1.4: `Financials — YCM`, `Operations — YCM`, `My Finances — YCM`, etc.

**Forbidden on hub pages and hub-adjacent sidebar labels:**

- The word **"Hub"** as a suffix, prefix, or descriptor in any nav label or hub page title. `/app/community-hub` is a pre-1.2 page, not a hub under this contract, and its label is flagged in §7 for rename under Phase 11.
- The word **"Overview"** as a suffix in any nav label EXCEPT the reserved cases below.

**"Overview" exceptions (reserved for dashboard-type pages):**

- **Operations Overview** at `/app/operations/dashboard` — locked by 0.1 AC 7. This is the Operations dashboard that coexists with the Operations hub (see §6 below).
- **Portfolio Health** at `/app/portfolio` — page title does not use "Overview" so no lint exemption required; kept here for clarity that "Health" is not an "Overview" variant.
- Any relocated workflow page placed under a hub at a sub-path whose rendered title ends in "Overview" (e.g., future `/app/governance/overview`, `/app/communications/overview`). These are dashboard sub-pages, not hub titles.

Put differently: the word "Overview" is legal **inside** a zone at a sub-path (where it is a dashboard sub-surface); it is illegal as the hub's own title.

The ESLint rule (see §8) enforces the "Hub" prohibition as a hard error. The "Overview" rule is enforced as a hard error with an explicit inline allow-comment at the one Operations Overview definition site.

## 5. Role-filtered hub sub-page lists (per 1.2 Q6)

Every hub renders its sub-page list filtered by the active persona.

- Hub sub-page lists use the **same role-gating model as the sidebar** — `canAccess(item, role)` in `app-sidebar.tsx` today, persona-access manifest (`shared/persona-access.ts`, Phase 9) once it ships.
- A user never sees a sub-page link on a hub to a page they cannot access.
- An Assisted Board user's Financials hub shows a different sub-page list than a Manager's Financials hub.

Phase 11 implementation note: the hub component accepts the adminRole via the same prop path as `AppSidebar`, and reuses the `canAccess` predicate. Do not invent a parallel role-filter function.

## 8. Naming rule ESLint enforcement (Phase 7 output)

The following ESLint rules are added to `eslint.config.js` (flat config) targeting `client/src/components/app-sidebar.tsx` and any future sidebar-adjacent nav config files. See `eslint.config.js` for the canonical implementation; this section documents the rule intent.

**Rule: no-hub-suffix-in-nav-labels (via `no-restricted-syntax`)**

- Selector matches object properties of shape `{ title: "… Hub" }` or `{ label: "… Hub" }` (any nav-label string ending in the word "Hub").
- Fires as an error on violation.
- No exception cases — "Hub" is globally forbidden as a nav label suffix.

**Rule: no-overview-suffix-in-nav-labels (via `no-restricted-syntax`)**

- Selector matches object properties of shape `{ title: "… Overview" }` or `{ label: "… Overview" }` (any nav-label string ending in the word "Overview").
- Fires as an error on violation.
- **Single inline exception:** the Operations Overview sidebar entry (`{ title: "Operations Overview" … }`) is locked by 0.1 AC 7. The definition site at `client/src/components/app-sidebar.tsx` uses an `eslint-disable-next-line` comment on the offending line to document the exception. No other "Overview" suffix is permitted.

See §4 above for the full allow-list rationale.

### 8.1 Scope of the lint rule

- **Scope:** `client/src/components/app-sidebar.tsx` (and any future `client/src/**/nav-*.ts[x]` files).
- **Out of scope:** breadcrumb label strings, tab labels (e.g., the Financial tab bar's "Overview" at `finance-tab-bar.tsx:11`), marketing copy on public pages, generic UI copy. The rule is about **navigation surfaces** that front zone-root hubs.
- **Tab bars are excluded** because "Overview" is the legitimate name for a tab within a tab-page container (e.g., Finance Foundation → Overview tab). This is a dashboard-sub-page use of "Overview" and is explicitly allowed by §4 and 1.2 Q7.

### 8.2 Manual verification required

Because the `no-restricted-syntax` selector runs against the client/src tree as a whole, it will also fire on tab labels and breadcrumb labels that happen to have the same object shape. The rule is deliberately narrow (title/label properties only) but may still produce false positives on non-nav surfaces. See §8.3 below for the handling.
