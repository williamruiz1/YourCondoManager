## Objective

Replace the current implicit-four-group sidebar (`client/src/components/app-sidebar.tsx:68-201`) with the six-zone canonical model (Home, Financials, Operations, Governance, Communications, Platform). Implement the **SUBSET-RENDER** model — Assisted Board and PM Assistant get a completely separate simplified sidebar; no greyed-out or hidden locked sections. Board Officer gets a Manager-equivalent sidebar minus the Platform zone, Portfolio Health, and Associations. Derive sidebar `roles` arrays and `<RouteGuard>` wrappers from a shared persona-access manifest (`shared/persona-access.ts`) sourced from the 0.2 matrix. Retire the sidebar association card and the Owner Portal launcher.

---

## Acceptance Criteria (verbatim / tightly paraphrased from decision doc)

1. Sidebar contains exactly six `SidebarGroup` blocks with labels Home, Financials, Operations, Governance, Communications, Platform; no ad-hoc strings; labels imported from a shared module; every item assigned to exactly one zone (Q1)
2. No `{activeAssociationName}` wrapper group; all six zones render as top-level peers; association-scoped zones render items scoped to active association; context shown in header switcher (Q2)
3. Zone label clicks navigate to the hub URL per 1.2 Q4 — Financials→`/app/financials`, Operations→`/app/operations`, Governance→`/app/governance`, Communications→`/app/communications`, Home→`/app`, Platform→`/app/platform/controls`; adjacent chevron toggles sub-list without navigating (Q3)
4. Home is the first SidebarGroup; Portfolio Health and Associations are sub-items under Home; Portfolio Health not visible to Board Officer / Assisted Board / PM Assistant; Associations not visible to Board Officer / Assisted Board (Q4)
5. SUBSET-RENDER: `assisted-board` and `pm-assistant` sidebars omit disallowed zones/items; no greyed-out, disabled, or visually-locked items — disallowed items absent from DOM; `board-officer` = Manager-minus-Platform-minus-Portfolio-Health-minus-Associations; `platform-admin` sees all six zones (Q5)
6. React Query hook (`usePersonaToggles` or equivalent) fetches the resolved toggle matrix keyed on `(activeAssociationId, adminRole)`; sidebar consumes it for Assisted Board / PM Assistant; loading state renders skeleton; platform/admin surfaces never in the toggle matrix (Q6)
7. Sidebar reserves a nav entry under Settings for PM toggle configuration, visible to `manager` + `platform-admin` only, linking to a placeholder `/app/settings/board-access` (Layer 4 delivers full UI) (Q7)
8. Platform zone visible only when `adminRole === "platform-admin"`; absent from DOM for all other personas; visibility uses the unified `canAccess()` predicate — no special-case branch; `canAccess()` returns `false` on nullish role (Q8)
9. A shared persona-access manifest module exports a canonical access manifest derived from the 0.2 matrix; sidebar `roles` arrays populated by imports; `<RouteGuard>` imports from the same module; no re-declared `AdminRole` (Q9)
10. Global association switcher renders in the top app-bar only; sidebar "Association" card at `app-sidebar.tsx:255-289` retired; single-association personas get a static label; multi-association personas get a dropdown; switching updates sidebar scoped content (Q10)
11. `app-sidebar.tsx:185` and `App.tsx:180` Owner Portal entries removed; no sidebar / footer / hover / inline help surface within `/app` links to `/portal`; Platform Admin uses portal-access-id in a separate session (Q11)
12. Every sidebar item assigned to exactly one zone (or footer for Settings / Help Center); zero orphans; Settings in footer gated `manager`/`board-officer`/`platform-admin`; Help Center in footer ungated; all Platform zone items gated to `platform-admin` only (Q12)
