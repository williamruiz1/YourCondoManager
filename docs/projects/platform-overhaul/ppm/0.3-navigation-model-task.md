## Objective

0.3 is a **governance-only card with no direct build**. It locks the canonical navigation model — single role-filtered sidebar (Q1), functional grouping (Q2), distinct landings on shared shell (Q3), shallow breadcrumbs (Q4), strict role isolation (Q5), 6–7 top-level sections for Manager/Board (Q6), header association switcher primary + portfolio page supporting (Q7), canonical home per persona (Q8), and central updates/inbox (Q9). Every Layer 1–3 module implementation verifies its relevant Q1–Q9 acceptance criteria as part of its own review. This card tracks that cross-cutting verification.

---

## Acceptance Criteria (consolidated from Q1–Q9, verified across Layers 1–3)

1. The navigation model defines one canonical zone structure; each persona sees only relevant sections; underlying taxonomy does not vary by persona (Q1)
2. All pages in 0.3 scope are assigned to a functional zone; sidebar grouping is not user-toggleable; zone labels are functional and stable across personas (Q2)
3. Manager entry remains `/app`; Board Officer / Assisted Board entry remains `/app`; Owner entry remains `/portal`; 0.3 introduces no new route prefixes that conflict with locked 0.2 (Q3)
4. Breadcrumbs never exceed three visible levels; association context appears in association-scoped views; labels match page titles; mobile may collapse (Q4)
5. Pages and navigation items are hidden when the role lacks access; read-only cross-role exposure requires explicit spec approval; role-gating rules documented per persona (Q5)
6. Manager and Board Officer / Assisted Board may exceed five top-level sections only when each has distinct functional purpose; Owner stays lean (Q6)
7. Manager workspace includes persistent association switcher in the header; portfolio/associations page exists as secondary nav surface; tab-style switching is not the core model (Q7)
8. Manager, Board Officer / Assisted Board, and Owner each have one canonical home/dashboard; navigation docs reference one default landing per persona (Q8)
9. A central updates/inbox concept exists in the navigation model; critical items from multiple zones surface there; inbox does not replace section-native workflows (Q9)
