## Nature of This Task

0.2 is a **cross-cutting governance spec**, not a single-chunk build. It locks the canonical persona boundary matrix that every Layer 2 (Role & Permission Model) and Layer 3 (Navigation Restructure) task must satisfy. This card anchors the locked matrix in pocketp.m. so downstream tasks can reference it and registers its acceptance criteria as gating for any Layer 2 / Layer 3 PR.

There is no direct build for this task in isolation. Criteria are verified when Layer 2 and Layer 3 modules implement their own specs. On build activation of each downstream module, the assignee must read 0.2's matrix and treat it as authoritative gating.

---

## Objective

Establish and preserve the three-persona boundary defined in 0.2:

- **Manager** — full operator workspace at `/app`, multi-association capable
- **Board Admin** — scoped operator workspace at `/app`, single-association only, no Portfolio Health / Settings
- **Owner** — Owner Portal at `/portal`, beta-scoped capabilities only; HOA payment processing and bill history are explicitly out of overhaul scope

The persona boundary matrix in the decision doc is the authoritative gating reference. Any routing, role check, or nav rendering that contradicts it is a bug.

---

## Acceptance Criteria (verbatim from decision doc)

1. No route inside `/app/*` is accessible to a user authenticated only through the Owner Portal portal-access-id flow
2. No route inside `/portal/*` requires an operator workspace session
3. A `board-admin` user navigating to `/app/portfolio` is redirected to `/app` — behavior preserved from 0.1
4. A `board-admin` user does not see a "Portfolio Health" nav item in the sidebar
5. A `board-admin` user does not see a Settings nav item in the sidebar
6. The Owner Portal sidebar contains exactly the sections listed under "Current live capabilities" — no nav items for beta-gap features (HOA payment processing, bill history/statements) are present
7. Layer 2 permission boundary corrections treat this matrix as the authoritative gating spec — any deviation requires a new decision doc, not an inline code judgment

---

## Gating Rules

- No downstream module may infer a persona boundary not present in 0.2's matrix. If a Layer 2/3 task surfaces a persona question the matrix does not answer, stop and file `ppm_human_task_create` — do not guess.
- A handoff doc is not required to *start* 0.2 because 0.2 is a governance spec. Each downstream Layer 2 / Layer 3 task verifies its relevant 0.2 criteria as part of its own review.
- Owner Portal scope is locked to current live capabilities in the decision doc. HOA payment processing end-to-end and formal bill history/statements are out of scope for Layers 1–5 without a new Layer 0 decision.
