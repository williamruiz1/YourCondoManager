## Objective

Land HV-3, the third and final phase of the hub-visibility rename. Drop the old `resident | owner | board | admin` enum values from `hub_visibility_level` pgEnum, recast both `hub_map_issues.visibility_level` and `community_announcements.visibility_level` columns to use the new enum exclusively, and remove the `HUB_VISIBILITY_RENAME` feature flag from `shared/feature-flags.ts`.

HV-1 (additive enum values + translation helper + flag) shipped in PR #23.
HV-2 (backfill + dual-read + per-association flag flip + default-ON) shipped in a follow-up.
HV-3 is the remaining track.

---

## Acceptance Criteria (verbatim from decision doc)

1. `0018_hub_visibility_rename_hv3.sql` migration drops the four old enum values and creates a new `hub_visibility_level` pgEnum with only the new vocab (`public | residents | unit-owners | board-only | operator-only`).
2. Both columns (`hub_map_issues.visibility_level` and `community_announcements.visibility_level`) are recast to use the new enum exclusively. Pre-cast row count is recorded in the PR body.
3. The `oldToNewVisibility()` and `newToOldVisibility()` translation helpers in `shared/hub-visibility.ts` are deleted; `normalizeHubVisibility()` becomes a no-op identity function.
4. `HUB_VISIBILITY_RENAME` is removed from `shared/feature-flags.ts` (`FeatureFlagKey` union + `DEFAULTS` + lifecycle comment).
5. Every `getFeatureFlag("HUB_VISIBILITY_RENAME")` call site is deleted.
6. The anonymous public-hub endpoint at `server/routes.ts:15319-15337` continues to return rows where `visibility_level = 'public'` with no behavior change.
7. No data loss: every row's `visibility_level` value before HV-3 maps cleanly to a value in the new enum (verified by HV-2 backfill).
8. `npm run check` + `npx vitest run` + `npm run build` all green.
