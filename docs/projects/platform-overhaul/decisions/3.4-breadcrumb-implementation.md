3.4 does not add any new breadcrumb semantics. Its job is to replace every hardcoded `breadcrumbs=` array and every parallel breadcrumb surface with a single component wired to route metadata, while preserving the leaf-title identity 1.4 requires.

## Conflict Analysis

All semantic conflicts were resolved in 1.3 and 1.4. The Conflict Analysis for 3.4 is restricted to implementation-shape questions:

1. Where the route-metadata map lives (co-located with routes vs. a dedicated module) and what it contains.
2. How association context is injected at runtime without coupling the component to a specific data-fetch strategy.
3. How the ≤3-level cap is enforced at build time (lint/test) vs. runtime (defensive truncation).
4. How the 39-row label catalog is produced during build and checked into the repo.
5. How the centralized component composes with `useDocumentTitle` so the leaf and the H1 cannot drift.
6. How mobile collapse is implemented (CSS-only breakpoint vs. `use-mobile` hook subscription).
7. How the two Class G parallel surfaces are deleted without regressing the views they live in.
8. How the `<RouteGuard>` pattern from 2.3 interacts with breadcrumb rendering order.
9. How hub pages derive their two-level breadcrumb from route metadata.
10. How the two Class A violations are sequenced relative to the main 3.4 landing.

No axis below re-opens a 1.3 or 1.4 lock.

## Selected Resolutions

### Q1 — Route metadata map: shape and location

**Resolution:** The route-metadata map lives in a dedicated module at `client/src/lib/route-metadata.ts`. It is a flat array of `RouteMetaEntry` objects, one per route path pattern. Each entry carries:

| Field | Type | Description |
|---|---|---|
| `path` | `string` | Route path pattern (e.g., `/app/financial/billing`) |
| `zone` | `string` | 1.1 zone label (e.g., `"Financials"`, `"Operations"`, `"Governance"`, `"Communications"`, `"Home"`, `"Platform"`) |
| `scope` | `"association" \| "portfolio"` | Determines Pattern A (`Association > Zone > Page`) vs. Pattern B (`Zone > Page`) per 1.3 Q2 |
| `title` | `string` | Canonical page title from 1.4 catalog — the leaf crumb label |
| `hub` | `boolean` | `true` for hub pages per 1.2 reclassification |
| `parent` | `string \| null` | Path of the parent page for mobile back-nav derivation; `null` for zone-root hubs |

The map is the single source of truth for breadcrumb derivation. No page reads or constructs breadcrumb content directly. The shared breadcrumb component reads from this map using the current route path as the lookup key.

**Derivation:** 1.3 Q5 requires centralized route metadata. The field set is the minimum needed to implement 1.3 Q1 (scope → root label), Q2 (scope + zone + title → chain), Q3 (hub flag → two-level crumb), Q4 (flat structure enforces ≤3 levels by construction), and Q7 (parent → mobile back target).

**Acceptance Criteria**
- `client/src/lib/route-metadata.ts` exists and exports the `RouteMetaEntry` type and the metadata array
- Every route in the 39-row catalog has exactly one entry in the metadata array
- No breadcrumb-relevant field (zone, title, scope, hub, parent) is hardcoded in any page component
- The metadata array is importable by both the breadcrumb component and `useDocumentTitle`

### Q2 — Association context injection

**Resolution:** The shared breadcrumb component reads the active association name via the existing `use-active-association` hook (`client/src/hooks/use-active-association`). When `scope === "association"`, the component prepends the association name as level one. When `scope === "portfolio"`, the component skips level one and starts at the zone label. No prop-drilling from layout wrappers is required; the hook subscription is internal to the breadcrumb component.

**Derivation:** 1.3 Q1 requires the root label to be the association name for association-scoped pages. The `use-active-association` hook already exists in the codebase and is the canonical source of association context. Consuming it inside the component keeps the breadcrumb self-contained per 1.3 Q5 (centralized, no per-page configuration).

**Acceptance Criteria**
- The shared breadcrumb component imports and calls `use-active-association` internally
- Association-scoped pages display the association name as the first breadcrumb level without any prop or context passed from the page
- Portfolio-scoped pages display the zone label as the first breadcrumb level
- If no active association is selected on an association-scoped route, the breadcrumb gracefully omits the root level (shows `Zone > Page` until an association is selected)

### Q3 — ≤3-level enforcement mechanism

**Resolution:** Enforcement is structural and verified by unit test. The route-metadata map shape (§Q1) structurally limits breadcrumbs to at most three segments: `association name` (from hook) + `zone` (from metadata) + `title` (from metadata). There is no mechanism to add a fourth level because the map carries no intermediate-level field. A unit test iterates the metadata array and asserts that every entry, when resolved through the breadcrumb derivation function, produces a chain of length ≤3. A runtime `console.error` in development mode fires if the derived chain exceeds 3 segments (defensive fallback — should never trigger if the metadata map is correct).

**Derivation:** 1.3 Q4 mandates ≤3 levels. Structural enforcement via the metadata shape is the simplest guarantee. The unit test catches regressions if someone adds a field or derivation path. The dev-mode runtime assertion is defense-in-depth.

**Acceptance Criteria**
- The breadcrumb derivation function never produces more than 3 segments for any entry in the metadata array
- A unit test (`route-metadata.test.ts` or equivalent) iterates all entries and asserts chain length ≤3
- In development mode, a `console.error` fires if the derived chain exceeds 3 segments
- No intermediate breadcrumb level field exists in the `RouteMetaEntry` type

### Q4 — 39-row canonical catalog artifact

**Resolution:** The canonical catalog is a snapshot-tested TypeScript constant exported from `client/src/lib/route-metadata.ts` — it IS the metadata map from Q1. The catalog artifact (the human-readable table of all routes with their breadcrumb chains) is generated by a script at `scripts/generate-breadcrumb-catalog.ts` that imports the metadata array, resolves each entry through the breadcrumb derivation function (using a placeholder association name `"{Association}"`), and writes a Markdown table to `docs/projects/platform-overhaul/artifacts/breadcrumb-catalog.md`. A Jest snapshot test (`route-metadata.test.ts`) snapshots the metadata array; any change to the array requires updating the snapshot, which triggers the developer to re-run the catalog generator.

The catalog contains 39 rows:
- 37 original surfaces from the 1.3 audit (36 `breadcrumbs=` pages + 1 dual-surface `financial-budgets.tsx`)
- +2 new routes from 3.2 Q3 (`/app/governance/overview`, `/app/communications/overview`)

**Derivation:** 1.3 Q4 AC requires "a list of all pages that required flattening." The generated Markdown table satisfies this. The snapshot test keeps the catalog in sync with the metadata map without requiring a CI-time generator step. The generator script is a convenience for human review.

**Acceptance Criteria**
- `client/src/lib/route-metadata.ts` exports a metadata array with exactly 39 entries (one per breadcrumb surface)
- `scripts/generate-breadcrumb-catalog.ts` produces a Markdown table with columns: Route, Scope, Zone, Breadcrumb Chain, Flattened (Y/N)
- `docs/projects/platform-overhaul/artifacts/breadcrumb-catalog.md` is generated (not hand-maintained)
- A snapshot test in `route-metadata.test.ts` captures the full metadata array; any diff requires explicit snapshot update
- The "Flattened" column identifies every page that dropped an intermediate level to meet the ≤3 cap

### Q5 — Coupling to `useDocumentTitle`

**Resolution:** Both the shared breadcrumb component and `useDocumentTitle` read the `title` field from the same `RouteMetaEntry` in the route-metadata map. The page component imports the metadata entry for its route, passes `entry.title` to `useDocumentTitle(entry.title)`, and the breadcrumb component independently reads `entry.title` as the leaf crumb. Because both consumers read the same field from the same object, leaf crumb == H1 == `document.title` by construction.

No wrapper hook (e.g., `usePageIdentity`) is introduced. The coupling is achieved through shared data (the metadata entry), not shared behavior. This keeps `useDocumentTitle` (1.4 Q2) a simple single-purpose hook and the breadcrumb component (1.3 Q5) a simple single-purpose renderer.

**Derivation:** 1.4 Q2 specifies `useDocumentTitle` as a single-argument hook accepting `title: string`. 1.4 Q3 requires H1 == the same title. 1.3 Q5 requires the breadcrumb leaf to match the page title. Shared-data coupling satisfies all three constraints without introducing a new abstraction.

**Acceptance Criteria**
- Every page component passes `routeMetaEntry.title` to `useDocumentTitle`
- The shared breadcrumb component reads `routeMetaEntry.title` as the leaf crumb label
- No page has a breadcrumb leaf that differs from its `<h1>` or its `document.title` page-title segment
- No `usePageIdentity` or equivalent wrapper hook exists — coupling is via the shared `RouteMetaEntry.title` field

### Q6 — Mobile collapse implementation

**Resolution:** The shared breadcrumb component uses the existing `use-mobile` hook (`client/src/hooks/use-mobile`) to detect viewport width below 768px. When mobile is detected, the component renders a single back-navigation affordance: `< {parent label}` where the parent label is derived from the `parent` field in the route-metadata entry. If `parent` is `null` (hub page or zone root), the mobile breadcrumb renders nothing (the user is already at the top level). The same DOM element is used; no separate `<BreadcrumbMobile>` component is created. The `use-mobile` hook drives a conditional render branch within the single `<BreadcrumbNav>` component.

The back chevron (`<`) is a functional link to the parent route, not decorative text.

**Derivation:** 1.3 Q7 mandates mobile collapse below 768px to `< Parent`. The `use-mobile` hook already exists in the codebase. Using it (rather than a CSS-only breakpoint) enables conditional rendering of the correct parent label and link target, which cannot be achieved with CSS `display: none` alone since the parent link target differs from the desktop breadcrumb links. A single component avoids duplication.

**Acceptance Criteria**
- Below 768px, the breadcrumb renders `< {parent label}` as a single clickable back link
- Above 768px, the breadcrumb renders the full trail (up to 3 levels)
- The `use-mobile` hook is the viewport detection mechanism; no CSS-only media query hides/shows breadcrumb levels
- The mobile back link navigates to the parent route specified in the metadata entry's `parent` field
- Hub pages at zone root (where `parent` is `null`) show no breadcrumb on mobile
- No separate `<BreadcrumbMobile>` component exists

### Q9 — 2.3 permission-boundary interaction

**Resolution:** Gating is strictly orthogonal to breadcrumb rendering. The route-metadata map does NOT include gating hints, role fields, or persona-conditional entries. Per 1.3 Q6, breadcrumb content is persona-invariant. Per 2.3 Q9, the `<RouteGuard>` component wraps routes at the router level, before the page (and therefore the breadcrumb) ever renders. If a persona cannot access a route, the `<RouteGuard>` redirects them before the breadcrumb component mounts. The breadcrumb component has zero awareness of roles or permissions.

**Derivation:** 1.3 Q6 locks persona-invariant breadcrumbs. 2.3 Q9 locks `<RouteGuard>` as the enforcement mechanism at the route level. These two locks together mean the breadcrumb component never needs to consider permissions — it renders the same output for any persona that reaches the page, and personas that should not reach the page never do.

**Acceptance Criteria**
- The `RouteMetaEntry` type contains no `roles`, `permissions`, or persona-related fields
- The shared breadcrumb component imports nothing from the permission/role system
- No conditional breadcrumb logic based on `adminRole`, persona, or permission exists anywhere in the codebase
- `<RouteGuard>` wraps the route before the breadcrumb component mounts in the component tree

### Q10 — Hub-crumb rendering per 1.2 + 1.3 Q3

**Resolution:** The shared breadcrumb component derives hub status from the `hub: true` flag in the route-metadata entry (§Q1). When `hub === true`:

- **Association-scoped hub:** Renders `Association Name > Zone Label` — two levels. The zone label is a non-linked current-page indicator (plain text, not an `<a>` tag).
- **Portfolio-scoped hub:** Renders `Zone Label` alone as a non-linked current-page indicator.

The component does NOT match against a list of registered hub paths from 1.2. The `hub` boolean in the metadata map is set during catalog construction based on 1.2's reclassification; at render time, the component only checks the flag.

**Derivation:** 1.3 Q3 mandates hub pages show a two-level breadcrumb with the zone label as non-linked current-page indicator. Using the metadata `hub` flag (rather than path-matching against 1.2's hub URL list) keeps the component decoupled from route-path specifics and consistent with the Q1 principle that all breadcrumb behavior is driven by the metadata map.

**Acceptance Criteria**
- Hub pages render exactly two breadcrumb levels (association-scoped) or one level (portfolio-scoped)
- The zone label on a hub breadcrumb is rendered as plain text, not as a clickable link
- Hub pages do not show a third breadcrumb level
- The `hub` field in the metadata entry is the sole input for hub-crumb logic; no path-matching against external hub URL lists occurs
- All hub routes defined in 1.2 (`/app/financials`, `/app/operations`, `/app/governance`, `/app/communications`, `/portal/finances`, `/portal/requests`, `/portal/community`) have `hub: true` in the metadata map
