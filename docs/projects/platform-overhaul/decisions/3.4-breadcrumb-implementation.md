## Conflict Analysis

All semantic conflicts were resolved in 1.3 and 1.4. The Conflict Analysis for 3.4 is restricted to implementation-shape questions:

1. Where the route-metadata map lives (co-located with routes vs. a dedicated module) and what it contains.
2. How association context is injected at runtime without coupling the component to a specific data-fetch strategy.
3. How the ≤3-level cap is enforced at build time (lint/test) vs. runtime (defensive truncation).
4. How the 37-row label catalog is produced during build and checked into the repo.
5. How the centralized component composes with `useDocumentTitle` so the leaf and the H1 cannot drift.
6. How mobile collapse is implemented (CSS-only breakpoint vs. `use-mobile` hook subscription).
7. How the two Class G parallel surfaces are deleted without regressing the views they live in.

No axis below re-opens a 1.3 or 1.4 lock.

## Open Questions

### Q1 — Route metadata map: shape and location
Where does the route-metadata map live, and what fields does each entry carry (path pattern, zone, scope `association|portfolio`, leaf title ref, hub flag, parent chain derivation)?

### Q2 — Association context injection
How does the shared component obtain the active association name for Pattern-A breadcrumbs — via `use-active-association` subscription inside the component, via a prop from a layout wrapper, or via route-metadata plus a context read?

### Q3 — ≤3-level enforcement mechanism
How is 1.3 Q4's ≤3-level cap enforced — a runtime assertion in dev, a unit test over the metadata map, an ESLint rule, or a build-time catalog generator that refuses to emit a 4-level entry?

### Q4 — 37-row canonical catalog artifact
How is the per-route label catalog (required by 1.3 Q4 AC "a list of all pages that required flattening") produced, where does it live in the repo, and how is it kept in sync with the metadata map (generator script, snapshot test, or hand-maintained with a CI check)?

### Q5 — Coupling to `useDocumentTitle`
How does 3.4 guarantee leaf crumb == H1 == `document.title` — by having the shared breadcrumb component read the same metadata entry the page passes to `useDocumentTitle`, by having `useDocumentTitle` itself read from the route metadata, or by a shared `usePageIdentity` wrapper that emits both?

### Q6 — Mobile collapse implementation
Is the mobile single-level back affordance (1.3 Q7) implemented via a CSS breakpoint over the same DOM, via `use-mobile` driving two render branches, or via a dedicated `<BreadcrumbMobile>` sibling component?

### Q7 — Class A pre-work sequencing
Are the two Class A violations (`dashboard.tsx:633`, `operations-dashboard.tsx:101`) remediated as part of the 3.4 landing PR, or as a separate pre-flight PR gated on 1.4's title renames (per 1.4 Q4 table) landing first?

### Q8 — Class G dual-surface retirement
How are the two parallel breadcrumb surfaces (`association-context.tsx:672`, `financial-budgets.tsx:232`) retired — deleted outright, replaced with a call to the shared component, or absorbed into a higher-level layout that hosts the shared component once?

### Q9 — 2.3 permission-boundary interaction
Per 2.3, some routes are unreachable for some personas. Does the metadata map include gating hints, or is gating strictly orthogonal (per 1.3 Q6) and enforced at the route layer before the breadcrumb ever renders?

### Q10 — Hub-crumb rendering per 1.2 + 1.3 Q3
1.3 Q3 mandates hub pages render `Association Name > Zone Label` with the zone label as a non-linked current-page indicator. Does the shared component derive "is hub" from the route metadata `hub: true` flag, or from the path matching a registered hub path in 1.2's reclassification?
