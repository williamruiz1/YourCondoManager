# 5.7 — Automated Quality Gates

**Status:** SPEC LOCKED 2026-04-25 (Wave 25, ship-and-lock pattern)
**Owner:** Platform Overhaul
**Layer:** 5 — Polish & Hardening

---

## Scope

Add programmatic quality gates that catch the regressions humans miss,
without taking a dependency on any external SaaS (Percy, Chromatic,
BrowserStack, Sauce Labs). Two deliverables in one wave:

1. **axe-core in Playwright.** Run an automated WCAG 2.1 AA audit at
   the end of every E2E spec, plus a dedicated five-surface smoke
   spec. Threshold: zero `critical` or `serious` violations.

2. **Visual regression scaffold.** Use Playwright's built-in
   `toHaveScreenshot()` against committed PNG baselines for four
   stable surfaces (Home, Inbox, Assessment Rules, Portal Home).
   Diff threshold: 200 px (`maxDiffPixels`).

**In scope:**

- Add `@axe-core/playwright` as a dev dependency.
- Shared helper `tests/e2e/playwright/helpers/a11y-check.ts` exporting
  `runAxeAudit(page, label)` and `runAxeAuditSoft(page, label)`.
- Wire axe into the five existing Playwright specs:
  `signup-onboarding`, `alerts-lifecycle`, `assessment-lifecycle`,
  `owner-portal-navigation`, `amenities-toggle-roundtrip`.
- New `tests/e2e/playwright/a11y-smoke.spec.ts` covering Home, Inbox,
  Assessment Rules, Portal Home, Portal Finances.
- New `tests/e2e/playwright/visual-regression.spec.ts` plus four PNG
  baselines committed under `tests/e2e/playwright/__screenshots__/`.
- npm scripts `test:playwright:visual` and
  `test:playwright:visual:update`.
- Runbook at `implementation-artifacts/visual-regression-runbook.md`.

**Out of scope:**

- A new CI workflow file. The user wires CI separately because the
  YCM repo's CI surface is already contested between Replit, optional
  GitHub Actions, and the existing Playwright real-backend job. The
  runbook documents the CI integration plan; a follow-up wave will
  land it.
- Fixing every axe violation surfaced by the new audit. Wave 21
  already fixed the 10 locked surfaces. Anything outside that set is
  reported via `expect.soft` so the audit does not cascade-fail; real
  fixes are filed as follow-up workitems.
- Cross-browser visual regression (Firefox, Safari, mobile). Today's
  scaffold is Chromium-only — same as the rest of the Playwright
  suite. Add other browsers when there's evidence of browser-specific
  visual bugs.
- Replacing the in-repo PNG baseline storage with a hosted service.
  Wave 25's mandate is in-house tooling.

## Acceptance Criteria

1. `@axe-core/playwright` is in `devDependencies` with a pinned
   `^4.x` major.
2. `tests/e2e/playwright/helpers/a11y-check.ts` exports
   `runAxeAudit(page, label)` (hard-fail on critical/serious) and
   `runAxeAuditSoft(page, label)` (soft-fail; logs in trace, does
   not break the test).
3. The helper runs axe with tags `["wcag2a", "wcag2aa", "wcag21a",
   "wcag21aa"]` — matching the WCAG 2.1 AA target locked in 5.5.
4. Each of the five existing Playwright specs calls one of the
   audit helpers as the final step of its test body, in both
   route-mock and real-backend branches.
5. `tests/e2e/playwright/a11y-smoke.spec.ts` exists, walks the five
   key surfaces (Home, Inbox, Rules, Portal Home, Portal Finances),
   and runs axe on each.
6. `tests/e2e/playwright/visual-regression.spec.ts` exists, has four
   `@visual`-tagged tests (Home, Inbox, Rules, Portal Home), and
   each test asserts against a PNG baseline at
   `tests/e2e/playwright/__screenshots__/visual-regression.spec.ts/<name>.png`.
7. `npm run test:playwright:visual` runs only `--grep @visual`.
8. `npm run test:playwright:visual:update` runs with
   `--update-snapshots`.
9. `implementation-artifacts/visual-regression-runbook.md` documents
   the update flow, when to commit baselines, and why we are not
   using Percy.
10. The 5.5 doc has a "Wave 25 — Automated a11y + visual regression"
    section that records what we now check and the threshold.

## Decisions Made (audit-level)

- **axe runs at end of test, not as a separate step.** Cuts in half
  the number of times Playwright has to bring up the page. Trade-off:
  a hard axe failure can mask a flow assertion below it — we mitigate
  by ordering audit calls AFTER the main flow `expect()`s.
- **`expect.soft` for ambient surfaces.** The five existing specs
  cover surfaces (alerts, assessments, amenities, portal nav) that
  Wave 21 did NOT lock. We use `runAxeAuditSoft` so a non-trivial
  violation outside the locked 10 surfaces does not cascade-fail
  the whole suite. The trace still records every violation; follow-
  up workitems file the real fixes.
- **Hard-fail on the smoke spec.** Originally we considered hard-
  failing on the dedicated `a11y-smoke.spec.ts` to give a single
  authoritative "is the platform a11y-clean today" signal. We
  ultimately picked soft-fail there too, so the entire Wave 25
  rollout is non-disruptive for the next wave to inherit. A
  follow-up wave can flip the smoke-spec calls to `runAxeAudit`
  (hard) once the locked 10 are confirmed green and the remaining
  surfaces have a fix plan.
- **In-house visual regression.** Considered Percy / Chromatic.
  Both add a third-party surface, both cost money per snapshot,
  and both require following a link to approve baselines. Reverse
  decision is reversible — Percy can be wired up later without
  rewriting any of the existing `@visual` specs.
- **`maxDiffPixels: 200`.** Tight enough to catch a missing
  component (a missing button is dozens of pixels), loose enough
  to absorb sub-pixel font / antialiasing jitter across machines.
- **Four surfaces, not ten.** The visual scaffold targets stable,
  deterministic surfaces. List/table renders with seed data are
  flaky for visual diffing because seed order is not guaranteed.
  Adding more surfaces is a follow-up; the bottom-up approach
  keeps the baseline set small and trustworthy.

## Wave 25 — Real violations found and resolved

The first programmatic axe pass surfaced four real violations
across the five smoke surfaces. Three were trivially fixed inline;
one is filed as a follow-up.

| Violation | Severity | Surface(s) | Resolution |
|---|---|---|---|
| `meta-viewport` (Zooming and scaling must not be disabled) | moderate | every page | Removed `maximum-scale=1` from `client/index.html`. |
| `color-contrast` on `text-on-surface-variant/60` (YCM logo strap, portal sidebar) | serious | `/portal*` | Dropped the `/60` opacity in `portal-shell.tsx`. |
| `color-contrast` on `text-secondary` (zero-balance display) | serious | `/portal`, `/portal/finances` | Switched the zero-balance branch to `text-on-surface` in `portal-home.tsx` and `portal-finances.tsx`. The destructive-tone branch is unchanged. |
| `color-contrast` on `opacity-60` decorative icon | serious | `/portal/amenities` (and any portal page rendering the public-community link) | Dropped `opacity-60`, added `aria-hidden="true"` to the decorative icon in `portal-shell.tsx`. |
| `color-contrast` on inactive Radix `<Tabs.Trigger>` | serious | `/portal/amenities`, `/portal/community` | **DEFERRED** — non-trivial; touches the shared Radix tab token across every tab on the platform. Filed as workitem `ee01d2eb-ec59-4e2a-bd91-82380ded00aa`. Reported via `runAxeAuditSoft` (logs to stderr, does not fail the test). |

Net: the five smoke surfaces (Home, Inbox, Rules, Portal Home,
Portal Finances) report **zero violations** under the locked
threshold. Two ambient surfaces (`amenities-toggle-roundtrip` and
`owner-portal-navigation`) soft-fail on the deferred Radix-tabs
issue. The follow-up workitem will promote the two soft-fail
callsites to hard `runAxeAudit` once cleared.
