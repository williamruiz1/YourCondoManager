## Conflict Analysis

The audit surfaced seven decision axes: root-label rule, association context crumb, hub-crumb behavior, flattening ≤3 levels, centralized-vs-per-page governance, persona variance, and mobile collapse. Each is resolved below.

---

## Selected Resolutions

### Q1 — What is the root label rule?

**Resolution:** The root label in a breadcrumb is the association name for association-scoped pages, and the zone label for portfolio-scoped pages. "Home," "Dashboard," "App," or any generic term is never used as a breadcrumb root label. If no association context exists, the breadcrumb starts at the zone label.

**Acceptance Criteria**
- Association-scoped breadcrumbs always show the association name as level one
- Portfolio-scoped breadcrumbs start at zone label
- No breadcrumb uses "Home," "Dashboard," or "App" as a root label

### Q2 — What is the canonical three-level breadcrumb pattern?

**Resolution:** `Association Name > Zone Label > Page Title` for association-scoped pages. `Zone Label > Page Title` for portfolio-scoped pages. These are the only two valid patterns.

**Acceptance Criteria**
- All 37 pages use one of these two patterns
- No other breadcrumb pattern exists in the codebase after implementation

### Q3 — Do hub pages get a breadcrumb, and if so, what does it show?

**Resolution:** Hub pages show a two-level breadcrumb: `Association Name > Zone Label` for association-scoped hubs. The hub page title is the same as the zone label, so level two and the page title are the same — the current page indicator is the zone label. Portfolio-scoped hubs show only the zone label as a non-linked current-page indicator.

**Acceptance Criteria**
- Hub pages show a breadcrumb with association name (if association-scoped) and zone label
- Zone label on a hub breadcrumb is not a link — it is the current page indicator
- Hub breadcrumbs do not show a third level

### Q4 — How do any pages that currently have four or more breadcrumb levels flatten to three?

**Resolution:** Any page currently showing four or more breadcrumb levels must drop intermediate levels, keeping only: association name (if applicable), zone label, and the immediate page title. Intermediate section labels are removed. If dropping an intermediate level makes the breadcrumb ambiguous, the page title must be made specific enough to stand alone.

**Acceptance Criteria**
- No page shows more than three breadcrumb levels after implementation
- Flattened pages have page titles specific enough to identify the page without the dropped level
- A list of all pages that required flattening is included in the implementation output

### Q5 — Centralized or per-page breadcrumb implementation?

**Resolution:** Centralized. One shared breadcrumb component reads from a route-level config or metadata object. No page hardcodes its own breadcrumb markup.

**Acceptance Criteria**
- One shared breadcrumb component exists
- All 37 pages use the shared component
- No page contains hardcoded breadcrumb markup outside the shared component
- Route metadata drives the breadcrumb content

### Q6 — Does breadcrumb content vary by persona?

**Resolution:** No. Breadcrumb content does not vary by persona. The same page shows the same breadcrumb regardless of role. Role-filtering affects navigation visibility, not breadcrumb content.

**Acceptance Criteria**
- Breadcrumb renders identically for all personas on any given page
- No persona-conditional logic exists in the breadcrumb component

### Q7 — How does the breadcrumb behave on mobile?

**Resolution:** On mobile (viewport width below 768px), the breadcrumb collapses to show only the immediate parent as a back-navigation affordance: `< Zone Label` or `< Association Name` depending on depth. The full trail is not shown on mobile.

**Acceptance Criteria**
- Below 768px, breadcrumb shows one level: the immediate parent with a back-chevron
- Above 768px, breadcrumb shows the full trail up to three levels
- The mobile breadcrumb is a functional back-navigation link, not decorative text
