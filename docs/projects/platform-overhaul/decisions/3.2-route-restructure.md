## Context

3.2 is the Layer 3 module that produces the canonical `/app/*` and `/portal/*` route map YCM ships after every Layer 0–2 decision is implemented. It consumes:

- **0.1 (LOCKED)** — `/app` = Home, `/app/portfolio` = Portfolio Health, `/app/operations/dashboard` = Operations Overview. Route paths are locked; no URL changes (0.1 §"What this resolution does NOT change").
- **0.2 (LOCKED, amended `f8dbf76`)** — four-persona model with Platform Admin as Persona 4. `/app/*` = Manager + Board Admin + Platform Admin; `/portal/*` = Owner only.
- **0.3 (LOCKED)** — Q3: Manager primary entry `/app`, Board Admin primary entry `/app`, Owner primary entry `/portal`. 0.3 introduces no new route prefixes that conflict with 0.2.
- **1.1 (LOCKED, amended `ac446c0`)** — six zones (Home, Financials, Operations, Governance, Communications, Platform) for `/app`; four zones (Home, My Finances, My Requests, My Community) for `/portal`.
- **1.2 (LOCKED)** — hub URLs at zone root: `/app/financials`, `/app/operations`, `/app/governance`, `/app/communications`; `/portal/finances`, `/portal/requests`, `/portal/community`. Home (`/app`) and Owner Home (`/portal`) are dashboards, not hubs (1.2 Q5).
- **1.2 Q4 cross-reference** — `/app/operations` (hub, title "Operations") coexists with `/app/operations/dashboard` (Operations Overview, locked by 0.1 AC 7).
- **2.3 (LOCKED)** — mandatory dual enforcement (server + `<RouteGuard>`), `AuthAwareApp:1051-1057` board-admin shunt retired (Q3), Portfolio redirect role-based for all board-admin sessions (Q5), Settings guard (Q6), `/app/admin*` all platform-admin only (Q11).
- **2.4 (LOCKED)** — `<RouteGuard roles={["platform-admin"]}>` on every `/app/platform/*`, `/app/admin/*`, `/app/ai/*` route (Q2). Owner Portal launcher inside platform-admin nav removed (Q5). No `/operator/*` route tree split (Q6).

3.2 does not re-decide any of the above. 3.2 produces the canonical route map that implements them, decides the fate of every legacy redirect currently at `client/src/App.tsx:249-254,270-278,288-291,365-461`, and specifies the migration strategy for deprecated URLs.

## Conflict Analysis

Seven decision axes surface from reconciling the current 52-route table with the locked Layer 0–2 spec. Each maps to an open question below.

1. **Hub URLs collide with current finance subtree.** 1.2 Q4 locks `/app/financials` as the Financials hub URL; the current subtree is rooted at `/app/financial/*` (singular, no hub page). Maps to **Q1**.
2. **Operations hub vs. Operations Overview coexistence.** 1.2 Q4 requires `/app/operations` (hub). 0.1 AC 7 locks `/app/operations/dashboard` (Operations Overview). Both must coexist and the hub page must be added. Maps to **Q2**.
3. **Governance hub URL vs. current route.** `/app/governance` already exists as a page (`App.tsx:286`). Under 1.2 Q4, `/app/governance` becomes the hub; the current page content either becomes the hub body or is moved to a sub-path. Maps to **Q3**.
4. **Communications hub URL vs. current route.** `/app/communications` exists (`App.tsx:295`). Same pattern as Governance. Maps to **Q3**.
5. **Legacy redirect retention policy.** 43 `RouteRedirect` instances exist. 1.2 How-This-Informs Layer 3 says "3.2 retires or repositions legacy redirects where they collide with hub URLs". Retention policy is undecided — keep all for back-compat? Retire in a cycle? Per-section decision? Maps to **Q4**.
6. **Owner Portal hub URL migration.** 1.2 Q4 locks `/portal/finances`, `/portal/requests`, `/portal/community`. Current `/portal/*` paths must be audited and a hub URL migration strategy defined. Maps to **Q5**.
7. **`/app/admin` alias route.** `App.tsx:259` maps `/app/admin` to `RoadmapPage` — an undocumented alias. 2.4 Q2 requires every `/app/admin*` route to have `<RouteGuard platform-admin>`. Retain the alias or retire it? Maps to **Q6**.
8. **PublicRouter `/app/*` catch-alls.** 26 `RouteRedirect` entries at `App.tsx:365-461` live outside `WorkspaceRouter`. Their purpose and retention under the 2.3 shunt-retirement model is undecided. Maps to **Q7**.

## Open Questions

### Q1 — Financials hub URL migration (`/app/financials` vs `/app/financial/*`)

Should the `/app/financial/*` subtree (`foundation`, `billing`, `payments`, `expenses`, `reports`) be moved to `/app/financials/*` to align with 1.2 Q4's zone-root hub URL (`/app/financials`), or should `/app/financials` be added as the hub while `/app/financial/*` sub-pages remain at their current singular-prefix paths with legacy redirects added from `/app/financials/*` → `/app/financial/*`?

**Tradeoff axis:** full migration (consistent URL vocabulary, largest back-compat surface to maintain) vs hub-only addition (minimum disruption, permanent plural/singular split) vs per-phase migration (hub now, subtree rename deferred).

PENDING

### Q2 — Operations hub page body

1.2 Q4 locks `/app/operations` as the Operations hub (title "Operations", per 1.2 Q7 zone-label-only). 0.1 AC 7 locks `/app/operations/dashboard` as "Operations Overview". 1.2 Q2 defines hub content (title, one-sentence description, sub-page link list, up to three quick actions, optional one summary widget). What is the hub body composition for Operations, given that the richest operational signal already lives on the Overview dashboard?

**Tradeoff axis:** minimal hub (just nav) vs hub with summary widget (one pulled from Overview, risks duplication) vs hub that redirects to Overview (retires the hub concept for Operations, violates 1.2 Q4).

PENDING

### Q3 — Governance and Communications hub migration

`/app/governance` (`App.tsx:286`) and `/app/communications` (`App.tsx:295`) are live pages today. 1.2 Q4 requires them to be hubs. Does the current page content become the hub body (if it already matches 1.2 Q2 hub rules), or does the current page move to a sub-path (e.g., `/app/governance/overview`) while a new hub takes the zone-root URL?

**Tradeoff axis:** convert in place (minimum routing churn, requires content audit to confirm hub-rule compliance) vs move-and-replace (cleanest hub/sub-page separation, adds two new routes).

PENDING

### Q4 — Legacy redirect retention policy

43 `RouteRedirect` entries exist today (9 finance legacy, 4 governance legacy, 2 persons/occupancy, 2 single-association, 26 PublicRouter `/app/*` catch-alls). 1.2 "How This Informs Downstream Layers" says "3.2 retires or repositions legacy redirects where they collide with hub URLs (e.g., `/app/financial/*` subtree coexists or migrates to `/app/financials/*` per build-phase decision)". Per-section policy — which groups retire, which persist as back-compat redirects, and for how long?

**Tradeoff axis:** retain-all (maximum back-compat, perpetual legacy surface) vs retire-all (cleanest tree, breaks bookmarks/links) vs phased retirement with a deprecation window (e.g., two release cycles, emit console warning, then retire).

PENDING

### Q5 — Owner Portal hub URL migration

1.2 Q4 locks `/portal/finances`, `/portal/requests`, `/portal/community` as Owner Portal hubs. What are the current `/portal/*` routes, and which migrate vs. redirect? Audit required before a resolution can be selected.

**Tradeoff axis:** same three options as Q1 applied to `/portal` — full migration, hub-only addition, or phased. Coupled to the Owner Portal feature scope locked by 0.2 "Current live capabilities" which bounds what sub-pages even exist.

PENDING

### Q6 — `/app/admin` alias disposition

`App.tsx:259` maps `/app/admin` → `RoadmapPage` with no sidebar entry and no inline role guard. 2.4 Q2 requires `<RouteGuard platform-admin>`. Options: (a) retain the alias with `<RouteGuard>` and a `<RouteRedirect to="/app/admin/roadmap" />`; (b) retire the alias entirely and 404 it; (c) promote `/app/admin` to a platform-admin landing hub parallel to `/app`.

**Tradeoff axis:** retire (cleanest, one less hidden URL) vs redirect (preserves inbound links if any exist) vs hub (adds a Platform-zone landing page, scope creep beyond 2.4).

PENDING

### Q7 — PublicRouter `/app/*` catch-all redirects

26 `RouteRedirect` entries at `App.tsx:365-461` live outside `WorkspaceRouter`, reachable via the public (unauthenticated) shell. After 2.3 Q3 retires the board-admin shunt, the authenticated-admin flow no longer needs a separate public fallback for `/app/*`. Do these 26 catch-alls remain as unauthenticated-to-login redirects, get consolidated into a single `/app/*` fallback, or get retired alongside the finance/governance legacy redirects in Q4?

**Tradeoff axis:** retain individually (clearest diagnostics) vs single fallback (cleanest tree) vs retire (smallest surface, risks silent 404s for bookmarked legacy URLs hit by unauthenticated traffic).

### Layer 5 — 5.1 (Cleanup) and 5.3 (Mobile nav)
