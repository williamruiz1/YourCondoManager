## Context

## Conflict Analysis

Seven decision axes surface from reconciling the current 52-route table with the locked Layer 0-2 spec. Each maps to a resolved question below.

1. **Hub URLs collide with current finance subtree.** 1.2 Q4 locks `/app/financials` as the Financials hub URL; the current subtree is rooted at `/app/financial/*` (singular, no hub page). Resolved in **Q1**.
2. **Operations hub vs. Operations Overview coexistence.** 1.2 Q4 requires `/app/operations` (hub). 0.1 AC 7 locks `/app/operations/dashboard` (Operations Overview). Both must coexist and the hub page must be added. Resolved in **Q2**.
3. **Governance hub URL vs. current route.** `/app/governance` already exists as a page (`App.tsx:286`). Under 1.2 Q4, `/app/governance` becomes the hub; the current page content moves to a sub-path. Resolved in **Q3**.
4. **Communications hub URL vs. current route.** `/app/communications` exists (`App.tsx:295`). Same pattern as Governance. Resolved in **Q3**.
5. **Legacy redirect retention policy.** 43 `RouteRedirect` instances exist. 1.2 How-This-Informs Layer 3 says "3.2 retires or repositions legacy redirects where they collide with hub URLs". Resolved in **Q4**.
6. **Owner Portal hub URL migration.** 1.2 Q4 locks `/portal/finances`, `/portal/requests`, `/portal/community`. Resolved in **Q5** — deferred to separate work.
7. **`/app/admin` alias route.** `App.tsx:259` maps `/app/admin` to `RoadmapPage` — an undocumented alias. 2.4 Q2 requires every `/app/admin*` route to have `<RouteGuard platform-admin>`. Resolved in **Q6**.

## Resolved Questions

### Q1 — Financials hub URL migration (`/app/financials` vs `/app/financial/*`)

**Decision:** Hub-only addition. Add `/app/financials` as a new hub page per 1.2 Q4. The existing `/app/financial/*` sub-pages (`foundation`, `billing`, `payments`, `expenses`, `reports`) remain at their current singular-prefix paths — no URL rename. Additionally, add legacy redirects from `/app/financials/*` to `/app/financial/*` so that any deep-link accidentally using the plural prefix (e.g., `/app/financials/billing`) is transparently redirected to the correct singular-prefix route (`/app/financial/billing`). This prevents 404s caused by the plural/singular split without requiring a full subtree migration.

**Rationale:** This is the minimum-disruption path. The Financials hub is a new navigational surface per 1.2 Q2 (zone title, description, sub-page links, optional quick actions, optional summary widget). Existing bookmarks and deep links to `/app/financial/*` continue to work. The plural-to-singular redirect layer catches accidental plural deep-links without a subtree rename. A full migration of `/app/financial/*` to `/app/financials/*` is explicitly not undertaken in this pass — that is a separate future decision if the permanent plural/singular split becomes untenable.

**New routes added:**
| Path | Handler | Purpose |
|---|---|---|
| `/app/financials` | `FinancialsHubPage` (new) | Financials zone hub per 1.2 Q2/Q4 |
| `/app/financials/foundation` | `RouteRedirect -> /app/financial/foundation` | Plural-to-singular legacy redirect |
| `/app/financials/billing` | `RouteRedirect -> /app/financial/billing` | Plural-to-singular legacy redirect |

### Q2 — Operations hub page

**Decision:** Create a new hub at `/app/operations`. The Operations Overview at `/app/operations/dashboard` is unchanged per 0.1 AC 7 — it becomes a sub-page of the Operations zone, not the hub itself. The hub page at `/app/operations` follows 1.2 Q2 composition rules: zone title "Operations" (per 1.2 Q7 zone-label-only), one-sentence description, sub-page link list (Operations Overview, Operations Records, Vendors, Work Orders, Resident Feedback, Maintenance Schedules, Inspections), optional quick actions, and optionally one lightweight summary widget.

**Rationale:** 1.2 Q4 locks `/app/operations` as the hub URL and 0.1 AC 7 locks `/app/operations/dashboard` as Operations Overview. Both coexist. The hub is a navigational and summary surface — it is not a redirect to the Overview dashboard, which would violate 1.2 Q4. The richest operational signal lives on the Overview dashboard; the hub page links to it but does not duplicate its content. This matches the pattern established by 1.2's Cross-Reference Notice: the hub at the zone root is distinct from the dashboard at `/app/operations/dashboard`.

**New routes added:**

| `/app/operations` | `OperationsHubPage` (new) | Operations zone hub per 1.2 Q2/Q4 |

**Routes preserved unchanged:**

### Q3 — Governance and Communications hub migration

**Decision:** Move to sub-path. `/app/governance` and `/app/communications` become hub pages per 1.2 Q4. The current page content at each URL moves to a new sub-path:

- Current `GovernancePage` content at `/app/governance` moves to `/app/governance/overview`.
- Current `CommunicationsPage` content at `/app/communications` moves to `/app/communications/overview`.

The zone-root URLs (`/app/governance`, `/app/communications`) are replaced with new hub page components that follow 1.2 Q2 composition rules: zone title (per 1.2 Q7 zone-label-only), one-sentence description, sub-page link list, optional quick actions, optional summary widget.

**Rationale:** The current `GovernancePage` and `CommunicationsPage` are workflow pages — they contain data tables, action buttons, and domain-specific UI that does not conform to the 1.2 Q2 hub content rules (hubs contain no forms and no deep data tables). Converting them in place would require stripping their content, which is destructive. Moving the current content to `/overview` sub-paths and replacing the zone-root with a clean hub produces the cleanest hub/sub-page separation with no content loss.

**New and changed routes:**

|---|---|---|
| `/app/governance` | `GovernanceHubPage` (new) | Replaces `GovernancePage` at this URL |
| `/app/governance/overview` | `GovernancePage` (existing, relocated) | Current governance page content moves here |
| `/app/communications` | `CommunicationsHubPage` (new) | Replaces `CommunicationsPage` at this URL |
| `/app/communications/overview` | `CommunicationsPage` (existing, relocated) | Current communications page content moves here |

**Governance legacy redirects updated:**
The four existing governance legacy redirects (`/app/governance/board-packages`, `/app/governance/meetings`, `/app/governance/compliance`, `/app/governance/elections`) currently point to `/app/governance`. Since `/app/governance` is now a hub, these redirects remain valid — users landing on the hub from a legacy bookmark can navigate to the correct sub-page from there.

### Q4 — Legacy redirect retention policy

**Decision:** Archive — keep all 43 legacy `RouteRedirect` rules for rollback safety. No redirects are deleted in this pass. The full set of 43 redirects (9 finance legacy, 4 governance legacy, 2 person/occupancy, 2 single-association role-based, 26 PublicRouter `/app/*` catch-alls) is preserved in the codebase. They serve as back-compatibility insurance for bookmarks, shared links, and external integrations that reference legacy URLs.

**Rationale:** Retiring redirects risks breaking bookmarks and external links with silent 404s. The redirect rules impose negligible runtime cost (they are static route declarations). Keeping all 43 rules provides maximum rollback safety and preserves the ability to trace legacy URL usage via analytics before making per-group retirement decisions in a future pass. 1.2 "How This Informs Downstream Layers" says "3.2 retires or repositions legacy redirects where they collide with hub URLs" — none of the 43 existing redirects collide with the new hub URLs (`/app/financials`, `/app/operations`, `/app/governance`, `/app/communications`), so no repositioning is required. The new plural-to-singular financials redirects added by Q1 are additive and do not conflict with any existing redirect.

**Future retirement.** Permanent removal of archived redirects is deferred to Layer 5 / 5.1 (Cleanup). A deprecation-tracking mechanism (e.g., console warning or analytics event on redirect hit) may be added in 5.1 to measure usage before retirement.

### Q5 — Owner Portal hub URL migration

**Decision:** Deferred. The Owner Portal restructure (`/portal/finances`, `/portal/requests`, `/portal/community` hub URLs per 1.2 Q4) is not bundled with the `/app` route pass. It will be addressed as separate work.

**Rationale:** The `/app/*` route restructure and the `/portal/*` route restructure have independent implementation surfaces (different routers, different components, different auth flows). Bundling them increases scope and risk for the current pass. The 1.2 Q4 hub URLs for `/portal` are locked and will be implemented when the Owner Portal work is scoped. 3.2 does not retire, rename, or redirect any existing `/portal/*` route.

### Q6 — `/app/admin` alias disposition

**Decision:** Retire. `/app/admin` returns a hard 404 (`NotFound` component). No redirect to `/app/admin/roadmap`. No deprecation banner. The route declaration at `App.tsx:259` that maps `/app/admin` to `RoadmapPage` is removed entirely.

**Rationale:** `/app/admin` is an undocumented alias with no sidebar entry, no nav link, and no known external consumers. It silently resolves to `RoadmapPage` without any route guard — a security surface flagged by 2.4 Q2. Rather than adding a `<RouteGuard>` and a redirect to an alias that should not exist, the cleanest resolution is removal. The catch-all `*` route at `App.tsx:308` will serve `NotFound` for any request to `/app/admin` after the dedicated route declaration is removed. All legitimate access to the admin roadmap goes through `/app/admin/roadmap`, which retains its `<RouteGuard roles={["platform-admin"]}>` per 2.3 Q11 / 2.4 Q2.

### Q7 — PublicRouter `/app/*` catch-all redirects

**Decision:** Archive — same retention policy as Q4. Keep all 26 public-side `RouteRedirect` entries at `App.tsx:365-461`. They are not consolidated into a single fallback and are not retired in this pass.

**Rationale:** After 2.3 Q3 retires the board-admin (old role) shunt, the `PublicRouter` catch-alls serve a different but still valid purpose: they handle unauthenticated traffic that hits `/app/*` URLs (e.g., expired sessions, shared links opened in a logged-out browser). Each individual redirect provides clear diagnostic signal about which legacy URL was requested, which is preferable to a single opaque fallback. The per-route entries impose negligible cost and aid in tracing redirect usage. Retirement is deferred to Layer 5 / 5.1 alongside the Q4 archive.
