## Summary

The `hubVisibilityLevelEnum` (`shared/schema.ts:2602`) uses role-coupled vocabulary (`public | resident | owner | board | admin`) that conflicts with 2.1 Q11 (LOCKED), which mandates decoupling hub visibility from the role enums. Target vocabulary must be role-agnostic while preserving 1:1 semantics with the current 5-value model.

This module owns the enum rename + text-column parallel rename + data migration + call-site update across 19 touch points (2 schema + 10 server + 7 client).

- **Origin:** deferred from 1.1 handoff AC 6 ("Small hub-visibility rename migration executed (labels only)" ‚Äî scope misdescribed at the time).
- **Policy upstream:** 2.1 Q11 (LOCKED) ‚Äî decouple hub visibility from role enums.

- **Public API implication:** `routes.ts:15319‚Äì15337` emits `visibility_level` verbatim on an anonymous public endpoint. Breaking the `"public"` string literal would silently regress any external consumer keyed on it. The target vocab below preserves `"public"` unchanged.

## Selected Resolution

### Target vocabulary
`public | residents | unit-owners | board-only | operator-only`

5 values, 1:1 mapping with the current enum. Matches the 2.1 Q11 illustrative list verbatim (making the illustrative choice authoritative). Preserves `"public"` unchanged to avoid a public-API break.

**Mapping:**

| Current | Target | Semantics |
|---|---|---|
| `public` | `public` | (preserved verbatim ‚Äî public-API safe) |
| `resident` | `residents` | any portal role with residency (tenants + owners + board) |
| `owner` | `unit-owners` | owner role + board-members with owner baseline |
| `board` | `board-only` | board-member role OR `portalHasBoardAccess` |
| `admin` | `operator-only` | operator-write-only; never surfaces in portal read paths |

Rationale: the `-only` suffix on the two gated tiers (`board-only`, `operator-only`) makes the gating intent explicit and eliminates the role-coupling that 2.1 Q11 flagged. `residents` and `unit-owners` are plural descriptive nouns instead of role names.

### Scope ‚Äî both columns migrate together

1. `hubMapIssues.visibility_level` ‚Äî pg enum-bound. Migration uses `ALTER TYPE hub_visibility_level ADD VALUE` for each new value (additive), then bulk `UPDATE` to new values, then a subsequent migration to rename the type and drop the old values after cutover.
2. `communityAnnouncements.visibility_level` ‚Äî plain `text()`. Zod validation added against the new vocabulary; backfill updates all non-null rows; NULLs remain (they read as "public-ish" per `routes.ts:15091`: `isNull || eq "public"` ‚Äî that NULL-equivalence preserved).

### Feature flag

`HUB_VISIBILITY_RENAME` in `shared/feature-flags.ts` ‚Äî default OFF; per-association override supported. Modeled after `PORTAL_ROLE_COLLAPSE`.

### Migration phases

- **HV-1 (additive, flag OFF).** `ALTER TYPE hub_visibility_level ADD VALUE` for all 5 new strings (pg allows add-not-drop in-place). Both old and new values legal. Zero behavior change. Deployable immediately.
- **HV-2 (backfill + dual-read, flag OFF ‚Üí per-association ON).** Data migration: `UPDATE hub_map_issues SET visibility_level = <new>` mapped per the table above; same for `community_announcements`. Read paths accept both old and new values with a translation layer. Manager can flip `HUB_VISIBILITY_RENAME` per association to cut over writes. Parity window runs until all associations flipped or an SLA passes.
- **HV-3 (retire old values, flag removed) ‚Äî SHIPPED 2026-04-26 (Wave 36).** Migration `0018_hub_visibility_rename_drop_old.sql` recreates `hub_visibility_level` with the 5 new values only via the recreate-and-recast pattern (`ALTER TYPE ‚Ä¶ RENAME TO ‚Ä¶_old` ‚Üí `CREATE TYPE ‚Ä¶ AS ENUM (new values)` ‚Üí column recast ‚Üí `DROP TYPE ‚Ä¶_old`). The text column `community_announcements.visibility_level` is constrained to the new vocab via a CHECK constraint added in the same migration (NULL still legal ‚Äî load-bearing for the public-read NULL-equivalence contract). The `HUB_VISIBILITY_RENAME` feature flag is removed from `shared/feature-flags.ts`. The `HubVisibilityOld` type and `toNewVocab` mapping function are removed from `shared/hub-visibility.ts`; `normalizeHubVisibility` is kept as a one-release identity shim to avoid touching all 19 call sites in the same PR as the schema change. Defensive `UPDATE ... WHERE visibility_level = <old>` statements precede the type recreate; per the 2026-04-24 prod audit they update zero rows in production but make the migration safe under any racing-write scenario. Defaults: `hub_map_issues.visibility_level` default flipped from `'board'` ‚Üí `'board-only'`. Public-API contract preserved verbatim.

### Call-site updates

All 19 sites (audit ¬ß1) updated from HV-2 onward: type declarations, form state, `visibilityLabels` record, `<SelectItem>` options, badge render, server filters, route reads. Test file added exercising the vocabulary explicitly.
