## Summary

The `hubVisibilityLevelEnum` (`shared/schema.ts:2602`) uses role-coupled vocabulary (`public | resident | owner | board | admin`) that conflicts with 2.1 Q11 (LOCKED), which mandates decoupling hub visibility from the role enums. Target vocabulary must be role-agnostic while preserving 1:1 semantics with the current 5-value model.

This module owns the enum rename + text-column parallel rename + data migration + call-site update across 19 touch points (2 schema + 10 server + 7 client).

## Context

- **Origin:** deferred from 1.1 handoff AC 6 ("Small hub-visibility rename migration executed (labels only)" — scope misdescribed at the time).
- **Policy upstream:** 2.1 Q11 (LOCKED) — decouple hub visibility from role enums.
- **Scope:** BOTH `hub_visibility_level` pgEnum AND `community_announcements.visibility_level` text column (the latter is NOT enum-bound; pure string column — see audit §1).
- **Public API implication:** `routes.ts:15319–15337` emits `visibility_level` verbatim on an anonymous public endpoint. Breaking the `"public"` string literal would silently regress any external consumer keyed on it. The target vocab below preserves `"public"` unchanged.

## Selected Resolution

### Target vocabulary
`public | residents | unit-owners | board-only | operator-only`

5 values, 1:1 mapping with the current enum. Matches the 2.1 Q11 illustrative list verbatim (making the illustrative choice authoritative). Preserves `"public"` unchanged to avoid a public-API break.

**Mapping:**

| Current | Target | Semantics |
|---|---|---|
| `public` | `public` | (preserved verbatim — public-API safe) |
| `resident` | `residents` | any portal role with residency (tenants + owners + board) |
| `owner` | `unit-owners` | owner role + board-members with owner baseline |
| `board` | `board-only` | board-member role OR `portalHasBoardAccess` |
| `admin` | `operator-only` | operator-write-only; never surfaces in portal read paths |

Rationale: the `-only` suffix on the two gated tiers (`board-only`, `operator-only`) makes the gating intent explicit and eliminates the role-coupling that 2.1 Q11 flagged. `residents` and `unit-owners` are plural descriptive nouns instead of role names.

### Scope — both columns migrate together

1. `hubMapIssues.visibility_level` — pg enum-bound. Migration uses `ALTER TYPE hub_visibility_level ADD VALUE` for each new value (additive), then bulk `UPDATE` to new values, then a subsequent migration to rename the type and drop the old values after cutover.
2. `communityAnnouncements.visibility_level` — plain `text()`. Zod validation added against the new vocabulary; backfill updates all non-null rows; NULLs remain (they read as "public-ish" per `routes.ts:15091`: `isNull || eq "public"` — that NULL-equivalence preserved).

### Feature flag

`HUB_VISIBILITY_RENAME` in `shared/feature-flags.ts` — default OFF; per-association override supported. Modeled after `PORTAL_ROLE_COLLAPSE`.

### Migration phases

- **HV-1 (additive, flag OFF).** `ALTER TYPE hub_visibility_level ADD VALUE` for all 5 new strings (pg allows add-not-drop in-place). Both old and new values legal. Zero behavior change. Deployable immediately.
- **HV-2 (backfill + dual-read, flag OFF → per-association ON).** Data migration: `UPDATE hub_map_issues SET visibility_level = <new>` mapped per the table above; same for `community_announcements`. Read paths accept both old and new values with a translation layer. Manager can flip `HUB_VISIBILITY_RENAME` per association to cut over writes. Parity window runs until all associations flipped or an SLA passes.
- **HV-3 (retire old values, flag removed).** Second `ALTER TYPE` migration drops old values via recreate-and-recast pattern (pg doesn't support enum value removal in-place; standard workaround is `CREATE TYPE … AS ENUM (new values)` → column re-cast → `DROP TYPE old`). Flag deleted. Translation layer removed.

### Call-site updates

All 19 sites (audit §1) updated from HV-2 onward: type declarations, form state, `visibilityLabels` record, `<SelectItem>` options, badge render, server filters, route reads. Test file added exercising the vocabulary explicitly.

## Acceptance Criteria

- `HUB_VISIBILITY_RENAME` flag present in `shared/feature-flags.ts`; defaults OFF.
- **Phase HV-1** migration adds the 5 new enum values to `hub_visibility_level` without removing old values. Deployable green with both vocabularies live.
- **Phase HV-2** data migration updates all `hub_map_issues.visibility_level` and `community_announcements.visibility_level` rows per the mapping table. Read paths accept both vocabularies via a translation helper (`shared/hub-visibility.ts`). Per-association flag flip tested.
- **Phase HV-3** migration drops the 5 old enum values via enum-recreate + column re-cast. Flag removed. Translation helper removed. All 19 call sites use new vocabulary exclusively.
- **Public API contract preserved:** `"public"` literal emitted unchanged at `routes.ts:15319–15337`; no external consumer regression.
- **Row-count audit:** `SELECT visibility_level, COUNT(*) FROM hub_map_issues GROUP BY 1;` and same on `community_announcements` run against production and captured in the HV-2 migration PR body (used to size the migration window).
- **Tests:** unit tests for `shared/hub-visibility.ts` translation helper; integration test that `GET /api/hub/:identifier/public` response `visibility_level` field equals `"public"` regardless of flag state; integration test asserting no `null` rows are converted to a non-null value during HV-2 backfill.
- **Zod validation** added to `communityAnnouncements` CRUD paths accepting only the new vocabulary (or null).
- **Deprecation doc** — `layer-2-primitives-handoff.md` AC 9 updated to reference this module; `1.1-zone-taxonomy-handoff.md` AC 6 struck (scope-misdescribed).
