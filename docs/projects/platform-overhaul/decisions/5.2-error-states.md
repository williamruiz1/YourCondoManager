# 5.2 — Error States

**Status:** SPEC LOCKED (Wave 14, 2026-04-24) · **IN BUILD (partial)**
**Owner:** Platform Overhaul
**Layer:** 5 — Polish & Hardening

---

## Scope

Consistent error-boundary + fallback UX, and a standardized `<ErrorState>` component for in-page error render paths (TanStack Query `error`, mutation failures, etc.).

Today:
- No top-level React error boundary exists — a render error in a lazy-loaded route can surface as a blank screen with a message only in the console.
- API errors are displayed inconsistently across pages: some surface `error.message` raw, some show a toast, some show nothing.
- There is no central `reportError(error, context)` helper; future Sentry/Datadog wiring has no single seam to hook into.

This module:
- Adds a root `<ErrorBoundary>` wrapping the `App` body (below `QueryClientProvider`, above routes) so any render error renders `<ErrorState>` instead of a blank page.
- Adds a shared `<ErrorState>` component for in-page error paths.
- Adds a `reportError(error, context)` helper that currently console-logs and leaves a TODO for future observability wiring. No new external deps (no Sentry / Datadog).

**In scope:**
- `client/src/components/error-boundary.tsx` (class component, React 18).
- `client/src/components/error-state.tsx` (shared fallback UI).
- `client/src/lib/error-reporting.ts` — `reportError(error, context)` helper.
- 3–5 page-level call sites moved onto `<ErrorState>` when a TanStack Query `error` is present.
- Tests.

**Out of scope:**
- External observability integration.
- Per-route error boundaries (root boundary is enough for Wave 14; per-route refinement filed as follow-up).
- Sentry replay / session replay.

## Acceptance Criteria

1. `client/src/components/error-state.tsx` exports `ErrorState` with props `{ title, description, retry?: () => void, details?: string }`. Renders a `<Card>` with an `AlertTriangle` icon, title, description, optional `<details>` collapse (for `details`), and optional "Retry" button.
2. `client/src/components/error-boundary.tsx` exports a class `ErrorBoundary` that catches render errors via `componentDidCatch`, stores `{ hasError, error }` in state, and renders `<ErrorState title="Something went wrong" …>` on error. Props include optional `onError?: (error, info) => void` and `fallback?: ReactNode` for customization.
3. `App.tsx` wraps its inner tree with `<ErrorBoundary>` below the TanStack Query provider and above routes so that a route-level render failure does not crash the full page. The trial banner + header chrome remain visible through the boundary.
4. `client/src/lib/error-reporting.ts` exports `reportError(error: unknown, context?: Record<string, unknown>)`. In Wave 14 it calls `console.error` with a `[reportError]` prefix and leaves a TODO for future observability. Safe to call at module scope (no side effects).
5. At least 3 call sites route query errors through `<ErrorState>` with a Retry button that calls `refetch()`. Wave 14 targets:
   - `portal-requests.tsx` — request-list query error.
   - `portal-notices.tsx` — notices query error.
   - `financial-rules.tsx` — run-history query error.
6. ≥6 tests under `tests/error-state.client.test.tsx` + `tests/error-boundary.client.test.tsx` cover: default render, retry callback fires, details toggle, boundary catches a throwing child, boundary renders `ErrorState`, and `reportError` is called from boundary.
7. No existing toast-based error paths are regressed — the new `<ErrorState>` is additive.

---

## Design Notes

- Error copy is generic: "Something went wrong" + a 1-sentence description that doesn't leak backend internals.
- Details (`<details>` / collapsible) renders the error message verbatim for engineering triage — but not automatically expanded. Defaults collapsed.
- Retry calls `refetch()` from TanStack Query at call sites; inside the boundary itself Retry sets `hasError: false` to attempt a re-render.

## Wave 18 — Round 2 adoption

**Date:** 2026-04-24

Wave 14 shipped the root `<ErrorBoundary>` and `<ErrorState>` adoption on 3 portal/operator surfaces. Wave 18 introduces **per-zone ErrorBoundary scoping** so a render error inside one zone does not take down sibling zones, and broadens `<ErrorState>` adoption on raw error displays.

### Per-zone ErrorBoundary policy (Wave 18)

`client/src/App.tsx` now wraps each zone's `<Route>` content in a `ZoneBoundary` helper that mounts an `<ErrorBoundary>` with zone-specific copy. The root `<ErrorBoundary>` in `App()` still catches errors that happen above the zone subtree (e.g. in the sidebar, header, association switcher, trial banner).

Zones with explicit boundaries (6 total):
1. **Financials** — wraps `/app/financial/foundation`, `/app/financial/rules`, `/app/financial/billing`, `/app/financial/payments`, `/app/financial/expenses`, `/app/financial/reports`.
2. **Operations** — wraps `/app/vendors`, `/app/work-orders`, `/app/resident-feedback`, `/app/maintenance-schedules`, `/app/inspections`.
3. **Governance** — wraps `/app/governance` and `/app/governance/elections/:id`.
4. **Communications** — wraps `/app/communications/inbox` and `/app/communications`.
5. **Platform** — wraps `/app/platform/controls` (platform-admin only).
6. **Portal** — wraps every `/portal/*` route in `PublicRouter`.

Each zone boundary renders `<ErrorState title="<Zone> hit an error" description="…">` on a render failure inside the zone subtree. Sidebar, header, trial banner, and the rest of the workspace shell remain interactive.

Other operator pages (Home, Portfolio Health, Settings, Admin, AI Ingestion, Community Hub, Amenities, Insurance, Announcements, Help Center, Onboarding) inherit the root `<ErrorBoundary>` from `App()` — no separate zone boundary needed because they are not part of a multi-page subtree.

### ErrorState adoptions in Wave 18

| Surface | Source pattern | Wave 18 change |
|---|---|---|
| `financial-foundation.tsx` (Account Activity) | `<p className="text-sm text-destructive">Failed to load account activity: {error.message}</p>` | Replaced with `<ErrorState retry={refetch} details={error.message} />` |
| `communications-inbox.tsx` | `<div className="...">Unable to load inbox: {error.message}</div>` | Replaced with `<ErrorState retry={refetch} details={error.message} />` (added `refetch` to `useCrossAssociationAlerts` return type) |
| `roadmap.tsx` (main payload) | `<Card><CardContent>Unable to load roadmap…<Button>Retry</Button></CardContent></Card>` | Replaced with `<ErrorState retry={refetch} details={error.message} />` (custom 403 description preserved). |
| `roadmap.tsx` (feature-tree) | Same pattern | Replaced. |
| `vendor-portal.tsx` | Material-symbols error icon + "Failed to load work orders. Please try again." | Replaced with `<ErrorState retry={refetchWorkOrders} details={error.message} />` |

Total Wave 18 ErrorState adoptions: **5 query-error paths**.

### Surfaces intentionally NOT migrated

- `boards-portal.tsx` — already uses purpose-built error fallback with a "Retry" button + an "Error loading board dashboard" headline; semantic match for ErrorState but copy is board-portal-specific. Filed as deferred.
- `community-hub-public.tsx` form-level inline error pills — these are form-validation errors, not query errors. Toast-equivalent UX is intentional.
- `vendor-portal.tsx` mutation-result inline `text-red-600` pills (4 instances) — same: per-action mutation errors, not query failures. Toast-equivalent UX is intentional.
- `new-association.tsx`, `onboarding-invite.tsx` — form-mutation error inline rendering is intentional; replacement to `<ErrorState>` would over-format.
- `AsyncStateBoundary` (`client/src/components/async-state-boundary.tsx`) already renders an inline error card with retry — surfaces using it (vendors, work-orders, documents, financial-ledger top-level) are not double-wrapped.
- `HubAlertWidget` ad-hoc inline error — spec-locked per Wave 14 (5.2 cross-references).

### Future / deferred

- Move `<AsyncStateBoundary>` internals onto shared `<ErrorState>` so a single change propagates to vendors / work-orders / documents / ledger top-level. Not done in Wave 18 to avoid scope creep.
- Sentry/observability wiring inside `reportError` — still console-only.
- Per-route boundaries inside `/portal/*` (already covered by zone boundary).
