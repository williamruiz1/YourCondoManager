# 5.8 — `prefers-reduced-motion` Audit

**Status:** SPEC LOCKED 2026-04-25 (ship-and-lock pattern, Wave 29)
**Owner:** Platform Overhaul
**Layer:** 5 — Polish & Hardening

---

## Scope

Honor the user's OS-level "Reduce motion" preference (`prefers-reduced-motion: reduce`) across the YCM client. Suppress non-essential animation while preserving animations that communicate state.

This module is shipped under the **ship-and-lock** pattern: spec is finalized concurrent with the implementation PR (Wave 29). No standalone IN SPEC review; the PR diff is the implementation surface.

### What is suppressed

When the OS reports `prefers-reduced-motion: reduce`:

- Decorative pulse animations (`animate-pulse` skeleton loaders).
- Slide-in / slide-out / fade / zoom animations on Radix-driven primitives — Dialog, AlertDialog, Sheet, Popover, Tooltip, HoverCard, Select, DropdownMenu, ContextMenu, Menubar, NavigationMenu, Toast, Accordion.
- Smooth scrolling — both global `html { scroll-behavior: smooth }` and imperative `element.scrollIntoView({ behavior: "smooth" })` calls.
- Any escaped CSS-driven `transition`/`animation` outside the component tree (covered by a global safety-net CSS rule).

### What is preserved

Animations that communicate state are NOT suppressed; they are slowed instead so they remain perceivable but non-distracting:

- **Loading spinners** (`animate-spin`) — kept rotating with a slowed 1.5s period. Removing the spin entirely would silently degrade the "in-progress" affordance.
- **Focus indicators** — never animated as decoration; native browser focus rings are unaffected.

---

## Approach

Three layers of defense, ordered from most-specific to most-general:

1. **Per-component Tailwind classes.** Every Radix-driven shadcn primitive that previously emitted `data-[state=open]:animate-in` (etc.) gets a paired `motion-reduce:transition-none motion-reduce:animate-none` override. This is the canonical fix and produces the most predictable result — no animation runs at all on these primitives when the user has reduce-motion set. The skeleton primitive's `animate-pulse` is suppressed the same way.

2. **Imperative scroll helper.** Smooth scrolling triggered by JavaScript (`scrollIntoView({ behavior: "smooth" })`) cannot be reached by Tailwind variants. We added `client/src/lib/prefers-reduced-motion.ts` exposing `prefersReducedMotion()` and `getScrollBehavior()` helpers. Existing call sites (`pages/units.tsx`, `pages/board-portal.tsx`) now route their `behavior` argument through `getScrollBehavior()`, which downgrades to `"auto"` when reduce-motion is set.

3. **Global CSS safety net.** In `client/src/index.css`, a `@media (prefers-reduced-motion: reduce)` block shortens any escaped `animation-duration` and `transition-duration` to ~1ms across all elements, and forces `scroll-behavior: auto`. This catches third-party CSS, Radix internal animations not yet hard-overridden, and any future `animate-*`/`transition-*` we forget to wrap. The `.animate-spin` selector is restored to a finite 1.5s infinite duration inside the same block to preserve loading-spinner semantics.

The matching positive case — `@media (prefers-reduced-motion: no-preference) { html { scroll-behavior: smooth } }` — applies smooth in-page scrolling only when the user has not opted out. Without reduce-motion, smooth scrolling remains active.

---

## Acceptance Criteria

1. **Spinners stay visible.** Any element with `animate-spin` continues to rotate when reduce-motion is active (slowed but present), so loading state remains communicated.
2. **Decorative pulses suppressed.** Skeleton loaders (`animate-pulse`) freeze at default opacity under reduce-motion — no flicker, no oscillation.
3. **Slide / fade / zoom suppressed.** Dialog, AlertDialog, Sheet, Popover, Tooltip, HoverCard, Select, DropdownMenu, ContextMenu, Menubar, NavigationMenu, Toast, Accordion all open/close instantly under reduce-motion.
4. **Smooth scrolling disabled.** `html { scroll-behavior: smooth }` only applies under `(prefers-reduced-motion: no-preference)`. Imperative `scrollIntoView({ behavior: "smooth" })` call sites use `getScrollBehavior()` to fall back to `"auto"`.
5. **No external dependencies added.** No `framer-motion`, `react-aria`, etc. (Note: `framer-motion` was already a transitive devDependency but is not used in the client tree; this audit confirms that and leaves the dependency untouched.)
6. **No functional regressions.** Existing animations remain visible on systems that have not opted out of motion. Tailwind's default `motion-reduce:` variant maps to `(prefers-reduced-motion: reduce)` per Tailwind v3+; no custom config is needed.
7. **Smoke test.** A client-side test verifies that mounting an animation-bearing component (Dialog overlay or Skeleton) emits class names containing the `motion-reduce:` overrides. The test does not exercise jsdom's media-query stub at runtime — it asserts class-list presence, which is what ships to the browser.

### Tailwind variant availability

Tailwind v3+ ships `motion-reduce:` and `motion-safe:` as core variants — no plugin or config change is required. Verified against `tailwind.config.ts` (no `corePlugins` exclusion of motion variants).

### Why a global CSS rule AND per-component overrides

- The per-component overrides are the **canonical** fix: they make the suppression visible in the component definition and survive future CSS resets.
- The global `@media (prefers-reduced-motion: reduce)` block is a **safety net** for: (a) third-party CSS we don't control, (b) Radix internal `data-state` animations we may have missed, (c) any developer who forgets to add `motion-reduce:` to a new component. Using `0.001ms` instead of `0` keeps the animation event lifecycle (`animationend`, etc.) firing — important for any component that listens for animation completion to trigger logic (e.g. Radix focus traps).

### What was deferred

- **Framer-motion deep audit.** `framer-motion` is in `package.json` but is not imported anywhere in `client/src/`. No work needed; flagged for periodic re-audit.
- **Per-page transition orchestration.** Wave 29 does not introduce a `<MotionConfig>` boundary or page-transition system. If we adopt one in future, it must consume the same `prefersReducedMotion()` helper.
- **Reduced-motion variants for charts.** Recharts was replaced in Wave 22 with hand-rolled SVG charts; chart entrance animations (if any) are static draws and need no suppression.
