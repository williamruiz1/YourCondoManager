## Context

The Financials zone (1.1 Q3: "All money-related pages. Budgets, assessments, owner ledgers, payment tracking, financial reports.") contains three existing assessment-related subsystems that operate independently:

1. **HOA fee schedules** (`shared/schema.ts:248-261`, `hoaFeeSchedules` table) — association-level fee definitions with frequency, amount, grace days, active flag, but **no execution engine**. Fee schedules are read by the portal financial dashboard (`server/routes.ts:11358-11365`) but never automatically posted to ledgers.

2. **Recurring charge schedules** (`shared/schema.ts:575-613`, `recurringChargeSchedules` + `recurringChargeRuns` tables) — the only subsystem with an automated execution engine. Schedules define a `chargeDescription`, `amount`, `frequency` (monthly/quarterly/annual), `dayOfMonth`, and optional unit scope. A batch sweep in `server/index.ts:109-168` (`runDueRecurringCharges`) runs as part of the automation sweep (`server/index.ts:255-270`, `runAutomationSweep`) and posts `ownerLedgerEntries` for each active schedule whose `nextRunDate <= now`. Retry tracking exists via `recurringChargeRuns` with status enum `pending | success | failed | skipped | retrying` (`shared/schema.ts:596-613`).

3. **Special assessments** (`shared/schema.ts:263-277`, `specialAssessments` table) — one-time or installment-based assessments with `totalAmount`, `installmentCount`, `startDate`, `endDate`, per-unit exclusions, and an `autoPostEnabled` flag. Execution is handled by `server/assessment-installments.ts:60-186` (`processSpecialAssessmentInstallments`), which posts installment ledger entries with `referenceType: "special_assessment_installment"`. Also runs in the automation sweep (`server/index.ts:261`).

These three subsystems share no common rule model, no unified execution pipeline, and no consolidated UI. The operator manages them across three different surfaces:

- **Fee schedules**: `client/src/pages/financial-foundation.tsx` (Foundation tab, alongside chart-of-accounts and categories — `client/src/pages/financial-foundation.tsx:27`)
- **Recurring charges**: `client/src/pages/financial-recurring-charges.tsx` (embedded in Foundation page via `FinancialRecurringChargesContent` — `client/src/pages/financial-foundation.tsx:27`)
- **Special assessments**: `client/src/pages/financial-assessments.tsx` (Assessments tab inside Billing page — `client/src/pages/financial-billing.tsx:5,23,30-31`)

### Downstream integration points

- **Ledger**: All three subsystems ultimately post to `ownerLedgerEntries` (`shared/schema.ts:536-548`) with `entryType` values `charge` or `assessment`.
- **Delinquency**: Delinquency tracking (`shared/schema.ts:2095-2131`, `delinquencyThresholds` + `delinquencyEscalations`) and delinquency notices (`shared/schema.ts:2961-3007`, `delinquencySettings` + `delinquencyNotices`) consume ledger balances. The delinquency notice sweep runs in the same automation batch (`server/index.ts:264`).
- **Late fees**: Late fee rules (`shared/schema.ts:279-284`, `lateFeeRules`) apply to overdue balances and are managed on the Billing page's Late Fees tab (`client/src/pages/financial-billing.tsx:33-34`).
- **Owner Portal**: The portal financial dashboard (`server/routes.ts:11350-11397`) returns `feeSchedules` (from `recurringChargeSchedules`), `nextDueDate`, `balance`, `paymentPlan`, and `recentEntries` to the owner. Owners see their assessment schedule read-only in the Financials tab (`client/src/pages/owner-portal.tsx:2265`).
- **Autopay**: Owner autopay enrollment (`shared/schema.ts:1519`) triggers payment against posted ledger charges.

## Conflict Analysis

Six decision axes emerge from the fragmented state:

1. **Three rule models, no unification.** `hoaFeeSchedules`, `recurringChargeSchedules`, and `specialAssessments` define overlapping but incompatible rule shapes. A recurring monthly assessment can be modeled in either fee schedules (no execution) or recurring charges (with execution). Resolved in **Q1**.

2. **Fee schedules have no execution engine.** `hoaFeeSchedules` rows are read by the portal dashboard but never posted to ledgers. Their `frequency` and `amount` fields imply recurring behavior that does not actually occur. Resolved in **Q2**.

3. **Execution is split across two code paths.** `runDueRecurringCharges` (`server/index.ts:109`) and `processSpecialAssessmentInstallments` (`server/assessment-installments.ts:60`) run independently in the same automation sweep, each with its own retry/skip logic. Resolved in **Q3**.

4. **No delinquency trigger from rule execution.** Recurring charge runs and assessment installment posting do not directly trigger delinquency evaluation. Delinquency detection relies on a separate sweep reading ledger balances. The gap between charge posting and delinquency detection is undefined. Resolved in **Q4**.

5. **Owner Portal shows `recurringChargeSchedules` but not `hoaFeeSchedules` or `specialAssessments`.** The portal financial dashboard (`server/routes.ts:11358-11365`) queries only `recurringChargeSchedules` for `feeSchedules`. Owners cannot see HOA fee schedule definitions or upcoming special assessment installments. Resolved in **Q5**.

6. **PM Toggle implications for Assisted Board rule visibility.** 0.2 grants Assisted Board read-only access to Financials by default. The PM can expand this via toggle, but there is no specification for what "read-only" means in the context of assessment rules — can Assisted Board see rule definitions, execution history, or just ledger outcomes? Resolved in **Q6**.

## Open Questions

### Q1 — Should the three rule models (fee schedules, recurring charges, special assessments) be unified into a single assessment rules engine, or remain separate subsystems?

**PENDING — resolution required.**

Current state:
- `hoaFeeSchedules` (`shared/schema.ts:248-261`): `name`, `amount`, `frequency` (monthly/quarterly/annually/one-time), `startDate`, `endDate`, `graceDays`, `isActive`
- `recurringChargeSchedules` (`shared/schema.ts:575-590`): `chargeDescription`, `amount`, `frequency` (monthly/quarterly/annual), `dayOfMonth`, `nextRunDate`, `status` (active/paused/archived), `unitId` (nullable = all units), `maxRetries`
- `specialAssessments` (`shared/schema.ts:263-277`): `name`, `totalAmount`, `installmentCount`, `startDate`, `endDate`, `autoPostEnabled`, `excludedUnitIdsJson`

Hypothesis A — **Unify**: Create a single `assessmentRules` table that generalizes all three. Rule type enum: `recurring_dues | special_assessment | one_time_fee`. Shared fields: frequency, amount model, unit scope, effective dates, execution configuration. Migrate existing data.

Hypothesis B — **Federate**: Keep the three tables. Add a shared `assessmentRuleRegistry` view or materialized index that presents a unified read surface for the UI and portal, but leave execution engines separate.

Hypothesis C — **Retire fee schedules, keep two**: Migrate `hoaFeeSchedules` into `recurringChargeSchedules` (adding execution capability). Keep `specialAssessments` separate because its installment model is structurally different.

**Resolution — Hypothesis C (retire fee schedules, keep two-table model).**

Rationale: Fee schedules and recurring charges are the same concept with different names and different levels of completeness. One has an execution engine; the other does not. Merging them eliminates the duplication and the Q2 data-integrity bug in one move. Special assessments retain their own table because they are semantically different — one-shot, board-voted, legal-document-backed, installment-based — and forcing them into the recurring charge model would require carrying installment-specific columns on a table most of whose rows never use them.

**Acceptance Criteria:**
- AC 1: No new `assessmentRules` unified table is introduced in this overhaul cycle.
- AC 2: `hoaFeeSchedules` is retired after data migration (AC 4–6 below).
- AC 3: `specialAssessments` remains its own table, unchanged in shape; its installment execution pipeline (`server/assessment-installments.ts`) remains distinct from the recurring charge pipeline.
- AC 4: `recurringChargeSchedules` is confirmed as the canonical home for both HOA-style recurring dues and ad-hoc recurring charges post-migration.

### Q2 — Should `hoaFeeSchedules` gain an execution engine, or should its data be migrated into `recurringChargeSchedules`?

**PENDING — resolution required.**

Current state: `hoaFeeSchedules` rows are stored and served to the portal dashboard via `recurringChargeSchedules` (the portal actually queries `recurringChargeSchedules`, not `hoaFeeSchedules` — `server/routes.ts:11359`). Fee schedules are effectively documentation-only records with no automated posting behavior.

Hypothesis A — **Add execution to fee schedules**: Extend `hoaFeeSchedules` with `nextRunDate`, `status`, and run-tracking columns. Build a parallel execution sweep.

Hypothesis B — **Migrate to recurring charges**: Map each active `hoaFeeSchedules` row into a `recurringChargeSchedules` row with matching frequency, amount, and dates. Deprecate the `hoaFeeSchedules` table. Update the fee-schedule CRUD endpoints (`server/routes.ts:2855-2877`) to write to `recurringChargeSchedules`.

Hypothesis C — **Leave as-is**: Fee schedules remain informational. Only `recurringChargeSchedules` executes. Document the distinction explicitly in the UI.

**Resolution — Hypothesis B (migrate to recurring charges, retire `hoaFeeSchedules`).**

Rationale: Consistent with Q1 = C. The existing `recurringChargeSchedules` execution engine already posts to `ownerLedgerEntries` on schedule with run-tracking and retry; building a parallel engine on `hoaFeeSchedules` (option A) would double maintenance cost for no gain. Option C is user-hostile — the portal surfaces fee schedule data implying active billing, and documenting the limitation does not fix the data-integrity issue.

**Acceptance Criteria:**
- AC 5: A one-time data migration script maps each active `hoaFeeSchedules` row to a new `recurringChargeSchedules` row with equivalent `frequency`, `amount`, `dayOfMonth` (derived from `startDate` if not explicit), `nextRunDate` (computed from the migration date + frequency rule), and `status: active`.
- AC 6: Per-unit scoping: `hoaFeeSchedules` applies at the association level (all units); the migration sets `unitId = null` on the target `recurringChargeSchedules` row.
- AC 7: Grace-period data (`hoaFeeSchedules.graceDays`) is not lost — migration records the value on an association-level billing config surface (exact target TBD with Q7 resolution; minimum-viable: store on the `recurringChargeSchedules` row via a new `graceDays` column or drop into association settings).
- AC 8: Fee-schedule CRUD endpoints (`server/routes.ts:2855-2877`) are redirected to read/write `recurringChargeSchedules` during the deprecation window, then removed after the client is migrated.
- AC 9: The Foundation page Fee Schedule surface (`client/src/pages/financial-foundation.tsx`) is updated to operate against `recurringChargeSchedules` (UI label may stay "Fee Schedule" for operator continuity).
- AC 10: Portal financial dashboard (`server/routes.ts:11358-11365`) already queries `recurringChargeSchedules` — no change needed, the data now reflects migrated rows.
- AC 11: After migration, `hoaFeeSchedules` table is marked deprecated in schema with a removal target (Layer 5 cleanup).

### Q3 — Should the recurring charge execution (`server/index.ts:109-168`) and special assessment installment posting (`server/assessment-installments.ts:60-186`) be consolidated into a single execution pipeline?

**PENDING — resolution required.**

Current state: Both run independently in `runAutomationSweep` (`server/index.ts:256`). They share the same target table (`ownerLedgerEntries`) but use different:
- Selection logic (recurring: `nextRunDate <= now` + `status: active`; assessment: `isActive + autoPostEnabled` with installment-number math)
- Run tracking (recurring: `recurringChargeRuns` table with retry/status; assessment: idempotency via `referenceId` dedup set, no run-log table)
- Error handling (recurring: per-unit run record with `failed`/`retrying` status; assessment: silent skip)

Hypothesis A — **Single pipeline**: Create a unified `assessmentExecution` service that dispatches to type-specific handlers but shares selection, run-logging, retry, and error reporting infrastructure.

Hypothesis B — **Keep separate, add shared run log**: Keep the two execution functions. Add a shared `assessmentRunLog` table that both write to, giving operators a single audit trail.

Hypothesis C — **Keep separate, no change**: Accept the current dual-path architecture. The two subsystems are structurally different enough that consolidation adds complexity without proportional benefit.

**Acceptance Criteria:** TBD after resolution.

### Q4 — How should rule execution trigger delinquency evaluation?

**PENDING — resolution required.**

Current state: Delinquency detection is decoupled from charge posting. The delinquency notice sweep (`server/index.ts:264`, `runAllDelinquencyNotices`) runs in the same automation batch as charge posting but reads ledger balances independently. Delinquency thresholds (`shared/schema.ts:2095-2108`) define stage-based escalation by `minimumBalance` and `minimumDaysOverdue`. Delinquency settings (`shared/schema.ts:2961-2975`) configure `gracePeriodDays`, `bucketBoundariesJson` (`[30,60,90]` default), and `autoLateFeeEnabled`.

The gap: when a recurring charge posts a new ledger entry, the delinquency system does not immediately re-evaluate that owner's status. Re-evaluation happens only on the next automation sweep cycle.

Hypothesis A — **Synchronous trigger**: After each ledger entry is posted by the rules engine, immediately invoke a lightweight delinquency check for that owner/unit. Escalate or de-escalate in the same transaction.

Hypothesis B — **Event-based**: Emit a `ledger.entry.posted` event after charge posting. A delinquency subscriber processes these events asynchronously but within the same sweep cycle, after all charges are posted.

Hypothesis C — **Status quo (sweep-based)**: Accept the current model where delinquency detection runs as a peer job in the automation sweep. The latency between charge posting and delinquency detection is bounded by the sweep interval.

**Resolution — Hypothesis C (status quo sweep-based) for Phase 1.**

Rationale: Within-24h delinquency detection is acceptable for condo/HOA billing — the industry moves on monthly cycles, not sub-hour. The existing automation sweep already runs delinquency notice generation alongside charge posting (`server/index.ts:264`), so the two jobs share a cadence. Synchronous re-evaluation (Hypothesis A) couples the rules engine and delinquency engine tightly and risks transaction-scope failures where a delinquency check error rolls back the ledger post. Event-based decoupling (Hypothesis B) is the architecturally right next step but requires new event infrastructure — out of proportion to the Phase 1 value. Hypothesis B is flagged as a post-lock enhancement; revisit if operator feedback shows the 24h latency is user-facing.

**Acceptance Criteria:**
- AC 12: No new synchronous delinquency-evaluation hook is introduced into the recurring charge or special-assessment execution paths.
- AC 13: The existing automation sweep ordering is preserved: charge/assessment posting runs before `runAllDelinquencyNotices` within `runAutomationSweep` (`server/index.ts:255-270`). Delinquency detection consumes the freshly-posted ledger entries on the same sweep cycle.
- AC 14: A post-lock enhancement ticket is queued: "event-based delinquency re-evaluation (Hypothesis B)" — revisit after Phase 1 telemetry.
- AC 15: Operator documentation for the Financials zone explicitly states the delinquency-detection cadence (bounded by sweep interval; currently daily).

### Q7 — What is the rule definition model for the unified/consolidated engine?

**PENDING — resolution required. Depends on Q1 resolution.**

Regardless of whether Q1 selects unification, federation, or partial migration, the rules engine needs a canonical data model for rule authoring. This question covers:

- **Frequency options**: Current enums differ — `hoaFeeSchedules` supports `monthly | quarterly | annually | one-time` (`shared/schema.ts:247`); `recurringChargeSchedules` supports `monthly | quarterly | annual` (`shared/schema.ts:573`). Should these be harmonized? Should `semi-annual`, `weekly`, or custom cron expressions be added?
- **Amount model**: Flat amount per unit (current) vs. percentage of unit assessed value vs. tiered by unit type/size vs. custom formula. Current `hoaFeeSchedules.amount` and `recurringChargeSchedules.amount` are both flat `real` values (`shared/schema.ts:252,581`).
- **Unit scope**: `recurringChargeSchedules.unitId` (`shared/schema.ts:578`) is nullable (null = all units). `specialAssessments.excludedUnitIdsJson` (`shared/schema.ts:274`) uses an exclusion list. Should the model support inclusion lists, unit-type filters, or building-level scope?
- **Effective dates**: `hoaFeeSchedules` has `startDate` + `endDate` (`shared/schema.ts:254-255`). `recurringChargeSchedules` has `nextRunDate` but no explicit end date. `specialAssessments` has `startDate` + `endDate` (`shared/schema.ts:268-269`). Should all rules support effective date ranges with explicit start/end?
- **Grace period**: Only `hoaFeeSchedules` has `graceDays` (`shared/schema.ts:256`). Should grace periods be a universal rule attribute that feeds into late fee and delinquency evaluation?
