## Scope

Targeted accessibility audit of the **top 10 user-facing surfaces** on the YCM client. Each surface receives a triage-level pass against WCAG 2.1 AA (the four criteria called out below) plus a small set of mechanical fixes. Full-app retrofit is out of scope for Wave 21.

**In scope:**
- `aria-label` / `aria-labelledby` audits on icon-only buttons and orphan controls.
- Keyboard navigation: Tab order, focus-visible outlines on links/buttons, Esc for modals (already handled by Radix `<Dialog>`, verified — no fix required).
- Focus-visible outlines (`focus-visible:outline-*` Tailwind utilities) on every interactive element that lacked one.
- Color contrast: spot-check against WCAG AA (4.5:1 body / 3:1 large text) — all in-scope surfaces use the existing token palette which already meets AA. No fixes needed.
- Semantic HTML: confirmed each shell renders exactly one `<main>`, no nested `<main>` after refactor; tables receive `<caption>` or `aria-labelledby`.
- Skip-links: added on both shells (`<a href="#main-content">` in `App.tsx`, `<a href="#portal-main-content">` in `portal-shell.tsx`).
- `role="status"` on shared `<EmptyState>` and inline empty/loading states; `role="alert"` on shared `<ErrorState>`.

**Out of scope:**
- The other ~50 list/table surfaces in the operator workspace and the remaining six portal surfaces. Tracked as backlog in `client/src/i18n/README.md` (the i18n module's deferred work overlaps the a11y backlog).
- Custom dropdown / combobox audits (no surfaces in this wave use a non-Radix dropdown).
- High-contrast / dark-mode contrast verification beyond the default tokens.
- Screen-reader transcripts (manual NVDA / VoiceOver runs are a follow-up).
- Replacing the iconography system or auditing every `<span className="material-symbols-outlined">` (only the user-facing ones in the 10 surfaces received `aria-hidden="true"`).
- Refactoring shadcn / Radix primitives — they already handle their own a11y semantics.

## Acceptance Criteria

1. Every icon-only button on the 10 in-scope surfaces has either visible text content, an `aria-label`, or a parent `aria-labelledby` providing the semantic name.
2. Every decorative icon inside a labeled control or text block carries `aria-hidden="true"` so it is not double-announced by screen readers.
3. Both shells render a "Skip to content" link at the top of the DOM. The links are visually hidden by default (`sr-only`) and become visible on `:focus`, jumping focus to the shell's `<main>` element via `id="main-content"` (operator) and `id="portal-main-content"` (portal).
4. The operator shell's `<main>` and the portal shell's `<main>` each receive `tabIndex={-1}` so the skip-link target is programmatically focusable.
5. Page-level wrappers under each shell use `<section>` with `aria-labelledby` (or `<div>` with no role) — no nested `<main>` elements.
6. Tables on `financial-rules.tsx` (Run History) and `portal-finances.tsx` (Recent Ledger, Full Ledger) include a `<caption className="sr-only">` or `aria-labelledby` referencing the heading immediately above.
7. Empty / loading / error states use semantic roles: `role="status"` for empty + loading containers, `role="alert"` for error states.
8. Shared `<EmptyState>` carries `role="status"`; shared `<ErrorState>` carries `role="alert"`. (Already-shipped components updated in place — no behavior change.)
9. Form controls in `financial-rules.tsx` (Rule type / Status / Date range / Custom window) have associated `<label htmlFor>` plus `aria-label` on the trigger, so screen readers announce both the field name and the current value.
10. Modal close buttons and dialog dismissal use the existing Radix `<Dialog>` semantics (verified — Esc closes, focus traps, focus restores). No code changes; documented for the audit record only.
11. Color contrast: all visible text on the 10 surfaces uses the existing `text-on-surface` / `text-muted-foreground` / `text-primary` token system, which already meets WCAG AA against its paired backgrounds. No new contrast violations introduced.

## Mechanical Fixes Applied (audit-level)

All ten surfaces received the following mechanical pass during the Wave 21 implementation. Per-surface diffs are catalogued in `implementation-artifacts/5.5-a11y-audit-findings.md`.

- Add `aria-label` to icon-only or short-text action buttons (`Review`, `Vote now`, `Set default`, `Remove`, etc.) so each button announces _what_ it does and _to which item_.
- Add `aria-hidden="true"` to every Lucide and Material icon rendered alongside text content, so the icon does not double-announce.
- Add `focus-visible:outline focus-visible:outline-2 focus-visible:outline-offset-2 focus-visible:outline-ring` to bare `<a>` and `<Link>` elements that previously had no focus ring (Tailwind `outline-ring` resolves to a visible color in both light and dark mode).
- Wrap loading / empty / error states with `role="status"` or `role="alert"` so assistive tech announces state changes without polling.
- Add `<caption className="sr-only">` to data tables, mirroring the visible `<h2>` immediately above.
- Add `aria-labelledby` linking each `<section>` to its in-section heading where the section was previously unlabeled.

## Deferred / Follow-up Items

These were identified during the audit but exceed the Wave 21 scope or require a deeper refactor than ship-and-lock allows. Each one is tracked here so the next a11y wave can pick them up without re-auditing.

- **Operator sidebar focus order** — the sidebar (in `App.tsx`'s `WorkspaceShell`) does not declare an `aria-label` on its top-level `<nav>`. Trivial one-line fix; deferred because it lives outside the 10 surfaces.
- **`<select>` semantics on Tier 2 surfaces** — surfaces such as `financial-billing.tsx` and `financial-payments.tsx` use bare HTML buttons styled as dropdowns. Replacing them with proper Radix `<Select>` (the pattern already used inside `financial-rules.tsx`) is a per-surface refactor, not a mechanical fix.
- **Reduced-motion** — none of the surfaces have animations, but the default Tailwind transitions (`transition-colors`) are used unconditionally. A `motion-reduce:` audit is deferred.
- **Skip-link on landing / public surfaces** — `landing.tsx`, `community-hub-public.tsx`, and `not-found.tsx` are unauthenticated surfaces that do not pass through either shell. They have no skip-link today; deferred with low impact (single `<main>` per page, short content).
- **Manual screen-reader pass** — Wave 21 is markup-only. A two-hour NVDA + VoiceOver verification on the 10 surfaces is filed as a follow-up so a human can confirm the announced text matches the visual labels.
- **Programmatic axe-core run in CI** — adding `@axe-core/react` to the test suite would catch regressions. Deferred because adding it is a new dev-dep with non-trivial flake rate; smoke test in `tests/a11y-smoke.client.test.tsx` covers the structural minimum (skip-link + `<main>`) for now.

## Wave 25 — Automated a11y + visual regression

**Added:** 2026-04-25 (companion to 5.7 Automated Quality Gates)
**Status:** SHIPPED

### What is now automated

`@axe-core/playwright` is wired into the Playwright suite via the
shared helper `tests/e2e/playwright/helpers/a11y-check.ts`. The
helper exports two entry points:

- `runAxeAudit(page, label)` — hard-fails on any `critical` or
  `serious` violation.
- `runAxeAuditSoft(page, label)` — same audit, but uses
  `expect.soft` so a non-trivial violation does not cascade-fail the
  test. Used today across all integrations because the surfaces
  exercised by the five flow specs (alerts, assessments, amenities,
  portal nav) extend beyond the Wave-21 locked 10.

Both entry points run axe with WCAG 2.1 AA tag set
`["wcag2a", "wcag2aa", "wcag21a", "wcag21aa"]`. The `best-practice`
and `experimental` tag sets are deliberately excluded — they surface
high-noise warnings that are not part of the locked Wave-21
acceptance criteria.

### Threshold

Per surface: **zero `critical` or `serious` violations** against the
WCAG 2.1 AA tag set. Lower-severity findings (`moderate`, `minor`)
are written to the trace via `console.error` but do not change the
test status.

### Coverage

- The five existing Playwright flow specs (`signup-onboarding`,
  `alerts-lifecycle`, `assessment-lifecycle`,
  `owner-portal-navigation`, `amenities-toggle-roundtrip`) each call
  `runAxeAuditSoft` as the final step of their test body, in both
  route-mock and real-backend branches.
- A dedicated smoke spec (`tests/e2e/playwright/a11y-smoke.spec.ts`)
  walks five surfaces and runs axe on each: Home (`/app`), Inbox
  (`/app/communications/inbox`), Assessment Rules
  (`/app/financial/rules`), Portal Home (`/portal`), Portal Finances
  (`/portal/finances`).

### Visual regression scaffold

Companion deliverable in Wave 25 — Playwright's built-in
`toHaveScreenshot()` against four PNG baselines committed under
`tests/e2e/playwright/__screenshots__/`. Surfaces: Home, Inbox,
Assessment Rules, Portal Home. Threshold: `maxDiffPixels: 200`. See
`implementation-artifacts/visual-regression-runbook.md` for the
update flow.

### What is still deferred

The 5.5 "Deferred / Follow-up Items" list previously called out
"Programmatic axe-core run in CI" as deferred — that item is now
**resolved** in repo, but the **CI workflow file is still deferred**
(documented in 5.7). Once the user wires the dedicated CI job, the
"axe in CI" follow-up closes fully.

Manual screen-reader verification (NVDA + VoiceOver) and the
remaining ~50 surface retrofit remain deferred to a future wave —
unchanged from the original Wave-21 spec.
