## Conflict Analysis

**Conflict class A — landing-order coupling.** Several 2.x corrections
are mutually entangled:
- Retiring the BoardPortal shunt (2.3 Q3) requires the Board Officer / Assisted Board surface
  to already be reachable through the unified shell, which requires
  `canAccessTab` to accept `board-officer` / `assisted-board` without the `null` short-circuit
  (2.1 Q7).
- Retiring `normalizeAdminRole` (2.1 Q5) requires every call site
  (`server/routes.ts:796` and 544 handlers via `requireAdminRole`) to
  already pass canonical strings — which means the portal role enum
  collapse (2.1 Q3 / 2.2 Q2) must precede it.
- Platform API narrowing (2.4 Q3 / Q7) removes `board-admin` (old role) from
  `requireAdminRole([...])` arrays on platform-only endpoints; doing this
  before the Owner Portal launcher removal (2.4 Q5) and tab-bar narrowing
  (2.4 Q4) creates user-visible 403s with no UI explanation.

> Cross-reference: Order-of-operations here is derived from, not a re-litigation of, 2.1 Q3/Q5/Q7, 2.2 Q2, 2.3 Q2/Q3, and 2.4 Q3/Q4/Q5/Q7.

**Conflict class B — test-gate availability.** RouteGuard rollout
(2.3 Q2) needs a parity harness against `canAccessWipRoute`
(`client/src/lib/wip-features.ts`) before it can replace per-route
inline checks. The harness itself is a 3.3 deliverable.

**Conflict class C — rollback atomicity.** Server-side narrowing
(2.4 Q3/Q7) and client-side surface removal (2.4 Q4/Q5) must roll
back together or users see either orphaned UI with 403s, or hidden UI
with still-open endpoints. Q8 owns the rollback bundling rule.

## Selected Resolutions

### Q1 — Landing-sequence backbone

**Resolution:** The canonical landing sequence is seven ordered steps, grouped into five zone-aligned landings per Q5. Each step cites its 2.x derivation source.

| Step | Name | Derives from | Zone landing |
|---|---|---|---|
| 1 | Portal role enum collapse | 2.1 Q3 / 2.2 Q2 | Financials |
| 2 | `canAccessTab` null-role fix | 2.1 Q7 | Financials |
| 3 | Portal `PortalRequest` retyping + `requirePortalBoard`/`requirePortalBoardReadOnly` retirement | 2.1 Q8 / 2.2 Q2 | Financials |
| 4 | `AdminRole` canonical export + type deduplication | 2.1 Q4 | Operations |
| 5 | RouteGuard rollout (zone-by-zone per Q5) | 2.3 Q2/Q9 | Financials → Operations → Governance → Communications → Platform |
| 6 | BoardPortal shunt retirement | 2.3 Q3 | Operations (rides with RouteGuard Operations zone landing) |
| 7 | `normalizeAdminRole` migration + deletion | 2.1 Q5 | Governance |
| 8 | Platform API + UI narrowing (server narrowing + tab-bar narrowing + Owner Portal launcher removal) | 2.4 Q3/Q4/Q5/Q7 | Platform |

**Dependency rationale:**
- Step 1 (enum collapse) must precede Step 7 (`normalizeAdminRole` removal) because the normalizer coerces non-canonical portal-adjacent role strings; collapsing the enum first eliminates the data that feeds the normalizer.
- Step 2 (null-role fix) must precede Step 6 (shunt retirement) because Board Officer / Assisted Board cannot enter the shared shell if `canAccessTab` returns `true` on null — they would see a full nav tree before session resolves.
- Step 5 (RouteGuard rollout) must precede Step 6 (shunt retirement) for Operations zone — Board Officer / Assisted Board needs per-route guards in place before the shunt is removed.
- Step 8 (platform narrowing) is last because removing `board-admin` from platform API role arrays before the client-side surfaces are guarded produces orphan 403s.

### Q7 — `normalizeAdminRole` migration (2.1 Q5)

**Lint rule:** A custom ESLint rule or a Drizzle-level check constraint prevents re-introduction of alias role strings. The rule fires on any `role =` assignment in `server/` that is not one of the canonical six values.

This step rides with the Governance zone landing (Q5 Step 3) because Governance exercises the broadest range of role-dependent behavior (elections with `viewer` gates, board-member governance surfaces).

### Q10 — Rollback playbook

**Resolution:** Per-landing (per zone). Each zone ships as its own PR per Q5. Rollback procedure:

**Revert trigger (stop-the-line signal):**
- Any 401/403 error on a route that should be permitted for any persona (Q11 gate 1 fails post-deploy).
- Any unguarded access discovered on a route that should be blocked (Q11 gate 2 fails).
- Two or more JS console errors that were not present in the baseline (Q11 gate 3 fails).
- A Tier 1 parity-harness test that was passing pre-deploy starts failing in CI.

**Revert procedure:**
1. Revert the zone's PR via `git revert` on the merge commit.
2. Deploy the revert.
3. Verify the revert restored pre-landing behavior (re-run the manual smoke for that zone).
4. File a post-mortem issue documenting what broke and why.
5. Fix forward: address the root cause, re-run all gates, re-land as a new PR.

**Scope:** Per zone only. If a zone landing breaks, revert that PR only. No wholesale rollback unless two or more consecutive zones fail — in which case, pause all zone landings, conduct a cross-zone root-cause analysis, and re-sequence.

**Data-side implications:**
- Portal role enum collapse (Q2 Step 1 migration) is not revertible by PR revert alone — if the migration must be rolled back, a reverse migration script restores the original role values from the migration log. This is why Q2 ships behind a feature flag.
- `normalizeAdminRole` deletion (Q7) is not revertible by PR revert alone — if needed, the normalizer function can be re-added from git history. The data migration (Q7 Step 2) is idempotent and does not need rollback.
- Feature flags (`PORTAL_ROLE_COLLAPSE`, `BOARD_SHUNT_ACTIVE`) provide independent rollback levers for Q2 and Q6 within a zone landing.

**Acceptance Criteria:**
- AC-Q10-1: Each zone landing is a single PR that can be reverted independently.
- AC-Q10-2: Revert trigger criteria are documented in the PR template.
- AC-Q10-3: Portal role migration (Q2) includes a reverse migration script.
- AC-Q10-4: Two or more consecutive zone failures trigger a full pause per the escalation rule.

### Q11 — Observability counters

**Resolution:** Three gates per zone landing, read manually post-deploy (no automated dashboard at this stage):

1. **Zero 401/403 on permitted routes.** After deploy, log in as each persona that should access the zone. Navigate to every route in the zone. Check browser Network tab for 401/403 responses. Zero tolerance — any 401/403 on a permitted route is a revert trigger.
2. **Zero unguarded access on blocked routes.** After deploy, log in as each persona that should NOT access the zone. Attempt to navigate to every route in the zone via direct URL. Confirm redirect or `<NotFound>` renders. For server-side: issue direct API calls to zone endpoints with the non-permitted persona's session — confirm 403. Zero tolerance.
3. **No increase in JS console errors.** Before deploy, record the JS console error count on a representative set of routes (one per zone) with each persona. After deploy, repeat the same traversal. Error count must not increase. Threshold: zero net-new console errors.

**Metrics read procedure:** Manual post-deploy checklist performed by the developer landing the zone. Results recorded in the zone landing PR as a post-merge comment. Ties to 2.4 Q3 platform-API audit logging — once the platform audit log exists, gate 2 (unguarded access) can be partially automated by querying the audit log for unexpected role/route combinations.

**Acceptance Criteria:**
- AC-Q11-1: Post-deploy checklist template includes all three gates.
- AC-Q11-2: Each zone landing PR has a post-merge comment with gate results.
- AC-Q11-3: Any gate failure triggers the Q10 revert procedure.

## How This Informs Downstream Layers

- **2.1 Q9 (544-handler classification table)** — 3.3 consumes the
  classification and produces the per-handler landing assignment (which zone PR each handler's gate change ships in).
- **2.3 RouteGuard pattern** — 3.3 Q4/Q5 operationalize the rollout: Q4 defines the parity harness, Q5 defines the zone-by-zone batching, Q6 coordinates the shunt retirement timing.
- **2.4 platform-only narrowing** — 3.3 Q8 bundles the atomic
  server+client landing as the Platform zone's single PR.
- **Layer 4 implementation modules** — inherit the ordered landing
  checklist from 3.3 Q1, the per-landing test gates from 3.3 Q9, and the handoff contract from 3.3 Q12. Layer 4 work on a zone cannot begin until that zone's handoff doc confirms "Ready for Computer."
- **Release notes / changelog** — 3.3 Q12 fixes the labelling
  convention (zone landing name + PR number) that downstream release tooling will key off.
