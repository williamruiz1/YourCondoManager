## Context

Layer 2 locked the *what* of role-gating:
- **0.2** fixed the four-persona boundary matrix (amended `f8dbf76`).
- **0.3 Q5** mandated strict persona isolation — no cross-persona surface leakage.
- **2.1** ratified the canonical `AdminRole` vocabulary and retired
  `normalizeAdminRole` as the permanent coercion layer.
- **2.2** drew the Owner Portal access boundary.
- **2.3** selected the `RouteGuard` pattern and retired the `BoardPortal` shunt.
- **2.4** narrowed the platform-admin surface (API, launcher, tab-bar).

Layer 3 module 3.3 owns the *how and when*: the order-of-operations that takes
the repo from its current mixed state (BoardPortal shunt still mounted at
`client/src/App.tsx:1106`; `normalizeAdminRole` still active at
`server/routes.ts:778`; `canAccessTab` still branches on a `null` role at
`client/src/App.tsx:649`; `requireAdminRole` still admits `board-admin` for
platform-only endpoints e.g. `server/routes.ts:1223`) to the 2.x-compliant
target, without breaking any of the 544 handlers catalogued in 2.1 Q9.

## Conflict Analysis

PENDING per Q. The skeleton flags the conflict *classes* that each Q must
resolve; concrete conflicts are enumerated at resolution time.

**Conflict class A — landing-order coupling.** Several 2.x corrections
are mutually entangled:
- Retiring the BoardPortal shunt (2.3 Q3) requires the board-admin surface
  to already be reachable through the unified shell, which requires
  `canAccessTab` to accept `board-admin` without the `null` short-circuit
  (2.1 Q7).
- Retiring `normalizeAdminRole` (2.1 Q5) requires every call site
  (`server/routes.ts:796` and 544 handlers via `requireAdminRole`) to
  already pass canonical strings — which means the portal role enum
  collapse (2.1 Q3 / 2.2 Q2) must precede it.
- Platform API narrowing (2.4 Q3 / Q7) removes `board-admin` from
  `requireAdminRole([...])` arrays on platform-only endpoints; doing this
  before the Owner Portal launcher removal (2.4 Q5) and tab-bar narrowing
  (2.4 Q4) creates user-visible 403s with no UI explanation.

> ⚠️ **CROSS-REFERENCE NOTICE** — Order-of-operations here is derived
> from, not a re-litigation of, 2.1 Q3/Q5/Q7, 2.2 Q2, 2.3 Q2/Q3, and
> 2.4 Q3/Q4/Q5/Q7. Any Q below that appears to change a *target gate*
> (rather than the *sequence* of landing it) is out of scope and must be
> kicked back to the owning Layer 2 module.

**Conflict class B — test-gate availability.** RouteGuard rollout
(2.3 Q2) needs a parity harness against `canAccessWipRoute`
(`client/src/lib/wip-features.ts`) before it can replace per-route
inline checks. The harness itself is a 3.3 deliverable.

**Conflict class C — rollback atomicity.** Server-side narrowing
(2.4 Q3/Q7) and client-side surface removal (2.4 Q4/Q5) must roll
back together or users see either orphaned UI with 403s, or hidden UI
with still-open endpoints. 3.3 Q8 owns the rollback bundling rule.

## Open Questions

- **Q1 — Landing-sequence backbone.** What is the canonical ordered list
  of landings (portal-enum-collapse → `canAccess` null-fix →
  RouteGuard-harness → RouteGuard-rollout → BoardPortal-shunt-retirement →
  `normalizeAdminRole`-migration → platform-API-narrowing →
  tab-bar-narrowing → Owner-Portal-launcher-removal, OR a re-ordering)?
  Each step must cite the 2.x question it derives from.

- **Q2 — Portal role enum collapse (2.1 Q3 / 2.2 Q2) landing.**
  What is the atomic unit — schema migration, server coercion removal,
  client type narrowing — and which ships first behind a flag?

- **Q3 — `canAccessTab` null-role fix (2.1 Q7) landing.** Does the
  `null` branch at `client/src/App.tsx:649` become a redirect-to-login,
  a hard error, or a typed-never? Gate: which test proves no live call
  site passes `null` post-landing.

- **Q4 — RouteGuard parity harness (precondition to 2.3 Q2).** What
  fixture covers `canAccessWipRoute` ↔ `RouteGuard` equivalence, and
  what is the coverage bar (every `/app/*` route, every persona × zone
  cell from 0.2) before rollout may begin?

- **Q5 — RouteGuard rollout (2.3 Q2) batching.** Route-by-route, zone-by-zone,
  or big-bang? How are the six 1.1 zones batched, and what is the
  per-batch rollback window?

- **Q6 — BoardPortal shunt retirement (2.3 Q3) cut-over.** Does the shunt
  at `client/src/App.tsx:1106` get removed in-place, or does it go
  dark-launched (feature-flagged off) for one release first? What
  session-auth migration steps ride with it?

- **Q7 — `normalizeAdminRole` migration (2.1 Q5) execution.** Call-site
  conversion order — `server/routes.ts:796` first, then 544 handlers via
  `requireAdminRole`, then delete `normalizeAdminRole` at
  `server/routes.ts:778`? What is the lint rule that prevents
  re-introduction?

- **Q8 — Platform-API narrowing + UI narrowing atomicity (2.4 Q3/Q4/Q5/Q7).**
  Which landings must bundle into a single PR to preserve rollback
  atomicity (Conflict class C)? Default proposal: server narrowing +
  launcher removal + tab-bar narrowing ship as one.

- **Q9 — Test gates per landing.** For each of Q2–Q8, what is the
  minimum test set (unit / integration / e2e / persona-matrix smoke)
  that must pass before the landing is accepted, and which fixture
  owns each gate?

- **Q10 — Rollback playbook.** Per-landing revert procedure, including
  data-side implications (session tokens, cached role claims, feature
  flags). What is the "stop-the-line" signal that triggers rollback vs.
  forward-fix?

- **Q11 — Observability hooks.** What telemetry counters / log fields
  must exist before rollout so that a regression in role-gating is
  detectable within one release cycle? Ties to 2.4 Q3 platform-API
  audit logging.

- **Q12 — Downstream handoff contract.** What does 3.3 hand to the
  4.x implementation modules? Landing checklist format, PR labelling
