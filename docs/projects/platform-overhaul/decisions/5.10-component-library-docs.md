# 5.10 — Storybook for the Shared Component Library

**Status:** SPEC LOCKED 2026-04-26 (Wave 37, ship-and-lock pattern)
**Owner:** Platform Overhaul
**Layer:** 5 — Polish & Hardening

---

## Scope

Stand up Storybook 10 against the existing client/src/components/
shared component library so designers and new engineers can browse
and interact with the YCM-built primitives in isolation. Stories
cover the loading, empty, error, and filled states each component
actually renders in production.

Storybook is install-only at this stage: no Chromatic, no published
preview, no per-PR snapshot diff. The static build runs locally
(`npm run storybook:build`) so the artifact can be picked up later
by a hosting wave (out of 5.10 scope) without further config work.

**In scope:**

- Storybook 10.3 + `@storybook/react-vite` framework + `@storybook/addon-a11y` addon, all as devDependencies.
- `.storybook/main.ts` + `.storybook/preview.ts` config; preview imports the global Tailwind stylesheet so stories render with production tokens.
- Vite path-alias mirroring (`@/*` → `client/src/*`, `@shared/*` → `shared/*`) so stories import exactly like app code.
- 10 story files spanning the 8 components specified in the wave brief plus the two charts.
- Per-component fixture files under `client/src/components/__fixtures__/` so the stories never reuse production payloads.
- `npm run storybook` and `npm run storybook:build` package scripts.
- `storybook-static/` added to `.gitignore`.
- Storybook 10 a11y addon registered with `test: "todo"` so axe findings surface in the panel without failing builds (5.5 owns remediation; this wave only makes the findings visible).
- Smoke test at `tests/storybook.client.test.tsx` that composes the EmptyState story file via `composeStories()` and renders each variant under jsdom.

**Out of scope (deferred):**

- Hosting Storybook (deploy preview, Chromatic, GitHub Pages).
- Story coverage for every shared component (started with the highest-leverage 10).
- Visual regression on stories (5.7 owns visual regression at the Playwright layer; adding a Storybook visual test is a future wave).
- Refactoring components to be "more Storybook friendly" (constraint § the wave brief).
- Generating MDX docs pages — only `.stories.tsx` files are added.

---

## Acceptance Criteria

1. `npm run storybook` boots the local Storybook dev server on port 6006.
2. `npm run storybook:build` produces a `storybook-static/` directory containing a fully self-contained static site.
3. Stories ship for at least 8 YCM-built shared components, each covering the meaningful state matrix (loading + empty + error + filled where the component supports those states; the appropriate subset where it does not).
4. Storybook config imports the Tailwind global stylesheet so component styling matches production.
5. `@storybook/addon-a11y` is registered and surfaces critical axe violations in the panel.
6. `tsc`, `eslint`, and the existing 712-test vitest suite continue to pass; the new storybook smoke test (3+ assertions) joins the suite.
7. The production app build (`npm run build`) is unaffected by the Storybook install.

## Implementation Notes

### Version pin

Storybook 10.3.5 — the first major Storybook line that supports Vite 7
peer-dep. Storybook 8 caps at Vite 5 and Storybook 9 caps at Vite 6, so
the wave brief's "Storybook 8+" target is satisfied at the closest stable
version compatible with our existing Vite 7 install. Storybook 10
folded the previous `addon-essentials` and `addon-interactions`
packages into the core, so the addon list intentionally only carries
`addon-a11y`.

|---|---|---|

The wave brief originally suggested `home-trial-banner.stories.tsx`,
but the file lives under `client/src/components/trial-banner.tsx` so
the story file follows the matching name. `<PortalShell>` was on the
suggested list but does not exist as a discrete shared file — the
portal layout is composed inline in `client/src/pages/portal/*` and
was skipped per the brief's "pick the closest 8–10" guidance.

### Decorator strategy

`<HubAlertWidget>`, `<HomeAlertsPanel>`, and `<SignupOnboardingChecklist>`
each call TanStack Query hooks internally. Per the wave brief constraint
"don't refactor components to make them more Storybook-friendly",
the stories pre-seed an isolated `QueryClient` cache with fixture
payloads keyed exactly the way the production hooks build their keys.
The component code is untouched. Decorators live inside the story
file — there is no Storybook-only context provider added to the app
root.

### Fixtures

Fixtures live under `client/src/components/__fixtures__/` so they are
co-located with the components they feed and can be ignored by
production bundlers via the existing `**/__fixtures__/**` patterns
(none currently in use, but the convention is established here for
future waves to lean on). Three fixture files cover the alert
surfaces, the chart surfaces, and the onboarding checklist surface.

### Smoke test

The wave brief asks for "smoke only — assert the story doesn't throw."
The smoke test imports the EmptyState story file and runs every named
export through React Testing Library inside jsdom. It is one
representative file, not the full inventory — a full per-file test
would duplicate the work of the dev server pass and add brittleness
without proportional value. If the Storybook setup ever drifts (broken
preview import, missing alias, framework upgrade), the smoke test
will surface the regression on the next CI run.
