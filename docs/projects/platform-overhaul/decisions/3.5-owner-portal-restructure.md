## Context

The Owner Portal is currently implemented as a single 3,935-line React file (`client/src/pages/owner-portal.tsx`) with 7 tab-switched views driven by local `activeTab` state. There are only two registered React routes for the entire portal surface: `/portal` and `/portal/amenities` (`client/src/App.tsx:356-357`). Browser back/forward does not work across the tab views. Deep-linking to a specific zone is impossible. No real route boundary exists between Financials, Maintenance, Documents, Notices, Communications, Voting, or Overview ‚Äî they are all conditional render branches in one mega-component.

Three Layer 1 locks presuppose a portal with real route boundaries:
- **1.1 Q5** locked four first-person zones: Home, My Finances, My Requests, My Community (`1.1-zone-taxonomy-corrections.md:112-117`).
- **1.2 Q4** locked three portal hub URLs: `/portal/finances`, `/portal/requests`, `/portal/community`, plus `/portal` as the owner dashboard (`1.2-section-hub-reclassification.md:87-90`).
- **1.4 Q7** locked portal title parity with `/app/*`: `useDocumentTitle` hook, `{Page Title} ‚Äî YCM` format, one `<h1>` per page, first-person labels per 1.1 (`1.4-page-title-consistency.md:105-113`).

None of these locks has a delivery vehicle in the repo today. 3.2 Q5 deferred the portal route restructure from the `/app` route pass to separate work (`3.2-route-restructure.md:205-207`). 4.2 Q1 closed the gap by creating this module. 4.2 Q4 further locked HYBRID Community integration ‚Äî `/portal/community` renders an inline preview (unread notices, upcoming events, recent posts) with click-through to the public `/community/:associationId` route for deep engagement. 4.2 Q5 locked that the 2.2 Q6 session-gate fix and the 1.4 Q7 title sprinkle ship as in-place patches to the mega-file; they will be superseded cleanly when 3.5 decomposes the portal.

3.5 runs in parallel with the `/app` Layer 3 work (3.1 sidebar redesign, 3.2 route restructure, 3.3 role-gating corrections, 3.4 breadcrumb implementation). It does not block `/app` Layer 3 and is not blocked by it. 3.5's start gate is functionally downstream of Phase 3 (1.4 hook shipping) and Phase 11 (hub scaffolds + sidebar shipping), because those phases establish the per-route title-hook pattern and the hub-page scaffolds that 3.5 mirrors for the portal surface.

## Current State

**Routes.** Two React routes under `/portal/*` (`client/src/App.tsx:356-357`):
- `/portal` ‚Üí `owner-portal.tsx` (renders all 7 tab views)
- `/portal/amenities` ‚Üí separate public route, currently with no session gate (2.2 Q6 fix is queued)

**Tab views.** Seven tab-switched views inside `owner-portal.tsx`, driven by `activeTab` local state (`client/src/pages/owner-portal.tsx:259`): Overview, Financials, Documents, Maintenance, Notices, Communications, Voting. Community Hub renders as an external link in the sidebar (not a tab).

**Navigation.** The portal sidebar currently renders 7 tab triggers (Overview, Financials, Documents, Maintenance, Notices, Communications, Voting) + one external link (Community Hub). None uses first-person labels; none matches the 1.1 Q5 four-zone vocabulary.

**File size.** `client/src/pages/owner-portal.tsx` is 3,935 LOC ‚Äî well beyond the threshold at which a single-file component is maintainable, testable, or guard-wrappable.

**Consequences.**
- No browser back/forward across portal zones.
- No deep-linking to a specific portal view.
- No per-zone route guard ‚Äî the whole portal is one gate.
- No per-route title ‚Äî 1.4 Q7 Phase 3 sprinkles `useDocumentTitle` at the 7 tab-switch render branches as a temporary compromise.
- No per-route parity test ‚Äî the portal parity harness described in 4.2 Q1 AC 3e cannot run against tab-switched views.

## Conflict Analysis

Eight decision axes surface. Each is a PENDING Q; none is pre-resolved in this skeleton.

**Axis 1 ‚Äî Route tree naming.** 1.2 Q4 already locked the portal URL shape (`/portal/finances`, `/portal/requests`, `/portal/community`). Question remaining: do we mirror `/app/*` URL patterns further (e.g., `/portal/finances/overview` sub-pages) or keep the portal flat (hub URL = zone landing)? This is about depth, not shape.

**Axis 2 ‚Äî Component split granularity.** One file per zone (4 files + shell) vs. finer (per-feature sub-components). Finer split reduces file size further but increases cross-file jumps; coarser split keeps zones coherent but leaves some files still 500‚Äì1000 LOC.

**Axis 3 ‚Äî Shared layout component.** Does the portal get its own `PortalShell` equivalent to `/app`'s `WorkspaceShell` (shared header, nav, session gate wrapper)? Alternative: a thinner `PortalLayout` that wraps only the nav and lets routes own their own shell chrome.

**Axis 4 ‚Äî Session gate placement.** `requirePortalSession` middleware currently applies server-side. On the client, `/portal` has session-redirect logic; `/portal/amenities` does not (2.2 Q6 fix queued). After restructure, does the shell wrapper own the session gate for all `/portal/*` routes, or does each route declare its own gate?

**Axis 5 ‚Äî Amenities inline vs. linked (coordinates with 4.2 Q3 PENDING).** 4.2 Q3 is still PENDING ‚Äî whether the amenity booking experience is upgraded from external-link to inline booking within My Community. 3.5 must not pre-decide Q3; but 3.5's PortalAmenities component shape depends on it. If Q3 resolves before 3.5 builds, 3.5 reflects the resolution; if not, 3.5 builds a placeholder that supports either outcome.

**Axis 6 ‚Äî Community Hub inline preview (coordinates with 4.2 Q4 LOCKED).** 4.2 Q4 locked HYBRID: `/portal/community` renders an inline preview (unread notices, upcoming events, recent posts) with click-through to `/community/:associationId` for deep engagement (`4.2-owner-portal-gaps.md:184-191`). Question for 3.5: which component renders the preview? A new `PortalCommunityPreview` inside `PortalCommunity.tsx`, or reuse of existing Community Hub widgets? The AC from 4.2 (AC-5 through AC-8) sets the behavior; 3.5 picks the implementation.

**Axis 7 ‚Äî Migration strategy.** Big-bang rewrite (one PR that decomposes the whole mega-file) vs. incremental per-zone migration (each zone ships in its own PR, mega-file shrinks over time). Big-bang is cleaner cutover but harder to revert; incremental preserves the mega-file during migration and risks merge conflicts on the shrinking file.

**Axis 8 ‚Äî Deep-link preservation.** If any existing links (email notices, help-desk URLs, shared links) point to the current `/portal` tab-deep-links (e.g., `/portal?tab=financials` or query-string tab indicators), do those redirect to the new URLs, or break on cutover? Redirect requires a legacy-URL compat layer; no-redirect is simpler but user-visible.

## Selected Resolution

Session 3.5 (2026-04-24) locks Q1‚ÄìQ8 as follows. Q5 and Q6 are PRE-RESOLVED by Session B (4.2 Q3/Q4 locks of 2026-04-24) and 3.5 reflects their outcomes rather than re-adjudicating.

### Q1 ‚Äî Route tree depth: sub-page mirroring of `/app/*` (Option B)

**Resolution.** The portal mirrors the sub-page structure of `/app/*`. Each zone hub (`/portal/finances`, `/portal/requests`, `/portal/community`) gets real sub-page routes, not flat render-everything hubs. Owners bookmark meaningful URLs; the deep-link compat layer in Q8 is easier to author against a real route tree; 1.4 Q7 per-route title parity works naturally when each view is a route.

**Examples of sub-page routes introduced:**
- `/portal/finances/payment-methods`
- `/portal/finances/ledger`
- `/portal/finances/assessments` (4.3 Q5 per-assessment drill-in surfaces as a page, not just a dialog)
- `/portal/requests/:requestId` (owner-side detail view)

### Q2 ‚Äî Component split granularity: one file per zone (Option A)

**Resolution.** One file per zone. ~8 files total under `client/src/pages/portal/`:
- `portal-shell.tsx`
- `portal-home.tsx`
- `portal-finances.tsx`
- `portal-requests.tsx`
- `portal-community.tsx`
- `portal-amenities.tsx`
- `portal-documents.tsx`
- `portal-notices.tsx`

Sub-components emerge organically as files exceed maintainable size; no premature fragmentation into 15+ per-feature files. Coherence per zone beats micro-split in the first landing.

### Q3 ‚Äî Shared layout: `PortalShell` mirror of `WorkspaceShell` (Option A)

**Resolution.** Portal gets a `PortalShell` that mirrors `WorkspaceShell`. Centralizes session-gate placement (Q4), nav rendering, breadcrumb wiring (1.3), and document-title scaffolding (1.4 Q7). Matches operator-side pattern and keeps portal + `/app` shells conceptually symmetric for future maintainers.

### Q4 ‚Äî Session gate: shell-owned (Option A)

**Resolution.** `PortalShell` owns the client session gate. All `/portal/*` routes inherit it via the shell wrapper; no route declares its own gate. Session B's 2.2 Q2 persona collapse (portal access scope is persona-uniform ‚Äî `owner` only) means there is no per-route persona branching the gate would need to express. `/portal/amenities` inherits the gate uniformly, superseding the in-place Phase 8b patch. 3.3's per-route `<RouteGuard>` pattern on `/app/*` is appropriate there because `/app` spans six personas; the portal does not need it.

### Q5 ‚Äî Amenities inline vs. linked: **PRE-RESOLVED BY SESSION B 4.2 Q3**

**Resolution.** Session B locked 4.2 Q3 on 2026-04-24 with the session-gate-only status quo plus the 3a amenities-toggle addendum (`associations.amenitiesEnabled` boolean, Manager + self-managed Board Officer togglable, defaults on, hides sidebar tab + 404s `/portal/amenities` when off). That lock preserves `/portal/amenities` as its own route; no inline integration into `/portal/community`. 3.5 reflects by keeping `portal-amenities.tsx` as an independent zone file and a standalone route under the PortalShell. Q5's originally enumerated tradeoff (inline vs. external) is settled upstream; 3.5 does not re-open it.

### Q6 ‚Äî Community inline preview component: **PRE-RESOLVED BY SESSION B 4.2 Q4**

**Resolution.** Session B locked 4.2 Q4 on 2026-04-24 as a zone-wrapper pattern: `/portal/community` renders a link-out card to `/community/:associationId`, not an inline preview of unread notices / events / posts. The original Q6 tradeoff (new `PortalCommunityPreview` component vs. reuse of Community Hub widgets) moots because no preview component exists ‚Äî `portal-community.tsx` is a thin link-out card referencing the public Community Hub. 3.5 reflects accordingly; no preview widget, no shared-render-lifecycle concern.

### Q7 ‚Äî Migration strategy: big-bang (Option A)

**Resolution.** Big-bang cutover. One large PR lands `PortalShell` + all 8 zone files + all new routes + the legacy-URL compat layer as a single atomic change, preceded by one small setup PR if scaffolding helpers (PortalShell skeleton, route-table wiring) need to land first for diff hygiene. The mega-file (`owner-portal.tsx`) is deleted in the cutover PR, not shrunk incrementally. Rationale: 3,935-LOC mega-file is a moving target under incremental migration (merge conflicts on every parallel PR touching portal), per-zone incremental PRs each have to straddle old-tabs-state-machine + new-route-boundary rendering, and reviewers struggle to hold half-migrated state in their head. A single honest diff is easier to QA, easier to revert, and easier to reason about.

### Q8 ‚Äî Deep-link preservation: legacy-URL compat layer (Option A)

**Resolution.** Ship a legacy-URL compat layer. Old query-string tab shapes (e.g., `/portal?tab=financials`, `/portal?tab=maintenance`) redirect with 301 to the new hub URLs (`/portal/finances`, `/portal/requests`, etc.). Constraint: live owner emails and notice templates in production reference the old shapes; a hard cutover would user-visibly break those links. Compat layer is a thin router-level redirect table in `PortalShell` (or equivalent layer); it is NOT a long-term surface. It is scheduled for removal in 5.1 (polish layer) once analytics confirm zero hits on old shapes for one full notice cycle. Removal of the compat layer is an AC of 5.1, not of 3.5.

**Route surface:**

6. `/portal/community` renders the zone-wrapper link-out card per 4.2 Q4 Session B lock; no inline preview.

**Component structure:**
7. 8 zone files exist under `client/src/pages/portal/`: `portal-shell.tsx`, `portal-home.tsx`, `portal-finances.tsx`, `portal-requests.tsx`, `portal-community.tsx`, `portal-amenities.tsx`, `portal-documents.tsx`, `portal-notices.tsx`.

9. `portal-shell.tsx` owns the portal nav (4 zones per 1.1 Q5: Home, My Finances, My Requests, My Community) and the breadcrumb wiring per 1.3.
10. Each `portal-*.tsx` file calls `useDocumentTitle` once with a first-person title per 1.4 Q7; the interim sprinkle inside `owner-portal.tsx` is deleted with the mega-file (4.2 Q1 AC 3d).

**Migration:**
11. Migration ships as a single large PR (preceded by at most one small setup PR for `PortalShell` scaffolding / route-table wiring). No incremental per-zone PRs that leave `owner-portal.tsx` in a half-migrated state.
12. Post-migration, `client/src/pages/owner-portal.tsx` (the 3,935-LOC mega-file) is deleted. Its removal is part of the cutover PR, not deferred to a follow-up.
13. Sidebar label migration to 1.1 Q5 four-zone vocabulary lands in the cutover (4.2 Q1 AC 3c).

**Legacy URL compat:**
14. Legacy URL compat layer handles `/portal?tab=<legacy>` ‚Üí 301 redirect to the new hub URL for every `tab=` value shipped in production emails or notice templates (`financials` ‚Üí `/portal/finances`, `maintenance` ‚Üí `/portal/requests`, `notices` ‚Üí `/portal/notices`, `documents` ‚Üí `/portal/documents`, `communications` ‚Üí TBD during grep sweep, `voting` ‚Üí TBD during grep sweep, `overview` ‚Üí `/portal`).
15. Legacy URL compat layer is scheduled for removal in 5.1 (polish) once analytics show zero hits on old shapes for one full notice cycle; 5.1 inherits removal as an AC, 3.5 does not.

**Parity + verification:**
16. Portal parity verification harness per 3.3 Q4 pattern is runnable against the new route-per-zone surface (4.2 Q1 AC 3e anchors this; the harness itself may land as a follow-up task).
