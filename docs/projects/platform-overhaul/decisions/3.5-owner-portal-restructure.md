## Context

The Owner Portal is currently implemented as a single 3,935-line React file (`client/src/pages/owner-portal.tsx`) with 7 tab-switched views driven by local `activeTab` state. There are only two registered React routes for the entire portal surface: `/portal` and `/portal/amenities` (`client/src/App.tsx:356-357`). Browser back/forward does not work across the tab views. Deep-linking to a specific zone is impossible. No real route boundary exists between Financials, Maintenance, Documents, Notices, Communications, Voting, or Overview — they are all conditional render branches in one mega-component.

Three Layer 1 locks presuppose a portal with real route boundaries:
- **1.1 Q5** locked four first-person zones: Home, My Finances, My Requests, My Community (`1.1-zone-taxonomy-corrections.md:112-117`).
- **1.2 Q4** locked three portal hub URLs: `/portal/finances`, `/portal/requests`, `/portal/community`, plus `/portal` as the owner dashboard (`1.2-section-hub-reclassification.md:87-90`).
- **1.4 Q7** locked portal title parity with `/app/*`: `useDocumentTitle` hook, `{Page Title} — YCM` format, one `<h1>` per page, first-person labels per 1.1 (`1.4-page-title-consistency.md:105-113`).

None of these locks has a delivery vehicle in the repo today. 3.2 Q5 deferred the portal route restructure from the `/app` route pass to separate work (`3.2-route-restructure.md:205-207`). 4.2 Q1 closed the gap by creating this module. 4.2 Q4 further locked HYBRID Community integration — `/portal/community` renders an inline preview (unread notices, upcoming events, recent posts) with click-through to the public `/community/:associationId` route for deep engagement. 4.2 Q5 locked that the 2.2 Q6 session-gate fix and the 1.4 Q7 title sprinkle ship as in-place patches to the mega-file; they will be superseded cleanly when 3.5 decomposes the portal.

3.5 runs in parallel with the `/app` Layer 3 work (3.1 sidebar redesign, 3.2 route restructure, 3.3 role-gating corrections, 3.4 breadcrumb implementation). It does not block `/app` Layer 3 and is not blocked by it. 3.5's start gate is functionally downstream of Phase 3 (1.4 hook shipping) and Phase 11 (hub scaffolds + sidebar shipping), because those phases establish the per-route title-hook pattern and the hub-page scaffolds that 3.5 mirrors for the portal surface.

## Current State

**Routes.** Two React routes under `/portal/*` (`client/src/App.tsx:356-357`):
- `/portal` → `owner-portal.tsx` (renders all 7 tab views)
- `/portal/amenities` → separate public route, currently with no session gate (2.2 Q6 fix is queued)

**Tab views.** Seven tab-switched views inside `owner-portal.tsx`, driven by `activeTab` local state (`client/src/pages/owner-portal.tsx:259`): Overview, Financials, Documents, Maintenance, Notices, Communications, Voting. Community Hub renders as an external link in the sidebar (not a tab).

**Navigation.** The portal sidebar currently renders 7 tab triggers (Overview, Financials, Documents, Maintenance, Notices, Communications, Voting) + one external link (Community Hub). None uses first-person labels; none matches the 1.1 Q5 four-zone vocabulary.

**File size.** `client/src/pages/owner-portal.tsx` is 3,935 LOC — well beyond the threshold at which a single-file component is maintainable, testable, or guard-wrappable.

**Consequences.**
- No browser back/forward across portal zones.
- No deep-linking to a specific portal view.
- No per-zone route guard — the whole portal is one gate.
- No per-route title — 1.4 Q7 Phase 3 sprinkles `useDocumentTitle` at the 7 tab-switch render branches as a temporary compromise.
- No per-route parity test — the portal parity harness described in 4.2 Q1 AC 3e cannot run against tab-switched views.

## Conflict Analysis

Eight decision axes surface. Each is a PENDING Q; none is pre-resolved in this skeleton.

**Axis 1 — Route tree naming.** 1.2 Q4 already locked the portal URL shape (`/portal/finances`, `/portal/requests`, `/portal/community`). Question remaining: do we mirror `/app/*` URL patterns further (e.g., `/portal/finances/overview` sub-pages) or keep the portal flat (hub URL = zone landing)? This is about depth, not shape.

**Axis 2 — Component split granularity.** One file per zone (4 files + shell) vs. finer (per-feature sub-components). Finer split reduces file size further but increases cross-file jumps; coarser split keeps zones coherent but leaves some files still 500–1000 LOC.

**Axis 3 — Shared layout component.** Does the portal get its own `PortalShell` equivalent to `/app`'s `WorkspaceShell` (shared header, nav, session gate wrapper)? Alternative: a thinner `PortalLayout` that wraps only the nav and lets routes own their own shell chrome.

**Axis 4 — Session gate placement.** `requirePortalSession` middleware currently applies server-side. On the client, `/portal` has session-redirect logic; `/portal/amenities` does not (2.2 Q6 fix queued). After restructure, does the shell wrapper own the session gate for all `/portal/*` routes, or does each route declare its own gate?

**Axis 5 — Amenities inline vs. linked (coordinates with 4.2 Q3 PENDING).** 4.2 Q3 is still PENDING — whether the amenity booking experience is upgraded from external-link to inline booking within My Community. 3.5 must not pre-decide Q3; but 3.5's PortalAmenities component shape depends on it. If Q3 resolves before 3.5 builds, 3.5 reflects the resolution; if not, 3.5 builds a placeholder that supports either outcome.

**Axis 6 — Community Hub inline preview (coordinates with 4.2 Q4 LOCKED).** 4.2 Q4 locked HYBRID: `/portal/community` renders an inline preview (unread notices, upcoming events, recent posts) with click-through to `/community/:associationId` for deep engagement (`4.2-owner-portal-gaps.md:184-191`). Question for 3.5: which component renders the preview? A new `PortalCommunityPreview` inside `PortalCommunity.tsx`, or reuse of existing Community Hub widgets? The AC from 4.2 (AC-5 through AC-8) sets the behavior; 3.5 picks the implementation.

**Axis 7 — Migration strategy.** Big-bang rewrite (one PR that decomposes the whole mega-file) vs. incremental per-zone migration (each zone ships in its own PR, mega-file shrinks over time). Big-bang is cleaner cutover but harder to revert; incremental preserves the mega-file during migration and risks merge conflicts on the shrinking file.

**Axis 8 — Deep-link preservation.** If any existing links (email notices, help-desk URLs, shared links) point to the current `/portal` tab-deep-links (e.g., `/portal?tab=financials` or query-string tab indicators), do those redirect to the new URLs, or break on cutover? Redirect requires a legacy-URL compat layer; no-redirect is simpler but user-visible.

## Open Questions

### Q1 — Route tree depth: flat hubs vs. sub-page mirroring of `/app/*`?

**Tradeoff axis:** 1.2 Q4 already locks the three hub URLs. The question is whether the portal also gets sub-pages (e.g., `/portal/finances/payment-methods`, `/portal/requests/:requestId`) or whether each zone lands as a single-route hub that renders all zone content inline. The `/app/*` side uses sub-pages heavily; the portal has a narrower capability set per 0.2 Persona 5 and may not need the depth. PENDING.

### Q2 — Component split granularity: one file per zone vs. finer?

**Tradeoff axis:** A one-file-per-zone split yields 4 zone files + a shell (+ Amenities + Documents + Notices + Voting as separate pages = ~8 files). Finer split (per-feature sub-components) could push to 15+ files. Finer is more testable; coarser is more coherent. PENDING.

### Q3 — Shared layout: `PortalShell` mirror of `WorkspaceShell`, or thinner `PortalLayout`?

**Tradeoff axis:** Mirroring `WorkspaceShell` gives portal consistency with `/app` and centralizes session-gate ownership. A thinner layout is less code but duplicates concerns (session gate, nav, breadcrumb) across zone components. PENDING.

### Q4 — Session gate: shell-owned vs. per-route?

**Tradeoff axis:** Shell-owned means `PortalShell` wraps the session check once and all `/portal/*` routes inherit it (2.2 Q6 fix applies uniformly). Per-route means each zone declares its own gate (more explicit, more boilerplate). 3.3's `<RouteGuard>` pattern on `/app/*` leans per-route; portal could mirror or diverge. PENDING.

### Q5 — Amenities inline vs. linked (coordinates with 4.2 Q3 PENDING)?

**Tradeoff axis:** 4.2 Q3 is the canonical decision. 3.5 cannot pre-decide it. If Q3 lands "inline" before 3.5 builds, 3.5's PortalAmenities component becomes a sub-component of PortalCommunity; if Q3 lands "external-link only", PortalAmenities stays as its own route at `/portal/amenities`. 3.5's interim answer: build PortalAmenities as a separate route component, with the session gate fix per 2.2 Q6, and defer inline integration to whenever 4.2 Q3 resolves. PENDING (coupled to 4.2 Q3).

### Q6 — Community inline preview component: new vs. reuse?

**Tradeoff axis:** 4.2 Q4 locked HYBRID (preview inline + click-through to `/community/:associationId`). The preview widget per AC-6 consumes the same data endpoints as the public Hub — so the question is whether `PortalCommunity.tsx` imports existing Community Hub widgets directly, or a new `PortalCommunityPreview` renders the summary. Direct reuse means coupled render lifecycles; a new preview component decouples but duplicates some render logic. PENDING.

### Q7 — Migration strategy: big-bang vs. incremental?

**Tradeoff axis:** Big-bang ships one PR that deletes the mega-file and lands all 8 new routes at once. Incremental ships one zone at a time; the mega-file shrinks each PR until the last zone migrates and the file is deleted. Big-bang is easier to revert (one PR) and easier to QA (one cutover). Incremental is safer per-zone but the mega-file is a moving target during migration. PENDING.

### Q8 — Deep-link preservation for existing `/portal` tab-deep-links?

**Tradeoff axis:** If the current portal supports query-string tab indicators (e.g., `/portal?tab=financials`), shipped links in outbound emails or help-desk notes may rely on them. A legacy-URL compat layer redirects old shapes to new hub URLs. Alternative: hard cutover, one-time user-visible breakage. Needs a grep sweep of outbound content to determine actual exposure. PENDING.
