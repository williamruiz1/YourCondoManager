# 5.9 — CI Pipeline Scaffold

**Status:** SPEC LOCKED 2026-04-25 (Wave 30, ship-and-lock pattern)
**Owner:** Platform Overhaul
**Layer:** 5 — Polish & Hardening

---

## Scope

Stand up a GitHub Actions CI pipeline that runs every quality gate
the YCM repo currently has — tsc, eslint, vitest, production build,
Playwright (route-mock + real-backend), axe-core a11y assertions,
and visual regression — on every pull request targeting `main` and
every push to `main`. Until this wave, every gate had to be run
manually before review; the gates from Wave 25 (axe-core) and Wave
17 (Playwright real-backend) were optional in practice.

Closes the deferral noted in `5.7-automated-quality-gates.md` (out-
of-scope item 1: "A new CI workflow file").

**In scope:**

- New workflow file at `.github/workflows/ci.yml`.
- Four jobs running in parallel where possible:
  - `check` — `npm run check` (`tsc`) + `npm run lint`.
  - `test` — `npx vitest run` (full vitest suite).
  - `build` — `npm run build` (sanity check the production bundle).
  - `playwright` — three Playwright modes in one job (route-mock,
    real-backend, visual regression).
- Cache `~/.npm` via the official `actions/setup-node` `cache: npm`
  helper.
- On Playwright failure, upload the traces + HTML report as a
  workflow artifact so a developer can drag-and-drop a `.zip` into
  `npx playwright show-trace`.
- `concurrency` group keyed by workflow + ref so the previous run
  on a branch is canceled when a new commit lands — saves CI
  minutes during rapid PR iteration.
- `paths-ignore` on the `pull_request` trigger so docs-only PRs
  skip the workflow entirely.
- Top-level `docs/ci-cd.md` documenting how to read failures, how
  to update visual baselines, how to skip CI for trivial doc-only
  PRs, and common troubleshooting.

**Out of scope:**

- Any deployment / CD step. The workflow validates that the
  artifact builds, but does not publish it. Replit + Fly stay as
  manual deploys.
- Code coverage uploads (Codecov / Coveralls etc.). Wave 25 already
  declined external SaaS for visual regression; same reasoning
  applies here.
- GitHub Pages or any other deploy target.
- Cross-browser Playwright coverage. The harness is Chromium-only
  per Wave 25 §123.
- Branch protection rules. The workflow is the prerequisite; the
  GitHub-side "require status checks before merging" toggle is a
  manual repo setting William flips after the first green run.
- Any new `SECRETS.*` references. CI runs entirely on public
  primitives — `npm ci`, ephemeral pglite, no third-party calls.
- A separate `skip-docs.yml` workflow. Replaced by `paths-ignore`
  on the same `ci.yml` workflow's `pull_request` trigger.

## Acceptance Criteria

1. `.github/workflows/ci.yml` exists and is valid YAML.
2. Workflow triggers on `pull_request` to `main` AND `push` to
   `main`. Both triggers run all four jobs.
3. Workflow uses `actions/checkout@v4` and `actions/setup-node@v4`
   pinned to Node 24 with `cache: 'npm'`.
4. Four jobs configured: `check` (tsc + lint), `test` (vitest),
   `build`, `playwright`.
5. The `playwright` job runs `npx playwright install --with-deps
   chromium` before invoking any spec.
6. The `playwright` job runs three Playwright modes sequentially:
   route-mock (`PLAYWRIGHT_STATIC=1 npm run test:playwright`),
   real-backend (`npm run test:playwright:real`), visual regression
   (`npm run test:playwright:visual`). The visual-regression step
   uses `if: ${{ !cancelled() }}` so it still attempts to run even
   if real-backend reports flakes.
7. On Playwright failure, traces and report are uploaded via
   `actions/upload-artifact@v4` named `playwright-traces` with
   `retention-days: 7`.
8. The workflow declares a `concurrency` group keyed by
   `${{ github.workflow }}-${{ github.ref }}` with
   `cancel-in-progress: true` so each new commit on a branch
   cancels the previous run.
9. Pull requests that touch only `docs/**` or `**/*.md` skip the
   workflow entirely (via `paths-ignore` on the `pull_request`
   trigger). Pushes to `main` always run.
10. `docs/ci-cd.md` exists and documents: what each job does, how
    to read a failure, how to update visual baselines, how the
    docs-only path-ignore works, common troubleshooting.
11. The `00-index.md` Layer 5 table has a 5.9 row with status
    `SPEC LOCKED · IN BUILD`.
12. The 5.9 PR's own CI run completes — every job either green or
    deterministically failing for a real, unrelated reason. (The
    workflow is its own validation; a self-test loop is part of
    the rollout plan.)

## Decisions Made

- **One workflow file, four jobs, parallel by default.** Considered
  splitting the four gates across separate workflow files (the
  classic monorepo pattern) but the YCM repo is a single Node
  package — every job needs `npm ci`, every job is small enough
  that the launch overhead of a separate workflow would dominate.
  Parallel jobs in one workflow give the same wall-clock latency
  with one source-of-truth file.

- **All three Playwright modes in one job.** Considered three
  separate Playwright jobs (route-mock, real-backend, visual). Each
  one needs `npm ci` + `playwright install --with-deps chromium`,
  which together account for ~3 minutes of setup per job. Running
  them sequentially in one job amortises the setup cost across all
  three modes. Trade-off: a route-mock failure short-circuits the
  later modes — the visual-regression `if: !cancelled()` mitigates
  this so we still get visual diffs when they're orthogonal to the
  flow failure.

- **`paths-ignore` on `pull_request` trigger, not a separate
  workflow.** The original Wave 30 prompt suggested an optional
  `skip-docs.yml`. That's strictly more files for the same
  behavior. `paths-ignore` does what the separate-workflow approach
  would, in five lines, in the same file. Pushes to `main` ignore
  the path filter so a docs-only merge does NOT skip the post-
  merge sanity run.

- **Node 24 to match `package.json` engines / Replit.** YCM's
  current dev environment runs Node 24 (Replit pin + `engines`
  inferred from current usage). Pinning the workflow to Node 24
  keeps the CI gate honest with what runs in production.

- **`actions/setup-node@v4` `cache: 'npm'`, NOT a manual
  `actions/cache@v4` step.** The official cache helper handles
  cache-key invalidation on `package-lock.json` changes correctly
  out of the box. Manual cache steps are an anti-pattern that
  bites when lockfile is stale.

- **`concurrency.cancel-in-progress: true`.** Every push to a PR
  branch cancels the previous run. Saves CI minutes when a PR
  iterates rapidly. Trade-off: the canceled run's logs are still
  available in the GitHub UI, just marked "canceled."

- **No code coverage upload.** Wave 25 declined external SaaS for
  visual regression; same reasoning applies. Coverage data is
  available locally via `npm run test:coverage` and that is
  sufficient for the spec-first overhaul. A future module can
  wire Codecov when there's a real need (e.g. external auditor).

- **No deploy step.** This is CI, not CD. Replit (preview) and Fly
  (production) stay as manual deploys, as documented in
  `docs/architecture-2026-04-25.md`. The workflow proves the
  artifact builds; the human pushes the deploy button.

- **`SESSION_SECRET` defaulted in `script/playwright-real-backend.ts`,
  not injected as a workflow secret.** That script already falls
  back to a deterministic test secret when `SESSION_SECRET` is
  unset (`process.env.SESSION_SECRET ?? "wave17-playwright-test-secret"`),
  so CI does not need any repo secrets to run the Playwright real-
  backend mode. Keeping the workflow zero-secrets makes it forkable
  / runnable on a fresh fork without setup ceremony.

- **Visual regression baselines committed to the repo.** Wave 25's
  decision: PNGs live under
  `tests/e2e/playwright/__screenshots__/`. CI consumes the
  baselines as-is; the update flow (`npm run
  test:playwright:visual:update` + commit + push) is documented in
  `docs/ci-cd.md`. The baselines are platform-specific (Chromium
  on Ubuntu); a developer running the suite locally on macOS will
  see diffs that disappear in CI.

## Tests

The workflow itself is the test. The 5.9 PR triggers its own CI run;
that run becomes the validation gate.

- `check` job: should pass. `npm run check` and `npm run lint` are
  green on `main` post-Wave 25.
- `test` job: should pass. Vitest suite is green on `main` post-
  Wave 26.
- `build` job: should pass. `npm run build` is green on `main`.
- `playwright` job: should pass. The five Playwright specs +
  visual-regression spec are green on `main` post-Wave 25.

If any job fails for a reason orthogonal to the workflow itself
(e.g. a flaky Playwright spec on Linux that passed on macOS dev),
the failure is filed as a follow-up workitem and the workflow is
adjusted (timeout, retry, environment) iteratively.
