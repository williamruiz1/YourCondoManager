## Gap Catalogue

### Gap 1 — HOA payment processing end-to-end

**Current state:** ACH payment initiation exists via Stripe Checkout (`0.2-persona-map.md:192`). The portal initiates a payment but full end-to-end HOA payment processing (real money movement, reconciliation, receipt generation, payment posting to owner ledger, failed-payment handling) is not implemented.

**What "done" looks like:** Complete payment lifecycle: owner initiates payment, payment clears or fails, ledger is updated, owner sees confirmation or failure with next steps, manager sees payment in financials.

**Scope status:** OUT OF SCOPE. Explicitly scoped out of this overhaul by 0.2 §Persona 5 Beta gaps (`0.2-persona-map.md:201`): "HOA payment processing (real money movement end-to-end, beyond initiation)." No nav slots, IA taxonomy entries, or feature specs may be written for this capability until it has its own Layer 0 decision doc (`0.2-persona-map.md:200`).

**Governing doc:** `0.2-persona-map.md:200-203`

### Gap 2 — Bill history / formal statements and invoices

**Current state:** Financials tab shows balance summary and ledger entry history (`0.2-persona-map.md:191`). No formal statement or invoice view exists — owners cannot view or download a monthly statement, an assessment invoice, or a payment receipt as a formatted document.

**What "done" looks like:** Owner can view a chronological list of statements/invoices, open each as a formatted document (PDF or rendered view), and download for records. Statement generation is an operator-side capability; the portal surfaces what the operator has published.

**Scope status:** OUT OF SCOPE. Explicitly scoped out of this overhaul by 0.2 §Persona 5 Beta gaps (`0.2-persona-map.md:202`): "Bill history / formal statement and invoice view." Requires its own Layer 0 decision doc before spec work can begin.

**Governing doc:** `0.2-persona-map.md:200-203`

### Gap 3 — Portal restructure from mega-file to real routes

**Current state:** The entire Owner Portal UI lives in a single 3,935-line file (`client/src/pages/owner-portal.tsx`) with 7 tab-switched views driven by local state (`activeTab` at `client/src/pages/owner-portal.tsx:259`). There are only 2 registered React routes: `/portal` and `/portal/amenities` (`client/src/App.tsx:356-357`). The portal has no real route boundaries — no `/portal/finances`, `/portal/requests`, or `/portal/community` hub URLs exist despite 1.2 Q4 locking them (`1.2-section-hub-reclassification.md:87-90`). Browser back/forward does not work across tabs. Deep-linking to a specific tab is impossible.

**What "done" looks like:** The mega-file is decomposed into route-per-zone components. 1.2 Q4 hub URLs are implemented: `/portal/finances`, `/portal/requests`, `/portal/community` with `/portal` as the dashboard. Each zone is a separate React route with its own component file. Browser navigation (back/forward/deep-link) works. 1.1 Q5 four-zone vocabulary (Home, My Finances, My Requests, My Community) is applied as sidebar labels (`1.1-zone-taxonomy-corrections.md:112-117`). 1.4 Q7 title rules are applied: `useDocumentTitle` hook, `{Page Title} — YCM` format, one `<h1>` per page, first-person labels (`1.4-page-title-consistency.md:105-113`).

**Scope status:** DEFERRED from 3.2. 3.2 Q5 explicitly defers the Owner Portal hub URL migration from the `/app` route pass to separate work (`3.2-route-restructure.md:205-207`). 3.2 AC 15-16 confirm no `/portal/*` route is added, removed, renamed, or redirected by 3.2 (`3.2-route-restructure.md:335-337`). The restructure is in-scope for the overhaul (it implements locked 1.1/1.2/1.4 specs) but has no assigned Layer 3 delivery vehicle. 4.2 flags this as a gap requiring its own implementation task or module.

**Governing docs:** `3.2-route-restructure.md:205-207` (deferral), `1.2-section-hub-reclassification.md:87-90` (hub URLs), `1.1-zone-taxonomy-corrections.md:112-117` (zone vocabulary), `1.4-page-title-consistency.md:105-113` (title parity)

### Gap 4 — Push notification preferences UI

**Current state:** Push notifications are a live capability — VAPID-based, with service worker subscription (`0.2-persona-map.md:195`). Subscribe and unsubscribe endpoints exist (`POST /api/portal/push/subscribe` at `server/routes.ts:10112`, `POST /api/portal/push/unsubscribe` at `server/routes.ts:10148`). The public VAPID key endpoint exists (`GET /api/portal/push/vapid-public-key` at `server/routes.ts:10105`). However, there is no dedicated preferences UI surface where an owner can view their current notification subscriptions, toggle notification categories (e.g., maintenance updates, financial alerts, community announcements), or manage per-device subscriptions.

**What "done" looks like:** A notification preferences section (within My Profile or as a standalone settings surface) where the owner can: (1) see which devices are subscribed, (2) toggle notification categories on/off, (3) manage per-device push enrollment. The UI integrates with the existing VAPID/service-worker infrastructure.

**Scope status:** PENDING. Push notifications are a live 0.2 capability but the preferences UI is not enumerated in the live capabilities list. 2.2 Q3 preserves status-quo on owner self-service mutations (`2.2-owner-portal-access-boundaries.md:283-285`) — push subscribe/unsubscribe commit directly. Whether a preferences UI is in-scope for this overhaul or deferred requires a founder decision.

**Governing doc:** `0.2-persona-map.md:195` (push notifications live), `2.2-owner-portal-access-boundaries.md:283-285` (Q3 status-quo)

### Gap 5 — Amenity booking inline

**Current state:** Amenity booking is listed as a live capability but is implemented as an external link to `/portal/amenities` (`0.2-persona-map.md:197`). The `/portal/amenities` route is registered as a separate public React route in `client/src/App.tsx:357` with no portal session gate — 2.2 Q6 resolved that it must inherit the `/portal` session-redirect pattern (`2.2-owner-portal-access-boundaries.md:295-296`). The amenity booking experience is a separate page outside the main portal tab structure, not an inline experience within the My Community zone.

**What "done" looks like:** Amenity booking is an inline experience within the portal zone structure (likely My Community per 1.1 Q5). The owner can browse available amenities, view availability, and book — all within the portal session without leaving to an external route. `/portal/amenities` either becomes a sub-route of `/portal/community` or is integrated into the community zone component. Session gating per 2.2 Q6 is enforced.

**Scope status:** PENDING. The 2.2 Q6 session-gating fix is in-scope (locked). Whether the UX is upgraded from external-link to inline booking experience is a separate question. The portal restructure (Gap 3) is a prerequisite — inline integration requires real route boundaries. Founder decision needed on whether inline amenity booking is in-scope or deferred.

**Governing docs:** `0.2-persona-map.md:197` (amenity booking link live), `2.2-owner-portal-access-boundaries.md:295-296` (Q6 session-required), `1.1-zone-taxonomy-corrections.md:112-117` (My Community zone)

### Gap 6 — Community Hub integration

**What "done" looks like:** Community Hub content is accessible within the portal zone structure under My Community. The owner navigates to the community zone and sees community content inline (announcements, discussions, events, shared resources) without leaving the portal session context. The `/community/:associationId` route either embeds within `/portal/community` or the portal community zone links to it with portal-session context preserved.

**Scope status:** PENDING. 1.1 Q5 locked "My Community" as a portal zone (`1.1-zone-taxonomy-corrections.md:116`), and 1.2 Q4 locked `/portal/community` as a hub URL (`1.2-section-hub-reclassification.md:89`). The portal restructure (Gap 3) is a prerequisite.

## Open Questions

### Q1 — Is the portal restructure (Gap 3) in-scope for this overhaul, and if so, which layer/module owns it?

**Tradeoff axis:** 3.2 Q5 deferred the portal restructure from the `/app` route pass. The restructure implements locked specs (1.1 Q5 zone vocabulary, 1.2 Q4 hub URLs, 1.4 Q7 title parity) — leaving it unimplemented means three locked Layer 1 specs have no delivery vehicle. A new Layer 3 or Layer 4 module could own it; alternatively, it ships as a standalone implementation task post-overhaul. Deferring risks the overhaul closing with locked specs that have no implementation path.

### Q2 — Is a push notification preferences UI (Gap 4) in-scope for this overhaul or deferred to a future capability spec?

**Tradeoff axis:** Push notifications are a live 0.2 capability, but 0.2 lists the infrastructure (VAPID, service worker) — not a preferences management surface. Adding a preferences UI is a UX enhancement to an existing capability, not a net-new feature. It could ship within the portal restructure (Gap 3) as part of the My Profile or My Community zone, or it could be deferred as a separate feature spec. The question is whether "production-ready" requires user-facing notification management or whether the current subscribe/unsubscribe-only model is sufficient.

### Q3 — Is amenity booking upgraded from external-link to inline experience (Gap 5) in this overhaul, or does the 2.2 Q6 session-gate fix suffice?

**Tradeoff axis:** 2.2 Q6 locks the session-gate fix — that is in-scope regardless. The UX upgrade (inline booking within My Community zone) depends on the portal restructure (Gap 3) and may also depend on the amenity booking feature surface itself (which lives in a separate routes module per `2.2-owner-portal-access-boundaries.md:102`). Inline integration adds scope; the session-gate-only fix is minimal and satisfies the locked 2.2 spec.

### Q4 — Is the Community Hub fully integrated into the portal (Gap 6) or does it remain an external link with a portal zone wrapper?

**Tradeoff axis:** Full integration means the `/community/:associationId` content renders inside the `/portal/community` zone. This requires either embedding the community app or rewriting its content into portal components. A zone wrapper (the `/portal/community` hub page links out to `/community/:associationId`) is lighter but leaves the owner session context when navigating to community content. The 1.2 Q4 hub URL `/portal/community` is locked and must exist — the question is what it contains.

### Q5 — What is the implementation sequencing: does the portal restructure (Gap 3) block all other portal gaps, or can session-gate fixes (2.2 Q6) and title-parity fixes (1.4 Q7) land independently?

**Tradeoff axis:** The portal restructure is a prerequisite for inline amenity booking (Gap 5) and community hub integration (Gap 6) because both require real route boundaries. However, session-gate enforcement (2.2 Q6 on `/portal/amenities`) and title/`<h1>` fixes (1.4 Q7) can land against the current mega-file without waiting for the restructure. Sequencing determines whether quick wins ship first or whether everything waits for the restructure.

### Q6 — Does the 7-tab sidebar label vocabulary (Overview, Financials, Documents, Maintenance, Notices, Communications, Voting) migrate to the 4-zone vocabulary (Home, My Finances, My Requests, My Community) as part of 4.2 or as part of the portal restructure (Gap 3)?

**Tradeoff axis:** The label migration is a 1.1 Q5 implementation requirement (`1.1-zone-taxonomy-corrections.md:112-117`). It could land as a standalone cosmetic change against the current mega-file (renaming tabs and regrouping them into four zones without changing routes). Alternatively, it ships jointly with the route restructure (Gap 3) to avoid two passes over `owner-portal.tsx`. The cosmetic-only approach risks merge conflicts with the restructure.

### Q2 — Push notification preferences UI (Gap 4)

### Q3 — Amenity booking UX (Gap 5)
**RESOLVED — 2026-04-24:** **Session-gate-only status quo preserved.** The 2.2 Q6 session-gate fix already shipped in PR #10 (`d6b193f`) satisfies 2.2 AC. Inline booking integration is deferred to 3.5 Owner Portal Restructure scope (Gap 3) where the portal route structure itself is rewritten — the inline experience is properly part of that workstream, not 4.2.

**Addendum — per-association amenities feature toggle (3a):** One gap that falls neither under 2.2 nor 3.5 surfaced during Session B: many complexes have no amenities, and there is currently no way for a Manager or self-managed Board Officer to hide the amenities surface. The `associations` table (`shared/schema.ts:6-19`) has no feature-enabled columns; the sidebar tab and `/portal/amenities` route always render. Resolution:

- Add `amenitiesEnabled: integer("amenities_enabled").notNull().default(1)` to the `associations` table (integer-boolean, matching the existing `isArchived` pattern). Default `1` (enabled) preserves current behavior on migration.
- Manager + self-managed Board Officer only: per-association toggle at `/app/settings` (exact placement deferred to 3.1/3.5 implementation). Owner persona cannot toggle.
- When `amenitiesEnabled = 0`:
  - The amenities sidebar entry in `owner-portal.tsx` + `app-sidebar.tsx` (if/when a manager-side amenities surface exists) is hidden.
  - `/portal/amenities` route returns `<NotFound/>` (not an empty state — the feature is off for this association).
  - Amenity-reservation API endpoints return 404 for this association. The amenities tables themselves remain (historical data preserved; no cascade delete).
- Migration is forward-only; no behavior change for existing complexes. All `associations` rows get `amenities_enabled = 1` at migration time.
- Filed as PPM workitem `amenities-toggle-3a` for build; implementation is scoped for a single PR after Session B locks. Shipping before 3.5 is fine — the toggle is independent of the portal restructure.

Full inline integration is deferred to a separate initiative tied to the Community app's own roadmap. The 1.2 Q4 URL remains valid; this resolution clarifies what the URL contains. No new scope for this overhaul beyond ensuring the hub page exists with a link-out.

### Q5 — Implementation sequencing
**RESOLVED — 2026-04-21:** Session-gate fix (2.2 Q6) and title-parity (1.4 Q7) land independently against the mega-file. The portal restructure (Gap 3 / 3.5) sequences separately. The 2.2 Q6 fix shipped in PR #10 (`d6b193f`); 1.4 Q7 shipped in PR #9 (`449b500`). 3.5 proceeds on its own track. See original Q5 entry in Decision Log.

### Q6 — 7-tab → 4-zone sidebar vocabulary migration
**RESOLVED — 2026-04-24:** **Bundle with 3.5 Owner Portal Restructure** (Gap 3), not a standalone cosmetic PR. Both passes touch `owner-portal.tsx` (a known mega-file flagged for restructure); doing them separately guarantees merge conflicts and doubles review cost.
