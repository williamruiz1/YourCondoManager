# 5.3 — Mobile Audit

**Status:** SPEC LOCKED (Wave 14, 2026-04-24) · **IN BUILD (partial)**
**Owner:** Platform Overhaul
**Layer:** 5 — Polish & Hardening

---

## Scope

Audit every authenticated operator and portal surface at **375px** (phone) and **768px** (tablet) breakpoints against `docs/projects/mobile-ui-rules.md`. Fix layouts that overflow, clip, or wrap in unreadable ways.

Wave 14 MVP:
- Inventory the surfaces with fixed-width CSS values or `flex`/`grid` layouts that are likely to overflow under 375px.
- Apply breakpoint fixes (mobile-first with `sm:` / `md:` Tailwind prefixes) to the highest-value surfaces: Home, `/portal`, `/app/financial/rules`, `/app/communications/inbox`, `/portal/finances`.
- Add one smoke-test asserting the Home page doesn't measurably overflow at 375px width. Note: `container.scrollWidth <= 375` is a rough proxy — real mobile layout correctness still requires a human pass.
- Ship `implementation-artifacts/5.3-mobile-audit-findings.md` with surfaces audited, fixes applied, and items deferred.

Everything else (automated visual regression, Percy/Chromatic integration, per-surface retrofits at 320px/430px) is out of scope.

---

## Acceptance Criteria

1. `implementation-artifacts/5.3-mobile-audit-findings.md` lists:
   - Surfaces audited (≥8).
   - Fixes applied in Wave 14 (file + 1-line summary each).
   - Items deferred to future work with rationale.
2. At least 3 mobile layout fixes committed in Wave 14 — each mobile-first with `sm:` / `md:` prefixes (no desktop-first `lg:` reverse patterns unless adopting existing convention).
3. `tests/mobile-audit.client.test.tsx` asserts a smoke expectation that the Home-panel-adjacent render path doesn't overflow at 375px width. Test acknowledges this is a proxy, not a pixel-perfect layout check.
4. `docs/projects/mobile-ui-rules.md` is NOT edited — Wave 14 follows the existing rules; any rule amendments are out of scope.
5. No regressions to desktop (`md:` and above) layouts.
