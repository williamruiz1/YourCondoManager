# 5.3 — Mobile Audit

**Status:** SPEC LOCKED (Wave 14, 2026-04-24) · **IN BUILD (partial)**
**Owner:** Platform Overhaul
**Layer:** 5 — Polish & Hardening

---

## Scope

Audit every authenticated operator and portal surface at **375px** (phone) and **768px** (tablet) breakpoints against `docs/projects/mobile-ui-rules.md`. Fix layouts that overflow, clip, or wrap in unreadable ways.

Wave 14 MVP:
- Inventory the surfaces with fixed-width CSS values or `flex`/`grid` layouts that are likely to overflow under 375px.
- Apply breakpoint fixes (mobile-first with `sm:` / `md:` Tailwind prefixes) to the highest-value surfaces: Home, `/portal`, `/app/financial/rules`, `/app/communications/inbox`, `/portal/finances`.
- Add one smoke-test asserting the Home page doesn't measurably overflow at 375px width. Note: `container.scrollWidth <= 375` is a rough proxy — real mobile layout correctness still requires a human pass.
- Ship `implementation-artifacts/5.3-mobile-audit-findings.md` with surfaces audited, fixes applied, and items deferred.

Everything else (automated visual regression, Percy/Chromatic integration, per-surface retrofits at 320px/430px) is out of scope.

---

1. `implementation-artifacts/5.3-mobile-audit-findings.md` lists:
   - Surfaces audited (≥8).

5. No regressions to desktop (`md:` and above) layouts.

## Wave 18 — Round 2 adoption

**Date:** 2026-04-24

Wave 14 shipped 3 layout fixes + 4 EmptyState refactors on hub placeholders and deferred 5 follow-up items in `implementation-artifacts/5.3-mobile-audit-findings.md`. Wave 18 closes 3 of those 5 follow-ups.

### Mobile fixes shipped in Wave 18

#### 5.3-F1 Operator table card-fallbacks (PARTIAL — Foundation ratified)

**Surface:** Account Activity table inside `client/src/pages/financial-foundation.tsx`.

**Change:** the 8-column "Account Activity" table previously rendered identically at every breakpoint, blowing past 375px on phones. Wave 18 introduces a stacked-card list rendered at `<md` (`hidden md:block` flips the desktop table off below the breakpoint, and a parallel `space-y-3 md:hidden` block renders mobile cards). Each card surfaces:

- Account name + inactive badge + account code/type meta line.
- Right-aligned utilization % badge.
- Three-up grid of Budget / Invoiced / Variance values (negative variance keeps `text-destructive`).

The desktop `<Table>` is unchanged structurally — only its parent now has `overflow-x-auto` so any future column additions stay confined to the card and don't push the page wider than the viewport at `md` and up.

**Recurring Charges + Special Assessments tables (the other two callouts in the deferred item):** already had isMobile-driven mobile-card renders shipped in earlier waves (Phase 8b). Verified in Wave 18 audit — no regression. The deferred item closes against Foundation only.

#### 5.3-F2 Header compression at <640px

**Surface:** the global operator workspace header in `client/src/App.tsx` (`<header>` inside `WorkspaceShell`).

**Change:** below `sm:` (640px) the header now uses `h-12 min-h-12 px-2 py-1.5 gap-1.5` (instead of `h-16 min-h-14 px-3 py-2 gap-2`) so the sidebar trigger + command palette + association switcher + account avatar fit on one row at 375px without the awkward second-line wrap. Above `sm:` the original sizing is preserved verbatim.

The "Notifications" + "Settings" icon buttons + association switcher already collapse to icon-only / w-full responsively inside `HeaderActions`, which keeps everything accessible at 375px after this change. No nav-drawer rewrite required.

#### 5.3-F4 320px floor

**Surfaces:** Home (`client/src/pages/dashboard.tsx`) and User Settings (`client/src/pages/user-settings.tsx`).

**Change:** outer wrapper padding tightened from `p-6` / `px-6 py-10` to `p-3 sm:p-6` / `px-3 py-6 sm:px-6 sm:py-10` so card content does not touch the viewport edge at 320px. Each Home/Settings card already constrains its own width via flexbox + grid — the outer padding was the offender at 320px.

`/portal/*` already used `px-4` in the portal shell at every breakpoint and was verified to render cleanly at 320px without changes. `/app/settings/billing` inherits the same outer padding fix via the `WorkspaceRouter` shell.

One narrow `w-[260px]` Input on `/app/ai/ingestion` (clause search) was relaxed to `w-full sm:w-[260px]` so it doesn't overflow at 320px.

## Wave 28 — Form-dialog mobile density

**Date:** 2026-04-24

Closes the Wave 14 deferred item **5.3-F5 Form dialog mobile density**. Wave 18 left F5 deferred (rationale: design-system level, would touch every `<Dialog>` form). Wave 28 ships the layout-only fixes across 10 high-traffic dialogs.

All fixes are layout-only — no validation, mutation, or form-state changes.

| 8 | Document Upload | `client/src/pages/documents.tsx` | DialogContent viewport clamp on the 3 dialogs (`max-w-3xl` upload + `max-w-3xl` view + `max-w-md` confirm). The internal `md:grid-cols-[1.1fr_0.9fr]` two-column was already responsive (stacks below md). |
| 9 | Admin User invite | `client/src/pages/admin-users.tsx` | DialogContent viewport clamp on the `max-w-lg` shell. |
| 10 | Election create + Tally + Ballot Options + Proxy | `client/src/pages/elections.tsx` | DialogContent viewport clamp on all 4 dialog shells. | 12 | Vendor create/edit | `client/src/pages/vendors.tsx` | DialogContent viewport clamp + `md:grid-cols-2` → `grid-cols-1 sm:grid-cols-2` for the 9-field vendor info grid. |
| 13 | Building / Association create | `client/src/pages/associations.tsx` | DialogContent viewport clamp on both dialogs (association create + view). Replaced `${isMobile ? "grid-cols-1" : "grid-cols-2"}` for City/State row with `grid-cols-1 sm:grid-cols-2`. |
| 14 | Persons (Owner / Resident) create + Board Role | `client/src/pages/persons.tsx` | DialogContent viewport clamp on both. Replaced 4 `isMobile` grid ternaries with `grid-cols-1 sm:grid-cols-2` (first/last name, role/unit, board start/end dates). Board-role action button → `w-full sm:w-auto`. |

That's **14 distinct dialogs across 10 files**, well above the 8–10 floor. The dialog count exceeds the file count because three pages (`financial-late-fees.tsx`, `elections.tsx`, `maintenance-schedules.tsx`) host multiple form dialogs that all needed the same treatment.

### Patterns applied

1. **Viewport clamp:** every fixed-width DialogContent now reads `max-w-[calc(100vw-2rem)] sm:max-w-{lg|xl|2xl|3xl}`. Below `sm:` (640px) the dialog can never exceed `viewport - 2rem` (the body has `p-6` ≈ 1.5rem × 2 = 3rem internal but the dialog itself sits at the viewport edge with a 1rem visual gutter).
2. **Responsive grids:** unconditional `grid-cols-2`, `md:grid-cols-2`, and `${isMobile ? "grid-cols-1" : "grid-cols-2"}` ternaries inside dialog forms were normalized to `grid-cols-1 sm:grid-cols-2`. This drops the `useIsMobile()` hook dependency for layout (one fewer re-render trigger) and gives Tailwind the same breakpoint semantics the rest of the codebase already uses.
3. **Action button rows:** `flex justify-end gap-2` → `flex flex-col-reverse gap-2 sm:flex-row sm:justify-end` so Cancel / Save become full-width stacked buttons on mobile (Save sits visually first because reversed-flex), with the desktop right-aligned row preserved at sm+.

The deferral is intentional — Wave 28 prioritized **dialogs that grep showed unconditional `grid-cols-2` or fixed widths > 380px without any viewport clamp**, plus the original target list.

### Patterns NOT applied (intentionally)

- **Sheet/Drawer migration on mobile:** the spec hints at swapping `<Dialog>` for `<Sheet>` (mobile drawer) at narrow widths. That's a structural rewrite that touches every dialog's open-state coordination, not a layout fix. Stays deferred.
- **Native HTML5 date pickers:** all date inputs already use `<Input type="date" />`, which delegates to the platform native picker on mobile. No change needed.
- **Form label shrinking:** the audited dialogs use Radix-driven labels at default sizing. Shrinking labels on mobile risks a11y regression (sub-12px text); we kept current sizing.
