## Conflict Analysis

Nine distinct conflicts surface from the IA audit docs and the live codebase. These are the decision surface 0.3 must cover.

### Conflict 1 — Zone visibility

**Evidence:** `docs/specs/ia-audit-spec.md` §Zone Model defines Z1/Z2/Z3 but the sidebar has no zone headers. Every current group is implicit.

**The question:** Should zones be visible as sidebar group headers ("Day-to-Day", "Setup & Reference", "Platform") or remain a pure taxonomy overlay invisible to users? This is a user-experience decision, not a taxonomy decision — the taxonomy is already locked in the audit spec.

**Tradeoff:** Visible zones give users a consistent mental model at the cost of extra chrome; invisible zones keep the sidebar dense but rely on implicit grouping.

### Conflict 2 — Section hub pattern

**Evidence:** `docs/specs/ia-audit-phase3-reconciliation.md` proposes "Operations Overview" as a Z1-6 section hub (locked in 0.1). Finance, Governance, and Board all currently have hub-like landing pages, but the pattern is inconsistent: some hubs are dashboards (Operations), some are tab containers (Finance), some are mixed (Governance).

**The question:** Is "section hub" a standard pattern — every Zone 1 section has a hub landing page — or is it per-section? If standard, what does a hub contain (KPI row, sub-tabs, recent activity, quick actions)?

### Conflict 3 — Board Admin sidebar presentation

**Evidence:** 0.2 matrix restricts Board Admin from Portfolio Health, Settings, and association switching. Current sidebar gates items with a `roles` array but the rendering model for a hidden item is unclear (invisible? disabled? replaced?).

**The question:** Does Board Admin see a reduced sidebar (only their owned sections rendered) or the full sidebar with restricted items hidden in place? If reduced, is there a single "Board Admin view" composition or per-item hiding?

### Conflict 4 — Active-association context switcher placement

**Evidence:** Manager persona (0.2) uses an active-association context switcher when managing multiple associations. Its current placement in the UI is not explicitly specced in 0.2 or in any audit doc.

**The question:** Where does the switcher live — top bar, above sidebar, inside sidebar as a header, on Home only? This affects sidebar real estate and the nav shell layout.

### Conflict 5 — Platform group (Zone 3) shell model

**Evidence:** `docs/specs/ia-audit-phase2-orphans.md` and `ia-audit-phase4-platform.md` confirm 5 Zone 3 pages (Platform Controls, AI Ingestion, Admin Roadmap, Admin Users, Executive Decks). Currently all five (minus admin-users and executive which are "orphan-surface") live under a single Platform Controls sidebar group. Owner Portal is incorrectly nested there.

**The question:** Is Zone 3 a sidebar group inside `/app` (current model, requires `platform-admin` gating) or does it move to a separate admin shell route (e.g., `/admin/*`) with its own sidebar? A separate shell cleanly separates operator and platform workspaces; a shared shell keeps all work in one place.

### Conflict 6 — Breadcrumb convention

**Evidence:** `breadcrumb-nav.tsx` is a primitive renderer; every page composes its own item array. No standard.

**The question:** Does the model want (a) a central `route → breadcrumb label` mapping derived from the route tree, (b) page-defined breadcrumbs with a convention document, or (c) auto-generated breadcrumbs from route segments with a label dictionary? And what is the root-label rule (is `/app` always "Home" per 0.1, or does the root vary by section)?

### Conflict 7 — Route depth and URL convention

**Evidence:** Mixed depth: some sections use `/app/section` (documents, insurance, board), others use `/app/section/page` (financial/billing, operations/dashboard, governance/elections). 43 redirects include both "missing section" and "wrong section depth" corrections.

**The question:** What is the canonical URL pattern? Options: (a) always `/app/section/page` (add depth where missing), (b) always `/app/section` with tabs (flatten), (c) tolerate both depending on whether a section has a hub.

### Conflict 8 — Legacy redirect policy

**Evidence:** 43 `RouteRedirect` rules, many pointing to hubs from pages that were merged (per `ia-audit-phase3-reconciliation.md`).

**The question:** Do we keep legacy redirects indefinitely as backward-compat for external bookmarks and Board Admin muscle memory, or plan a deprecation cycle (e.g., keep through 2026-Q3, then remove)?

### Conflict 9 — Mobile nav scope

**Evidence:** 0.2 sets a "low cognitive load" target for Owner Portal (mobile-first) but no mobile directive for Manager/Board Admin. Layer 5 (5.3 Mobile audit) is QUEUED.

**The question:** Does 0.3 lock the mobile nav model for the operator workspace (drawer/bottom nav/collapse behavior) or is that explicitly deferred to Layer 5? Locking here constrains Layer 3 sidebar redesign; deferring risks 5.3 revealing conflicts with 3.1.

## Open Questions

Each question below is a decision slot. None are answered in this skeleton. Format mirrors 0.1's Q1–Q3 pattern — one-line question, followed by the tradeoff axis the decision lives on.

**Q1 — Zone visibility.** Should Z1/Z2/Z3 be visible as sidebar group headers, or invisible (taxonomy-only)?
_Tradeoff axis:_ UX legibility vs. chrome/density.

**Q2 — Section hub pattern.** Is every Zone 1 section required to have a hub page, or is the hub pattern per-section?
_Tradeoff axis:_ consistency vs. over-engineering single-tab sections.

**Q2a — Hub composition.** If hubs are standard, what must every hub contain? (KPI row? Sub-tabs? Recent activity? Quick actions?)
_Tradeoff axis:_ rigid template vs. per-section tailoring.

**Q3 — Board Admin sidebar.** Reduced sidebar (only owned sections rendered), or same sidebar with restricted items hidden in place?
_Tradeoff axis:_ persona clarity vs. implementation simplicity.

**Q4 — Association context switcher.** Placement: top bar, above sidebar, inside sidebar header, or Home-only?
_Tradeoff axis:_ discoverability vs. sidebar real estate.

**Q5 — Zone 3 shell.** Sidebar group inside `/app` (current), or separate admin shell at `/admin/*` with its own sidebar?
_Tradeoff axis:_ unified workspace vs. clean separation of operator and platform work.

**Q6 — Breadcrumb source.** Central route-label mapping, per-page composition with convention doc, or auto-derived from route segments + label dictionary?
_Tradeoff axis:_ central control vs. per-page flexibility.

**Q6a — Breadcrumb root.** Is `/app` always labeled "Home" per 0.1, or does the root label vary by section context?
_Tradeoff axis:_ consistency vs. context-aware clarity.

**Q7 — Canonical URL depth.** Always `/app/section/page`, always `/app/section` with tabs, or mixed per-section?
_Tradeoff axis:_ uniformity vs. tailored URL shape.

**Q8 — Legacy redirect lifecycle.** Keep indefinitely, or plan a deprecation cycle (and when)?
_Tradeoff axis:_ backward-compat burden vs. routing-code cleanliness.

**Q9 — Mobile scope.** Lock operator mobile nav here, or defer to Layer 5 (5.3 Mobile audit)?
_Tradeoff axis:_ prevent Layer 3/5 rework vs. avoid blocking 0.3 on mobile research.
