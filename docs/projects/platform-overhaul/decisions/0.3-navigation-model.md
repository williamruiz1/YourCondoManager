## Context

0.1 fixed the three-command-center conflict at the page level. 0.2 locked the three-persona map and the surface-ownership matrix. 0.3 locks the **navigation model** — the rules by which routes, sidebar groups, breadcrumbs, entry points, and section hubs are composed into a coherent workspace for the personas defined in 0.2.

Every Layer 1–3 module depends on 0.3 being locked:
- **Layer 1** (IA Taxonomy Cleanup) uses the functional zone model to correct section labels.
- **Layer 2** (Role & Permission Model) uses the strict role-isolation principle and the 0.2 boundary matrix as its gating source of truth.
- **Layer 3** (Navigation Restructure) IS the implementation of this decision.

This document does not design pages. It designs the *framework* pages live inside: sidebar taxonomy, route preservation rules, breadcrumb convention, entry-point model, section count ceiling, cross-association navigation, default landing behavior, and the updates/inbox concept.

## Conflict Analysis

Nine distinct conflicts surface from the IA audit and live codebase. Each maps to a locked question below.

1. **Primary navigation shape** — is the sidebar one canonical structure filtered by role, or separate shells per persona? Maps to **Q1**.
2. **Sidebar grouping taxonomy** — functional domains (Finance, Operations, …) vs lifecycle stages (Setup → Operate → Govern) vs audience framing? Maps to **Q2**.
3. **Entry points by persona** — do personas share `/app` with different default landings, or get separate route trees? Maps to **Q3**.
4. **Breadcrumb convention** — deep hierarchy, shallow wayfinding pattern, or auto-generated? Maps to **Q4**.
5. **Role-gating principle** — show-but-disable, hide-on-no-access, or strict role isolation? Maps to **Q5**.
6. **Top-level section count** — strict cap (≤5), moderate (6–7), or unbounded? Maps to **Q6**.
7. **Cross-association navigation** — header switcher, tab pattern, portfolio page, or mix? Maps to **Q7**.
8. **Default landing behavior** — single canonical home per persona, or "last visited / custom home"? Maps to **Q8**.
9. **Updates / inbox model** — central notification center across zones, per-section inboxes only, or no cross-cutting updates surface? Maps to **Q9**.

## Selected Resolutions

### Q1 — Primary navigation shape → **A: Single global sidebar, role-filtered**

YCM uses one canonical navigation model across the platform. Sidebar items are filtered by persona and permission rather than maintaining fully separate structural shells per persona. This keeps the platform legible and reduces IA drift while still allowing Owner, Board Admin, and Manager experiences to differ in visible scope.

### Q2 — Sidebar grouping / zone taxonomy → **A: Functional grouping**

Primary sidebar taxonomy is grouped by function, not by lifecycle stage or audience. Canonical top-level zones are functional domains such as Financials, Operations, Governance, and Communications. Lifecycle or audience framing may appear in supporting copy but not as the canonical sidebar taxonomy.

### Q3 — Entry points by persona → **Distinct landings, shared shell (per 0.2)**

> **Resolution adjusted to honor 0.2.** The original Q3 option tree proposed separate route prefixes per persona (`/app`, `/board`, `/owner-portal`). 0.2 is locked with Manager and Board Admin sharing `/app` and Owner at `/portal`. 0.3 does not amend 0.2.

YCM maintains **distinct primary landing experiences by persona** while preserving the locked route model from 0.2. Manager and Board Admin share the `/app` workspace shell but land into role-appropriate experiences with different navigation scope and permissions. Owner remains in `/portal` as a separate self-service workspace.

- Manager primary entry: `/app` → broad operator experience
- Board Admin primary entry: `/app` → single-association, board-scoped experience within the same shell
- Owner primary entry: `/portal` → separate Owner Portal

### Q4 — Breadcrumb convention → **B: Shallow breadcrumbs**

Breadcrumbs follow a shallow hierarchy in the form:

```
Association > Section > Page
```

Breadcrumbs are a UI wayfinding pattern, not a URL rule. They remain concise and avoid deep, noisy nesting — breadcrumbs work best when they reflect a clear single path and do not wrap on mobile.

### Q5 — Role-gating principle → **B: Strict isolation**

YCM defaults to strict role isolation. Users do not see surfaces they cannot meaningfully act on, except where explicitly approved by spec. This reduces confusion, support burden, and cross-role leakage, and aligns with RBAC least-privilege principles.

### Q6 — Top-level section count → **B: 6–7 for Manager/Board, tighter for Owner**

Manager and Board Admin workspaces may use up to 6–7 top-level sections where justified by functional complexity. Owner Portal remains more compact and focused.

### Q7 — Cross-association navigation → **A primary, B supporting**

**Primary:** a global association switcher in the header.
**Supporting:** a portfolio/associations page exists as a secondary hub for browsing and entering association-scoped work.

Organization-switcher patterns are standard for multi-tenant SaaS because they change context cleanly while keeping actions scoped to the active organization. Tab-style association switching is not the core navigation model.

### Q8 — Default landing behavior → **A: One canonical home per persona**

Each primary persona has one default home/dashboard that acts as the standard landing page for that workspace. Manager lands on `/app` "Home" (per 0.1). Board Admin lands on `/app` "Home" in single-association scope (per 0.2). Owner lands on the Owner Portal home at `/portal`.

### Q9 — Updates / inbox model → **A: Central updates/inbox**

YCM uses a central updates model for cross-cutting alerts, notices, and important communications, while detailed work remains in its native section. This creates one reliable place for users to check what needs attention. The central inbox is a notification center, not a replacement for section-native workflows.

## Rationale

- **Simplicity first.** Chosen options favor operational clarity over architectural cleverness.
- **Single taxonomy, multiple entry experiences.** YCM gets consistency without forcing identical user experiences.
- **Strict role boundaries.** Cleaner support model, fewer permission ambiguities.
- **Association context is primary.** Especially for Board and Manager flows.
- **Owner stays compact.** Owner Portal remains focused and does not inherit admin complexity.

---

## What this resolution does NOT change

- **Route paths locked by 0.1 and 0.2.** `/app`, `/app/portfolio`, `/app/operations/dashboard`, `/portal`, and the single-association redirect behavior are preserved. 0.3 introduces no new route prefixes that conflict with locked 0.2.
- **0.2 persona boundary matrix.** 0.3 does not widen or narrow any cell of the 0.2 matrix.
- **Owner Portal scope.** Beta-scoped capabilities per 0.2 stand. HOA payment processing and bill history remain out of overhaul scope.
- **API contracts.** No backend changes required by 0.3 itself — API behavior is downstream of Layer 2 permission corrections.

## Acceptance Criteria

Consolidated from Q1–Q9. These criteria are verified during Layer 1, Layer 2, and Layer 3 implementation.

### Q1 — Single canonical navigation
1. The navigation model defines one canonical zone structure for the platform.
2. Each persona sees only the sections and pages relevant to that role.
3. Owner, Board Admin, and Manager experiences may differ in visible items but not in the underlying taxonomy.
4. No module may introduce a persona-specific top-level taxonomy without an explicit decision doc update.

### Q2 — Functional grouping
5. All pages in 0.3 scope are assigned to a functional zone.
6. Sidebar grouping is not user-toggleable.
7. Zone labels are functional and stable across personas.
8. Lifecycle or audience framing may appear in supporting copy but not as canonical sidebar taxonomy.

### Q3 — Distinct landings, shared shell
9. Manager primary entry remains `/app`.
10. Board Admin primary entry remains `/app`.
11. Owner primary entry remains `/portal`.
12. Board Admin lands into a single-association, board-scoped experience within `/app`.
13. Manager lands into the broader operator experience within `/app`.
14. Owner remains in a separate portal experience.
15. 0.3 does not introduce new route prefixes that conflict with locked 0.2.

### Q4 — Shallow breadcrumbs
16. Breadcrumbs never exceed three visible levels.
17. Association context appears when the user is in an association-scoped view.
18. Breadcrumb labels are human-readable and match page titles/section labels.
19. Mobile views may collapse or suppress breadcrumbs where space is constrained.
20. Breadcrumb structure does not imply multiple alternative parent paths for the same page.

### Q5 — Strict role isolation
21. Pages and navigation items are hidden when the role lacks access.
22. Read-only exposure to cross-role content requires explicit approval in spec.
23. Permissions are role-based, not individually customized by default.
24. Role-gating rules are documented for each primary persona.

### Q6 — Section count
25. Owner navigation target stays lean relative to Manager/Board.
26. Manager and Board Admin may exceed five top-level sections only when each section has a distinct functional purpose.
27. No persona accumulates redundant or overlapping top-level sections.
28. Section count decisions are justified by task separation, not legacy structure.

### Q7 — Cross-association navigation
29. Manager workspace includes a persistent association switcher in the header.
30. Switching association updates the active app context.
31. A portfolio/associations page exists as a secondary navigation surface.
32. Cross-association views are clearly distinguished from association-scoped views.
33. Tab-style association switching is not the core navigation model.

### Q8 — Canonical home per persona
34. Manager has one canonical home/dashboard.
35. Board Admin has one canonical home/dashboard.
36. Owner has one canonical home/dashboard.
37. Navigation docs reference one default landing page per persona.
38. Any future "last visited" or custom-home behavior is out of scope unless separately specified.

### Q9 — Central updates/inbox
39. A central updates/inbox concept exists in the navigation model.
40. Critical items from multiple zones can surface there.
41. The inbox does not replace detailed section-native workflows.
42. Owner notices/communications and broader platform alerts can be reasoned about within one shared model.
43. The final design clearly distinguishes "notification center" behavior from deep workflow pages.

## How This Informs Downstream Layers

### Layer 1 — IA Taxonomy Cleanup
Sections are named and grouped by functional domain (Q2). Zone labels corrected to match the functional taxonomy. The "section hub" pattern (Layer 1 module 1.2) is scoped per-section — a hub is introduced where a section has enough surface area to warrant one; otherwise the section goes directly to its primary page.

### Layer 2 — Role & Permission Model
The Layer 2 role-gating audit (2.4) and permission boundary corrections (2.3) enforce strict isolation (Q5) against the 0.2 boundary matrix. Any route or nav item that exposes a surface to a persona not gated for it in 0.2 is a bug.

### Layer 3 — Navigation Restructure
Layer 3 implements Q1 (single role-filtered sidebar), Q2 (functional groups), Q4 (shallow breadcrumbs), Q6 (6–7 section ceiling), Q7 (header switcher + portfolio page), Q8 (canonical homes), and Q9 (central inbox). The sidebar design must gate items per-role per the 0.2 matrix and the Q5 isolation principle.

### Layer 4 — Feature Gaps
Module 4.1 (Cross-association alert engine) aligns with Q9 — alerts surface in the central updates/inbox for cross-cutting signals, not as per-section noise. Module 4.2 (Owner Portal gaps) remains bounded by 0.2's "Current live capabilities" list.

### Layer 5 — Polish & Hardening
Mobile nav behavior (5.3) respects Q4 (breadcrumbs may collapse on mobile) and Q1 (the same canonical nav model is adapted, not replaced, for mobile).
