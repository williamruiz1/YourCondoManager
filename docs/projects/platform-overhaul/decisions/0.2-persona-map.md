## Context

Every structural decision in Layers 1–5 — nav topology, route gating, section ownership, feature scope — must be grounded in a clear understanding of who uses each surface and what they need. Without an explicit persona map locked at Layer 0, nav restructure and IA taxonomy work will make implicit persona assumptions that conflict with each other.

This document defines the six active YCM personas, their workspace boundaries, their current capabilities, and how they inform downstream IA and navigation decisions. It is the canonical source of truth for persona scope across the overhaul.

---

## The Six Active Personas

### Persona 1 — Manager

**Who they are:** The professional or self-managing volunteer responsible for day-to-day operation of one or more condo associations. In Track 1 (self-managed association), this is the board president or designated administrator. In Track 2+ (multi-association portfolio), this is a management company employee or owner.

**Primary workspace:** `/app` — the full operator workspace

**Role values in system:** `manager`

**What they own:**
- All associations in their account
- Full access to every section of the operator workspace: People, Units, Financials, Documents, Governance, Operations, Communications, Elections, Settings
- Portfolio Health (`/app/portfolio`) — the only persona with multi-association aggregated view
- Cross-association alert triage from the Home page
- **PM-managed associations only:** Assisted Board and PM Assistant feature toggle configuration — the PM sets per-feature View and Write access for Assisted Board members and PM Assistants on a per-association basis (see PM Toggle Configuration Model below)

**What they do not own:**
- Owner Portal (they configure it but do not use it as an owner)
- Board Officer / Assisted Board workspace (they operate above it — the Manager view contains a superset of what Board Officer / Assisted Board can see)

**Session context:** Operates with an active-association context switcher when managing multiple associations. Cross-association signals (alerts, portfolio health) are visible regardless of active context.

**Cognitive load target:** High — this persona operates a complex tool and expects density. Simplification is for clarity, not feature removal.

---

### Persona 2 — Board Officer

**Who they are:** An elected or appointed board member (president, treasurer, secretary, member-at-large) in a **self-managed** association where no property manager exists. The board IS the manager.

**Primary workspace:** `/app` — full Manager-equivalent access

**Role values in system:** `board-officer`

**What they own:**
- One association (enforced by redirect: `board-officer` with ≤1 association → `/app`, not `/app/portfolio`)
- Full access to every section of the operator workspace — equivalent to Manager: People, Units, Financials, Documents, Governance, Operations, Communications, Elections, Settings
- No feature restrictions apply — all sections, all write capabilities, all settings are available

**What they do not own:**
- Portfolio Health — redirected away from `/app/portfolio` (behavior confirmed intentional in 0.1)
- Owner Portal (they access it separately as a unit owner if applicable, through the portal login flow — not through the operator workspace)
- Platform / admin surfaces (never)

**Session context:** No association context switcher. Single-association focus throughout. The workspace visually and functionally scopes to one association only.

**Cognitive load target:** Medium — board members are volunteers, not operators. However, in self-managed mode they have full access and are expected to function as the operator.

---

### Persona 3 — Assisted Board

**Who they are:** An elected or appointed board member (president, treasurer, secretary, member-at-large) in a **PM-managed** association. A property manager handles all day-to-day operations. The Assisted Board member's role is **governance and oversight only** — they approve, review, and decide. They do not execute operational tasks.

**Primary workspace:** `/app` — scoped view governed by PM-Managed Default Access Table + PM toggles

**Role values in system:** `assisted-board`

**What they own:**
- One association (enforced by redirect: `assisted-board` with ≤1 association → `/app`, not `/app/portfolio`)
- Access to the operator workspace sections defined by the PM-Managed Default Access Table below, plus any expansions configured by PM toggles

**What they do not own:**
- Portfolio Health — redirected away from `/app/portfolio` (behavior confirmed intentional in 0.1)
- Settings — PM toggle cannot grant access to `/app/settings/*`
- Unit management — PM operational function only
- Platform / admin surfaces (never, regardless of PM toggle)
- Owner Portal (they access it separately as a unit owner if applicable, through the portal login flow — not through the operator workspace)

**Session context:** No association context switcher. Single-association focus throughout. Simplified sidebar (subset-render per 0.3 Q1).

**Cognitive load target:** Medium — board members are volunteers, not operators. Section visibility should be scoped to their relevant areas. Sections they cannot act on should not be prominent.

**Known gap (flagged by 0.1):** The `assisted-board` silent redirect from `/app/portfolio` to `/app` was confirmed intentional in 0.1 and must be preserved with an explicit code comment in `App.tsx`.

### PM-Managed Default Access Table

This is the canonical default access baseline for Assisted Board in a PM-managed association. Every row is authoritative. The PM may expand or restrict any row via the toggle model below.

| Domain | View (default) | Write (default) | Rationale |
|---|---|---|---|
| Financials — reports & statements | ✅ Yes | ❌ No | Board reviews; PM executes |
| Budget — annual approval | ✅ Yes | ✅ Approve action only | Board votes on annual budget |
| Meetings & minutes | ✅ Yes | ✅ Yes | Board sets agenda, presides, approves minutes |
| Governance documents (bylaws, rules, resolutions) | ✅ Yes | ❌ No | Board is the authority but PM files |
| Violations — view & appeals | ✅ Yes | ❌ No | Board hears appeals; PM issues notices |
| Maintenance requests — view only | ✅ Yes | ❌ No | Board approves capital projects; PM coordinates |
| Vendor contracts — view & approve | ✅ Yes | ✅ Approve action only | Board approves contracts over threshold |
| Announcements | ✅ Yes | ✅ Yes | Board communicates policy to owners |
| Owner directory — limited view | ✅ Yes (limited) | ❌ No | Board communicates with owners; no sensitive data |
| Reports & dashboards | ✅ Yes | ❌ No | Board monitors community health |
| Unit management | ❌ No | ❌ No | PM operational function only |
| Platform / admin surfaces | ❌ No | ❌ No | Operator-level only; not configurable via toggle |

### PM Toggle Configuration Model

Targets: Assisted Board and PM Assistant.

In PM-managed associations, the PM configures Assisted Board and PM Assistant access beyond the defaults above. The following rules are binding:

1. **Per-feature, per-permission-level** — View and Write are separate toggles, never bundled
2. **Association-scoped** — toggles apply to all Assisted Board members (or all PM Assistants) in a given association; per-member granularity is a future question, not decided here
3. **PM can expand access** — e.g., allow board to write violations if the association charter requires it
4. **PM can restrict access** — e.g., hide financials from board members if association governance is tighter
5. **YCM defaults** — the table above is the default; PM only overrides when needed
6. **PM-only action** — Assisted Board members and PM Assistants cannot self-configure their own access
7. **Platform and admin surfaces are never configurable** — no toggle may grant Assisted Board or PM Assistant access to `/app/platform/*` or `/app/admin/*`

### Persona 4 — PM Assistant

**Who they are:** PM-company junior staff member with controlled access to the operator workspace. Not a board member. Works under the Manager's supervision with a configured subset of Manager surfaces.

**Primary workspace:** `/app` — Manager-subset access configured by Manager via PM-internal toggles

**Role values in system:** `pm-assistant`

**What they own:**
- Configured subset of Manager surfaces (per PM toggle configuration)
- Access scope determined by Manager via the PM Toggle Configuration Model

**What they do not own:**
- Settings (unless Manager grants via toggle)
- Platform / admin surfaces (never, regardless of toggle)
- Portfolio Health — redirected away from `/app/portfolio`
- Owner Portal

**Session context:** Operates within a single association or multi-association scope depending on Manager configuration. PM-internal toggles govern surface visibility.

**Cognitive load target:** Medium — PM Assistants are operational staff, not strategic decision-makers. Surface visibility should match their configured responsibilities.

---

### Persona 5 — Owner (Owner Portal)

**Who they are:** A condo unit owner (or occupant with portal access) who accesses YCM through the Owner Portal — a separate, authenticated surface distinct from the operator workspace. They are not managers or board members (though a board member who is also an owner may access both surfaces through separate logins).

**Primary workspace:** `/portal` — Owner Portal (standalone, not inside `/app`)

**Role values in system:** `owner` (portal access, not operator role)

**Current live capabilities (Beta):**
- OTP login with multi-association selection
- Overview / KPI summary (unit, balance, open maintenance count)
- Owner profile management (contact info, emergency contact, communication preferences, SMS opt-in)
- Occupancy reporting (owner-occupied vs. tenant, tenant contact info)
- Maintenance request submission and status tracking (with progress timeline)
- Notices (read/unread, full-text expand)
- Communications inbox
- Elections: voting, proxy designation/revocation, self-nomination
- Documents (view and download)
- Financials: balance summary, ledger entry history
- ACH payment initiation (via Stripe Checkout)
- Payment methods management (add bank account via Stripe setup flow)
- Autopay enrollment, pause, cancel
- Push notifications (VAPID, service worker)
- Amenity booking link (external route `/portal/amenities`)
- Community Hub link (external route `/community/:associationId`)

**Beta gaps — explicitly scoped OUT of this overhaul:**
The following capabilities are acknowledged as future work but are **not** in scope for Layers 1–5 of this overhaul. No nav slots, IA taxonomy entries, or feature specs should be written for them until they have their own Layer 0 decision doc.
- HOA payment processing (real money movement end-to-end, beyond initiation)
- Bill history / formal statement and invoice view
- Any Owner Portal integrations not listed in "Current live capabilities" above

**Session context:** Separate authentication from the operator workspace. Portal access ID is the session token. No operator workspace nav is visible. The Owner Portal is a fully isolated surface.

**Cognitive load target:** Low — owners are residents, not operators. The portal must be simple, mobile-first, and scannable. Dense tables and administrative controls are not appropriate here.

### Persona 6 — Platform Admin

**Who they are:** YCM staff or PM employee with controlled internal access. Not a customer persona. Platform Admin operates above the customer role tiers — they do not manage a single association's day-to-day work (that is Manager / Board Officer / Assisted Board); they administer YCM platform configuration, cross-tenant operations, ingestion pipelines, admin-user lifecycle, and executive reporting. A PM employee who also manages a specific association holds both `platform-admin` and `manager` roles on the relevant tenant contexts; the two are separate capability boundaries and gate independently.

**Primary workspace:** `/app/platform/*`, `/app/admin/*`, `/app/ai/*` — the operator-tier surfaces inside `/app` that are invisible to customer personas.

**Role values in system:** `platform-admin` (operator role, `shared/schema.ts:161` `adminUserRoleEnum`).

**Scope and intent:**
- Prevent association sabotage and human error. Platform Admin is the only persona that can reach tenant-config, permission envelopes, ingestion lifecycle controls, admin-user role changes, executive decks, and platform diagnostics.
- Separate capability boundary from Manager. Manager owns a customer association's `/app/*` workspace (People, Units, Financials, Documents, Governance, Operations, Communications, Elections, Settings). Platform Admin owns YCM-internal operator tooling. The two personas overlap on a single person only when that person holds both roles; they are not additive by default.

**What they own:**
- `/app/platform/*` — Platform Controls (tenant config, permission envelopes, email/SMS/push provider configuration, policy, logs, diagnostics)
- `/app/admin/*` — Admin Roadmap (`/app/admin`, `/app/admin/roadmap`), Admin Users (`/app/admin/users`), Executive Deck (`/app/admin/executive`)
- `/app/ai/ingestion` — AI Ingestion lifecycle and review queue
- Any diagnostic/observability endpoint surfaced on `/api/platform/*`, `/api/admin/*`, or `/api/ai/*` namespaces

**What they do not own:**
- Customer association content — People, Units, Financials, Governance, Operations, Communications, Elections in the customer tenant scope. A Platform Admin who also holds `manager` on a tenant accesses customer content via the `manager` role, not `platform-admin`.
- Owner Portal — no operator-side launcher or shortcut to `/portal`. Platform Admin users who need to inspect Owner Portal content authenticate via the same portal-access-id flow as any Owner, in a separate session.
- Settings (`/app/settings/*`) — Settings is owned by Manager (customer-tenant scope). Platform Admin's configuration surfaces live under `/app/platform/*`, not `/app/settings/*`.

**Session context:** Operator session identical to Manager / Board Officer / Assisted Board (same auth cookie, same `admin_users` table). No separate login surface. Gate is the `platform-admin` role value alone.

**Cognitive load target:** High — Platform Admin operates a complex, consequential surface. No simplification at the expense of observability or control.

**Viewer capability variant:** The `viewer` role (`adminUserRoleEnum.viewer` at `shared/schema.ts:161,165`) is a read-only capability variant. When applied to a `platform-admin`-context user, `viewer` scopes to read-only access across the Platform Admin surfaces. See 2.1 Q2 for the capability-variant framing.

## How This Informs Downstream Layers

### Layer 1 — IA Taxonomy Cleanup
Zone and section labels must distinguish between Manager-only sections, Board Officer / Assisted Board-visible sections, and PM-toggle-configurable sections. Taxonomy entries must not imply Owner Portal content belongs inside the operator workspace.

### Layer 2 — Role & Permission Model
The permission boundary corrections (2.3) and role-gating audit (2.4) use this matrix as the source of truth. Any gating in `App.tsx` that contradicts this matrix is a bug, not a feature. The PM Toggle Configuration Model is a new system capability that must be architectured in Layer 2 — specifically: where toggle state is stored, how it is evaluated at route and component level, and how it is administered in the PM settings surface.

### Layer 3 — Navigation Restructure
The Manager sidebar and the Assisted Board sidebar may share a shell but must scope nav items to what each persona owns. In PM-managed mode, Assisted Board sidebar items must reflect the default access table and be responsive to PM toggle state. Assisted Board must not see Portfolio Health in the sidebar. Owner Portal is a separate nav system entirely — it has its own sidebar defined in `owner-portal.tsx` and is not part of the operator workspace sidebar redesign.

### Layer 4 — Feature Gaps
Module 4.2 (Owner Portal gaps) must use the "Beta gaps" list in this document as its starting scope. No Owner Portal feature gap spec may be written for a capability not in the current live list without a new Layer 0 decision authorizing it.

## Acceptance Criteria

These criteria are verified during Layer 2 and Layer 3 implementation:

1. No route inside `/app/*` is accessible to a user authenticated only through the Owner Portal portal-access-id flow
2. No route inside `/portal/*` requires an operator workspace session
3. A `board-officer` or `assisted-board` user navigating to `/app/portfolio` is redirected to `/app` — behavior preserved from 0.1
4. An `assisted-board` user does not see a "Portfolio Health" nav item in the sidebar
5. An `assisted-board` user does not see a Settings nav item in the sidebar
6. An `assisted-board` user sees only the sections defined in the PM-Managed Default Access Table by default, with PM toggle state applied at runtime
7. A `board-officer` user has full Manager-equivalent access — no section restrictions apply, including Settings
8. No PM toggle may grant `assisted-board` or `pm-assistant` access to `/app/platform/*`, `/app/admin/*`, or `/app/settings/*`
9. The Owner Portal sidebar contains exactly the sections listed under "Current live capabilities" — no nav items for beta-gap features (HOA payment processing, bill history/statements) are present
10. Layer 2 permission boundary corrections treat this matrix as the authoritative gating spec — any deviation requires a new decision doc, not an inline code judgment
11. A `platform-admin` user is the only persona with access to `/app/platform/*`, `/app/admin/*`, and `/app/ai/*`; Manager, Board Officer, Assisted Board, PM Assistant, and Owner have ❌ on these surfaces
12. No `/app/platform/*`, `/app/admin/*`, or `/app/ai/*` API endpoint accepts a role other than `platform-admin` without an explicit spec approval in a new locked decision doc (0.3 Q5 AC 22 requirement)
13. No operator-side nav surface contains a link to `/portal` — Platform Admin users needing Owner Portal content authenticate via portal-access-id in a separate session
14. A `pm-assistant` user sees only the surfaces configured by Manager via PM toggle — unconfigured surfaces are hidden
15. A `pm-assistant` user does not see Settings unless Manager explicitly grants access via toggle
