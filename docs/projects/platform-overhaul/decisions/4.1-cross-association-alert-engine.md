## Context

0.1 AC 4 (LOCKED) mandates: "The Home page alerts panel surfaces overdue work orders and due maintenance schedule instances regardless of the currently active association context" (`docs/projects/platform-overhaul/decisions/0.1-dashboard-resolution.md:126,169`). 0.3 Q9 (LOCKED) established a central updates/inbox model: "YCM uses a central updates model for cross-cutting alerts, notices, and important communications, while detailed work remains in its native section. This creates one reliable place for users to check what needs attention" (`docs/projects/platform-overhaul/decisions/0.3-navigation-model.md:116-119`). 0.3 Q9 AC 39-43 define the inbox contract: a central updates/inbox concept exists, critical items from multiple zones surface there, it does not replace section-native workflows, and it clearly distinguishes notification-center behavior from deep workflow pages (`docs/projects/platform-overhaul/decisions/0.3-navigation-model.md:199-205`).

1.1 (LOCKED, amended `ac446c0`) locked six functional zones for `/app`: Home, Financials, Operations, Governance, Communications, Platform (`docs/projects/platform-overhaul/decisions/1.1-zone-taxonomy-corrections.md:44-57`). Alerts must route to zones — each alert type is owned by a zone and surfaces in the zone's hub page and/or the central inbox. 1.2 Q2 (LOCKED) allows hubs to contain "optionally one lightweight summary widget where it adds genuine value (e.g., outstanding balance in Financials, open requests in Operations)" (`docs/projects/platform-overhaul/decisions/1.2-section-hub-reclassification.md:59`). This optional hub widget is the mechanism by which zone-specific alerts reach zone hub pages.

3.1 Q5 (LOCKED) established the SUBSET-RENDER sidebar model: Assisted Board and PM Assistant get a completely separate simplified sidebar where only permitted zones render; no greyed-out or hidden locked sections (`docs/projects/platform-overhaul/decisions/3.1-sidebar-redesign.md:157-159`). Alerts are visible only for zones the active persona is permitted to see. A Manager sees alerts across all six zones; an Assisted Board user sees alerts only for zones their 0.2 default access + PM toggle state permits.

0.2 (LOCKED, amended `f8dbf76`) defines six personas — Manager, Board Officer, Assisted Board, PM Assistant, Owner, Platform Admin — with an explicit surface-ownership matrix (`docs/projects/platform-overhaul/decisions/0.2-persona-map.md:249-267`). Alert visibility per persona must conform to this matrix. 0.3 Q5 (LOCKED) mandates strict role isolation: "Pages and navigation items are hidden when the role lacks access" (`docs/projects/platform-overhaul/decisions/0.3-navigation-model.md:97-99`).

Module 4.1 designs the cross-association alert engine that:
1. Surfaces alerts on the Home page (`/app`) regardless of active association context (0.1 AC 4).
2. Feeds the central inbox / notification center (0.3 Q9).
3. Routes per-zone signals to optional zone hub summary widgets (1.2 Q2).
4. Respects the SUBSET-RENDER sidebar model (3.1 Q5) — alerts for hidden zones are invisible.

## Conflict Analysis

Nine decision axes surface from the locked foundations. Each maps to an open question below.

1. **Alert source catalog.** Which API endpoints and data models constitute the canonical alert sources? The Home alerts panel, zone hubs, and central inbox all consume alerts — but no locked doc enumerates the full source list beyond 0.1's two explicit signals (overdue work orders, due maintenance). Maps to **Q1**.

2. **Zone ownership of alert types.** Each alert type must be assigned to exactly one zone (per 1.1 Q2 intent rubric). Some signals (e.g., a financial delinquency that triggers a governance violation) cross zones. The routing rule is unresolved. Maps to **Q2**.

3. **Central inbox vs Home alerts panel.** 0.1 AC 4 mandates Home alerts. 0.3 Q9 mandates a central inbox. Whether these are the same surface, overlapping surfaces, or complementary surfaces with different content is undefined. Maps to **Q3**.

4. **Central inbox zone assignment and URL.** 1.1 Q3 places "updates inbox" under Communications zone. 0.3 Q9 AC 42 says "Owner notices/communications and broader platform alerts can be reasoned about within one shared model." Whether the inbox lives at a Communications sub-URL, a standalone `/app/inbox` route, or is a panel/drawer on Home is unresolved. Maps to **Q4**.

5. **Per-persona alert visibility filtering.** The SUBSET-RENDER model (3.1 Q5) hides zones Assisted Board and PM Assistant cannot access. Must the alert engine also filter individual alert items within a visible zone (e.g., Assisted Board can see Operations zone but only maintenance-request alerts, not vendor alerts)? Or is zone-level filtering sufficient? Maps to **Q5**.

6. **Cross-association aggregation model.** 0.1 AC 4 says alerts surface "regardless of the currently active association context." For a Manager with 15 associations, does the Home alerts panel fetch and merge alerts from all 15 simultaneously, or does it use a server-side aggregation endpoint? Performance and staleness implications differ. Maps to **Q6**.

7. **Alert deduplication policy.** An alert may appear on the Home panel, the zone hub widget, and the central inbox. Should it appear in all three places simultaneously, or should dismissing/reading it in one place clear it everywhere? Maps to **Q7**.

8. **Alert staleness and refresh model.** Cross-association alerts may change frequently (work orders completed, maintenance performed). What is the staleness tolerance — real-time push (websocket/SSE), polling interval, or fetch-on-navigate? Maps to **Q8**.

9. **Zone hub summary widget alert contract.** 1.2 Q2 allows "optionally one lightweight summary widget" per hub. 3.2 created four hub pages. What data does each hub widget display, and how does it relate to the central alert engine vs. the hub's own section APIs? Maps to **Q9**.

## Open Questions

### Q1 — What is the canonical alert source catalog?

0.1 AC 4 (`docs/projects/platform-overhaul/decisions/0.1-dashboard-resolution.md:169`) names two explicit sources: overdue work orders and due maintenance schedule instances. The current `dashboard.tsx` also renders active elections (`docs/projects/platform-overhaul/decisions/0.1-dashboard-resolution.md:24`). The Operations Dashboard aggregates six summary totals, work-order aging buckets, vendor status counts, recent work orders, due maintenance, and recent inspections (`docs/projects/platform-overhaul/decisions/0.1-dashboard-resolution.md:53`). 0.3 Q9 says "critical items from multiple zones can surface there" (`docs/projects/platform-overhaul/decisions/0.3-navigation-model.md:200`).

**What is the exhaustive list of alert sources (API endpoints and data models) the alert engine consumes?** Which of the sources listed in §Current State D are in scope for 4.1, which are deferred to future modules, and which are explicitly excluded? Are there additional sources not yet catalogued?

**Tradeoff axis:** A broad catalog (all 10+ sources) creates a comprehensive alert surface but increases API surface area, complexity, and staleness management. A narrow catalog (only the two 0.1-mandated signals) is safe to ship but underdelivers on 0.3 Q9's central inbox vision.

### Q2 — Which zone owns which alert type?

Per 1.1 Q2 (`docs/projects/platform-overhaul/decisions/1.1-zone-taxonomy-corrections.md:59-66`), assignment follows the intent rubric: "What is the user trying to accomplish on this page? Which zone label best describes that intent?" For alerts, the question becomes: "What zone does this alert's resolution action belong to?" A financial delinquency alert resolves in Financials; a governance-document expiry alert resolves in Governance. But some alerts are cross-zone (e.g., a vendor contract approval that is both Operations and Governance per 0.2 PM-Managed Default Access Table `docs/projects/platform-overhaul/decisions/0.2-persona-map.md:127`).

**What is the canonical zone-to-alert-type mapping table?** How are cross-zone alerts handled — primary zone assignment with a secondary zone reference, or duplicate entries?

**Tradeoff axis:** Single-zone ownership (each alert assigned to exactly one zone) keeps the model clean but may hide cross-zone relevance. Dual-zone tagging increases complexity but surfaces alerts in all relevant contexts.

### Q3 — What is the relationship between the Home alerts panel and the central inbox?

0.1 AC 4 (`docs/projects/platform-overhaul/decisions/0.1-dashboard-resolution.md:169`) mandates cross-association alerts on Home. 0.3 Q9 (`docs/projects/platform-overhaul/decisions/0.3-navigation-model.md:116-119`) mandates a central updates/inbox. 0.3 Q9 AC 41 says "The inbox does not replace detailed section-native workflows" (`docs/projects/platform-overhaul/decisions/0.3-navigation-model.md:201`). 0.3 Q9 AC 43 says "The final design clearly distinguishes 'notification center' behavior from deep workflow pages" (`docs/projects/platform-overhaul/decisions/0.3-navigation-model.md:204`).

**Are the Home alerts panel and the central inbox the same component, overlapping surfaces, or separate surfaces with distinct content scopes?** If separate, what is the content boundary — Home shows high-priority/actionable alerts while inbox shows the full history?

**Tradeoff axis:** Single surface (Home IS the inbox) minimizes implementation but may overload Home. Dual surface (Home shows a summary, inbox shows full history) adds a navigation destination but clearly separates triage from archive. Overlapping content (inbox is a superset of Home) risks confusion about where to go.

### Q4 — Where does the central inbox live (URL, zone, rendering model)?

1.1 Q3 (`docs/projects/platform-overhaul/decisions/1.1-zone-taxonomy-corrections.md:76`) places "updates inbox" under the Communications zone definition. 3.2 created `/app/communications` as the Communications hub and `/app/communications/overview` for the current communications content (`docs/projects/platform-overhaul/decisions/3.2-route-restructure.md:188-190`). No inbox route exists today.

**Should the inbox be (a) a sub-page under Communications (`/app/communications/inbox`), (b) a top-level route outside any zone (`/app/inbox`), (c) a persistent drawer/panel accessible from the header, or (d) absorbed into the Home alerts panel with no separate route?**

**Tradeoff axis:** (a) is consistent with 1.1 Q3 zone assignment but buries a cross-zone surface inside one zone. (b) creates a standalone route but requires a zone exception or a seventh "zone" slot. (c) provides ambient access without navigation but adds UI complexity. (d) is simplest but may not satisfy 0.3 Q9's "one reliable place" vision.

### Q5 — Is zone-level alert filtering sufficient, or do individual alert items need per-feature filtering for Assisted Board and PM Assistant?

3.1 Q5 SUBSET-RENDER (`docs/projects/platform-overhaul/decisions/3.1-sidebar-redesign.md:157-159`): Assisted Board sees only permitted zones. 0.2 PM-Managed Default Access Table (`docs/projects/platform-overhaul/decisions/0.2-persona-map.md:117-131`) grants Assisted Board View access to "Maintenance requests — view only" but NOT "Unit management." Both are in the Operations zone per 3.1 Q12 (`docs/projects/platform-overhaul/decisions/3.1-sidebar-redesign.md:288-297`). If an Operations alert is about a unit record (e.g., "Unit 4B insurance expired"), Assisted Board should not see it even though they can see the Operations zone.

**Must alert filtering operate at the feature-domain level (per PM-Managed Default Access Table row) in addition to zone level?** Or is zone-level filtering acceptable as the minimum viable model, with feature-domain filtering deferred?

**Tradeoff axis:** Zone-level filtering is simple and aligns with the sidebar model. Feature-domain filtering is more precise but requires the alert engine to evaluate PM toggle state per alert item, which couples it deeply to the 0.2 toggle model. 3.1 Q6 (`docs/projects/platform-overhaul/decisions/3.1-sidebar-redesign.md:183-188`) already specifies a `usePersonaToggles` hook — the alert engine could reuse it.

### Q6 — What is the cross-association aggregation model?

0.1 AC 4 (`docs/projects/platform-overhaul/decisions/0.1-dashboard-resolution.md:126`): "cross-association operational alerts on the Home page — overdue work orders, due maintenance schedules, and any other operational signals must surface here regardless of which association is currently active in context." For a Manager with N associations, the alert engine must aggregate across all N.

**Does the alert engine (a) make N parallel API calls client-side and merge, (b) call a single server-side aggregation endpoint that returns cross-association alerts, or (c) use a server-side push model (websocket/SSE) that streams alerts as they arise?**

**Tradeoff axis:** (a) is simplest but scales poorly (N API calls on every Home load). (b) requires a new server endpoint but is the standard SaaS pattern — one call returns all alerts the persona can see. (c) provides real-time updates but adds infrastructure complexity. The existing Operations Dashboard uses pattern (b) — a single `GET /api/operations/dashboard` endpoint (`docs/projects/platform-overhaul/decisions/0.1-dashboard-resolution.md:53`).

### Q7 — What is the alert deduplication and read-state policy?

An "overdue work order" alert may appear on (1) the Home alerts panel, (2) the Operations hub summary widget, and (3) the central inbox. 0.3 Q9 AC 41 (`docs/projects/platform-overhaul/decisions/0.3-navigation-model.md:201`) says "The inbox does not replace detailed section-native workflows." This implies the inbox is supplementary — the same signal appears in both the inbox and the zone-native surface.

**When a user reads or dismisses an alert in one surface, does it clear in all surfaces? Is there a global alert read-state, or are the three surfaces (Home, hub widget, inbox) independent views of the same underlying data?**

**Tradeoff axis:** Global read-state (read-once-clears-everywhere) provides a clean UX but requires a shared state layer. Independent views (each surface shows the raw signal regardless of interaction in other surfaces) are simpler but may show stale "unread" badges after the user has already triaged an item in a different surface.

### Q8 — What is the alert staleness and refresh model?

Cross-association alerts change as work orders are completed, maintenance is performed, elections close, payments are received. The alert engine must decide how fresh alerts are.

**What is the refresh model — (a) fetch-on-navigate (alerts refresh when the user navigates to Home or opens the inbox), (b) polling at a fixed interval (e.g., every 60s), (c) server-sent events / websocket push, or (d) a hybrid (fetch-on-navigate + polling while the page is active)?**

**Tradeoff axis:** (a) is simplest and cheapest but alerts go stale if the user sits on Home for 30 minutes. (b) adds background load but ensures freshness. (c) is the most responsive but requires infrastructure (the existing push notification system at `POST /api/portal/push/subscribe` is VAPID-based for the Owner Portal — a different model). (d) balances freshness and cost.

### Q9 — What alert signal does each zone hub summary widget display?

1.2 Q2 (`docs/projects/platform-overhaul/decisions/1.2-section-hub-reclassification.md:59`) allows "optionally one lightweight summary widget" per hub. 3.2 Q2 (`docs/projects/platform-overhaul/decisions/3.2-route-restructure.md:159-162`) notes the Operations hub may include "optionally one lightweight summary widget." The same applies to Financials, Governance, and Communications hubs.

**For each of the four zone hubs, should a summary widget be included? If so, what data does it display — a count badge (e.g., "3 overdue work orders"), a mini-list (top 3 alerts), or a status indicator (e.g., "All clear" / "Attention needed")? Does the widget pull from the central alert engine or from the zone's own section API?**

**Tradeoff axis:** Count badges are minimal and cheap but low-information. Mini-lists provide actionable context but may duplicate Home alerts panel content. Status indicators are simple but binary. Pulling from the alert engine creates a single data source; pulling from zone APIs is simpler per-hub but fragments the alert model.

## How This Informs Downstream Layers

### Layer 4 — 4.2 (Owner Portal gaps)

### Layer 4 — PM toggle configuration surface

If Q5 resolves to feature-domain-level alert filtering, the toggle configuration surface must also govern which alerts Assisted Board and PM Assistant see — the toggle model controls both sidebar visibility (3.1 Q5/Q6) and alert visibility (4.1 Q5).

### Layer 5 — Mobile / polish (5.3)

### Layer 5 — Performance (5.x)
Q6 (aggregation model) and Q8 (refresh model) have direct performance implications. If 4.1 introduces a server-side aggregation endpoint, 5.x must benchmark it against multi-association portfolios (10+ associations).
