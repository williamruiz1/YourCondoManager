## Context

1.4 defines the canonical title catalog per route, the `document.title` format, the single title-setter hook, and the governance rules that keep titles and breadcrumbs in sync. No title-setter hook exists today; every `/app/*` and `/portal/*` tab shares the static marketing string in `client/index.html:6`. Three competing title surfaces coexist (`WorkspacePageHeader`, raw `<h1>`, dynamic greetings). 1.4 collapses all of this onto a single convention.

---

## Current State

- **No `useDocumentTitle` hook exists.** `client/src/hooks/` contains only `use-active-association`, `use-mobile`, `use-residential-dataset`, `use-toast`, `use-user-settings`.
- **`document.title` is static.** `client/index.html:6` sets the marketing string; no `/app/*` or `/portal/*` page mutates it.
- **Three title surfaces:** `WorkspacePageHeader` (most `/app/*`), raw `<h1>` (`portfolio.tsx:263`, `user-settings.tsx:352`, `ai-ingestion.tsx:1204`, `associations.tsx:285`, `persons.tsx:401`, `new-association.tsx:149`, `onboarding-invite.tsx:127`, all `/portal/*`), dynamic data-driven titles (election-detail, election-ballot).
- **Live 0.1 violations:** `dashboard.tsx:630` renders "Dashboard"; `portfolio.tsx:263` renders "Portfolio at a Glance"; `operations-dashboard.tsx:98` renders "Operations Dashboard".
- **Pages audited: 44** (35 `/app/*` operator pages + 9 `/portal/*` sub-views).

## Selected Resolutions

### Q1 тАФ What is the document title format string?

**Resolution:** `{Page Title} тАФ YCM`. Em dash, not hyphen. One space on each side of the dash. The word "YCM" is always uppercase. No other format is valid.

**Acceptance Criteria**
- Every page sets `document.title` in the format `{Page Title} тАФ YCM`
- The separator is an em dash (тАФ), not a hyphen (-) or en dash (тАУ)
- "YCM" is always uppercase
- No page deviates from this format

### Q2 тАФ What is the `useDocumentTitle` hook specification?

**Resolution:** The hook is named `useDocumentTitle`. It accepts one argument: `title` (string). It sets `document.title` to `${title} тАФ YCM` on mount and whenever `title` changes. It lives at `client/src/hooks/useDocumentTitle.ts`. It is the only mechanism in the codebase for setting `document.title`.

**Acceptance Criteria**
- Hook is named `useDocumentTitle`
- Hook lives at `client/src/hooks/useDocumentTitle.ts`
- Hook accepts one `title: string` argument
- Hook sets `document.title` to `${title} тАФ YCM`
- Hook updates on title change
- No other mechanism sets `document.title` in the codebase

### Q3 тАФ What is the display title (`<h1>`) rule?

**Resolution:** Every page has exactly one `<h1>`. The `<h1>` content matches the `title` argument passed to `useDocumentTitle` exactly. Display title and document title are always in sync.

**Acceptance Criteria**
- Every page has exactly one `<h1>`
- `<h1>` content matches the `useDocumentTitle` title argument exactly
- No page has a `<h1>` that differs from its document title

### Q4 тАФ What titles replace the six "Dashboard"-labeled pages?

**Resolution:**

|---|---|---|
| Dashboard (association-level) | Association Overview | Home |
| Dashboard (portfolio-level) | Portfolio Overview | Home |
| Dashboard (operations) | Operations Overview | Operations |

| Dashboard (board) | Board Overview | Home |

If the actual page content does not match the proposed title, VS Code agent surfaces the discrepancy before renaming. Do not rename to a title that does not reflect page content.

**Acceptance Criteria**
- All six "Dashboard"-labeled pages are renamed to their canonical titles
- Renamed titles match actual page content
- Any mismatch between proposed title and actual content is flagged before commit
- Breadcrumb labels on these six pages are updated to match (unblocking 1.3)

### Q5 тАФ Is "Dashboard" prohibited as a page title going forward?

**Resolution:** Yes. "Dashboard" is prohibited as a standalone page title in YCM. It may appear in component names or internal code references but never as a user-facing `<h1>` or `document.title`. Any future page that would be titled "Dashboard" must use a specific descriptive title instead.

**Acceptance Criteria**
- No page uses "Dashboard" as its `<h1>` or `document.title` after implementation
- This prohibition is noted in the decision doc for future reference
- Lint or code review process should catch violations тАФ a comment in `useDocumentTitle.ts` notes this rule

### Q6 тАФ Is the hook the only mechanism for setting document title?

**Resolution:** Yes. `useDocumentTitle` is the only permitted mechanism. Direct `document.title = ...` assignments are forbidden outside the hook. Any existing direct assignments are removed during 1.4 implementation.

**Acceptance Criteria**
- No `document.title =` assignment exists outside `useDocumentTitle.ts`
- All three live violations are remediated to use the hook
- A grep for `document.title` returns only the hook file after implementation

### Q7 тАФ Do portal pages follow the same title rules?

**Resolution:** Yes. All `/portal/*` pages follow identical rules: `useDocumentTitle` hook, `{Page Title} тАФ YCM` format, one `<h1>` matching the title. Portal page titles use first-person zone labels where appropriate (e.g., "My Finances," "My Requests") consistent with the Owner zone taxonomy in 1.1.

**Acceptance Criteria**
- All `/portal/*` pages use `useDocumentTitle`
- Portal document titles follow `{Page Title} тАФ YCM` format
- Portal `<h1>` matches document title
- Portal titles use first-person labels consistent with 1.1 Owner zone taxonomy
