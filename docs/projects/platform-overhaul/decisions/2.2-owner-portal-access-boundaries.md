## Context

0.2 locked Persona 3 (Owner) as a self-service persona whose primary workspace is `/portal`, whose role value in the system is `owner`, and whose live Beta capabilities are enumerated verbatim in 0.2 §Persona 3. 0.2 §Persona Boundary Matrix asserts that `/portal` is owned exclusively by the Owner persona and that Manager and Board Admin do NOT access `/portal` through the operator workspace — Board members who are also unit owners "access it separately as a unit owner if applicable, through the portal login flow — not through the operator workspace" (`0.2:62`). 0.3 Q5 (Strict role isolation) forbids cross-persona surfaces unless explicitly approved in spec.

1.1 Q5 locked the four Owner Portal zones as **Home, My Finances, My Requests, My Community** (`1.1:92-100`). 1.2 Q4 locked portal hub URLs as `/portal/finances`, `/portal/requests`, `/portal/community`, with `/portal` as the dashboard (not a hub) (`1.2:87-90`). 1.4 Q7 locked portal title parity with `/app/*` — `useDocumentTitle`, `{Page Title} — YCM` format, one `<h1>`, first-person labels per 1.1.

Module 2.2 audits whether the live `/portal/*` surface matches the locked 0.2 Owner persona scope and whether it conforms to the 1.x IA locks. It does not design access fixes — it surfaces the boundary decisions required before Layer 2 can close. The scope is (1) every `/portal/*` client route and owner-portal view, (2) every `/api/portal/*` endpoint and its auth/role enforcement, (3) every portal-originated mutation that changes association-scoped data, (4) the relationship between `portal_access.role` and the 0.2 persona enum (Owner only).

The audit's headline finding is that the live portal contains a full Board-Admin-in-portal subsurface (Board workspace UI and 29 `/api/portal/board/*` endpoints) that is not present in the 0.2 Beta capability list and is not modeled in 0.2 §Persona Boundary Matrix. Whether this subsurface is in scope for the overhaul, out of scope for the overhaul, or a boundary violation that must be removed is an open question — this skeleton does not resolve it.

## Conflict Analysis

### Conflict 1 — Board-in-portal subsurface vs 0.2 Persona Boundary Matrix

0.2 §Persona Boundary Matrix (`0.2:127`) lists `/portal` as `❌` for Board Admin ("separate login") and `✅ Full (beta scoped)` for Owner only. 0.2 Persona 3 §Current live capabilities (Beta) (`0.2:80-97`) does not include a Board workspace, board-scoped governance/meetings/documents/communications, or owner-ledger access. The live portal ships all of these via 29 `/api/portal/board/*` endpoints (§C2) and the `portal_access_role` enum value `board-member` (`shared/schema.ts:1084`).

> **⚠️ CROSS-REFERENCE NOTICE**
> **AC (0.2 AC 1, LOCKED):** "No route inside `/app/*` is accessible to a user authenticated only through the Owner Portal portal-access-id flow."
> **Conflicting evidence:** `/api/portal/board/*` endpoints expose operator-surface capabilities (association edit, meetings CRUD, owner-ledger entry creation, person/unit edit) to users authenticated via portal-access-id when that row carries `boardRoleId`. The surface is not in `/app/*` path-wise, but it delivers `/app/*` capabilities through a portal session — which contradicts the spirit of 0.2 AC 1 (portal sessions must not reach operator capabilities) and directly contradicts 0.2 §Persona 3 Beta capability list (board workspace is not listed).
> **AC (0.2 AC 6, LOCKED):** "The Owner Portal sidebar contains exactly the sections listed under 'Current live capabilities' — no nav items for beta-gap features (HOA payment processing, bill history/statements) are present."
> **Conflicting evidence:** The live owner-portal.tsx sidebar (§B) contains exactly the seven Owner tabs. It does NOT expose the board subsurface in the `/portal` client. The board subsurface is reachable only via API. Whether an unshown-but-reachable subsurface violates 0.2 AC 6 is a founder question (Q1). Read/write of the Board endpoints currently depends on client code we did not locate in `owner-portal.tsx` — the board dashboard query `["portal/board-dashboard"]` (`client/src/pages/owner-portal.tsx:699`) suggests a hidden UI or an incomplete merge.

Maps to **Q1**.

### Conflict 2 — `portal_access_role` enum vs 0.2 Owner-only persona

0.2 Persona 3 defines one portal role: `owner`. `shared/schema.ts:1084` defines four: `owner`, `tenant`, `readonly`, `board-member`. Three of the four are undocumented in 0.2. Whether tenants and read-only viewers are implicit Owner-persona sub-types, or are separate personas that require Layer 0 decisions of their own, is unresolved. Maps to **Q2**.

### Conflict 3 — Direct-commit owner mutations vs manager approval gates

0.2 Persona 3 describes owner-scope capabilities (profile edit, occupancy report, maintenance submit, payment, elections, etc.) but does not state whether any owner mutation requires manager approval before taking effect. The live portal uses a review-gate pattern for `contact-updates` (§E) but commits directly for occupancy, payment methods, payments, proxy/nominate, and `PATCH /me`. Whether the approval-gate pattern should expand, stay status-quo, or contract is a product decision. Maps to **Q3**.

### Conflict 4 — Portal reads-through-board-flag leak operator data

§F notes that `GET /api/portal/maintenance-requests` branches on `req.portalHasBoardAccess` and returns association-wide data to board-flagged portal sessions. If Q1 resolves to "board-in-portal is out of scope and must be removed," this branching logic is a bug; if Q1 resolves "board-in-portal is in scope but gated," the branch is fine but must be documented. Maps to **Q1** (primary) and **Q4** (branching-on-role policy).

### Conflict 5 — Unauthenticated public payment-link flow inside `/api/portal/*`

`/api/portal/payments/link/:token` (§C4) is intentionally public — a one-time token-authenticated payment surface. It predates the portal-access-id session model. Whether it belongs in `/api/portal/*` or should be relocated to a `/api/pay/*` or `/pay/:token` namespace is an IA question, not a boundary-security question. Maps to **Q5**.

### Conflict 6 — `/portal/amenities` is public, not gated

`client/src/App.tsx:357` registers `/portal/amenities` as a public route with no auth gate in App.tsx. Gating is presumably in-page. This mirrors the `/portal` pattern (in-page OTP login), but the distinct route means a bookmarked `/portal/amenities` link reaches the amenities component without establishing portal session context. Whether this is intentional (deep-link-then-login) or a boundary gap is unresolved. Maps to **Q6**.

### Conflict 7 — Portal nav zone labels vs 1.1 locked zone vocabulary

1.1 Q5 (`1.1:92-100`) locked portal zones as **Home, My Finances, My Requests, My Community**. The live sidebar labels are **Overview, Financials, Documents, Maintenance, Notices, Communications, Voting** (§B). None of these are first-person; none match the 1.1 four-zone vocabulary.

## Open Questions

### Q1 — What is the authoritative scope of `/portal`? Specifically, is the Board-in-portal subsurface (29 endpoints + board dashboard query in `owner-portal.tsx:699`) in scope, out of scope and retained read-only, or out of scope and removed?

**Tradeoff axis:** Board members who are also unit owners currently have a single-login experience with board read-only visibility. Removing the subsurface honors 0.2 strictly and pushes board members back to the operator workspace (`/app`) with a separate login. Retaining it requires amending 0.2 to introduce a second portal persona. The live `requirePortalBoardReadOnly` middleware (`server/routes.ts:1104-1106`) already hard-returns 403 on writes — the subsurface is effectively a read-only board view at the API layer, which could be the de facto resolution.

### Q2 — Does the `portal_access_role` enum (`owner | tenant | readonly | board-member`) require a 0.2 amendment, a narrowing to `owner` only, or a separate Layer 0 decision for non-owner portal personas?

**Tradeoff axis:** Narrowing breaks live `tenant` and `board-member` access records (migration cost). Amending 0.2 acknowledges multiple portal personas and requires Layer 1/2/3 rework to match. A separate Layer 0 decision keeps 0.2 unchanged but pushes the scope expansion into a new doc.

### Q3 — Which owner mutations require a manager approval gate (review-queue pattern) and which may commit directly?

**Tradeoff axis:** A strict approval gate (all mutations queued) increases manager workload and delays owner action but tightens association data integrity. A permissive model (direct commit) preserves owner UX but lets an owner alter shared records (occupancy, payment method, proxy designation) without manager awareness. Candidate gating policies: (a) status-quo (only contact-update is gated); (b) expand gate to occupancy and payment-method changes; (c) expand gate to all writes except owner's own profile and self-scoped records.

### Q4 — How are portal reads that branch on `portalHasBoardAccess` (e.g., `GET /api/portal/maintenance-requests`) governed?

**Tradeoff axis:** Inline branching on `portalHasBoardAccess` couples owner and board reads into the same endpoint with a role-conditional response. Splitting into dedicated `/api/portal/maintenance-requests` (owner) and `/api/portal/board/maintenance-requests` (board) endpoints makes the surface self-documenting but adds N endpoints. This question is downstream of Q1 — if board-in-portal is removed, Q4 is moot.

### Q5 — Does the public payment-link flow (`/api/portal/payments/link/:token`) belong inside `/api/portal/*`, or should it be relocated to a non-portal namespace?

**Tradeoff axis:** Leaving it in place avoids a breaking change to live payment-link emails. Relocating (e.g., to `/api/pay/:token`) clarifies that portal namespace == portal-access-id session, and the token-link is a separate one-shot flow. This is an IA question, not a security one — the token itself is the credential in either location.

### Q6 — Is `/portal/amenities` a deliberately public deep-link (auth-in-page) or a boundary gap? Should it inherit the `/portal` session-redirect pattern?

**Tradeoff axis:** Public deep-link preserves the ability to share amenity-booking links (e.g., in a notice). Session-required blocks share-links but guarantees portal-scope context before any UI renders.

### Q7 — When do 1.1/1.2/1.4 portal conformance (four first-person zones, hub URLs, `useDocumentTitle`) land relative to 2.2? Joint PR, 1.x-first, or 2.2-first?

**Tradeoff axis:** 2.2 access rework that changes URLs (Q5, Q6) should not collide with 1.2 hub URL introduction. If 2.2 lands first and creates new route boundaries, 1.2 implementation must rebase. Sequencing is a Layer-3 build-order question, flagged here for visibility.

## Selected Resolution

### Q1 — Selected Resolution

Q1 Selected Resolution is populated. Module status remains SKELETON pending full Q-set resolution.

Board Admin is an `/app` persona, not a `/portal` persona. There are two distinct Board Admin modes in YCM and they must be treated differently:

**Self-Managed Association:**
- The board IS the manager. No PM exists.
- Board Admin has full `/app` access equivalent to the manager role.
- No feature restrictions apply.
- In self-managed associations, Board Admin has Settings access — the restriction in 0.2 AC 4 applies to PM-managed mode only.

**PM-Managed Association:**
- A property manager handles all day-to-day operations.
- Board Admin's role is governance and oversight only — they approve, review, and decide. They do not execute.
- Board Admin gets a scoped default access set (see PM-Managed Default Access Table below), and the PM can configure per-feature View and/or Write access via toggles.

**PM-Managed Default Access Table** (authoritative baseline; embed verbatim wherever this access model is referenced):

| Domain | View (default) | Write (default) | Rationale |
|---|---|---|---|
| Financials — reports & statements | ✅ Yes | ❌ No | Board reviews; PM executes |
| Budget — annual approval | ✅ Yes | ✅ Approve action only | Board votes on annual budget |
| Meetings & minutes | ✅ Yes | ✅ Yes | Board sets agenda, presides, approves minutes |
| Governance documents (bylaws, rules, resolutions) | ✅ Yes | ❌ No | Board is the authority but PM files |
| Violations — view & appeals | ✅ Yes | ❌ No | Board hears appeals; PM issues notices |
| Maintenance requests — view only | ✅ Yes | ❌ No | Board approves capital projects; PM coordinates |
| Vendor contracts — view & approve | ✅ Yes | ✅ Approve action only | Board approves contracts over threshold |
| Announcements | ✅ Yes | ✅ Yes | Board communicates policy to owners |
| Owner directory — limited view | ✅ Yes (limited) | ❌ No | Board communicates with owners; no sensitive data |
| Reports & dashboards | ✅ Yes | ❌ No | Board monitors community health |
| Unit management | ❌ No | ❌ No | PM operational function only |
| Platform / admin surfaces | ❌ No | ❌ No | Operator-level only |

**PM Toggle Configuration Model:**
1. Per-feature, per-permission-level — View and Write are separate toggles, never bundled.
2. Association-scoped — toggles apply to all board-admins in a given association. Per-board-member toggle granularity is explicitly deferred (see Decision Log).
3. PM can expand or restrict access from defaults.
4. YCM defaults are the table above — PM only overrides when needed.
5. Toggles are a PM-only action — board admins cannot self-configure their own access.

**Runtime Detection.** Board Admin mode is determined at runtime by `tenant_configs.managementType` (`shared/schema.ts:1189`). This field is the single source of truth for mode branching. Self-managed = full access. PM-managed = default access table + PM toggle state evaluated per association. No other signal should be used to determine which mode applies.

**Legacy / containment.** Legacy `/portal/board/*`, `portal_access.role = "board-member"`, and `/portal/board-dashboard` constructs are non-canonical not merely because `/portal` is the wrong surface, but because the correct governed model is toggle-scoped access within `/app`. These legacy artifacts MUST NOT be expanded and are queued for containment or removal in a downstream implementation task.
