# 4.1 Tier 3 — Alert push + email notifications

> **Status:** SPEC LOCKED 2026-04-25 (ship-and-lock pattern)
> **Wave:** 32
> **Branch:** `wave/32-alert-notifications`
> **Builds on:** Waves 2 / 3 / 4 / 5 (cross-association alert engine), 15a (auto-dismiss/source-resolution), 33 (Tier 2 sources)
> **Spec parent:** `4.1-cross-association-alert-engine.md`

## Why

Today alerts only show inside the app. Users miss high-severity events when
not actively using the app. This wave wires push + email so critical alerts
get delivered out-of-band.

## Scope

Out-of-band delivery (push + email) for `severity: critical` alerts only.
Lower severities (`high`, `medium`, `low`) stay in-app — the cross-association
endpoint, hub widget, and home panel continue to surface them.

The Wave 33 source set ships only `critical | high | medium | low` (the
Wave 2 type union has no `serious` value). Wave 32 gates on `severity ===
"critical"` exclusively. If Tier 4 expands the union to add `serious`, the
gate is a one-line edit in `notifications.ts` (`CRITICAL_SEVERITIES`).

### Operator-only

Portal users (residents, owners) do **not** receive alert push/email today.
The cross-association alert engine is operator-only (Manager / Board /
Platform-Admin), so notification fan-out is keyed on `admin_users`. The
existing portal `push_subscriptions` table is **untouched** — it remains the
owner-portal channel for assessment / notice / parcel notifications. This
wave adds a parallel `admin_push_subscriptions` table.

## Per-user opt-in

Two new boolean columns on `admin_user_preferences` (the existing per-admin
preferences row, not `admin_users` proper — keeping prefs in the prefs table
preserves a single update surface and avoids the N+1 column drift on
`admin_users`):

- `notify_alerts_email integer NOT NULL DEFAULT 1` — email default ON for
  critical alerts. Existing `email_notifications` controls everything else.
- `notify_alerts_push integer NOT NULL DEFAULT 0` — push default OFF until
  the user enrolls a subscription via `Notification.requestPermission()` +
  `pushManager.subscribe()`.

Users with no `admin_user_preferences` row use the defaults (email ON, push
OFF).

The spec says "new columns on `admin_users`"; we ship them on
`admin_user_preferences` as the structurally cleaner location. Spec intent
("per-user opt-in for alert email/push") is fully honored.

## Frequency cap & dedup

Two layers:

1. **Per-`(alertId, adminUserId)` dedup.** A row in `alert_notifications`
   uniquely keyed `(alertId, adminUserId, channel)`. Any non-failed
   row blocks future fan-out for that tuple. Because `alertId` is
   deterministic (`${ruleType}:${recordType}:${recordId}` per Wave 2 Q7),
   re-running fan-out is idempotent.
2. **Per-user hourly rate limit.** At most 5 sends per `adminUserId` per
   rolling 60 minutes (counted across both channels). When the cap is hit,
   the user is skipped for this fan-out cycle; eligible alerts get picked
   up the next time the cap window slides.

## Suppression — auto-dismiss before fan-out

The fan-out queries the cross-association resolver pipeline as a live read
— it does not consume a queue. If a critical alert was generated at T and
the underlying source record is resolved at T+90s (Wave 15a auto-dismiss),
the next fan-out cycle (T+5min) sees the resolver no longer emit that
alert and therefore never inserts an `alert_notifications` row. No
notification fires. This is the spec's "auto-dismissed before notification"
behavior, achieved at zero cost — no separate suppression table needed.

## Don't-fire-retroactively

A migration ships an empty `alert_notifications` table. The first fan-out
cycle after deploy will find every existing critical alert as "no row in
alert_notifications" and would naively send mass notifications to opted-in
users.

To prevent this we seed `alert_notifications` once, on the first
fan-out cycle, with `delivery_status='suppressed-pre-existing'` rows for
every currently-emitted critical alert. A future fan-out finds those rows
and dedupes. The seeding is idempotent — gated on the absence of any
row in the table.

## Delivery model

A single helper `fanOutCriticalAlerts()` runs every 5 min as part of the
existing automation sweep (`server/index.ts:runAutomationSweep`). It does
not run inside the request lifecycle.

```
1. Run cross-association orchestrator across the union of all opted-in
   admin users' permitted associations (deduped). Filter to severity ===
   'critical'.
2. For each (alertId, adminUserId) where the user has the alert in their
   permitted set AND no row exists in alert_notifications for that tuple
   AND user's hourly rate-limit count < 5:
     a. Pick channel: push if user opted in AND has an active
        admin_push_subscriptions row → push; else if email opted in →
        email; else skip.
     b. Insert alert_notifications row (status 'pending'), send via
        channel, update row to 'sent' / 'failed' + errorMessage.
3. On the first ever cycle (table empty), seed all current critical
   alerts as 'suppressed-pre-existing' instead of sending.
```

Errors during send are captured on the row but never throw out of the
sweep — the next cycle retries failed rows by deleting failed rows older
than 1h before scanning. (Failed rows < 1h block re-send to avoid hot
loops.)

## Schema

- `admin_user_preferences.notify_alerts_email` — integer NOT NULL DEFAULT 1
- `admin_user_preferences.notify_alerts_push` — integer NOT NULL DEFAULT 0
- `alert_notifications` — new table, columns:
  - `id varchar PK`
  - `alert_id text NOT NULL`
  - `admin_user_id varchar NOT NULL → admin_users(id)`
  - `channel text NOT NULL` (`'email' | 'push' | 'suppressed-pre-existing'`)
  - `delivery_status text NOT NULL` (`'pending' | 'sent' | 'failed' | 'suppressed-pre-existing'`)
  - `error_message text`
  - `sent_at timestamp NOT NULL DEFAULT now()`
  - `created_at timestamp NOT NULL DEFAULT now()`
  - unique `(alert_id, admin_user_id, channel)`
  - btree `(admin_user_id, sent_at)` for the 1-hour rate-limit count
- `admin_push_subscriptions` — new table parallel to portal-side
  `push_subscriptions`, columns:
  - `id varchar PK`
  - `admin_user_id varchar NOT NULL → admin_users(id)`
  - `endpoint text NOT NULL UNIQUE`
  - `p256dh_key text NOT NULL`
  - `auth_key text NOT NULL`
  - `user_agent text`
  - `is_active integer NOT NULL DEFAULT 1`
  - `created_at`, `updated_at` timestamps
  - unique `(endpoint)`

## Endpoints

- `GET /api/admin/notification-preferences` →
  `{ email: boolean, push: boolean }` — narrow shape carrying only the two
  alert-channel toggles. Distinct from existing `GET
  /api/admin/me/preferences` which carries the full notification preference
  surface — kept additive so portal-side persona-map doesn't break.
- `PATCH /api/admin/notification-preferences` — body
  `{ email?: boolean, push?: boolean }` — partial updates. Returns the
  updated narrow shape.
- `POST /api/admin/push/subscribe` — body
  `{ endpoint, keys: { p256dh, auth } }`. Inserts an
  `admin_push_subscriptions` row (or no-ops if endpoint already exists).
- `DELETE /api/admin/push/subscribe` — body `{ endpoint }`. Marks
  `is_active = 0` (soft-delete; matches portal behaviour).
- `GET /api/admin/push/vapid-public-key` — returns the VAPID public key for
  the operator-side. Reuses the same key as the portal — the spec calls
  out parallel infra but a separate key is only needed if the keys are
  ever revoked independently. Today they are not.

## Frontend

The existing `/app/settings` (`client/src/pages/user-settings.tsx`) gets a
new "Critical alert delivery" card that:

- Reads `GET /api/admin/notification-preferences`.
- Renders two switches (email, push).
- For push: gates the toggle on `Notification.requestPermission()` and
  `serviceWorker.pushManager.subscribe()`. On enroll-success, calls
  `POST /api/admin/push/subscribe` with the endpoint + keys, then PATCHes
  the preference.
- On disable-push, calls `DELETE /api/admin/push/subscribe` with the
  current endpoint, then PATCHes `push: false`.

The existing service worker (`client/public/service-worker.js`) already
handles the `push` event — payload shape is `{ title, body, url }`.

## Out of scope

- SMS — separate channel, not in 4.1 Tier 3.
- Resident/portal notifications — those go through a separate roadmap.
- Quiet hours / digest interaction — Wave 32 sends every cycle if eligible.
  Quiet hours support deferred; the prefs already exist on the existing
  prefs row but are not consulted by `fanOutCriticalAlerts()` yet.
- Per-association notification suppression — global per-user only.
- HTML-rich email — body is plain text + a deep link.
