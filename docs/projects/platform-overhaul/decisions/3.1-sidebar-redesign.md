## Context

3.1 is the Layer 3 module that implements the new `/app` sidebar: the single canonical, role-filtered navigation surface locked by `docs/projects/platform-overhaul/decisions/0.3-navigation-model.md:69-72` (Q1 Resolution A — "Single global sidebar, role-filtered"). It consumes the six functional zones locked in `docs/projects/platform-overhaul/decisions/1.1-zone-taxonomy-corrections.md:44-57` (Home, Financials, Operations, Governance, Communications, Platform), the hub-at-zone-root URL convention from `docs/projects/platform-overhaul/decisions/1.2-section-hub-reclassification.md:79-95` (Q4), the four-persona model from `docs/projects/platform-overhaul/decisions/0.2-persona-map.md:32-200`, the canonical `AdminRole` + RouteGuard policy from `docs/projects/platform-overhaul/decisions/2.1-role-model-audit.md:172-173` (Q4) and `docs/projects/platform-overhaul/decisions/2.3-permission-boundary-corrections.md:153-173` (Q1–Q11), and the Platform-zone exclusivity from `docs/projects/platform-overhaul/decisions/2.4-platform-admin-surface-audit.md:125-151` (Q1–Q7).

The new sidebar must replace the current implicit-four-group structure in `client/src/components/app-sidebar.tsx:68-201` with the six-zone model, retire the Board Officer / Assisted Board shell shunt at `client/src/App.tsx:1051-1057` per 2.3 Q3, remove the Owner Portal launcher from platform nav per 2.4 Q5, surface the PM toggle configuration entry point for Assisted Board / PM Assistant access per 0.2 amendment, and render coherently across Manager (multi-assoc), Board Officer (self-managed = Manager-equivalent), Assisted Board (PM-managed = governance-scoped defaults + toggle state), PM Assistant (Manager-subset), and Platform Admin.

3.1 does not design hub pages themselves (1.2), does not gate route accessibility (2.3), does not change API role width (2.4 Q3/Q7), and does not define breadcrumbs (1.3) or page titles (1.4). 3.1 designs the sidebar component tree, the role-filter algorithm, the zone data source, and the PM toggle-driven Assisted Board / PM Assistant item visibility logic.

## Conflict Analysis

Twelve decision axes surface from the audit. Each maps to an open question below.

1. **Zone group label vocabulary.** Live sidebar groups are `Overview`, `{activeAssociationName}`, `Platform`. 1.1 locks six zone labels: **Home, Financials, Operations, Governance, Communications, Platform**. The live groups do not partition to these labels — Finance (1.1: Financials), Board + Governance + Communications + Announcements split across Board and Operations ambiguously, Insurance / Documents / Amenities / Community Hub unassigned to a zone. Maps to **Q1**.

   > ⚠️ **CROSS-REFERENCE NOTICE — 1.1 Q1 AC.** `docs/projects/platform-overhaul/decisions/1.1-zone-taxonomy-corrections.md:52-57` (Q1 Acceptance Criteria, LOCKED): "The sidebar in `/app` contains up to six top-level zones... Zone labels are: Home, Financials, Operations, Governance, Communications, Platform." Conflicting evidence: `client/src/components/app-sidebar.tsx:295, 327, 380` render group labels `Overview`, `{activeAssociationName}`, `Platform` — only `Platform` matches. Flag only. Do not resolve.

2. **Flat six-zone list vs preserved "Overview + Association + Platform" split.** 0.3 Q1 locks a single canonical sidebar filtered by role. 1.1 locks six top-level zones. The live sidebar's three-group chrome (Overview / Association-name / Platform) is a visual grouping, not a zone vocabulary. Maps to **Q2**.

3. **Hub link vs zone-label-as-group-header.** 1.2 Q4 locks hubs at zone-root URLs (`/app/financials`, `/app/operations`, `/app/governance`, `/app/communications`). The zone label in the sidebar is simultaneously (a) the group header and (b) the navigable link to the hub page. Current sidebar does not have a hub concept — clicking "Finance" takes the user to `/app/financial/foundation` (`client/src/components/app-sidebar.tsx:102`), a tabbed page, not a hub. Maps to **Q3**.

4. **Home placement.** 1.1 Q1 names `Home` as one of six zones; 1.2 Q5 says Home is a dashboard, not a hub, and exempt from hub content rules. 0.1 fixed Home at `/app`. Portfolio Health lives "subordinate to Home" per 0.1 §Selected Resolution. Does the sidebar render Home as a peer zone in the six-zone list, or as a special root item above the zone list with Portfolio Health nested under it? Maps to **Q4**.

5. **Board Officer vs Assisted Board sidebar branching.** 0.2 §Persona 2 (Board Officer) and §Persona 3 (Assisted Board) define two distinct board personas: Board Officer (self-managed, Manager-equivalent) and Assisted Board (PM-managed, governance-scoped defaults + PM toggle). The live sidebar does not distinguish the two personas. Persona determination per 2.3 Q6 uses `tenant_configs.managementType` (`shared/schema.ts:1189`). The sidebar must branch its role-filter on this value for `board-officer` / `assisted-board` sessions. Maps to **Q5**.

6. **PM toggle runtime evaluation.** 0.2 PM Toggle Configuration Model specifies per-feature View/Write toggles targeting Assisted Board and PM Assistant, association-scoped, PM-configurable, never applicable to Platform / Admin surfaces. 2.3 Q7 resolution introduces `useIsReadOnly()` for write-gating. For the sidebar, toggle state drives item **visibility** (View=false → hide item). The sidebar needs a data source for current toggle state by active association. Maps to **Q6**.

7. **PM toggle configuration surface location.** 0.2 makes Manager (PM) responsible for authoring toggle state; the UI surface for this is undesigned. Candidates: `/app/settings/toggle-access`, `/app/settings/associations/:id/toggle-access`, `/app/platform/*` (would violate 2.4 Q6 — Platform zone is Platform Admin only), or a new surface under an association-scoped zone. The sidebar needs a nav entry to this surface that is visible only to Manager + Platform Admin. Maps to **Q7**.

8. **Platform zone visibility on shared shell.** 2.4 Q1 (LOCKED) Platform zone = Platform Admin persona only; Q4 narrowed `platformSubPages` to `["platform-admin"]`; Q5 removes Owner Portal launcher. 2.3 Q3 retires the Board Officer / Assisted Board shell shunt, so all personas share the same `AppSidebar` component — the Platform group must render conditionally on `adminRole === "platform-admin"`. The six-zone filter must collapse the Platform zone to nothing for every non-Platform-Admin session. Maps to **Q8**.

   > ⚠️ **CROSS-REFERENCE NOTICE — 2.4 Q1 AC + 0.2 Persona 4.** `docs/projects/platform-overhaul/decisions/2.4-platform-admin-surface-audit.md:156-159` (Q1 AC, LOCKED): `/app/platform/*`, `/app/admin/*`, `/app/ai/*` = Full for Platform Admin; `❌` for all customer personas. Conflicting evidence: `client/src/components/app-sidebar.tsx:209-213` `canAccess()` returns `true` for nullish roles — a pre-auth render caller sees every zone including Platform. 2.1 Q7 AC 22 (LOCKED) mandates `canAccess()` returns `false` on nullish role. 3.1 must consume the 2.1 Q7 fix; it is not a 3.1-originated change. Flag only.

9. **RouteGuard vs sidebar `roles` as source of truth.** 2.3 Q9 (LOCKED) makes `<RouteGuard roles={[...]}>` the security-coherent route-level gate; sidebar `roles` arrays are nav-visibility only. 3.1 must not re-implement route gating — the sidebar calls into the same persona/role predicate used by RouteGuard, or reads a shared manifest. Maps to **Q9**.

10. **Multi-association association-context header.** 0.3 Q7 locks a global association switcher in the header as primary cross-association nav. Live sidebar renders an "Association" card at `client/src/components/app-sidebar.tsx:255-289` that doubles as context display + link to `/app/association-context`. The 3.1 redesign must decide: does the association switcher remain in the sidebar header card, move to the top app-bar per 0.3 Q7, or render in both? Maps to **Q10**.

11. **Owner Portal launcher retirement in Platform group.** 2.4 Q5 (LOCKED) removes `{ title: "Owner Portal", url: "/portal", roles: ["platform-admin"] }` at `client/src/components/app-sidebar.tsx:185` and the equivalent `client/src/App.tsx:180` `platformSubPages` entry. 3.1 implements this removal. Maps to **Q11**.

12. **Legacy sidebar items not mapped to any 1.1 zone.** Associations (`/app/associations`), Buildings & Units (`/app/units`), People (`/app/persons`), Documents (`/app/documents`), Insurance (`/app/insurance`), Amenity Booking (`/app/amenities`), Community Hub (`/app/community-hub`), Announcements (`/app/announcements`), Settings (`/app/settings`), Help Center (`/app/help-center`) — 1.1 does not pre-assign these to a zone; 1.2 Q3 flags the mapping as required implementation output that has not yet been authored. The sidebar design depends on these assignments. Maps to **Q12**.

## Open Questions

### Q1 — Zone group label vocabulary

Does the sidebar render exactly six `SidebarGroup` blocks with labels verbatim from 1.1 (`Home`, `Financials`, `Operations`, `Governance`, `Communications`, `Platform`), does it preserve a separate `Overview` chrome group (Home + Portfolio Health + Associations), or does it collapse Home + Portfolio Health into a non-zone root-level affordance above the zone list?

**Tradeoff axis:** strict 1.1 vocabulary (one source of truth, unambiguous mapping) vs preserved-chrome (lower visual disruption, retains an "overview" affordance from the current sidebar) vs root-affordance (elevates Home as the canonical workspace landing and keeps the zone list pure).

### Q2 — Active-association grouping

Does the sidebar retain the current "association-scoped" visual cluster under an `{activeAssociationName}` group header (grouping Financials/Operations/Governance/Communications visually by association), or flatten all zones as top-level peers with no association-name wrapper?

**Tradeoff axis:** visual context for multi-association Managers (keep wrapper) vs strict 1.1 six-zone flatness (remove wrapper; rely on header switcher per 0.3 Q7 for context).

### Q3 — Zone-label click behavior

When a user clicks a zone label in the sidebar (e.g., "Financials"), does the sidebar (a) navigate to the zone hub at `/app/financials` per 1.2 Q4 and expand an inline sub-page list on hover/active, (b) expand the sub-page list without navigating, or (c) both — click navigates to hub; separate chevron expands sub-list?

**Tradeoff axis:** discoverability (click-navigates surfaces the hub as the canonical zone landing per 1.2) vs density (expand-only preserves compact sidebar footprint) vs both (richer but requires an affordance the current component does not have).

### Q4 — Home + Portfolio Health placement

Does Home render as (a) a peer zone in the six-zone list with Portfolio Health as a sub-item, (b) a root-level item above the zone list (with Associations list + Portfolio Health adjacent), or (c) something else? Per 0.1 §Selected Resolution, Portfolio Health is "subordinate to Home, not a peer."

**Tradeoff axis:** 1.1 purity (Home is a zone, sub-items nest under it) vs 0.1 spirit (Home is the workspace landing, not a zone peer) vs hybrid (Home outside the zone list, Portfolio Health under Home).

### Q5 — Board Officer vs Assisted Board branching algorithm

For `adminRole === "board-officer"` or `adminRole === "assisted-board"` sessions, does the sidebar filter:
- (a) read `tenant_configs.managementType` for the active association and branch — self-managed → Manager-equivalent zone list; PM-managed → PM-Managed Default Access Table applied;
- (b) always apply PM-managed defaults and let 2.3 Q6 `RouteGuard` exceptions (self-managed exemption) cover the gap;
- (c) render an identical zone list with per-item toggle-evaluation handling both modes uniformly?

**Tradeoff axis:** mode-aware sidebar (accurate, aligns with 0.2 AC 4/5/7) vs uniform-default (simpler, defers mode logic to RouteGuard) vs uniform-toggle (makes Mode A a special case of "all toggles enabled").

### Q6 — PM toggle runtime evaluation source

Where does the sidebar read PM toggle state from at render time — a new `/api/portal/toggles` or `/api/board-access/toggles` endpoint joined on active association, a hydrated-in-session blob attached to `AuthSession`, or a React Query hook that fetches toggle state keyed on `(activeAssociationId, adminRole)`?

**Tradeoff axis:** session-hydration (one fetch per session, but stale if PM edits toggles mid-session) vs per-association query (always fresh, N+1 on association switch) vs endpoint-on-session (hybrid, requires server change).

### Q7 — PM toggle configuration surface location

Where does the Manager (PM) author toggle state? Options: (a) `/app/settings/associations/:id/board-access` (association-scoped settings surface), (b) `/app/settings/board-access` (global settings surface that lists all PM-managed associations and their toggle matrix), (c) a new top-level zone-adjacent surface, or (d) deferred — a Layer 4 feature-gap module.

**Tradeoff axis:** proximity-to-association (a = natural extension of per-association settings) vs central-audit (b = PM sees all their associations at once) vs new-surface (c = architecturally clean but adds a zone) vs defer (d = unblocks 3.1 but leaves the 0.2 amendment without a UI).

### Q8 — Platform zone visibility mechanism

How does the sidebar hide the Platform zone for non-Platform-Admin personas — (a) static filter at module-array build time (Platform group omitted from the JSX unless `adminRole === "platform-admin"`), (b) per-item role filter via the same `canAccess()` predicate used for all zones, or (c) a new shared persona-manifest that encodes Platform zone as explicitly `platform-admin`-only?

**Tradeoff axis:** static (fastest, clearest at read time) vs unified (one filter for all zones, matches 2.3 Q4 sidebar hygiene) vs manifest (scales to future persona additions, largest refactor).

### Q9 — RouteGuard ↔ sidebar `roles` source of truth

Are sidebar `roles` arrays derived from the same persona/role manifest consumed by `<RouteGuard>` (single source of truth, e.g., `shared/persona-access.ts`), or does the sidebar keep per-module `roles: [...]` literals as a separate declaration reviewed against 0.2 via lint/audit?

**Tradeoff axis:** derived (impossible to drift from RouteGuard gating, requires new shared module) vs independent-literals (matches 2.3 Q4 "retain and normalize" — each sidebar item explicitly declares its roles in situ) vs hybrid (shared per-route manifest, sidebar reads it).

### Q10 — Association switcher placement

Does the global association switcher (0.3 Q7 AC 29) render (a) in the top app-bar only (sidebar "Association" card retired), (b) in the sidebar header card only (current behavior, elevated to canonical), or (c) in both — app-bar switcher primary, sidebar card shows active-association context read-only?

**Tradeoff axis:** app-bar-only (matches 0.3 Q7 SaaS convention, frees sidebar space) vs sidebar-card-only (lower implementation risk, keeps active context visible next to zones) vs both (redundancy, but richer context on multi-association Manager sessions).

### Q11 — Owner Portal launcher retirement

2.4 Q5 AC mandates removal of `client/src/components/app-sidebar.tsx:185` and `client/src/App.tsx:180`. Does 3.1 additionally forbid any operator-side link to `/portal` anywhere in the sidebar (footer CTA, hover action, inline help), or does 2.4 Q5 AC 3 ("No `/app/*` nav surface contains a link to `/portal`") fully cover the removal with no additional 3.1 restriction needed?

**Tradeoff axis:** 3.1 restates (explicit in the sidebar spec) vs 3.1 inherits (avoid duplication with 2.4).

### Q12 — Legacy item zone assignment

Which 1.1 zone (if any) does each of the following items belong to:
- `/app/associations` (Associations list — currently Overview group) — Home zone, or no zone (operator-chrome item)?
- `/app/units` / `/app/persons` (Buildings & Units / People) — Operations? Home? A new "Records" concept?
- `/app/documents` — Governance (per 0.2 matrix "Governance documents")? Communications? Standalone?
- `/app/insurance` — Governance? Operations? Home?
- `/app/amenities` — Operations? Communications? Home?
- `/app/community-hub` — Communications? Home?
- `/app/announcements` — Communications (obvious), but currently nested under Board group — confirm?
- `/app/settings` — footer-only (not a zone), or a hidden zone?
- `/app/help-center` — footer-only?
- `/app/new-association` — footer CTA (Platform Admin only per `client/src/components/app-sidebar.tsx:430-436`), or moved into Platform zone?

**Tradeoff axis:** 1.1 Q2 assignment rubric (intent-based, one zone per page) vs operator-chrome exemption (some items are workspace chrome, not zone content) vs defer to 1.2 implementation output table (blocks 3.1 until 1.2 mapping lands).
