## Context

3.1 is the Layer 3 module that implements the new `/app` sidebar: the single canonical, role-filtered navigation surface locked by `docs/projects/platform-overhaul/decisions/0.3-navigation-model.md:69-72` (Q1 Resolution A — "Single global sidebar, role-filtered"). It consumes the six functional zones locked in `docs/projects/platform-overhaul/decisions/1.1-zone-taxonomy-corrections.md:44-57` (Home, Financials, Operations, Governance, Communications, Platform), the hub-at-zone-root URL convention from `docs/projects/platform-overhaul/decisions/1.2-section-hub-reclassification.md:79-95` (Q4), the six-persona model from `docs/projects/platform-overhaul/decisions/0.2-persona-map.md:32-268`, the canonical `AdminRole` + RouteGuard policy from `docs/projects/platform-overhaul/decisions/2.1-role-model-audit.md:172-173` (Q4) and `docs/projects/platform-overhaul/decisions/2.3-permission-boundary-corrections.md:153-173` (Q1-Q11), and the Platform-zone exclusivity from `docs/projects/platform-overhaul/decisions/2.4-platform-admin-surface-audit.md:125-151` (Q1-Q7).

The new sidebar must replace the current implicit-four-group structure in `client/src/components/app-sidebar.tsx:68-201` with the six-zone model, retire the Board Officer / Assisted Board shell shunt at `client/src/App.tsx:1051-1057` per 2.3 Q3, remove the Owner Portal launcher from platform nav per 2.4 Q5, surface the PM toggle configuration entry point for Assisted Board / PM Assistant access per 0.2 amendment, and render coherently across Manager (multi-assoc), Board Officer (self-managed = Manager-equivalent), Assisted Board (PM-managed = governance-scoped defaults + toggle state), PM Assistant (Manager-subset), and Platform Admin.

3.1 does not design hub pages themselves (1.2), does not gate route accessibility (2.3), does not change API role width (2.4 Q3/Q7), and does not define breadcrumbs (1.3) or page titles (1.4). 3.1 designs the sidebar component tree, the role-filter algorithm, the zone data source, and the PM toggle-driven Assisted Board / PM Assistant item visibility logic.

## Conflict Analysis

Twelve decision axes surface from the audit. Each maps to a resolved question below.

1. **Zone group label vocabulary.** Live sidebar groups are `Overview`, `{activeAssociationName}`, `Platform`. 1.1 locks six zone labels: **Home, Financials, Operations, Governance, Communications, Platform**. The live groups do not partition to these labels. Maps to **Q1**.

2. **Flat six-zone list vs preserved "Overview + Association + Platform" split.** 0.3 Q1 locks a single canonical sidebar filtered by role. 1.1 locks six top-level zones. The live sidebar's three-group chrome is a visual grouping, not a zone vocabulary. Maps to **Q2**.

3. **Hub link vs zone-label-as-group-header.** 1.2 Q4 locks hubs at zone-root URLs. The zone label in the sidebar is simultaneously the group header and the navigable link to the hub page. Maps to **Q3**.

4. **Home placement.** 1.1 Q1 names `Home` as one of six zones; 1.2 Q5 says Home is a dashboard, not a hub. Maps to **Q4**.

5. **Board Officer vs Assisted Board / PM Assistant sidebar branching.** 0.2 defines distinct personas with different access scopes. Maps to **Q5**.

6. **PM toggle runtime evaluation source.** 0.2 PM Toggle Configuration Model specifies per-feature View/Write toggles. Maps to **Q6**.

7. **PM toggle configuration surface location.** 0.2 makes Manager responsible for authoring toggle state. Maps to **Q7**.

8. **Platform zone visibility on shared shell.** 2.4 Q1 locks Platform zone to Platform Admin only. Maps to **Q8**.

9. **RouteGuard vs sidebar `roles` as source of truth.** 2.3 Q9 makes `<RouteGuard>` the security gate; sidebar `roles` are nav-visibility only. Maps to **Q9**.

10. **Multi-association association-context header.** 0.3 Q7 locks a global association switcher in the header. Maps to **Q10**.

11. **Owner Portal launcher retirement in Platform group.** 2.4 Q5 removes the `/portal` link. Maps to **Q11**.

12. **Legacy sidebar items not mapped to any 1.1 zone.** Several items need zone assignment. Maps to **Q12**.

## Selected Resolutions

### Q1 — Zone group label vocabulary

**Resolution:** The sidebar renders exactly six `SidebarGroup` blocks with labels verbatim from 1.1: **Home, Financials, Operations, Governance, Communications, Platform**. No other group labels are used. The current ad-hoc labels (`Overview`, `{activeAssociationName}`, `Platform`) are retired. Every sidebar item belongs to exactly one of these six zones. The sidebar component declares these six labels as constants imported from a shared zone-vocabulary module; no per-component string literals.

**Derivation:** 1.1 Q1 (LOCKED) — "Six canonical zones. The six zones are: Home, Financials, Operations, Governance, Communications, Platform." 0.3 Q2 AC 7 — "Zone labels are functional and stable across personas." 1.1 Q1 AC — "Zone labels are: Home, Financials, Operations, Governance, Communications, Platform."

**Acceptance Criteria**
1. The sidebar contains exactly six `SidebarGroup` blocks with labels: Home, Financials, Operations, Governance, Communications, Platform.
2. No sidebar group uses the labels `Overview`, `{activeAssociationName}`, or any other ad-hoc string.
3. Zone labels are imported from a shared module — no inline string literals in the sidebar component.
4. Every sidebar item is assigned to exactly one of the six zones.

### Q5 — Assisted Board / PM Assistant mode branching (SUBSET-RENDER)

**Resolution (founder decision):** Assisted Board and PM Assistant in PM-managed mode get a **completely separate simplified sidebar**. Only permitted zones render. No greyed-out or hidden locked sections. This is the **SUBSET-RENDER** model.

Implementation: the sidebar component evaluates the active persona (`adminRole`) and, for `assisted-board` and `pm-assistant`, renders only the zones and items their 0.2 boundary matrix row permits. Disallowed zones are omitted entirely from the render tree — they are not greyed out, not collapsed, not hidden behind a toggle. The same canonical six-zone taxonomy from 1.1 is used; the rendered subset varies by persona.

Board Officer (`board-officer`) = Manager-equivalent sidebar. Same full six-zone view as Manager, minus Platform zone (per 0.2 boundary matrix — Board Officer has ❌ on `/app/platform/*`, `/app/admin/*`, `/app/ai/*`).

Persona sidebar views:
- **Manager:** All six zones. Portfolio Health visible. Associations visible. All sub-items visible.
- **Board Officer:** Five zones (Home, Financials, Operations, Governance, Communications). No Platform zone. Portfolio Health NOT visible. Associations NOT visible. Settings visible. All other sub-items identical to Manager.
- **Assisted Board:** Subset of zones per 0.2 PM-Managed Default Access Table + PM toggle state. No Platform zone. No Portfolio Health. No Associations. No Settings. Sub-items filtered by toggle state.
- **PM Assistant:** Subset of zones per Manager PM-internal toggle configuration. No Platform zone. No Portfolio Health (unless Manager grants). No Settings (unless Manager grants). Sub-items filtered by toggle state.
- **Platform Admin:** All six zones (full operator view including Platform zone).

**Derivation:** Founder binding decision (2026-04-16). 0.3 Q1 AC 1 — "one canonical zone structure" (taxonomy is canonical; rendered subset varies). 0.3 Q5 — strict isolation (hidden when no access). 0.2 §Persona 2 (Board Officer = Manager-equivalent), §Persona 3 (Assisted Board = scoped), §Persona 4 (PM Assistant = Manager-subset), §Persona 6 (Platform Admin).

**Acceptance Criteria**
19. `assisted-board` sidebar omits zones and items not permitted by the 0.2 PM-Managed Default Access Table and active PM toggle state.
20. `pm-assistant` sidebar omits zones and items not permitted by the Manager's PM-internal toggle configuration.
21. No greyed-out, disabled, or visually-locked sidebar items exist for any persona — disallowed items are absent from the DOM.
22. `board-officer` sidebar is identical to `manager` sidebar minus the Platform zone, Portfolio Health, and Associations.
23. `platform-admin` sidebar includes all six zones.
24. The sidebar render function accepts persona and toggle state as inputs and produces a filtered zone tree — no imperative branching on persona names in the JSX.

### Q6 — PM toggle runtime evaluation source

**Resolution:** PM toggles are read from `tenant_configs` (the existing association-scoped configuration table at `shared/schema.ts:1189`) joined with toggle state, evaluated per association at runtime. The sidebar reads toggle state via a React Query hook keyed on `(activeAssociationId, adminRole)` that fetches the resolved toggle matrix for the active persona on the active association. Toggle state changes by the PM are reflected on next sidebar render (query invalidation on association switch or toggle mutation).

This is not a session-hydrated blob — toggle state is fetched per association context to ensure freshness when a Manager edits toggles mid-session. The hook returns the resolved permission set: for each feature domain, whether View and Write are enabled for the active persona.

**Derivation:** 0.2 PM Toggle Configuration Model rules 1-7 — per-feature, per-permission-level, association-scoped, PM-only. 2.3 Q7 — `useIsReadOnly()` hook for write-gating (the sidebar uses the View toggle for visibility; `useIsReadOnly()` handles write gating at the page level).

**Acceptance Criteria**
25. A React Query hook `usePersonaToggles(activeAssociationId, adminRole)` (or equivalent) fetches the resolved toggle matrix for the active persona and association.
26. The sidebar consumes this hook's output to filter items for `assisted-board` and `pm-assistant` personas.
27. Toggle state refreshes on association switch (query key includes `activeAssociationId`).
28. If the toggle query is loading, the sidebar renders a loading skeleton — not the full unfiltered nav tree.
29. Platform and admin surfaces are never included in the toggle matrix (0.2 PM Toggle Configuration Model rule 7).

### Q8 — Platform zone visibility mechanism

**Resolution:** The sidebar uses the unified `canAccess()` role-filter predicate (the same function consumed by `<RouteGuard>` per Q9) to determine Platform zone visibility. The Platform zone's items carry `roles: ["platform-admin"]` in the sidebar data structure. The `canAccess()` predicate evaluates the session's `adminRole` against each item's `roles` array. For non-`platform-admin` sessions, every item in the Platform zone returns `false`, and the zone group is omitted from the render tree entirely (consistent with Q5 subset-render: empty zones do not render).

No special-case static filter is needed. The unified role-filter predicate handles Platform zone hiding as a natural consequence of the `roles` arrays.

**Derivation:** 2.4 Q1 — Platform zone = Platform Admin only. 2.4 Q2 — `<RouteGuard roles={["platform-admin"]}>` on every platform route. 0.3 Q5 — strict isolation. 2.1 Q7 — `canAccess()` returns `false` on null role.

**Acceptance Criteria**
34. Platform zone is visible only when `adminRole === "platform-admin"`.
35. For all other personas (Manager, Board Officer, Assisted Board, PM Assistant), the Platform zone group is absent from the DOM — not hidden, not collapsed, not greyed out.
36. The visibility mechanism uses the same `canAccess()` predicate used for all other zone filtering — no special-case `if (adminRole === "platform-admin")` branch for Platform alone.
37. `canAccess()` returns `false` when role is nullish (per 2.1 Q7 AC 22) — no Platform zone leak on pre-auth render.

### Q9 — RouteGuard and sidebar roles source of truth

**Resolution:** Both sidebar `roles` arrays and `<RouteGuard>` roles derive from the **0.2 Persona Boundary Matrix** as the single source of truth. Implementation: a shared module (e.g., `shared/persona-access.ts`) exports a canonical role-filter function and a per-route/per-feature access manifest derived from the 0.2 matrix. The sidebar imports this manifest to populate each item's `roles` array. `<RouteGuard>` imports the same manifest to gate route access.

The sidebar `roles` arrays are NOT independent literals reviewed via lint — they are programmatically derived from the shared manifest. This eliminates the drift between sidebar visibility and route accessibility that exists today (e.g., sidebar grants `board-admin` access to Portfolio at `app-sidebar.tsx:76` while the route redirects).

**Derivation:** 2.3 Q4 — "Retain and normalize [sidebar roles] to 0.2 matrix." 2.3 Q9 — `<RouteGuard>` is the enforcement standard. 2.1 Q4 — single canonical `AdminRole` from `shared/`.

**Acceptance Criteria**
38. A shared module exports a canonical access manifest derived from the 0.2 Persona Boundary Matrix.
39. Sidebar `roles` arrays are populated by importing from this shared module — no per-item inline `roles: [...]` string literals in the sidebar component.
40. `<RouteGuard>` imports from the same shared module.
41. Any change to persona access requires updating the shared manifest — sidebar and RouteGuard stay in sync automatically.
42. No `AdminRole` type is re-declared in the sidebar component — it is imported from `shared/` per 2.1 Q4.
