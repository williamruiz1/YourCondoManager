## Context

0.1 locked one concrete example of a section hub: `/app/operations/dashboard` is re-classified from Z1-1 to **Z1-6 "Operations Overview"** and stays as the landing page for the Operations section. That is the reference template for what a "section hub" is in YCM's IA: a functional zone's aggregate landing surface — sub-section index, status signals, recent activity, quick actions — that sits above the section's working pages.

The rest of `/app/*` has not been reviewed through that lens. Some top-level sections land on an implicit hub (Operations, Finance tab-container), some redirect to a primary working page (`/app/financial` → `/app/financial/billing`), some are flat single-page sections with no hub at all (Documents, Insurance, Community Hub), and some are tab containers masquerading as hubs (Governance, Communications). The result is an inconsistent surface where the user cannot predict whether clicking a sidebar group label will show a landing view or dump them into a working page.

This module decides, per top-level section, which pattern each section uses — **Z1-6 section hub**, **direct-to-primary-page redirect**, or **tab container** — and reclassifies any section whose current behavior does not match its intended pattern. It is coupled to 1.1 (Zone tag reclassification) because zone tags are the input to the hub/direct decision, and to 0.3 Q8 (canonical home per persona) because section hubs are explicitly **not** the persona's default landing — Home is.

## Conflict Analysis

Seven distinct conflicts surface from the inventory above and the audit sources. Each maps to a locked question in the next section.

1. **Hub universality.** 0.1 establishes a Z1-6 hub for Operations. 0.3 Q2 locks functional grouping as the sidebar taxonomy. It is unresolved whether every functional group gets a hub or whether hubs are optional per-section based on surface area. Maps to **Q1**.
2. **Hub composition.** `operations-dashboard.tsx` carries KPI totals, aging buckets, recent activity, recommended actions, and audit trail. Whether that exact composition is the canonical template, or a subset/superset is appropriate per section, is undefined. Maps to **Q2**.
3. **Low-surface-area sections.** Documents, Insurance, Amenity Booking, and Community Hub each have only 1–2 sub-pages (some none). A hub over a single page is ceremony; directness may serve better. Maps to **Q3**.
4. **Persona-adapted hubs.** 0.3 Q3 locks distinct persona landings within a shared shell. It is unresolved whether a given section hub is persona-uniform or whether Manager, Board Admin, and (where applicable) Owner see different hub compositions. Maps to **Q4**.
5. **Per-section hub-yes / hub-no verdicts.** The Current State table enumerates 13 sections without a per-row classification. That classification is the core output of this module. Maps to **Q5**.
6. **Hub vs. redirect coexistence.** Finance today redirects `/app/financial` → `/app/financial/foundation` (→ or into billing). If Finance gets a hub, the redirect must retire or coexist as a deep-link fallback. Maps to **Q6**.
7. **Naming convention.** 0.1 uses "Operations **Overview**" as the Z1-6 hub label. Other candidates are "Hub" (per `community-hub.tsx`), "Home" (reserved by 0.1 for `/app`), or the bare section name. A canonical suffix rule is needed for consistency. Maps to **Q7**.

## Open Questions

Each question is stated with its tradeoff axis. No answers are proposed in this skeleton.

### Q1 — Hub universality
**Question:** Is a Z1-6 section hub required for every Zone 1 functional group, or optional per-section based on surface area and task density?
**Tradeoff axis:** Taxonomy consistency (every group behaves the same) ↔ surface-area efficiency (no hub over a single page).

### Q2 — Hub composition
**Question:** What is a section hub required to contain — some subset of {KPI row, sub-section index, recent activity feed, quick actions panel, alerts subset, recommended actions}? Is the `operations-dashboard.tsx` composition the canonical template, or a menu to draw from?
**Tradeoff axis:** Template rigidity (uniform scannability) ↔ per-section fit (surface what each zone actually needs).

### Q3 — Low-surface-area sections
**Question:** For sections with only 1–2 sub-pages (Documents post-merge, Amenity Booking, Insurance if retained, Community Hub), is a hub warranted, or should the section go directly to its primary page?
**Tradeoff axis:** Pattern uniformity across the sidebar ↔ avoiding empty ceremony for thin sections.

### Q4 — Persona-adapted hubs
**Question:** Where a hub serves more than one persona (Manager + Board Admin in `/app`), is the hub composition identical per persona, or persona-adapted within the shared shell per 0.3 Q3?
**Tradeoff axis:** Shared-shell simplicity ↔ persona-specific task support (Board Admin sees different signals than Manager).

### Q5 — Per-section classification
**Question:** For each of the 13 rows in the Current State table, what is the hub verdict — hub-yes (Z1-6), hub-no (direct to primary page), or ambiguous (needs further evidence)?
**Tradeoff axis:** Per-row — captured implicitly in Q1/Q2/Q3/Q4.
**Note:** Resolution table is PENDING below.

### Q6 — Hub vs. default redirect
**Question:** When a section has both a hub and a redirect rule (e.g., `/app/financial` → `/app/financial/billing`), does the hub replace the redirect, coexist with it as deep-link behavior, or supersede only for specific persona/context combinations?
**Tradeoff axis:** Clean cutover (retire redirects) ↔ back-compat for bookmarks and external links.

### Q7 — Naming convention
**Question:** What is the canonical suffix for a section hub label — "Overview" (per 0.1 Operations Overview), "Hub" (per `community-hub.tsx`), or bare section name?
**Tradeoff axis:** Explicit signaling ("Overview" tells the user this is a landing) ↔ brevity and existing nomenclature.

## How This Informs Downstream Layers

### Layer 1 — IA Taxonomy Cleanup (1.1, 1.3)
1.2 consumes zone tags produced by 1.1 and produces the per-section hub/direct verdict. Any section whose Z-tag shifts in 1.1 may force a revisit of its 1.2 verdict. 1.3 (label corrections) consumes the naming convention answered by Q7.

### Layer 2 — Role & Permission Model
Hubs that surface alerts or signals across sub-pages must respect the 0.2 boundary matrix — a hub may not expose a signal sourced from a sub-page the persona cannot access. 2.3 (permission boundary corrections) is the enforcement layer.

### Layer 3 — Navigation Restructure
Sidebar group behavior changes per 1.2's verdicts: sections reclassified hub-yes get a landing route wired in `App.tsx` and a sidebar group-label click that targets the hub, not the first child tab. Sections reclassified hub-no retire their landing-redirect or landing page accordingly. Breadcrumb convention (0.3 Q4) applies identically to hub pages — "Association > Section" (hub is the terminal segment).

### Layer 4 — Feature Gaps
Module 4.1 (cross-association alert engine) is a hub-relevant signal source. Hub-yes sections may surface a subset of 4.1 alerts in their hub; the central updates/inbox (0.3 Q9) remains the canonical aggregator.

### Layer 5 — Polish & Hardening
Mobile nav (5.3) must handle hub pages — a hub with a KPI row + sub-section index must collapse gracefully on mobile without burying the sub-section entry points.
