# 5.6 — i18n Scaffolding

**Status:** SPEC LOCKED 2026-04-25 (Wave 21, ship-and-lock pattern)
**Owner:** Platform Overhaul
**Layer:** 5 — Polish & Hardening

---

## Scope

Establish a minimal, dependency-free string registry for the YCM client and migrate the **same 10 surfaces** covered by 5.5 onto it.

The point of this module is _extraction discipline_: every visible English literal on the in-scope surfaces lives in `client/src/i18n/strings.en.ts`, and pages call `t(key)` instead of writing the literal inline. Wave 21 does not translate — it only locks the registry pattern and the ten highest-leverage surfaces. A follow-up wave introduces a real translation engine and a locale switcher.

**In scope:**
- `client/src/i18n/strings.en.ts` — flat `as const` registry of ~150 string keys covering the 10 in-scope surfaces.
- `client/src/i18n/use-strings.ts` — `t(key): string` lookup helper, plus `useStrings()` hook form for future locale-aware reads.
- `client/src/i18n/README.md` — pattern documentation (how to add a string, how to add a locale, why these 10 first, backlog).
- Refactor the 10 surfaces to call `t()` instead of inline literals.
- Tests under `tests/i18n/strings.test.ts` proving uniqueness, lookups, and the missing-key fallback.

**Out of scope:**
- Adding `i18next`, `formatjs`, or any other translation library. The goal is a registry; the engine is a future wave.
- Translating any string. The English locale is the single source of truth this wave.
- Plural rules, gender, or interpolation. Strings stay flat.
- Locale switching, `Accept-Language` parsing, or a user-preference field. Not in scope.
- Migrating Tier 1 / Tier 2 / Tier 3 surfaces beyond the 10 called out above. Backlog tracked in the README.

## Acceptance Criteria

1. `client/src/i18n/strings.en.ts` exists, exports a flat `strings` object marked `as const`, and exports the derived `StringKey` union type.
2. `client/src/i18n/use-strings.ts` exists, exports `t(key: StringKey): string` and `useStrings()`. Both return the literal value for known keys; unknown keys return the key itself (safe fallback).
3. Every key in `strings.en.ts` is unique. (Tested.)
4. The ten in-scope surfaces import from `@/i18n/use-strings` and call `t()` for every user-visible literal that was inline before Wave 21:
   - `dashboard.tsx`
   - `communications-inbox.tsx`
   - `hubs/financials-hub.tsx`
   - `hubs/operations-hub.tsx`
   - `hubs/governance-hub.tsx`
   - `hubs/communications-hub.tsx`
   - `financial-rules.tsx`
   - `portal/portal-home.tsx`
   - `portal/portal-finances.tsx`
   - `settings-billing.tsx`
5. `useDocumentTitle()` calls on the in-scope surfaces use `t()` for the page-title input.
6. Status / plan enum labels in `settings-billing.tsx` (`PLAN_LABELS`, `STATUS_LABELS`) are replaced by helper functions that route each enum value through `t()`. The status / plan enums themselves are not translated — only their human-readable labels.
7. Run-history filter values, billing status enums, and inbox tab labels are extracted (these are common high-traffic short strings).
8. `client/src/i18n/README.md` documents the four required topics: how to add a string, how to add a locale, why these 10 surfaces first, and a tiered backlog of remaining surfaces.
9. No new runtime dependencies introduced. `package.json` is unchanged in this wave.
10. Tests under `tests/i18n/strings.test.ts` verify: (a) `strings.en.ts` exports an object, (b) every key resolves to a non-empty string, (c) `t()` returns the registered value for known keys, (d) `t()` falls back to the key for unknown keys, (e) all keys are unique.

## Naming Conventions

- Flat dotted keys: `surface.context.element` (`home.alerts.title`, `inbox.empty.unread`).
- `*.title`, `*.subtitle`, `*.body`, `*.cta` for headings, supporting copy, and primary action.
- `*.empty.title` / `*.empty.body` for empty states; `*.error.title` / `*.error.body` for error states.
- `common.*` only when at least two surfaces share the literal. Single-use strings stay scoped under their surface.
- Do **not** translate test IDs, query keys, route paths, or feature-flag names — those are infrastructure, not user copy.

---

## Migration Path

The README under `client/src/i18n/README.md` is the canonical migration playbook. Summary:

1. Future wave: pick a tier (Tier 1 high-traffic surfaces first), extract literals to `strings.en.ts`, ship.
2. Future wave (when product commits): introduce `strings.es.ts` (or equivalent) with the same key shape; TypeScript will refuse partial coverage.
3. Future wave: add a locale provider, swap `t()` to a locale-aware implementation, and (only if needed) swap the registry to `i18next` for plural / interpolation. The `t(key) → string` API is intentionally a subset so that swap is one wrapping layer.
