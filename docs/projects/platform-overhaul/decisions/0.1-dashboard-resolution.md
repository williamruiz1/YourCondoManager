## Context

Three pages in the YCM operator workspace each exhibit the behavioral signature of a Z1-1 Command Center — a landing surface that aggregates cross-functional health signals, stats, and actions. Having three such pages without a clear ownership boundary creates navigation ambiguity for every user persona that operates inside `/app`, and it produces duplicated data fetches, competing entry points, and an unclear hierarchy for any future nav restructure.

This conflict was flagged independently by two Pass 1 auditors (A1 and A6) and cannot be resolved by taxonomy alone — it requires a product decision about what the primary entry point of the operator workspace is, and what role each of the three surfaces plays relative to that entry point.

This decision memo documents the three candidates, the conflict analysis, the options considered, and the selected resolution. All Layer 1+ work that touches navigation, nav labels, or command-center placement depends on this decision being locked first.

---

## The Three Candidates

### Candidate A — `dashboard.tsx` (`/app`)
**Current role:** Primary app landing page. Serves the `manager`, `board-admin`, and `viewer` personas. Renders portfolio-level stat cards (total associations, units, owners, tenants, board members, documents), a live alerts panel, a quick-actions panel, an active elections widget, and a setup wizard for new accounts.

**Behavioral signature:** Action-oriented. The user arrives here to see what needs attention today and take quick actions. The setup wizard makes it function as an onboarding surface as well.

**Known issues from A1:**
- Stat cards (associations, units, owners) duplicate totals visible in `portfolio.tsx`
- Quick Actions panel contains a stale link (`/app/financial/ledger` → redirect to `/app/financial/billing`) with outdated label
- Conflates portfolio-level counts with per-association alert triage — the purpose is blurred

**Cognitive load:** High

---

### Candidate B — `portfolio.tsx` (`/app/portfolio`)
**Current role:** Cross-association analytical view. Manager-only. Renders a KPI table per association, risk and threshold alerts, financial summary, and a recent activity feed across all associations.

**Behavioral signature:** Analytical/comparative. The user arrives here to assess relative health across the portfolio — not to take action but to understand state.

**Known issues from A1:**
- Currently subordinate in nav to Dashboard despite being the more powerful cross-association view
- No clear label differentiating it from Dashboard's portfolio-level stat cards
- Silently redirects `board-admin` users with ≤1 association to `/app` — effectively invisible to a large user segment
- A1 verdict: RENAME-MOVE, promote to primary Command Center or clearly differentiate as "Portfolio Health" beneath Dashboard

**Cognitive load:** High

---

### Candidate C — `operations-dashboard.tsx` (`/app/operations/dashboard`)
**Current role:** Section-scoped operations health view. Manager-only. Renders a single aggregate endpoint (`GET /api/operations/dashboard`) returning six summary totals, work-order aging buckets, vendor status counts, recent work orders, due maintenance instances, recent inspection records, and an audit trail. Includes a recommended-actions panel.

**Behavioral signature:** Section hub. The user arrives here for a real-time operational status check within the Operations section. It is the parent nav entry for the Operations sidebar group.

**Known issues from A6:**
- Named "Operations Dashboard" in the sidebar, suggesting peer status with the main "Dashboard" rather than a sub-section view
- Breadcrumb shows `/app` as "Dashboard" then "Operations Dashboard" — implies sibling relationship
- No link back from `dashboard.tsx` to this page
- Audit trail section duplicates functionality of a generic audit log
- A6 verdict: PATCH — re-classify as Z1-6 section hub, rename to "Operations Overview"

**Cognitive load:** High

## Conflict Analysis

The core problem is a **nav hierarchy mismatch**, not a content overlap problem. The three pages serve genuinely different purposes but have been given equivalent visual weight in the sidebar, which implies they are peers when they should be in a clear parent-child-sibling relationship.

| Dimension | dashboard.tsx | portfolio.tsx | operations-dashboard.tsx |
|---|---|---|---|
| Scope | Portfolio + per-association | Portfolio only | Operations section only |
| Primary action | Act on alerts, quick nav | Analyze comparative health | Monitor operational queue |
| User type | manager, board-admin, viewer | manager only | manager only |
| Nav position | `/app` (root landing) | `/app/portfolio` (peer of root) | `/app/operations/dashboard` (section parent) |
| Redirect behavior | Destination for board-admin redirects | Redirects board-admin w/ ≤1 assoc to `/app` | No redirect behavior |
| Setup/onboarding role | Yes — setup wizard lives here | No | No |
| Cross-section reach | Yes | Yes | No — Operations only |

**The real conflict is between A and B.** Candidates A and B both claim the cross-association portfolio-level niche. Candidate C is structurally a section hub and is not a true peer — it only appears as a Command Center conflict because its name includes the word "Dashboard" and its nav placement implies sibling status.

**Candidate C is the easiest to resolve:** rename it and reclassify it. It does not need a structural change.

**Candidates A and B require a product decision:** which one is the primary entry point and what does the other become?

## Options Considered

### Option 1 — Dashboard as primary, Portfolio as analytical sub-view (SELECTED)
`dashboard.tsx` remains the root landing page (`/app`). It is refocused as the "today" action surface — alerts, elections, quick actions, setup wizard — and its portfolio-level stat cards are removed. Portfolio Health becomes the dedicated home for all aggregate counts.

`portfolio.tsx` is renamed "Portfolio Health" and repositioned as a clearly subordinate analytical view, accessible from the Home page and from the sidebar as a second-level item under the Home nav group. It is no longer a nav peer of Home.

`operations-dashboard.tsx` is renamed "Operations Overview" and reclassified as the Operations section hub (Z1-6). No structural changes to its content or API.

**Why this was selected:**
- Dashboard is already the root landing page. Keeping it stable is low-risk for all existing users.
- Portfolio's analytical purpose is genuinely different from the action purpose of Home. Subordinating it as a named drill-down preserves both surfaces without requiring a merge.
- The `board-admin` silent redirect to `/app` becomes coherent: single-association board admins land on the action surface that serves them, not the multi-association portfolio view that is irrelevant to them.
- Lowest implementation risk of the three options.

---

### Option 2 — Portfolio as primary landing, Dashboard becomes per-association "Today" view
Not selected. Requires changing the root route `/app`, which affects every existing user and all deep links. Portfolio is manager-only — making it the root creates an immediate role-gating problem for board-admins and viewers. Single-association managers (Track 1 persona) would land on a one-row portfolio page. Higher implementation risk, no user experience gain for the primary track.

---

### Option 3 — Merge Portfolio into Dashboard as a collapsible section
Not selected. Dashboard is already high cognitive load. Adding a full KPI table increases complexity rather than reducing it. Retires a complete, working page unnecessarily.

## Selected Resolution — Full Directives

### `dashboard.tsx` → renamed "Home"

- Remains at `/app` as the root landing page
- Nav label: **"Home"**
- **Remove** portfolio-level stat cards (total associations, units, owners, tenants, board members, documents) — these move to Portfolio Health
- **Retain:** alerts panel, quick actions, active elections widget, setup wizard
- **Add:** compact "Portfolio Health" summary widget or link row pointing to `/app/portfolio`
- **Add:** cross-association operational alerts on the Home page — overdue work orders, due maintenance schedules, and any other operational signals must surface here regardless of which association is currently active in context. The active-association context switcher must not suppress cross-association alerts on this page.
- **Fix:** stale quick-action link — update label from "Post Ledger Entry" / `/app/financial/ledger` to "Billing" pointing to `/app/financial/billing`
- Cognitive load target: medium (down from high)

### `portfolio.tsx` → renamed "Portfolio Health"

- Nav label: **"Portfolio Health"**
- Repositioned in sidebar: subordinate to Home, not a peer
- **Receives** the stat cards removed from Home (total associations, units, owners, tenants, board members, documents) — these become the header summary row of Portfolio Health
- No other content changes — content is correct, placement and label are wrong
- The `board-admin` redirect behavior (≤1 association → `/app`) is confirmed **intentional and correct**. Add an explicit inline comment in `App.tsx` at the redirect logic documenting this behavior so future agents do not treat it as a bug.

### `operations-dashboard.tsx` → renamed "Operations Overview"

- Nav label: **"Operations Overview"**
- Page title: **"Operations Overview"**
- Breadcrumb: updated to read "Operations Overview" — the word "Dashboard" must not appear in page title or breadcrumb
- Reclassified from Z1-1 to Z1-6 in the IA taxonomy
- No content changes to the page itself
- A link or alert row from the Home page alerts panel to Operations Overview closes the "no link back" gap flagged by A6 — this is handled by the cross-association alerts directive on Home above

### What this resolution does NOT change
- Route paths — no URL changes (`/app`, `/app/portfolio`, `/app/operations/dashboard` remain)
- API contracts — no backend changes required
- Content of Portfolio Health or Operations Overview beyond labels
- The `board-admin` silent redirect behavior — confirmed intentional

## Acceptance Criteria

These criteria apply when the nav restructure (Layer 3) implements this decision:

1. Navigating to `/app` renders a page with nav label and header "Home" — the word "Dashboard" does not appear in the nav label, page title, or breadcrumb for this page
2. The Home page does not contain portfolio-level aggregate stat cards (total associations, units, owners, tenants, board members, documents)
3. The Home page contains a visible link or summary widget pointing to `/app/portfolio`
4. The Home page alerts panel surfaces overdue work orders and due maintenance schedule instances regardless of the currently active association context
5. Navigating to `/app/portfolio` renders a page titled "Portfolio Health" that includes the aggregate stat cards removed from Home
6. "Portfolio Health" appears in the sidebar as a subordinate item to "Home" — not at the same nav level
7. Navigating to `/app/operations/dashboard` renders a page titled "Operations Overview" — the word "Dashboard" does not appear in the page title, breadcrumb, or sidebar nav label for this page
8. A `board-admin` user with exactly one association navigating to `/app/portfolio` is redirected to `/app` — behavior preserved
9. `App.tsx` contains an explicit inline comment at the board-admin redirect logic explaining the intent
10. The quick-actions panel on Home does not contain any link to a legacy redirect path; the billing quick action points directly to `/app/financial/billing`
