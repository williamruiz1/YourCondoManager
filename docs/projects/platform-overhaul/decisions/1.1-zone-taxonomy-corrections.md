## Conflict Analysis

## Selected Resolutions

### Q1 — How many top-level zones should the `/app` workspace have?

**Resolution:** Six canonical zones. The six zones are: **Home, Financials, Operations, Governance, Communications, Platform.** No additional top-level zones may be added without a spec update.

The first five zones (Home, Financials, Operations, Governance, Communications) serve customer personas (Manager, Board Officer, Assisted Board, Owner-adjacent content within `/app`). The sixth zone, **Platform**, serves the Platform Admin persona only and covers `/app/platform/*`, `/app/admin/*`, and `/app/ai/*` — the operator-tier surfaces locked by 0.2 Persona 6 (amended). Customer personas are strictly excluded from the Platform zone per 0.3 Q5 strict isolation and 0.2 boundary matrix rows for `/app/platform/*`, `/app/admin/*`, `/app/ai/*`.

**Acceptance Criteria**
- The sidebar in `/app` contains up to six top-level zones — five for customer personas, one (Platform) for Platform Admin only
- Zone labels are: Home, Financials, Operations, Governance, Communications, Platform
- No seventh zone exists without a decision doc amendment
- Customer personas (Manager, Board Officer, Assisted Board, Owner) do not see the Platform zone in the sidebar per 0.3 Q5 strict isolation + 0.2 Persona 6 boundary matrix
- Platform Admin sees the Platform zone; other zones are also visible only to the extent their 0.2 matrix row grants access (generally ❌ for customer-tenant content under Platform Admin scope — see 0.2 matrix)

### Q2 — What is the assignment rubric for tagging a page to a zone?

**Resolution:** Pages are assigned by primary user intent. The rubric is: (1) What is the user trying to accomplish on this page? (2) Which zone label best describes that intent? If a page serves two zones, it belongs to the zone of its primary action, not its secondary context. Ambiguous cases are escalated — not silently assigned.

**Acceptance Criteria**
- Every page is assigned to exactly one zone
- Assignment follows the intent rubric, not page location in the file tree
- Pages serving two zones are assigned to their primary-action zone
- Any page where intent is genuinely ambiguous is flagged as an open item before assignment

### Q3 — What are the six zone definitions in plain language?

**Resolution:**
- **Home** — Portfolio-level and association-level overview. Landing pages, summary dashboards, health indicators.
- **Financials** — All money-related pages. Budgets, assessments, owner ledgers, payment tracking, financial reports.
- **Operations** — All work and property-related pages. Maintenance requests, work orders, vendors, inspections.
- **Governance** — All board and compliance pages. Meetings, elections, violations, governing documents, rules.
- **Communications** — All messaging and notice pages. Announcements, notices, updates inbox, direct messaging.
- **Platform** — All YCM-internal operator tooling. Platform Controls (tenant config, permission envelopes, provider configuration and diagnostics), Admin Roadmap, Admin Users, Executive Deck, AI Ingestion lifecycle. Platform Admin persona only.

**Acceptance Criteria**
- Each zone definition is documented verbatim in 1.1
- No page is assigned to a zone whose definition does not match its content
- Zone definitions are referenced in the zone tag format comment on each page file
- Platform zone is never assigned to a page that serves a customer persona; `zone: Platform` implies `persona: Platform Admin` and nothing else

### Q4 — What is the zone tag format applied to each page file?

**Resolution:** Every page file carries a metadata comment block in the following format:
```
// zone: [Zone Label]
// persona: [comma-separated list: Manager | Board Officer | Assisted Board | PM Assistant | Owner | Platform Admin]
```
Example:
```
// zone: Financials
// persona: Manager, Board Officer, Assisted Board
```
```
// zone: Platform
// persona: Platform Admin
```
This comment appears at the top of the page component file, below any license or import block.

**Acceptance Criteria**
- All untagged `/app/*` pages (40 customer-tenant surfaces + 5 Platform-zone surfaces) receive a zone tag comment in this format
- Zone label in the comment matches one of the six canonical zone labels exactly
- Persona list reflects which personas can access that page per 0.2 boundary matrix (including the Platform Admin column added in the 0.2 `f8dbf76` amendment)
- No page file is left untagged after 1.1 implementation
- `zone: Platform` files always have `persona: Platform Admin` as the only persona entry

### Q5 — What are the Owner Portal zones?

**Resolution:** Four zones for `/portal`: **Home, My Finances, My Requests, My Community.** These are distinct from the `/app` zones and use first-person labeling to reflect the Owner's self-service context.

**Acceptance Criteria**
- The Owner Portal sidebar contains exactly four top-level zones
- Zone labels are: Home, My Finances, My Requests, My Community
- Portal zone tags use the same comment format as `/app` with `persona: Owner`
- Portal zones do not mirror `/app` zone labels verbatim

## How This Informs Downstream Layers

### Layer 1 — 1.2 (Section hub reclassification)
1.2 consumes the five-zone model to decide which zones get hubs and which sections collapse into those zones. 1.2 Q4's hub URL convention uses the zone labels directly (`/app/financials`, `/app/operations`, `/app/governance`, `/app/communications`).

### Layer 2 — Role & Permission Model
2.3 and 2.4 use the zone tag comments on page files as a batch-review handle. Permission boundary corrections enforce 0.2 against the per-page `persona:` list recorded in each tag.

### Layer 3 — Navigation Restructure
3.1 builds the five-zone sidebar for `/app` (four-zone for `/portal`) per Q1 and Q5. Sidebar group labels are taken verbatim from the zone labels — no separate user-visible vocabulary is introduced.

### Layers 4 and 5
4.1 (cross-association alert engine) and 5.x polish pages consume the canonical zone table for any signal grouping, filtering, or inbox routing.
