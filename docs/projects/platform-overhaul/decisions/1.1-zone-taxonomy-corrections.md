## Context

0.3 locked the navigation model: a single canonical sidebar filtered by role (Q1),
grouped by **functional domain** (Q2 → Financials, Operations, Governance,
Communications, …), with strict role isolation (Q5). 0.1 locked the page-level
identity of the three command centers (`/app` = "Home", `/app/portfolio` =
"Portfolio Health", `/app/operations/dashboard` = "Operations Overview") and
reclassified the last of those from **Z1-1 Command Center** to **Z1-6 Service &
Maintenance** section hub. 0.2 locked the three personas and the surface-ownership
matrix.

1.1 is the Layer 1 module that reconciles the **Z1 / Z2 / Z3 zone taxonomy** used
throughout the IA audit artifacts with the **functional grouping** taxonomy locked
by 0.3 and the **page-level labels** locked by 0.1. It is downstream of 0.3
because 0.3 decided the user-visible sidebar grouping principle (functional, not
zone-based); 1.1 now has to decide whether zones survive as (a) a second-level
taxonomy layered inside the functional groups, (b) an internal-only documentation
overlay with no UI expression, or (c) a structure that collapses entirely into
the functional groups.

1.1 does **not** design sidebars, pages, or routes. It produces a canonical
zone-tag assignment for every `/app/*` page and a policy for how those tags are
expressed (or not expressed) in the product. Layer 3 (Navigation Restructure)
consumes 1.1's output when it builds the sidebar.

## Conflict Analysis

1.1's corrective scope surfaces five distinct conflict classes. Each maps to a
question enumerated below.

1. **Zone-vs-functional taxonomy overlap.** 0.3 Q2 locked functional grouping
   (Financials, Operations, Governance, Communications, …) as the canonical
   sidebar taxonomy. The audit artifacts use Z1 / Z2 / Z3 + sub-tag as the
   canonical classification. These two vocabularies are not interchangeable —
   functional groups cross zone boundaries (e.g., "Documents" spans Z2-2
   reference vault behavior while sitting next to Z1 operational surfaces in
   any functional sidebar), and zone sub-tags describe intent (Command Center,
   Section Hub) that the functional vocabulary does not carry. 1.1 must decide
   the relationship between the two vocabularies. Maps to **Q1, Q3**.

2. **Zone tag ambiguity on specific pages.** The audit artifacts leave several
   pages with disputed or unresolved zone tags even after Phase 3 and Phase 4:
   `operations-dashboard.tsx` Z1-1 vs Z1-6 taxonomy-definition question
   (`docs/specs/ia-audit-phase3-reconciliation.md:132-142`), `board-portal.tsx`
   Z1-1 board-admin variant vs full re-architecture
   (`docs/specs/ia-audit-phase4-platform.md:218-223`), `workspace-preview.tsx`
   Z3-3 vs out-of-scope (`docs/specs/ia-audit-phase4-platform.md:41-45`),
   `executive.tsx` Z1-1 vs Z3-2 (spec-level ambiguity at
   `docs/specs/ia-audit-spec.md:134`). 0.1 locked `operations-dashboard.tsx` at
   the page-label level but did not rewrite the Z1-1 taxonomy definition. Maps
   to **Q2, Q4**.

3. **Zone-3 population after Phase 4.** Z3-3 "Platform Diagnostics" ends up
   empty after `board-portal.tsx` and `workspace-preview.tsx` are removed from
   it (`docs/specs/ia-audit-phase4-platform.md:66-67`). The taxonomy spec
   (`docs/specs/ia-audit-spec.md:124-130`) lists three Z3 sub-categories; one
   is now empty. 1.1 must decide whether to keep Z3-3 as a provisional slot,
   collapse Zone 3 to two sub-tags, or treat the empty sub-category as
   acceptable taxonomy. Maps to **Q4**.

4. **Visibility treatment of zone labels in UI.** 0.3 Q1 resolved sidebar
   persona filtering and Q2 resolved sidebar grouping taxonomy, but 0.3 does
   **not** explicitly rule on whether zone tags surface anywhere user-visible
   (sidebar group headers, breadcrumbs, page titles). The IA audit treats
   zones as documentation overlays; the functional-grouping resolution in
   0.3 implies functional groups are the visible taxonomy, but the skeleton
   must make the internal-vs-external status of zones explicit so Layer 3 has
   a clear directive. Maps to **Q1, Q5**.

5. **Pages that 0.1 forbade the word "Dashboard" on but the audit still tagged
   Z1-1 Command Center.** 0.1 forbids "Dashboard" in nav labels, page titles,
   and breadcrumbs for `/app`, `/app/portfolio`, and `/app/operations/dashboard`.
   The Z1-1 "Command Center" category name is unaffected (it is a taxonomy
   label, not a user-visible string) — but only if 1.1 confirms zones are not
   surfaced in user-visible copy. If zones become visible (Q5 → visible), 0.1's
   forbidden-word constraint propagates into the zone-label vocabulary and
   "Command Center" may need separate user-visible phrasing. Maps to **Q5**.

- **Q3 — Zone vs functional-grouping relationship.** How are zone labels
  reconciled with 0.3's functional grouping taxonomy (Financials, Operations,
  Governance, Communications, …)? Three shapes:
  (a) **Two-layer** — functional groups visible, zones as a second-level
  internal taxonomy tagged per page;
  (b) **Collapse** — zones deprecated and fully replaced by functional groups;
  (c) **Overlay** — zones retained verbatim, functional groups treated as
  implementation labels that render on top of zone tags.
  *Tradeoff axis: preserving audit-artifact vocabulary (overlay) vs converging
  on a single canonical taxonomy (collapse) vs carrying both (two-layer).*

- **Q4 — Ambiguous-zone policy for ORPHAN-SURFACE and DEMOTE-ADMIN pages.**
  What is the policy for pages where the audit flagged ORPHAN-SURFACE or
  DEMOTE-ADMIN but where the correct zone is ambiguous or disputed —
  specifically `board-portal.tsx` (TD-2) and `workspace-preview.tsx` (TD-3),
  and secondarily any page left with an unpopulated Q2 row at resolution time?
  *Tradeoff axis: assign-a-zone-provisionally vs park-out-of-zone-taxonomy vs
  defer-to-later-layer.*

- **Q5 — User-visible vs internal-only zone labels.** Are zone tags visible to
  end users anywhere (page titles, breadcrumbs, section headers, sidebar
  group labels) or are they purely internal documentation / code-level labels?
  *Tradeoff axis: users see the zone vocabulary (risk of collision with 0.1
  forbidden-word rules and with 0.3 functional-group labels) vs zones are an
  agent/code/docs overlay only (cleaner UI, lossier surface signalling).*

## How This Informs Downstream Layers

### Layer 1 — Module 1.2 (Section hub consolidation)

1.2 decides which sections get a dedicated hub page and which route directly to
their primary page. 1.2 inherits the canonical Z1-sub-tag assignments from 1.1
Q2 — pages tagged as section-level parents (e.g., the Z1-6 section hub re-label
for `operations-dashboard.tsx` locked in 0.1) are hub candidates.

### Layer 2 — Role & Permission Model (2.3, 2.4)

2.3 (permission boundary corrections) and 2.4 (role-gating audit) apply the 0.2
matrix against routes. The zone sub-tag from 1.1 does not change gating but is
the handle 2.3 uses when batch-reviewing pages in a zone (e.g., "all Z3 pages
require `platform-admin`").

### Layer 3 — Navigation Restructure

Layer 3 consumes 1.1 Q3 (zone ↔ functional-group relationship) and Q5
(user-visible vs internal) directly. If Q1 resolves "visible", Layer 3 adds the
corresponding sidebar group-label copy. If "overlay", Layer 3 ignores zones in
UI and uses only the 0.3 functional vocabulary.

### Layer 4 — Feature Gaps

Module 4.1 (cross-association alert engine) and 4.2 (Owner Portal gaps) are
largely insensitive to 1.1 but inherit the canonical zone-tag table for any
audit or analytics surface that groups issues by zone.

### Layer 5 — Polish & Hardening

Mobile nav (5.3) and any future inbox taxonomy (0.3 Q9 central inbox) use the
Q2 canonical table as the source of truth when routing items by zone in
condensed surfaces.
