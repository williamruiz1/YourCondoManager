## Context

Module 2.3 audits every `/app/*` route in YCM and catalogs how each one is gated today. It is the bridge between the locked persona boundary matrix in 0.2 and the Layer 3 navigation restructure work that must actually hide, redirect, or reject traffic consistent with that matrix. 0.3 Q5 fixed the binding principle — **strict role isolation** (`docs/projects/platform-overhaul/decisions/0.3-navigation-model.md:97-99,173-177`): "Pages and navigation items are hidden when the role lacks access. Read-only exposure to cross-role content requires explicit approval in spec." Anything short of hide-or-redirect for out-of-scope personas is a bug per the locked foundation.

The YCM code today uses at least six distinct gate mechanisms in parallel — (1) a shell-level branch that sends `board-admin` into a wholly different React tree (`client/src/App.tsx:1043-1057`, `1051-1057`), (2) the single-association redirect (`client/src/App.tsx:192-194`, `243`, `300-302`), (3) inline `adminRole ===` ternaries inside `WorkspaceRouter` (`client/src/App.tsx:293`, `297`), (4) the sidebar `roles: […]` allowlist (`client/src/components/app-sidebar.tsx:51-201`), (5) the `canAccessWipRoute` allowlist (`client/src/lib/wip-features.ts:3-12`), and (6) server middleware (`server/routes.ts:980-1047`). These layers do not agree with each other on what each persona can reach, nor do they agree with the 0.2 matrix. The most consequential divergence: `board-admin` never reaches `WorkspaceRouter` at all — `AuthAwareApp` shunts them to `BoardPortal` (`client/src/App.tsx:1051-1057`, `client/src/pages/board-portal.tsx:186`) — which means the board-admin entries sprinkled through the sidebar `roles` arrays and the `board-admin` redirects inside `WorkspaceRouter` are dead code at the UI level, even though the server middleware still honors `board-admin` as a role (`server/routes.ts:1025-1047`).

This skeleton does not decide which mechanism is canonical, which gates are bugs, or which server endpoints need to be widened or tightened. It enumerates evidence and surfaces conflicts for YCM CC + founder to resolve.

The coupling to 2.1 (role enumeration) is structural: 2.3 can only describe which roles can reach what; 2.1 must confirm whether the four declared roles (`platform-admin`, `board-admin`, `manager`, `viewer` per `client/src/App.tsx:89`, `client/src/components/app-sidebar.tsx:51`, `client/src/lib/wip-features.ts:1`) are exhaustive and how they map to the three 0.2 personas (0.2 has no `platform-admin` or `viewer` persona).

The tab group (`App.tsx:177-179`) conflicts with sidebar (`app-sidebar.tsx:186`) on who sees `roadmap` (tab includes `board-admin`; sidebar says platform-admin-only).
6. **Read-only vs. full access not enforced client-side.** 0.2 distinguishes "Full" vs "Read-only ledger" vs "Read + triage" vs "Read-only" — nothing in `/app/*` UI today enforces the read/write split at the route level. 9. **`board-admin` shell swap is undocumented at code level.** 0.2 §Known gap notes a redirect from `/app/portfolio` to `/app` must preserve an explicit code comment in `App.tsx` — but the broader shell-level swap to `BoardPortal` (`App.tsx:1051-1057`) is the de-facto strict isolation mechanism, not the single-association redirect. ## Conflict Analysis

1. **Gate ownership** — Which layer owns the authoritative gate for a `/app/*` page: the router, the sidebar, inline component checks, server middleware, or the shell-level branch? Today all five are in play and disagree (e.g., Portfolio, Settings, financial posting). Maps to **Q1**.
2. **Client-side vs server-side enforcement policy** — Should every sensitive route require both a client-side route guard AND server middleware, or is server-only acceptable? Today the majority of pages (31 of the enumerated routes) rely on server 403s for access control. Maps to **Q2**.
3. **Shell-level branch vs per-route gating for `board-admin`** — Is the global `AuthAwareApp` swap to `BoardPortal` (`App.tsx:1051-1057`) the canonical mechanism, or should Board Admin use the same `/app` shell with per-route gates as 0.3 Q1/Q3 describes ("single global sidebar, role-filtered" and "Manager and Board Admin share the `/app` workspace shell")? Maps to **Q3**.

   > ⚠️ **CROSS-REFERENCE NOTICE** — 0.3 Q1 (AC 1–4, `0.3:146-149`): "The navigation model defines one canonical zone structure for the platform." 0.3 Q3 (AC 9–15, `0.3:158-164`): "Manager and Board Admin share `/app` shell but land into role-appropriate experiences." **Conflicting evidence:** `client/src/App.tsx:1051-1057` shunts `board-admin` into a separate React component tree (`BoardPortal`) that does NOT use `AppSidebar`, does NOT use `WorkspaceRouter`, and does NOT share the `/app` shell. Per 0.3 this is a bug; per live behavior it's the only reason strict isolation works today.
   >
   > **2.2 Q1 Resolution update (2026-04-15):** The BoardPortal shunt is now confirmed as implementation drift from the correct governance-scoped `/app` model (per 2.2 Q1 + 0.2 amendment at commit `2070b7e`). Board Admin is an `/app` persona in both self-managed and PM-managed associations. The shunt does not represent an approved alternate architecture. In PM-managed mode, the correct model is a role-scoped `/app` view governed by the PM-Managed Default Access Table and PM toggle configuration — not a separate shell. Mode determination uses `tenant_configs.managementType` (`shared/schema.ts:1189`). Unwind of `App.tsx:1051-1057` and resurrection of board-admin sidebar entries from dead code are queued as downstream implementation work pending 2.3 LOCKED status.

4. **Sidebar `board-admin` entries** — Dead code that should be removed, scaffolding for a future Board Admin slice within `/app`, or contradictions that should be reconciled to match 0.2? Maps to **Q4**.

   > ⚠️ **CROSS-REFERENCE NOTICE** — 0.2 Persona Boundary Matrix (`0.2:114-127`) allows Board Admin on most `/app/*` rows. **Conflicting evidence:** `client/src/App.tsx:1051-1057` makes those rows unreachable for Board Admin via the shell. Either the shell-swap is wrong (Board Admin should reach `/app`) or the sidebar/matrix expectations are wrong (Board Admin should only see `BoardPortal`). Flag, do not resolve.

5. **Portfolio gate** — Sidebar says Board Admin can see it; matrix says redirect; redirect only fires for ≤1 association. Maps to **Q5**.

   > ⚠️ **CROSS-REFERENCE NOTICE** — 0.2 AC 3 (`0.2:164`): "A `board-admin` user navigating to `/app/portfolio` is redirected to `/app` — behavior preserved from 0.1." 0.2 AC 4: "A `board-admin` user does not see a 'Portfolio Health' nav item in the sidebar." **Conflicting evidence:** `client/src/components/app-sidebar.tsx:76` includes `board-admin` in the Portfolio `roles` array; `client/src/App.tsx:300-302` redirects only when `singleAssociationBoardExperience` is true. A multi-association `board-admin` (theoretical edge case) would reach Portfolio if the shell branch at `App.tsx:1051-1057` were removed.

6. **Settings gate** — No client gate at all; matrix says Manager-only. Maps to **Q6**.

   > ⚠️ **CROSS-REFERENCE NOTICE** — 0.2 AC 5 (`0.2:166`): "A `board-admin` user does not see a Settings nav item in the sidebar." **Conflicting evidence:** `client/src/components/app-sidebar.tsx:195-201` has no `roles` array on Settings; `client/src/App.tsx:574-580` renders the header settings icon unconditionally.
