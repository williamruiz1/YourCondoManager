## Context

Module 2.3 audits every `/app/*` route in YCM and catalogs how each one is gated today. It is the bridge between the locked persona boundary matrix in 0.2 and the Layer 3 navigation restructure work that must actually hide, redirect, or reject traffic consistent with that matrix. 0.3 Q5 fixed the binding principle — **strict role isolation** (`docs/projects/platform-overhaul/decisions/0.3-navigation-model.md:97-99,173-177`): "Pages and navigation items are hidden when the role lacks access. Read-only exposure to cross-role content requires explicit approval in spec." Anything short of hide-or-redirect for out-of-scope personas is a bug per the locked foundation.

The YCM code today uses at least six distinct gate mechanisms in parallel — (1) a shell-level branch that sends `board-admin` into a wholly different React tree (`client/src/App.tsx:1043-1057`, `1051-1057`), (2) the single-association redirect (`client/src/App.tsx:192-194`, `243`, `300-302`), (3) inline `adminRole ===` ternaries inside `WorkspaceRouter` (`client/src/App.tsx:293`, `297`), (4) the sidebar `roles: […]` allowlist (`client/src/components/app-sidebar.tsx:51-201`), (5) the `canAccessWipRoute` allowlist (`client/src/lib/wip-features.ts:3-12`), and (6) server middleware (`server/routes.ts:980-1047`). These layers do not agree with each other on what each persona can reach, nor do they agree with the 0.2 matrix. The most consequential divergence: `board-admin` never reaches `WorkspaceRouter` at all — `AuthAwareApp` shunts them to `BoardPortal` (`client/src/App.tsx:1051-1057`, `client/src/pages/board-portal.tsx:186`) — which means the board-admin entries sprinkled through the sidebar `roles` arrays and the `board-admin` redirects inside `WorkspaceRouter` are dead code at the UI level, even though the server middleware still honors `board-admin` as a role (`server/routes.ts:1025-1047`).

## Selected Resolution

**Q1 — Gate ownership:** Server middleware (`requireAdmin` + `requireAdminRole`) is the authoritative security boundary. Client-side route guards are required for UX coherence but are secondary. Both layers are mandatory per Q2.

**Q2 — Client + server, or server-only?** Mandatory dual enforcement. Every sensitive `/app/*` route requires both a `<RouteGuard>` client wrapper and server middleware. The 31 server-only routes listed in Section E are implementation debt resolved in Layer 3.

**Q3 — Board Admin shell model:** The `AuthAwareApp:1051-1057` shunt is confirmed as implementation drift. **Board Admin migrates into the shared `/app` shell.** `BoardAdminPortalShell` and `BoardPortal` are retired as the Board Admin entry point. Per-route `<RouteGuard>` gates per the 0.2 matrix replace the global shunt. Implementation is a Layer 3 task — no code changes until 2.3 reaches LOCKED status. A code comment citing `docs/projects/platform-overhaul/decisions/2.3-permission-boundary-corrections.md` must be added at `App.tsx:1051-1057` at time of retirement.

**Q4 — Sidebar `roles` hygiene:** Retain and normalize. All sidebar `roles` arrays are audited against the 0.2 Persona Boundary Matrix and corrected in Layer 3 / 3.1. Delete only entries that contradict 0.2 (e.g., `board-admin` on Portfolio). Do not delete correct entries even if currently dead code — they become live once the shunt is removed.

**Q5 — Portfolio gate scope:** Role-based, not count-based. The portfolio redirect fires for **all** `board-admin` sessions regardless of association count. Sidebar `board-admin` entry for Portfolio (`app-sidebar.tsx:76`) is a bug and is removed. The `singleAssociationBoardExperience` count-conditional at `App.tsx:300-302` is replaced with a role-based redirect: any `adminRole === "board-admin"` → `/app`.

**Q6 — Settings gate:** `<RouteGuard roles={["manager", "platform-admin"]}>` wraps `/app/settings` in Layer 3. The header settings icon at `App.tsx:574-580` is rendered conditionally by role. Exception: self-managed `board-admin` has Settings access per 0.2 Mode A / AC 7 — the RouteGuard must evaluate `managementType` for `board-admin` sessions.

**Q7 — Financial read-only for Board Admin:** Per-page read-only mode. A `useIsReadOnly()` hook evaluates `adminRole` + PM toggle state and returns a boolean. All write affordances (post, edit, delete buttons) gate on `!isReadOnly`. No separate read-only routes. Hook is implemented in Layer 3 alongside the PM toggle evaluation logic from 0.2.

**Q8 — Viewer role:** Deferred to 2.1. Working assumption in 2.3: `viewer` = read-only-of-Manager. No gating changes targeting `viewer` are introduced in Layer 3 before 2.1 locks.

**Q9 — Enforcement consistency standard:** `<RouteGuard roles={[...]}>` wrapper is mandatory for every sensitive `/app/*` route. The `canAccessWipRoute` pattern at `App.tsx:292-294` is the reference implementation. The generalized `<RouteGuard>` component replaces all inline ternaries and sidebar-visibility-only patterns. Sidebar `roles` arrays remain for nav visibility but are not the security gate.

**Q10 — Document the shunt:** Code comment at `App.tsx:1051-1057` is added before removal. This decision doc (`2.3-permission-boundary-corrections.md`) is the canonical reference. An end-to-end test verifies that after migration, a `board-admin` session correctly reaches only routes permitted by the 0.2 matrix — not that the old shunt holds. The test suite replaces shunt-reliance with per-route assertion.

**Q11 — Admin family routes:** Platform-admin-only, strictly. All `/app/admin*` routes (`/app/admin`, `/app/admin/roadmap`, `/app/admin/users`, `/app/admin/executive`) receive `<RouteGuard roles={["platform-admin"]}>`. Tab group entries that include `board-admin` on `roadmap` and `executive` (`App.tsx:177-179`) are bugs — removed in Layer 3. Sidebar is already correct (`app-sidebar.tsx:186`).

**Q12 — Ambient pages:** Added to 0.2 matrix via amendment committed alongside this doc. Canonical posture per route:
- `/app/insurance` → Manager + Board Admin (read-only for PM-managed Board Admin)
- `/app/community-hub` → Manager + Board Admin
- `/app/amenities` → Manager + Board Admin
- `/app/help-center` → All roles, no gate (intentional)
- `/app/association-context` → Manager only (PM operational surface)

## Acceptance Criteria

1. Every sensitive `/app/*` route has both a `<RouteGuard>` client wrapper and server middleware — no route relies on sidebar visibility or shell-swap alone (Q2)
2. `board-admin` sessions reach the shared `/app` shell via `WorkspaceRouter`; `BoardAdminPortalShell` is retired; `AuthAwareApp:1051-1057` branch contains a code comment citing this doc before removal (Q3)
3. All sidebar `roles` arrays match the 0.2 Persona Boundary Matrix — no entries contradict the matrix (Q4)
4. Any `board-admin` session navigating to `/app/portfolio` is redirected to `/app` — role-based, not count-based (Q5)
5. `/app/settings` is wrapped with `<RouteGuard roles={["manager", "platform-admin"]}>` except for self-managed `board-admin` (Mode A) who retains full access per 0.2 AC 7; header settings icon renders conditionally (Q6)
6. Financial pages render in read-only mode for `board-admin` in PM-managed associations; `useIsReadOnly()` hook gates all write affordances; no write action is reachable by read-only Board Admin via UI (Q7)
7. All `/app/admin*` routes are wrapped with `<RouteGuard roles={["platform-admin"]}>` (Q11)
8. The `canAccessWipRoute`-style pattern is generalized into a reusable `<RouteGuard>` component; all ad-hoc inline ternary gates are replaced (Q9)
