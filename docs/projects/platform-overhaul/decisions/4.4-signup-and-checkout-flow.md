## Context

4.4 is the Layer 4 module that designs the signup-to-first-association onboarding flow for new YCM customers. It covers: plan selection on the public `/pricing` page, account creation, Stripe Checkout integration, first association setup, role assignment, onboarding wizard completion, and the transition from public marketing pages to the authenticated `/app` workspace.

The signup persona is the **Manager** (Persona 1, `shared/schema.ts:161` `adminUserRoleEnum` — PM-company employee or self-managing volunteer) per `docs/projects/platform-overhaul/decisions/0.2-persona-map.md:37-58`. For self-managed associations, the signup user is the **Board Officer** (Persona 2, `docs/projects/platform-overhaul/decisions/0.2-persona-map.md:62-83`) — the board president or designated administrator who functions as the manager.

The authenticated workspace entry point is `/app` (Home zone, `docs/projects/platform-overhaul/decisions/1.1-zone-taxonomy-corrections.md:72` — "Portfolio-level and association-level overview. Landing pages, summary dashboards, health indicators"). The canonical route table for `/app/*` is locked in `docs/projects/platform-overhaul/decisions/3.2-route-restructure.md:226-291`.

### Workspace provisioning (`server/routes.ts:12580-12620`)

`provisionWorkspace()` is called after successful checkout:
1. Reads `associationId` and `adminUserId` from Stripe session metadata (`server/routes.ts:12582-12584`)
2. Idempotency check: skips if `platformSubscriptions` row exists for the association (`server/routes.ts:12588-12590`)
3. Activates the admin user (`server/routes.ts:12602`: sets `isActive: 1`)
4. Creates `platformSubscriptions` row with plan, Stripe IDs, trial dates (`server/routes.ts:12604-12613`)
5. Creates `tenantConfigs` row if missing (`server/routes.ts:12616-12619`)

### Role assignment at signup

The current signup flow creates the admin user with `role: "platform-admin"` at `server/routes.ts:12731`:
```
const [adminUser] = await db.insert(adminUsers).values({ email: ..., role: "platform-admin", isActive: 0 }).returning();
```
This contradicts 0.2: the signup persona should be `manager` (PM companies) or `board-officer` (self-managed boards). The `platform-admin` role grants access to `/app/platform/*`, `/app/admin/*`, `/app/ai/*` per 0.2 boundary matrix — surfaces the signup user should not own.

### Stripe integration pattern

The live implementation uses raw `fetch()` calls to `api.stripe.com/v1` via `stripeRequest()` helper at `server/routes.ts:12561-12577`. No Stripe SDK is installed. The platform Stripe secret key is loaded from the secrets store at `server/routes.ts:12562`. Webhook signature verification uses `createHmac("sha256")` at `server/routes.ts:12787`.

### Post-signup success page flow

`PlanSignupSuccessPage` at `client/src/pages/plan-signup-success.tsx:53-228`:

### Known gaps

1. **Role assignment bug**: signup creates `platform-admin` role instead of `manager` / `board-officer` (`server/routes.ts:12731`)
2. **No onboarding wizard**: after checkout success, user lands at `/app` with an empty association (name = organizationName, address = "TBD", city = "TBD", state = "TBD" per `server/routes.ts:12730`). No guided first-association setup exists.
3. **No `admin_user_associations` link**: the stub admin user is created but not linked to the stub association via any junction table. The code sets association ID as Stripe metadata only.
4. **Pricing stale**: client-side plan data at `plan-signup.tsx:30-87` is annotated stale. The Phase 0 `planCatalog` table exists but is not consumed.
5. **No upgrade/downgrade path**: the `/api/admin/billing/portal-session` endpoint opens Stripe Customer Portal but there is no in-app upgrade/downgrade UI or plan-change flow.
6. **Enterprise flow is a mailto link**: `plan-signup.tsx:278` redirects to `mailto:sales@yourcondomanager.org` — no form submission or lead capture.
7. **Success page does not authenticate**: after provisioning, the user sees "Go to Workspace" but may not have an active session. The success page does not set an auth cookie.
8. **Phase 0 billing tables unused**: `planCatalog`, `billingAccounts`, `billingSubscriptions`, `signupPlanSelections` exist in schema but are not wired into the signup or subscription lifecycle.

## Conflict Analysis

Seven decision axes surface from reconciling the current signup flow with the locked Layer 0-2 decisions and the identified gaps.

1. **Role assignment at signup.** The signup endpoint creates `platform-admin` instead of `manager` or `board-officer`. 0.2 defines Manager as the signup persona for PM companies and Board Officer for self-managed associations. Resolved in **Q1**.
2. **Onboarding wizard after checkout.** The success page transitions to `/app` with an empty association. No guided setup for first association details, unit import, or owner invitation. Resolved in **Q2**.
3. **Pricing source of truth.** Client hardcodes plan data at `plan-signup.tsx:30-87` (annotated stale). The `planCatalog` table exists but is unused. The signup endpoint reads price IDs from a secrets store JSON blob. Resolved in **Q3**.
4. **Phase 0 billing migration.** The live signup flow uses `platformSubscriptions` directly. The Phase 0 schema (`planCatalog`, `billingAccounts`, `billingSubscriptions`, `signupPlanSelections`) is designed but unwired. Resolved in **Q4**.
5. **Trial and free-tier model.** The current flow creates a 14-day trial via `subscription_data[trial_period_days]: "14"` at `server/routes.ts:12741`. No free tier exists. No trial expiration handling in the app UI. Resolved in **Q5**.
6. **Upgrade/downgrade paths.** Stripe Customer Portal is available via `/api/admin/billing/portal-session` (`server/routes.ts:12635-12648`) but no in-app plan change surface exists. Resolved in **Q6**.
7. **Post-checkout session continuity.** The success page calls `/api/public/signup/complete` which provisions the workspace but does not establish an authenticated session. The "Go to Workspace" link may land on a login wall. Resolved in **Q7**.

### Cross-reference notices

> **CROSS-REFERENCE NOTICE — 0.2 Persona Boundary Matrix.** Manager owns the full `/app` workspace (0.2 matrix rows 1-11). Board Officer has Manager-equivalent access in self-managed mode. The signup flow must assign the correct role per account type — not `platform-admin`, which owns `/app/platform/*`, `/app/admin/*`, `/app/ai/*` only.

> **CROSS-REFERENCE NOTICE — 3.2 Route Table.** The canonical route table at `docs/projects/platform-overhaul/decisions/3.2-route-restructure.md:226-291` lists `/app` (Home) as the first authenticated surface. The signup-to-workspace transition must land at `/app` per 3.2 row 1. No `/app/onboarding` or `/app/setup` route exists in 3.2 — if an onboarding wizard is introduced, it requires a 3.2 amendment or must be implemented as a modal/overlay on `/app`.

> **CROSS-REFERENCE NOTICE — 2.1 Q4 AdminRole canonicalization.** The canonical `AdminRole` type at `shared/schema.ts:161` enumerates `["platform-admin", "board-officer", "assisted-board", "pm-assistant", "manager", "viewer"]`. The signup endpoint must use one of these values. 2.1 Q7 mandates that a nullish or unrecognized role is denied access.

> **CROSS-REFERENCE NOTICE — 1.1 Q3 Home zone definition.** Home zone is "Portfolio-level and association-level overview. Landing pages, summary dashboards, health indicators" (1.1 Q3). The post-signup landing at `/app` must render a meaningful Home experience even when the association is in setup state.

## PENDING Resolutions

### Q1 — What role should the signup endpoint assign to new users?

**PENDING.** The current code at `server/routes.ts:12731` assigns `platform-admin`. Candidate resolution: assign `manager` for `property-manager` plan key, `board-officer` for `self-managed` plan key. Enterprise is handled separately (no self-serve signup). This aligns with 0.2 Persona 1 (Manager: "PM-company employee or self-managing volunteer") and Persona 2 (Board Officer: "self-managed association where no property manager exists").

Key sub-questions:
- P1: Should self-managed signups get `board-officer` (matching their persona) or `manager` (the superset role that includes multi-association capability)?
- P2: If the signup user later adds additional associations, does the role need to change?
- P3: How does role assignment interact with the admin user activation flow (`isActive: 0` -> `isActive: 1` at `server/routes.ts:12602`)?

### Q2 — Should an onboarding wizard exist after checkout, and what does it contain?

**PENDING.** Currently the post-checkout flow lands at `/app` with a stub association (name from form, address/city/state = "TBD" per `server/routes.ts:12730`). The user sees the full workspace with no guided setup.

Key sub-questions:
- P1: Is onboarding a separate route (e.g., `/app/onboarding`) or a modal/banner on `/app` Home? If a separate route, 3.2 requires amendment.
- P2: What steps does the wizard contain? Candidates: association details (address, city, state), unit import, first owner invitation, portal configuration, branding.
- P3: Can onboarding be skipped/deferred? What is the minimum viable first-login experience?
- P4: Does the wizard differ between self-managed (Board Officer) and PM (Manager) signup tracks?

### Q3 — Where should plan pricing data live and how should the signup page consume it?

**PENDING.** Three pricing sources currently exist:
1. Client-side hardcoded PLANS object at `client/src/pages/plan-signup.tsx:30-87` (stale)
2. `planCatalog` table at `shared/schema.ts:2795-2818` (Phase 0, unused)
3. Stripe price IDs in secrets store JSON blob at `server/routes.ts:12702-12703`

Key sub-questions:
- P1: Should the pricing page render from `planCatalog` via an API, or remain a static client-side component?
- P2: Should `signupPlanSelections` (`shared/schema.ts:2892-2905`) be wired in to capture plan selection state before checkout?
- P3: How does the self-managed two-tier pricing (under/over 30 units, `server/routes.ts:12706-12712`) map to `planCatalog.unitMin`/`unitMax` fields?

### Q4 — Should the signup flow migrate from `platformSubscriptions` to the Phase 0 billing tables?

**PENDING.** The Phase 0 billing data model (`planCatalog`, `billingAccounts`, `billingSubscriptions`, `billingSubscriptionItems`, `signupPlanSelections`) at `shared/schema.ts:2787-2905` is a more complete schema that supports versioned pricing, billing intervals, and line-item tracking. The live flow uses only `platformSubscriptions` (`shared/schema.ts:2555-2573`).

Key sub-questions:
- P1: Is this migration in scope for the platform overhaul or deferred to a separate billing initiative?
- P2: If migrated, does `platformSubscriptions` become a view/wrapper over `billingSubscriptions`, or is it retired?
- P3: What is the data migration strategy for existing `platformSubscriptions` rows?

### Q5 — What is the trial/free-tier model?

**PENDING.** The current implementation sets `subscription_data[trial_period_days]: "14"` at `server/routes.ts:12741` and `payment_method_collection: "if_required"` at `server/routes.ts:12737` — a 14-day trial that does not collect payment info upfront. The `platformSubscriptionStatusEnum` includes `"trialing"` at `shared/schema.ts:2545-2547`.

Key sub-questions:
- P1: Does the trial period remain 14 days?
- P2: Is there a free tier (e.g., limited feature set, single association, read-only after trial)?
- P3: What happens at trial expiration? Options: hard lock (redirect to billing), soft lock (read-only mode), grace period.
- P4: Should the app UI display trial status and days remaining? Where — Home banner, sidebar badge, settings?
- P5: The success page shows "14-day free trial" and "no credit card required" (`plan-signup.tsx:155-158,365-369`). Is this still the desired messaging?

### Q6 — What upgrade/downgrade paths should exist?

**PENDING.** The Stripe Customer Portal is accessible via `POST /api/admin/billing/portal-session` at `server/routes.ts:12635-12648`, which redirects to Stripe's hosted portal with `return_url: /app/platform/controls`. No in-app plan comparison or upgrade CTA exists.

Key sub-questions:
- P1: Should an in-app billing/plan management surface exist under `/app/settings` or `/app/platform/controls`?
- P2: Is Stripe Customer Portal sufficient for upgrade/downgrade, or does YCM need a custom plan-change flow?
- P3: Where does the billing management link live in the sidebar? Currently the return URL points to `/app/platform/controls` — per 0.2 this is Platform Admin only. Manager/Board Officer need a billing surface they can access.
- P4: Should self-managed users be able to upgrade to property-manager in-app? What triggers the upgrade (adding a second association)?

### Q7 — How should post-checkout session continuity work?

**PENDING.** After Stripe Checkout completes, the browser redirects to `/signup/success?session_id=...` (`server/routes.ts:12742`). The success page calls `GET /api/public/signup/complete` (`server/routes.ts:12760-12775`) which provisions the workspace. But:
- The `provisionWorkspace()` function at `server/routes.ts:12580-12620` does not set a session cookie
- The "Go to Workspace" button links to `/app` (`plan-signup-success.tsx:206-209`)
- The user may land on a login wall since no authenticated session was established
- Google OAuth pre-auth during signup (`plan-signup.tsx:242-268`) may or may not persist

Key sub-questions:
- P1: Should `/api/public/signup/complete` set an authenticated session cookie for the newly created admin user?
- P2: If Google OAuth was used during signup, is the Google session still valid after the Stripe redirect?
- P3: Should the success page auto-redirect to `/app` after a brief delay, or require the user to click?
- P4: If the user lands on `/app` without a session, what login method do they use? Magic link, Google OAuth, or password?
