## Context

4.4 is the Layer 4 module that designs the signup-to-first-association onboarding flow for new YCM customers. It covers: plan selection on the public `/pricing` page, account creation, Stripe Checkout integration, first association setup, role assignment, onboarding wizard completion, and the transition from public marketing pages to the authenticated `/app` workspace.

The signup persona is the **Manager** (Persona 1, `shared/schema.ts:161` `adminUserRoleEnum` — PM-company employee or self-managing volunteer) per `docs/projects/platform-overhaul/decisions/0.2-persona-map.md:37-58`. For self-managed associations, the signup user is the **Board Officer** (Persona 2, `docs/projects/platform-overhaul/decisions/0.2-persona-map.md:62-83`) — the board president or designated administrator who functions as the manager.

The authenticated workspace entry point is `/app` (Home zone, `docs/projects/platform-overhaul/decisions/1.1-zone-taxonomy-corrections.md:72` — "Portfolio-level and association-level overview. Landing pages, summary dashboards, health indicators"). The canonical route table for `/app/*` is locked in `docs/projects/platform-overhaul/decisions/3.2-route-restructure.md:226-291`.

## Conflict Analysis

Seven decision axes surface from reconciling the current signup flow with the locked Layer 0-2 decisions and the identified gaps.

1. **Role assignment at signup.** The signup endpoint creates `platform-admin` instead of `manager` or `board-officer`. 0.2 defines Manager as the signup persona for PM companies and Board Officer for self-managed associations. Resolved in **Q1**.
2. **Onboarding wizard after checkout.** The success page transitions to `/app` with an empty association. No guided setup for first association details, unit import, or owner invitation. Resolved in **Q2**.
3. **Pricing source of truth.** Client hardcodes plan data at `plan-signup.tsx:30-87` (annotated stale). The `planCatalog` table exists but is unused. The signup endpoint reads price IDs from a secrets store JSON blob. Resolved in **Q3**.
4. **Phase 0 billing migration.** The live signup flow uses `platformSubscriptions` directly. The Phase 0 schema (`planCatalog`, `billingAccounts`, `billingSubscriptions`, `signupPlanSelections`) is designed but unwired. Resolved in **Q4**.
5. **Trial and free-tier model.** The current flow creates a 14-day trial via `subscription_data[trial_period_days]: "14"` at `server/routes.ts:12741`. No free tier exists. No trial expiration handling in the app UI. Resolved in **Q5**.
6. **Upgrade/downgrade paths.** Stripe Customer Portal is available via `/api/admin/billing/portal-session` (`server/routes.ts:12635-12648`) but no in-app plan change surface exists. Resolved in **Q6**.
7. **Post-checkout session continuity.** The success page calls `/api/public/signup/complete` which provisions the workspace but does not establish an authenticated session. The "Go to Workspace" link may land on a login wall. Resolved in **Q7**.

### Q1 — What role should the signup endpoint assign to new users? — **RESOLVED**

**Resolution (founder-decided, 2026-04-16):** The default role for public signup is always `manager`. No public signup code path may assign `platform-admin`, `board-officer`, `assisted-board`, or `pm-assistant`.

- **Primary:** `manager` — the person signing up is the manager of that association, regardless of plan type (self-managed or property-manager).
- **Secondary Board role:** If a board-side role is needed at signup, use `board-officer` explicitly — never infer `pm-assistant` or `assisted-board` during signup.
- **Platform-admin:** Creation of any `platform-admin` account is an explicit, separate path (internal tooling or platform-admin seeding only), never reachable via public signup.
- **Bug fix applied:** `server/routes.ts:12731` changed from `role: "platform-admin"` to `role: "manager"` in the same commit as this resolution.
- **Test added:** `tests/signup-role-assignment.ts` — creates a user via the public signup INSERT and asserts `adminRole === "manager"`; asserts no code path in the signup handler can produce `platform-admin`.

Sub-questions resolved:
- P1: Self-managed signups get `manager` (not `board-officer`). The signup persona is always Manager; Board Officer is assigned later by the Manager when adding board members to the association.
- P2: Role does not change when the user adds additional associations. `manager` is multi-association-capable by definition (0.2 §Persona 1).
- P3: Role is set at INSERT (`isActive: 0`); activation at `provisionWorkspace` (`:12602`) flips `isActive` to 1 but does not change role.

**Acceptance Criteria:**
- `/api/public/signup/start` creates admin users with `role: "manager"` (verified by test)
- No code path in `/api/public/signup/start` or `provisionWorkspace()` can produce `role: "platform-admin"` (verified by source-scan test)
- `platform-admin` accounts are created only through explicit internal tooling paths, never through public signup

### Q5 — What is the trial/free-tier model? — **RECOMMENDED / LOCKED BY AGENT (pending founder confirmation)**

**Resolution (agent-recommended 2026-04-24, Wave 13):** 14-day trial, no free tier, hard lock after a 7-day grace window, single-surface trial-status UI (Home banner only). No credit-card capture upfront.

- **Trial duration:** **14 days**. Keeps the current messaging and `subscription_data[trial_period_days]: "14"` (`server/routes.ts:13496`) — no change.
- **Free tier:** **None.** Feature parity during trial; the product locks to billing after trial + grace.
- **Trial expiration → grace window:** Stripe transitions `trialing → active` (paid) or `trialing → canceled`/`unpaid`/`past_due` (no PM or PM failed). Grace is **7 days** of continued access after `trialing` ends — surfaced with escalated copy in the TrialBanner. After grace elapses (or Stripe flips status to `canceled`/`unpaid`), all `/app/*` routes redirect to `/app/settings/billing` via the client-side `SubscriptionLockScreen` modal (already shipped — `client/src/components/subscription-lock-screen.tsx`). Past_due keeps access during the grace window; canceled/unpaid hard-lock immediately.
- **Trial status UI (one surface only):** `TrialBanner` at the top of `/app` (already mounted in `WorkspaceShell`, `client/src/App.tsx:911-917`). Banner shows days remaining, an "Upgrade Now" button that opens the Stripe Customer Portal in a new tab, and a per-session dismiss (not per-day — session-storage, cleared on logout). The sidebar is NOT cluttered with trial state; settings-level billing detail lives at `/app/settings/billing`.
- **Payment-method collection:** stays `"if_required"` during trial (`server/routes.ts:13492`) — no CC upfront.
- **Hard-lock redirect trigger:** `platformSubscriptions.status === "canceled"` OR `"unpaid"` → `SubscriptionLockScreen` blocks the workspace. Status `past_due` surfaces the lock modal only after grace elapses (the server-side `requireActiveSubscription` middleware at `server/routes.ts:1061-1086` already returns 402 for non-active statuses; the client lock modal is the user-visible half of the pair).
- **Messaging updates:** `plan-signup-success.tsx` and `plan-signup.tsx` reference the 14-day / 7-day-grace / hard-lock flow in the fine print.

Sub-questions resolved:
- P1: 14 days.
- P2: No free tier.
- P3: Hard lock after 7-day grace.
- P4: Home banner only, per-session dismiss.
- P5: Keep "14-day free trial · no credit card required" — copy unchanged, add a brief note about the 7-day grace on the success page.

**Acceptance Criteria:**
- `TrialBanner` visible when `subscription.status === "trialing"` with a valid `trialEndsAt`.
- `SubscriptionLockScreen` visible when `subscription.status` is `canceled` or `unpaid` (hard lock). `past_due` surfaces the lock after grace expires.
- Dismiss key is session-storage-scoped: `ycm:trial-banner-dismissed` cleared at logout/session-end.
- `/app/settings/billing` is the canonical settings entry point for billing management.

**Pending founder confirmation:** the numeric values (14-day trial, 7-day grace) and the "no credit card upfront" stance are agent-recommended defaults pending founder sign-off. Filed as PPM human task.
