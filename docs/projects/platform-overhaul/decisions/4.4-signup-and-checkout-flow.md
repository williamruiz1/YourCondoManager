## Context

4.4 is the Layer 4 module that designs the signup-to-first-association onboarding flow for new YCM customers. It covers: plan selection on the public `/pricing` page, account creation, Stripe Checkout integration, first association setup, role assignment, onboarding wizard completion, and the transition from public marketing pages to the authenticated `/app` workspace.

The signup persona is the **Manager** (Persona 1, `shared/schema.ts:161` `adminUserRoleEnum` — PM-company employee or self-managing volunteer) per `docs/projects/platform-overhaul/decisions/0.2-persona-map.md:37-58`. For self-managed associations, the signup user is the **Board Officer** (Persona 2, `docs/projects/platform-overhaul/decisions/0.2-persona-map.md:62-83`) — the board president or designated administrator who functions as the manager.

The authenticated workspace entry point is `/app` (Home zone, `docs/projects/platform-overhaul/decisions/1.1-zone-taxonomy-corrections.md:72` — "Portfolio-level and association-level overview. Landing pages, summary dashboards, health indicators"). The canonical route table for `/app/*` is locked in `docs/projects/platform-overhaul/decisions/3.2-route-restructure.md:226-291`.

## Conflict Analysis

Seven decision axes surface from reconciling the current signup flow with the locked Layer 0-2 decisions and the identified gaps.

1. **Role assignment at signup.** The signup endpoint creates `platform-admin` instead of `manager` or `board-officer`. 0.2 defines Manager as the signup persona for PM companies and Board Officer for self-managed associations. Resolved in **Q1**.
2. **Onboarding wizard after checkout.** The success page transitions to `/app` with an empty association. No guided setup for first association details, unit import, or owner invitation. Resolved in **Q2**.
3. **Pricing source of truth.** Client hardcodes plan data at `plan-signup.tsx:30-87` (annotated stale). The `planCatalog` table exists but is unused. The signup endpoint reads price IDs from a secrets store JSON blob. Resolved in **Q3**.
4. **Phase 0 billing migration.** The live signup flow uses `platformSubscriptions` directly. The Phase 0 schema (`planCatalog`, `billingAccounts`, `billingSubscriptions`, `signupPlanSelections`) is designed but unwired. Resolved in **Q4**.
5. **Trial and free-tier model.** The current flow creates a 14-day trial via `subscription_data[trial_period_days]: "14"` at `server/routes.ts:12741`. No free tier exists. No trial expiration handling in the app UI. Resolved in **Q5**.
6. **Upgrade/downgrade paths.** Stripe Customer Portal is available via `/api/admin/billing/portal-session` (`server/routes.ts:12635-12648`) but no in-app plan change surface exists. Resolved in **Q6**.
7. **Post-checkout session continuity.** The success page calls `/api/public/signup/complete` which provisions the workspace but does not establish an authenticated session. The "Go to Workspace" link may land on a login wall. Resolved in **Q7**.

### Q1 — What role should the signup endpoint assign to new users? — **RESOLVED**

**Resolution (founder-decided, 2026-04-16):** The default role for public signup is always `manager`. No public signup code path may assign `platform-admin`, `board-officer`, `assisted-board`, or `pm-assistant`.

- **Primary:** `manager` — the person signing up is the manager of that association, regardless of plan type (self-managed or property-manager).
- **Secondary Board role:** If a board-side role is needed at signup, use `board-officer` explicitly — never infer `pm-assistant` or `assisted-board` during signup.
- **Platform-admin:** Creation of any `platform-admin` account is an explicit, separate path (internal tooling or platform-admin seeding only), never reachable via public signup.
- **Bug fix applied:** `server/routes.ts:12731` changed from `role: "platform-admin"` to `role: "manager"` in the same commit as this resolution.
- **Test added:** `tests/signup-role-assignment.ts` — creates a user via the public signup INSERT and asserts `adminRole === "manager"`; asserts no code path in the signup handler can produce `platform-admin`.

Sub-questions resolved:
- P1: Self-managed signups get `manager` (not `board-officer`). The signup persona is always Manager; Board Officer is assigned later by the Manager when adding board members to the association.
- P2: Role does not change when the user adds additional associations. `manager` is multi-association-capable by definition (0.2 §Persona 1).
- P3: Role is set at INSERT (`isActive: 0`); activation at `provisionWorkspace` (`:12602`) flips `isActive` to 1 but does not change role.

**Acceptance Criteria:**
- `/api/public/signup/start` creates admin users with `role: "manager"` (verified by test)
- No code path in `/api/public/signup/start` or `provisionWorkspace()` can produce `role: "platform-admin"` (verified by source-scan test)
- `platform-admin` accounts are created only through explicit internal tooling paths, never through public signup
- The test `tests/signup-role-assignment.ts` passes

### Q5 — What is the trial/free-tier model? — **RESOLVED**

**Resolution (founder-decided, 2026-04-21):** 14-day trial + 7-day grace period + hard lock. No free tier.

Rationale: Balances conversion against giving away product. Trial gets the prospect in the door (no credit card upfront); grace period handles the "I meant to pay but got distracted" case without giving the product away indefinitely; hard lock forces the payment decision. Cleaner than soft-lock read-only mode (which invites users to work against a paywalled view — bad UX on both sides). No free tier because an HOA SaaS is not a product-led growth product — signup already requires real association setup; the trial + grace is enough runway to evaluate.

Sub-questions resolved:
- P1: Trial period remains 14 days.
- P2: No free tier in Phase 1.
- P3: Sequence: Day 0–14 = full access (trial). Day 15–21 = full access + persistent "your trial has ended" banner on every page with CTA to add payment method (grace period). Day 22 = hard lock — `/app` redirects to a paywall page with "Add payment method" CTA; Stripe Customer Portal link for payment; all API writes rejected with 402 Payment Required. Read access to core data (ledger, documents) is NOT preserved post-lock — hard lock means hard lock.
- P4: Trial status UI: a persistent header chip showing "Trial: X days left" (dismissible to a tray icon) during Days 0–14; the chip turns red and persists non-dismissibly during Days 15–21 with text "Trial ended — N days grace remaining." The Manager-only billing surface (Q6 `/app/settings/billing`) shows full trial/grace/subscription state.
- P5: Success-page messaging remains "14-day free trial" and "no credit card required" — consistent with the resolved model. Post-trial messaging ("your trial has ended, X days grace") is added to the billing surface and the header chip.

**Acceptance Criteria:**
- AC 6: Trial period remains 14 days; no code change to `server/routes.ts:12741`.
- AC 7: A new 7-day grace period is introduced — when `platformSubscriptions.status` transitions from `"trialing"` to `"past_due"` (or Stripe-equivalent expired-trial state), the workspace remains fully accessible for 7 calendar days with a persistent banner.
- AC 8: After the 7-day grace, the workspace enters a hard-lock state: `/app/*` redirects to a paywall page; API writes return 402 Payment Required.
- AC 9: A persistent header chip component shows trial/grace status on all `/app/*` pages; it is dismissible during trial and non-dismissible during grace.
- AC 10: No free tier is introduced — `planCatalog` is not extended with a free plan entry in this overhaul cycle.
- AC 11: Signup success-page messaging at `plan-signup-success.tsx:206-209` remains unchanged.

### Q7 — How should post-checkout session continuity work? — **RESOLVED**

**Resolution (founder-decided, 2026-04-21):** Auto-authenticate on signup complete. The `/api/public/signup/complete` endpoint sets the authenticated session cookie for the newly created admin user during `provisionWorkspace()` — the user lands on `/app` logged in, with no interstitial login wall.

Rationale: The founder experience should be "pay → you're in." Anything else (magic link, password entry, OAuth re-prompt) is user-hostile immediately after a successful payment. The implementation detail is careful session-token ordering during the Stripe webhook → provision flow; this is a known-solvable problem. If auto-authentication proves infeasible for a specific flow edge (e.g., webhook race condition), fall back to a magic-link email as a compensating mechanism rather than forcing password entry.

Sub-questions resolved:
- P1: Yes. `/api/public/signup/complete` establishes an authenticated session via the existing admin-session mechanism before returning to the client. Session cookie is set in the response to the `/api/public/signup/complete` GET call, so the subsequent navigation to `/app` carries the session.
- P2: If Google OAuth was used during signup, the OAuth session persists through the Stripe redirect (the Stripe Checkout flow does not invalidate same-site cookies). Provisioning reconciles the OAuth-authenticated user with the newly-created admin record — no double-session.
- P3: Success page auto-redirects to `/app` after a brief delay (configurable, ~3 seconds) with a visible "Taking you to your workspace…" message. A manual "Go to Workspace" button remains as a fallback for users who want to click immediately.
- P4: If auto-authentication fails for any reason (webhook race, cookie-setting failure, session store error), the fallback is a magic-link email sent to the signup email address with a one-click authentication URL. Password-based login is NOT the fallback — new users do not have a password set during OTP-first signup. This preserves the no-password signup path.

**Acceptance Criteria:**
- AC 18: `/api/public/signup/complete` sets an authenticated admin session cookie in its response; the session is valid for the newly-provisioned admin user with `role: "manager"` per Q1.
- AC 19: The success page (`plan-signup-success.tsx`) auto-redirects to `/app` after ~3s with a visible status message; "Go to Workspace" button remains clickable as an immediate manual override.
- AC 20: If session-establishment fails during `provisionWorkspace()`, the system sends a magic-link email to the signup address with a 15-minute-expiry authentication URL — not a password-reset prompt.
- AC 21: Google OAuth sessions established during signup (`plan-signup.tsx:242-268`) are reconciled with the admin record on `/api/public/signup/complete` — no duplicate admin records, no OAuth re-prompt.
- AC 22: The OTP-first signup path does not gain a password requirement; fallback auth is magic-link only.
