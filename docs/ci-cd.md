## What runs on every PR

Every pull request targeting `main` (and every push to `main`) runs
four jobs in parallel:

| Job          | Command(s)                                                                                         | Purpose                                                                |
|--------------|----------------------------------------------------------------------------------------------------|------------------------------------------------------------------------|
| `check`      | `npm run check` then `npm run lint`                                                                | TypeScript compile + ESLint.                                           |
| `test`       | `npx vitest run`                                                                                   | Full Vitest suite (client + server + integration).                     |
| `build`      | `npm run build`                                                                                    | Production bundle sanity check (catches build-only regressions).       |
| `playwright` | `playwright install --with-deps chromium` then route-mock, real-backend, and visual-regression runs | End-to-end. Bundles axe-core a11y assertions (Wave 25) into each spec. |

All four jobs gate merge. If any job fails, the PR cannot be merged
until either the failure is fixed or the gate is explicitly waived
on the GitHub side (admin-only).

The `playwright` job runs the three Playwright modes sequentially in
one job so the npm install + browser install setup is amortised
across all three modes:

1. **route-mock** — `PLAYWRIGHT_STATIC=1 npm run test:playwright`
   uses the static-build server and route-level mocks. Fastest
   feedback loop.
2. **real-backend** — `npm run test:playwright:real` boots an
   ephemeral pglite (in-process Postgres in WASM) behind a
   pg-gateway and runs the dev server against it. Catches
   integration-level bugs the route-mock pass misses.
3. **visual regression** — `npm run test:playwright:visual` runs
   only the `@visual`-tagged specs against committed PNG baselines
   under `tests/e2e/playwright/__screenshots__/`. Tagged with
   `if: ${{ !cancelled() }}` so it still runs even if the
   real-backend pass reports a flake, since visual diffs are
   typically orthogonal to flow assertions.

## Reading a CI failure

When a job fails, click into the run from the PR's "Checks" tab.
GitHub will show:

- The failing step's command output (stdout + stderr).
- For Playwright failures: a downloadable `playwright-traces.zip`
  artifact at the bottom of the run summary.

To debug a Playwright failure locally:

```bash
# Download the artifact ZIP from the failed run.
unzip playwright-traces.zip -d /tmp/ci-traces

# Open the HTML report.
npx playwright show-report /tmp/ci-traces/playwright-report

# Or open a specific trace (best for failure forensics — shows
# screenshots, network, console at every step).
npx playwright show-trace /tmp/ci-traces/test-results/<spec-name>/trace.zip
```

The trace UI lets you scrub through the test, see DOM snapshots at
every step, and replay network requests. This is the fastest way to
diagnose a Linux-only failure (e.g. a fontFamily diff that a macOS
dev never saw).

Artifacts are retained for 7 days then garbage-collected.

## Updating visual regression baselines

The visual-regression suite asserts that four stable surfaces (Home,
Inbox, Assessment Rules, Portal Home) render pixel-identical to
committed PNG baselines under
`tests/e2e/playwright/__screenshots__/visual-regression.spec.ts/`.

When you intentionally change one of those surfaces (e.g. swap a
component, change a color token), the visual job will fail with
"Screenshot comparison failed: <N> pixels (ratio M.MMM) differ."
The fix:

1. Run the suite locally with `--update-snapshots`:
   ```bash
   npm run test:playwright:visual:update
   ```
   This regenerates the PNGs from the current build.

2. Inspect the diff. The previous baseline is in git; the new one
   is the freshly written PNG.
   ```bash
   git diff --stat tests/e2e/playwright/__screenshots__/
   ```

3. Commit the updated PNG(s) on the same branch as the visual
   change. CI will re-run with the new baseline.

**Important:** baselines are platform-specific. The committed PNGs
are generated by Chromium on Ubuntu in CI. A baseline regenerated on
macOS will differ slightly (font-rendering, antialiasing) and break
CI. The right workflow when you only have a macOS dev box:

- Make the visual change.
- Push the PR.
- Let CI fail with a screenshot diff.
- Download the `playwright-traces.zip` artifact, extract the
  freshly-rendered "actual" PNG from `test-results/`.
- Commit that PNG as the new baseline. Push.

The Wave 25 runbook
([`implementation-artifacts/visual-regression-runbook.md`](projects/platform-overhaul/implementation-artifacts/visual-regression-runbook.md))
covers this in more detail.

## Skipping CI for docs-only changes

The workflow's `pull_request` trigger has a `paths-ignore` filter:

```yaml
on:
  pull_request:
    branches: [main]
    paths-ignore:
      - "**/*.md"
      - "docs/**"
  push:
    branches: [main]
```

A pull request whose every changed file matches one of those globs
will skip the workflow entirely — the GitHub Checks tab will show
the workflow as "no relevant matches" and the merge gate is
considered satisfied vacuously.

**Pushes to `main` always run the workflow**, even for docs-only
merges. This is intentional: the post-merge sanity run is cheap
insurance against a docs-only commit accidentally including a code
change.

If you have a mixed PR (docs + code), the workflow runs in full.
There is no in-between "partial-skip" mode.

## Common failures and fixes

### `npm ci` fails with "lockfile drift"
- Cause: `package.json` and `package-lock.json` are out of sync.
- Fix: run `npm install` locally, commit the updated lockfile.

### `npm run check` fails with TS errors
- Run `npm run check` locally. CI runs the exact same command.

### `npm run lint` fails
- Run `npm run lint` locally. The repo's ESLint config is in
  `eslint.config.js`. There is no `--fix` step in CI; fix locally
  and commit.

### `npx vitest run` fails on a flaky test
- Re-run the failed job from the GitHub UI ("Re-run failed jobs").
- If it fails twice in a row, file a flake follow-up workitem and
  fix the underlying determinism bug. Do NOT add `.skip` or
  retries to silence flakes — every flake is a real bug.

### `npm run build` fails
- Cause: usually a TS error that `tsc` (in `check`) catches
  separately. If `check` is green but `build` fails, suspect a
  Vite-only or esbuild-only issue (e.g. a top-level await pattern
  that `tsc` accepts but the bundler rejects).

### Playwright real-backend hangs
- Cause: pglite + drizzle-kit-push timing in CI is occasionally
  slow on a cold runner. The `webServer.timeout` is 180_000 ms
  which has been sufficient on `main` so far.
- If a real-backend hang becomes recurring, file a follow-up to
  bump the timeout or add a startup-readiness probe.

### Playwright visual regression fails on every PR
- See "Updating visual regression baselines" above.

### Playwright fails with "browser not installed"
- The job runs `npx playwright install --with-deps chromium`
  before any spec. If this step itself fails (network blip), re-
  run the failed job.

### "axe found N critical/serious violations"
- The Wave 25 a11y harness is hard-failing on a real WCAG 2.1 AA
  violation that landed in your branch. Open the trace, find the
  axe call's stderr output (it logs every violation with selector +
  fix description), fix the markup, push.
- The five locked Wave 25 surfaces (Home, Inbox, Rules, Portal
  Home, Portal Finances) hard-fail; ambient surfaces soft-fail
  (logged but non-blocking). If your PR touches one of the five
  locked surfaces and you can't immediately fix the violation,
  flag the PR as blocked and file a follow-up.

## Cost / minutes notes

- Each full run uses approximately:
  - `check`: ~2 min
  - `test`: ~2-3 min
  - `build`: ~3 min
  - `playwright`: ~10-15 min (browser install dominates the
    setup; the three modes themselves run in ~5-7 min combined)
- The four jobs run in parallel, so wall-clock latency is gated
  on the slowest job (`playwright`).
- `concurrency.cancel-in-progress: true` cancels the previous run
  on a branch when a new commit lands, saving minutes during
  rapid PR iteration.
- A docs-only PR uses zero CI minutes (skipped via `paths-ignore`).

## Future work

Out of scope for Wave 30 (5.9), filed as follow-ups when needed:

- **Branch protection rules.** Manual repo setting; flip after
  the first green run on `main`. Settings → Branches → Add rule
  → require status checks: `check`, `test`, `build`, `playwright`.
- **Deploy / CD step.** YCM deploys are manual (Replit preview,
  Fly production). A future module can wire a `release` workflow
  if/when there's a need for a tagged-version release pipeline.
- **Cross-browser Playwright.** Chromium-only today (Wave 25
  §123). Add Firefox / WebKit projects when there's evidence of
  browser-specific bugs.
- **Code coverage upload.** Wave 25 + Wave 30 declined external
  SaaS. Local coverage via `npm run test:coverage` is sufficient
  today.
- **Self-hosted runner.** Public Actions runners are sufficient.
  If CI minutes become a budget constraint, a single self-hosted
  Mac mini or Linux VPS would amortise heavy Playwright runs.
